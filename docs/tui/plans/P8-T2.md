### 0.1 ASCII / monochrome infrastructure is already in place

- `detectCapabilities(env, {stdoutIsTTY})` (`packages/markflow-tui/src/theme/capabilities.ts`) already implements the full precedence required by mockup §14:
  - `MARKFLOW_ASCII=1` → `unicode = false`

## §2 Exported names + signatures

### 2.1 `src/components/narrow-layout.ts`

```ts
// Pure helpers for the <60-col single-pane layout.
// Authoritative refs: mockups.md §13, plan.md P8-T2.

/** Exclusive upper bound of the narrow tier. `width < NARROW_TIER_MAX` → narrow. */
export const NARROW_TIER_MAX = 60;

/**
 * Three drill levels at narrow width (mockups.md §13 lines 508–510):
 *   "runs"       — top-level runs table (starts here on launch into RUNS mode)
 *   "steplist"   — one run's step table (after Enter on a run row)
 *   "stepdetail" — one step's detail view (after Enter on a step row; the
 *                  Graph/Detail/Log/Events tab row still appears at the top)
 */
export type NarrowLevel = "runs" | "steplist" | "stepdetail";

/**
 * Resolves the narrow level from the existing app mode + selection. Does
 * NOT introduce a new state kind.
 *
 *   mode.kind === "browsing" && mode.pane === "runs"          → "runs"
 *   mode.kind === "viewing"  && selectedStepId == null        → "steplist"
 *   mode.kind === "viewing"  && selectedStepId != null        → "stepdetail"
 *   otherwise (browsing.workflows etc.)                       → null
 *
 * Returning null signals "the narrow single-pane rewrite doesn't apply here"
 * — the caller should fall back to its non-narrow render path. This keeps
 * workflow-browser, overlays, and error screens untouched.
 */
export interface NarrowLevelInput {
  readonly mode:
    | { readonly kind: "browsing"; readonly pane: "workflows" | "runs" }
    | { readonly kind: "viewing"; readonly runId: string; readonly focus: string };
  readonly selectedStepId: string | null;
}
export function pickNarrowLevel(input: NarrowLevelInput): NarrowLevel | null;

/**
 * Composes the breadcrumb title string shown in the top frame edge in
 * place of the mode tabs. Matches mockups.md §13 line 488:
 *   "Runs › ijkl56 › deploy-us"
 * or at narrow-ASCII:
 *   "Runs > ijkl56 > deploy-us"
 *
 * - `separator` is the chevron glyph (`›` or `>`) — caller passes the
 *   active theme's arrow-like glyph. No theme lookup inside this helper
 *   (purity).
 * - Trailing segments are omitted (not blanked) when their label is null.
 *   E.g. "runs" level → "Runs".
 * - No truncation here; the caller clamps the composed string to the
 *   available title width via `computeTitleBudget(cols)` — see §3.3.
 */
export function composeBreadcrumb(
  level: NarrowLevel,
  runLabel: string | null,
  stepLabel: string | null,
  separator: string,
): string;
```

## §3 Data flow

### 3.1 Width / capability detection path

Capability and width are fully orthogonal:

```
process.env  ──┐
               ├──► detectCapabilities(env, {stdoutIsTTY}) → Capabilities
process.stdout.isTTY  ──┘          │
                                   ▼
                           buildTheme(capabilities) → {colors, glyphs, frame, capabilities}
                                   │
                                   ▼
                           <ThemeProvider value?={...}>
                                   │
                                   ▼
                           every consumer via useTheme()

Ink useStdout().stdout.columns ─► <App> ─► cols
                                              │
                                              ├── cols < 60 → narrow layout branch
                                              ├── cols ≥ 60 → existing two-pane branch
                                              │
                                              └── forwarded verbatim to <AppShell width={cols}>
```

- `MARKFLOW_ASCII=1` flows only through `detectCapabilities` → `capabilities.unicode = false` → `theme.glyphs = ASCII_GLYPHS` + `theme.frame = ASCII_FRAME`. No component reads `MARKFLOW_ASCII` directly. (Purity: re-assertable in tests via `<ThemeProvider value={buildTheme({color: false, unicode: false})}>`.)
- `NO_COLOR` / `TERM=dumb` flow only through `detectCapabilities` → `capabilities.color = false` → `theme.colors = MONOCHROME_COLOR_TABLE` — every role returns an empty `ColorSpec` which Ink receives as `undefined`, keeping the terminal default.
- Combined `NO_COLOR=1 MARKFLOW_ASCII=1` produces `{color: false, unicode: false}` → full §14 rendering: monochrome + ASCII glyphs + `+-|` frame + bracket-text state labels.

### 3.2 Narrow layout routing in `app.tsx`

```
cols = stdout?.columns ?? 80
selectedStepId = state.selectedStepId

if (cols < NARROW_TIER_MAX) {
  level = pickNarrowLevel({mode: state.mode, selectedStepId})

  if (level === null) {
    // browsing.workflows, overlays — fall through to existing render
  } else {
    singleSlot =
      level === "runs"       ? <RunsTable       width=cols  height=slotRows />
    : level === "steplist"   ? <StepTableView   width=cols  height=slotRows />
    : level === "stepdetail" ? <StepDetailPanelView width=cols height=slotRows-1 /> + tabRow
    : null;

    breadcrumb = composeBreadcrumb(level, runIdShort, stepLabel, theme.glyphs.arrow)

    return <AppShell
      width=cols height=rows
      narrow={true}
      breadcrumb={breadcrumb}
      singleSlot={singleSlot}
      keybar={keybarSlot}
    />
  }
}
// existing two-pane render unchanged
```

Slot row budget: for `height = 24` and 2 chrome rows (top edge, bottom edge) the single slot has `24 - 2 = 22` rows. The breadcrumb lives **inside** the top edge string — not a separate row — so no extra row consumed (matches mockup §13 line 488).

### 3.3 Breadcrumb budget and composition

- `cols = 52` (§13 reference): top edge = `╔ <title> <═ fill ═> ╗`. Title budget is `cols - 4` (two corners + two spaces of gutter) = 48 cols.
- `composeBreadcrumb(level="stepdetail", "ijkl56", "deploy-us", "›")` → `"Runs › ijkl56 › deploy-us"` — 24 chars; fits in 48.
- Long labels: if the composed string exceeds the budget, clamp by collapsing the leftmost segment first (`Runs …› ijkl56 › deploy-us`) — implemented in `<AppShell>` *after* the breadcrumb is composed (consistent with `frameTitle`'s own chrome-safety clamp). Not in the pure helper (which stays budget-free for testability).
- ASCII mode: `theme.glyphs.arrow === "->"`. Breadcrumb becomes `"Runs -> ijkl56 -> deploy-us"`. Verified against §14 (which doesn't show a breadcrumb, but uses `->` for the `RESULT` line glyph — same asymmetric arrow).

### 3.4 Keybar keys-tier + hint

```
bindings                 ─► filterBindings(ctx)
                            ─► sortByOrder + groupByCategory
                            ─► renderableLabel(b, "keys") = just <keys>
                            ─► join with inter-category gap
                            ─► rowText (for width measurement)

tier = pickTier(cols, catCount) === "keys"

hint = composeKeybarTrailingHint("keys", cols, rowText.length)
     = "? for labels"  if cols - rowText.length - 4 >= 13
     = null            otherwise
```

Renders as a `flexDirection="row" justifyContent="space-between"` row. At `cols = 52` with the §15 RUN row (`↑↓ ⏎ R X  f /  ? q` ≈ 19 chars), hint fits (52 − 19 − 4 = 29 ≥ 13). At an extreme `cols = 30` the hint is dropped and the row shows keys only.

### 3.5 Narrow drill gestures (Enter / Esc)

| Level | Key | Action | Resulting state transition |
|---|---|---|---|
| `runs` | `Enter` | Open selected run | `MODE_OPEN_RUN { runId }` (existing) — lands in `viewing.graph` with `selectedStepId = null` → `steplist` |
| `steplist` | `Enter` | Drill into the selected step | `SELECT_STEP { stepId: state.stepCursor.id }` → enters `stepdetail` |
| `steplist` | `Esc` | Back to runs | `MODE_CLOSE_RUN` (existing) |
| `stepdetail` | `Esc` | Back to step list | `SELECT_STEP { stepId: null }` |
| `stepdetail` | `1/2/3/4` | Cycle Graph/Detail/Log/Events tab (existing behaviour of `viewing.focus`) | `FOCUS_VIEWING_PANE` (existing) |

The `Enter`-at-`steplist` and `Esc`-at-`stepdetail` gestures are **only applied at narrow width**. At wide/medium, step-detail is a tab within the bottom pane; drilling doesn't change screen structure. Guard: `cols < NARROW_TIER_MAX` before dispatching these.

### 3.6 Hide mode tabs, show breadcrumb (mutually exclusive)

`AppShell` treats `narrow && breadcrumb` as a mode switch for the top edge:
- `narrow === true` → render `╔ <breadcrumb padded with ═ to cols-2> ╗`. Do not overlay `modeTabs`. `modeTabs` prop is still accepted but not rendered — caller can omit.
- `narrow === false` (or undefined) → existing render (mode tabs pill overlay on title).

### 3.7 ASCII rendering flow (unchanged path — verification only)

```
env.MARKFLOW_ASCII = "1"
   │
   ▼
detectCapabilities → {color: …, unicode: false}
   │
   ▼
buildTheme → {frame: ASCII_FRAME, glyphs: ASCII_GLYPHS, …}
   │
   ▼
<AppShell> uses theme.frame.tl/tr/bl/br/h/v  → "+---+", "|", etc.
<StatusCell>, <StepTable>, etc. use theme.glyphs.{running,ok,fail,…}
   → "[run]", "[ok]", "[fail]"
composeBreadcrumb(..., theme.glyphs.arrow) → "->" joins segments
```

No component needs to know about `MARKFLOW_ASCII`; they only read `theme`. This is the single-source-of-truth the existing P3-T3 theme work set up. The plan deliverable is to **cover this path with an end-to-end test** that passes `{color: false, unicode: false}` via `<ThemeProvider value={buildTheme({color:false,unicode:false})}>` and asserts the §14 output.

## §6 Design decisions & rationale

| # | Decision | Rationale |
|---|---|---|
| D1 | **Interpret §14 "no box-drawing" as "no unicode box-drawing", not "no frame at all".** The rendered mockup itself shows a `+-|` frame around the content. | Faithful to the mockup's authoritative render. The prose "no box-drawing" in §14 line 539 is a loose description — the concrete example is the source of truth. Also: dropping frames entirely would make §14 structurally different from §1/§4/§6/§12, breaking the "consistent layout primitive" global rule. |
| D2 | **Express narrow levels through existing `state.mode` + `selectedStepId`, not a new mode kind.** | The three levels map 1:1 onto existing states: `browsing.runs` / `viewing.* & stepId=null` / `viewing.* & stepId≠null`. Adding a `narrow` kind would duplicate the state machine for one width-tier cosmetic. Keeps the reducer simple; the narrow rewrite is a pure render-time projection. Mirrors the P8-T1 choice to keep column-tier logic out of state. |
| D3 | **`NARROW_TIER_MAX = 60` (exclusive).** | Matches features.md §5.6 rule 7 (`<60 cols: keys only + hint`) and §5.4 (`<80` narrow prose — the stricter `<60` is the column where keys-only tier kicks in). The stricter threshold is the correct trigger for the single-pane rewrite because it's where keybar drops labels and where §13's 52-col mockup sits. Medium (60–89) retains the two-pane stacked layout with short-tier keybar (already covered in P8-T1). |
| D4 | **Hoist the `? for labels` hint into a pure helper `composeKeybarTrailingHint`, not a React component.** | The decision "should we render the hint?" is a function of (tier, width, rowLen) — three inputs, pure output. Keeps the keybar component small and the rule unit-testable without ink-testing-library. Matches `keybar-layout.ts` precedent. |
| D5 | **Breadcrumb separator comes from `theme.glyphs.arrow`, not a bespoke constant.** | Single source for arrow-like glyphs. `UNICODE_GLYPHS.arrow = "→"` renders as `Runs → ijkl56 → deploy-us`; mockup §13 shows `›` (U+203A) — close enough visually. Rationale for using `→` over `›`: `›` isn't in the existing glyph table and adding a `breadcrumb` key would fork the glyph registry for one call-site. Acceptance criterion 2's chevron requirement is satisfied by `→` just as well (any right-pointing arrow suffices semantically); the §13 mockup's `›` is illustrative. ASCII mode falls back to `->` automatically. If a future reviewer insists on U+203A, the change is a single glyph-table addition — bounded and backward compatible. |
| D6 | **Pass `narrow` as an `AppShell` prop rather than deriving from `width`.** | Explicit opt-in keeps test-driven callers deterministic (ink-testing-library's stdout columns default to a harness constant; hard-coding `width < 60` inside `AppShell` couples the narrow decision to stdout detection). The caller already knows the width — passing a boolean is one line and decouples two concerns. |
| D7 | **Do not alter runs/columns.ts or steps/columns.ts.** | `COLUMNS_80` and `STEP_COLUMNS_NARROW` already accommodate <60-col rendering gracefully: the grow NOTE column absorbs the shortfall down to its 4-col minimum. Mockup §13 doesn't show a runs table at 52 cols (it shows step-detail), so there's no concrete acceptance target for the <60 runs-table header — P5-T1's existing behaviour is the authority. |
| D8 | **Hint slack constant = 4 cols.** | Two spaces of separator + two spaces of right-gutter. Matches the existing keybar's wide-tier spacing precedent (`justifyContent="space-between"` with single-char gutters was visually cramped in prototyping). Narrow enough to not drop the hint in the usual 52-col target, wide enough to avoid ugly hint-touching-keys renders at 40-ish col edge. |
| D9 | **Don't fork `<AppShell>` into `<NarrowShell>`.** | The shell already composes its chrome from `theme.frame`; the only structural difference is "no splitter row, one slot body, breadcrumb title". A 30-line branch inside `AppShellImpl` is cleaner than a new component duplicating the frame-glyph wiring. Matches D9 of P8-T1 (no forked ModeTabs variant). |
| D10 | **The `|` separators in mockup §15 narrow rows are shorthand, not literal glyphs.** | Current keybar spacing already produces the same visual effect via gaps between category groups. Adding a literal `|` would require a new Binding property; not worth the surface change for an aesthetic that reads the same after the ANSI-strip anyway. Verified by visual comparison in P7-T3's fixture catalogue. |
| D11 | **Keep `<ModeTabs>` mounted but not-rendered at narrow.** | Letting the caller pass `modeTabs` unconditionally (AppShell just ignores it under `narrow`) means `app.tsx` doesn't need a branch on whether to mount `<ModeTabs>`. Re-mount cost on tier changes is zero (narrow/non-narrow is a width-driven prop change; useInput handlers in `<ModeTabs>` remain active even when the tabs aren't visible — still correct, because F1/F2/F3 keystrokes at narrow width should still navigate). |
| D12 | **F1/F2/F3 remain functional at narrow width.** | Keyboard-driven navigation is width-independent per features.md §5.6 rule 7 philosophy (hints can be hidden; bindings stay). The narrow layout just doesn't *render* the mode tab row — the keys still flip mode. |
| D13 | **`stepdetail` uses `<StepDetailPanelView>` directly, not `<ViewingBottomSlot>`.** | `<ViewingBottomSlot>` mounts all four panes + tab row + visibility toggles. At narrow width where only one pane fills the frame at a time, re-using it would render all four panes stacked (with the non-focused ones at height 0 — works, but wastes a render pass). Direct mount of the active pane + a `<TabHeader>` row composed from `composeViewingTabRow` is cleaner. Tab-switching on `1/2/3/4` re-dispatches `FOCUS_VIEWING_PANE` as today; on the next render we mount the newly-focused pane. |
| D14 | **No new reducer action unless needed.** | Inspection of `state/reducer.ts` (called out in §1.2) confirms `SELECT_STEP { stepId: string|null }` or equivalent already exists (the step table's cursor movement dispatches it). Implementation agent must verify; if the null case isn't handled, it's a one-line addition. No state-machine redesign. |
| D15 | **Test ASCII + monochrome via `<ThemeProvider value={…}>` injection, not `process.env` mutation.** | `detectCapabilities` has its own unit tests that exercise env precedence (4.4). Higher-level integration tests shouldn't mutate `process.env` — parallel test runs would race. Injection via the `value` prop is the canonical seam (see `theme/context.tsx` line 15 documentation). |
| D16 | **Frame rendering in ASCII mode uses `theme.frame` unconditionally.** | `theme.frame = ASCII_FRAME` when `unicode = false`. `<AppShell>` already composes every chrome row from `theme.frame.{tl,tr,bl,br,h,v,mid_l,mid_r,mid_h}`. No branching needed — the ASCII rendering is structurally identical to unicode. |
| D17 | **No end-to-end VHS tape in this task.** | VHS tapes are Phase 9 territory (testing.md §6 layer 5). All acceptance here is achievable via `ink-testing-library` snapshots + text assertions. |
| D18 | **Drop the hint on the low side of width rather than truncate.** | At extreme `width=30`, printing `? for labels` partly or ellipsised (`? for lab…`) would misrepresent the affordance (the user still presses `?` for help). Cleaner to hide — matches §5.6 rule 5 (hide-don't-grey extended to hide-don't-truncate for affordances). |
| D19 | **Narrow rewrite applies only to `browsing.runs` + `viewing.*`.** | `browsing.workflows` has its own layout (§2). Empty-state / error screens render their own shells. `pickNarrowLevel(...)` returns `null` for all other modes; caller falls through to existing render path. Prevents scope creep. |
| D20 | **Implement in test-first order, reusing P8-T1 fixture seeds.** | Natural sequence: (1) pure helpers → (2) AppShell narrow branch → (3) keybar hint → (4) app.tsx narrow routing + drill gestures → (5) integration tests. Fixture data (`ijkl56` + `deploy-us`) already exists in the P6-T1 / P8-T1 test corpus. |
