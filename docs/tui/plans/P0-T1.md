## 5. Risks & unknowns

### Risks with mitigations

1. **Lockfile collision.** npm workspaces require a single root `package-lock.json`. The existing `package-lock.json` was generated for the non-workspace single-package layout.
   - *Mitigation.* **Delete** `/Users/boyd/wip/markdown-flow/package-lock.json` before running `npm install` at the new root. Do **not** move it into `packages/markflow/` — npm workspaces disallow per-package lockfiles and will ignore or warn about them. The root `npm install` regenerates a clean lockfile that encodes the workspace structure.
2. **Stale `dist/` and `node_modules/`.** Moving these would carry over paths/links that are wrong post-reorg. Leaving the old root `node_modules/` in place with no `package.json` next to it will confuse npm.
   - *Mitigation.* **Delete** both `/Users/boyd/wip/markdown-flow/dist/` and `/Users/boyd/wip/markdown-flow/node_modules/` **before** creating the `packages/` structure. `npm install` at the new root regenerates `node_modules/` with workspace links; `npm run build -w packages/markflow` regenerates `dist/`.
3. **`tsconfig.json` paths / baseUrl.** If the tsconfig had `paths` or `baseUrl` aliases, they could break when the config moves.
   - *Mitigation.* Verified (§3.2): the current tsconfig has **no** `paths` or `baseUrl`. All paths (`rootDir`, `outDir`, `include`) are relative and still resolve when the file is moved into the package dir. No action needed.
4. **`tsup.config.ts` entries.** Entries are `src/core/index.ts` and `src/cli/index.ts`.
   - *Mitigation.* Verified (§3.2): entries are relative, they resolve correctly when `tsup` runs with `packages/markflow/` as cwd (which it does when invoked via `npm run build -w packages/markflow`). No action needed.
5. **Root `.npmrc` with `@jsr:registry`.** The `@jsr:registry=https://npm.jsr.io` config is critical for the `@emily/mermaid-ast` dep (`npm:@jsr/emily__mermaid-ast@^0.8.2`).
   - *Mitigation.* `.npmrc` stays at the repo root — npm reads root `.npmrc` for all workspace operations. Verify post-install that `packages/markflow` resolves `@emily/mermaid-ast` correctly (it's the first dep listed in `packages/markflow/package.json#dependencies`).
6. **`workspace:*` protocol in published tarball.** When `markflow-tui` is eventually published, `"markflow": "workspace:*"` must be rewritten to a real semver range. This is not a P0-T1 concern (package is `private: true`) but flagging so the later publish task doesn't get caught out.
   - *Mitigation.* Out of scope for P0-T1. Will be handled when publishing is set up — typically via `npm publish --workspaces` which rewrites `workspace:*` to the current version.
7. **`markflow-tui/dist/cli.js` vs gitignore.** The stub file needs to be checked in so the `bin` symlink has something to point to immediately after clone+install, but `packages/*/dist/` is gitignored.
   - *Mitigation.* Use a negation entry (`!packages/markflow-tui/dist/cli.js`) in the root `.gitignore` — documented in §3.3. Alternative if the team dislikes negation: move the bin target to `packages/markflow-tui/bin/cli.js` (outside `dist/`) and update the `bin` field accordingly. The negation is cleaner because it preserves the spec's `bin: { "markflow-tui": "./dist/cli.js" }` wording verbatim.
8. **Runtime `runs/` directory location.** `runs/` is created on demand by `markflow run`. Today users may have a `runs/` at the repo root (gitignored).
   - *Mitigation.* `runs/` stays at the repo root — it's a user/runtime artifact, not a package file. The `RunManager` resolves paths relative to `process.cwd()`, so as long as users invoke `markflow` from the repo root or their own workspace dir, behavior is unchanged.
9. **`./markflow` root shell wrapper.** Its hardcoded `src/cli/index.ts` path becomes invalid post-move.
   - *Mitigation.* Modify in place per §3.4 to use an absolute path resolved from the script's own location. Alternative: delete the wrapper and tell users to run `npm run dev -w packages/markflow`. Modifying is safer — existing muscle-memory (`./markflow run foo.md`) keeps working.
10. **Docs drift.** `CLAUDE.md` and `README.md` reference `src/core/…` paths that move under `packages/markflow/`.
    - *Mitigation.* **Out of scope for P0-T1** per spec ("Leave `docs/` and `README.md` at the repo root (unchanged)"). File a follow-up task to sweep docs after the workspace reorg lands. Flag in the PR description.

### Unknowns

- **None blocking.** All ambiguities in the spec ("current version", "`workspace:*`", "stub binary") have a single reasonable interpretation documented above. See §6 for anything requiring the reviewer's explicit sign-off.
