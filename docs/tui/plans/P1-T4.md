## 0. Summary

Add `watch(options?)` to the public `RunManager` factory result. It returns an
`AsyncIterable<RunEvent>` that first emits an `added` event for every run
directory already present on disk (alphabetical by runId), then continues to
emit `added` / `updated` / `removed` as the filesystem changes, until the
caller aborts via an `AbortSignal` (or `break`s out of the `for await`). Each
event carries a freshly-projected `RunInfo` snapshot (via the existing
`getRun` → `readEventLog` → `replay` path), so consumers never have to call
`listRuns()` themselves.

Backed by `fs.watch` — one watcher on the runs directory plus one watcher per
existing run (targeting that run's `meta.json`). `meta.json` is the
write-through cache whose writes trigger a 50 ms debounced re-projection.

The TUI run list (Phase 6) consumes this; nothing in this task wires it up.

## 1. Grounding — current `RunManager` shape

From `/Users/boyd/wip/markdown-flow/packages/markflow/src/core/run-manager.ts`

```ts

  createRun(workflowDef: WorkflowDefinition): Promise<RunDirectory>;
  openExistingRun(id: string): Promise<ResumeHandle>;
  listRuns(): Promise<RunInfo[]>;
  getRun(id: string): Promise<RunInfo | null>;
  completeRun(id: string, status: RunStatus): Promise<void>;
}

export function createRunManager(runsDir = "./runs"): RunManager { ... }
```

**Key facts this plan relies on:**

- `RunManager` is an **interface** returned by the `createRunManager(runsDir)`
  **factory**. All state (`runsDir`) is captured in the closure, not on
  `this`. The new `watch()` method is added both to the interface type and to
  the object literal the factory returns. No class refactor is required.
- `getRun(id)` already does the authoritative projection: reads `meta.json`,
  reads `events.jsonl` via `readEventLog`, folds through `replay()`, returns
  the composed `RunInfo` or `null`. **We reuse it verbatim** — every `added`
  and `updated` event emitted by `watch()` calls `getRun(id)` to obtain the
  `RunInfo`. This guarantees parity with `ls` / `show` and avoids duplicating
  the projection logic.
- `meta.json` is written in three places only: `createRun` (initial), the
  `openExistingRun` `status: "running"` rewrite, and `completeRun`. Watching
  `meta.json` therefore catches every status transition that matters to the
  run-list UI. (Tokens flipping inside `events.jsonl` do **not** change
  `meta.json`; for the TUI run list that is fine — P1-T3's `tailEventLog` is
  the per-run event stream.)
- `listRuns()` already sorts by `entries.sort().reverse()` (reverse
  alphabetical). For startup `watch()` emission we deliberately choose **plain
  alphabetical by runId** (documented below, §4).

## 3. Implementation strategy

`watch()` is an async generator implemented as a private helper inside
`createRunManager`, returned from the `watch` method. It closes over `runsDir`
and has access to `getRun` via the same closure (not `this`, since
`getRun` is defined as an object-literal method — we hoist `getRun` into a
local `const` so both `listRuns` and `watch` share the same reference).

### 3.1 State kept by one `watch()` invocation

```ts
// local to the generator invocation
const runWatchers = new Map<string, fs.FSWatcher>();   // runId -> meta.json watcher
const debounceTimers = new Map<string, NodeJS.Timeout>();
const seen = new Set<string>();                         // runIds for which `added` has already been emitted
const queue: RunEvent[] = [];                           // events ready to yield
let resolveNext: (() => void) | null = null;            // one waiter at a time
let done = false;
```

A single queue + one waiter is sufficient: `AsyncIterable` is a single
consumer (one `for await`). Multiple consumers would each call `watch()`
and get independent generators.

### 3.2 Emission helper

```ts
function emit(event: RunEvent): void {
  if (done) return;
  queue.push(event);
  if (resolveNext) {
    const r = resolveNext;
    resolveNext = null;
    r();
  }
}
```

### 3.3 Attaching the top-level watcher FIRST, then listing

This order is critical to avoid the startup race (§8). Concretely:

1. `await mkdir(runsDir, { recursive: true })` — create the runs dir if
   absent so `fs.watch` doesn't throw `ENOENT`. Idempotent.
2. Attach top-level `fs.watch(runsDir, { persistent: false })`. Its handler
   is `onTopLevelChange(eventType, filename)` (see §3.4).
3. Enumerate existing runs via `readdir(runsDir)`. For each, call
   `handleAdded(runId)` — which is the same path the top-level watcher uses,
   guarded by the `seen` set so the race window is harmless.

Using `persistent: false` prevents the watcher itself from keeping the Node
event loop alive — the caller controls lifetime via their own work plus the
`AbortSignal`.

`{ recursive: false }` is the default; stated for clarity — we explicitly do
NOT use `{ recursive: true }` because its behaviour differs across platforms
(macOS supports it, Linux historically did not until v20).

### 3.4 Top-level change handler

```ts
async function onTopLevelChange(eventType: string, filename: string | null) {
  if (!filename) return; // platforms may coalesce; best-effort
  const runId = filename;
  const runPath = join(runsDir, runId);
  try {
    await access(runPath);
    // directory exists (or was just created)
    if (!seen.has(runId)) await handleAdded(runId);
  } catch {
    // directory missing -> removal
    if (seen.has(runId)) handleRemoved(runId);
  }
}
```

Rationale for the `access` probe: `fs.watch` on a directory fires `rename`
for both creation and deletion of immediate children (and `filename` is
unreliable on some platforms — we tolerate a null by returning; we accept
we may miss an unlabeled rename event, which is extremely rare on the
platforms Node officially supports).

### 3.5 `handleAdded(runId)`

```ts
async function handleAdded(runId: string): Promise<void> {
  if (seen.has(runId)) return;
  seen.add(runId);

  // Attach per-run meta.json watcher BEFORE projecting, so any write that
  // lands after projection but before we subscribe still wakes us up.
  const metaPath = join(runsDir, runId, "meta.json");
  try {
    const watcher = fsWatch(metaPath, { persistent: false }, () =>
      scheduleUpdate(runId),
    );
    watcher.on("error", () => {
      // meta.json disappeared mid-watch — treat as removal on next tick.
    });
    runWatchers.set(runId, watcher);
  } catch {
    // meta.json not yet written; best-effort. The top-level watcher will
    // refire when the file lands (rename on the parent dir).
  }

  const snapshot = await getRunLocal(runId);
  if (snapshot) emit({ kind: "added", runId, snapshot });
  // If snapshot is null (meta.json / events.jsonl not yet flushed), skip
  // emission. The next meta.json write will trigger scheduleUpdate and
  // eventually emit an `updated` once the snapshot is readable. We rely on
  // the consumer not caring whether the first event is `added` vs
  // `updated` for a brand-new run — but for correctness we upgrade the
  // first successful projection for an unseen-snapshot run to `added` by
  // tracking a `firstEmit` flag per runId (see §3.7).
}
```

### 3.6 `handleRemoved(runId)`

```ts
function handleRemoved(runId: string): void {
  seen.delete(runId);
  const w = runWatchers.get(runId);
  if (w) {
    w.close();
    runWatchers.delete(runId);
  }
  const t = debounceTimers.get(runId);
  if (t) {
    clearTimeout(t);
    debounceTimers.delete(runId);
  }
  emit({ kind: "removed", runId });
}
```

### 3.7 `scheduleUpdate(runId)` — the debouncer

```ts
function scheduleUpdate(runId: string): void {
  const existing = debounceTimers.get(runId);
  if (existing) clearTimeout(existing);
  const t = setTimeout(async () => {
    debounceTimers.delete(runId);
    const snapshot = await getRunLocal(runId);
    if (!snapshot) return; // file may have been removed mid-flight
    // If we never successfully emitted `added` for this run, promote this
    // to `added` instead of `updated`.
    if (!emittedAddedFor.has(runId)) {
      emittedAddedFor.add(runId);
      emit({ kind: "added", runId, snapshot });
    } else {
      emit({ kind: "updated", runId, snapshot });
    }
  }, DEBOUNCE_MS);
  debounceTimers.set(runId, t);
}
```

With:

```ts
const DEBOUNCE_MS = 50;
const emittedAddedFor = new Set<string>(); // tracks runs whose `added` was emitted
```

`handleAdded` sets `emittedAddedFor.add(runId)` after a successful emit.

### 3.8 Generator body

```ts
async function* watch(opts?: WatchOptions): AsyncIterable<RunEvent> {
  const signal = opts?.signal;
  if (signal?.aborted) return;

  // setup (mkdir, attach top-level watcher, enumerate, emit startup `added`s)
  await setup();

  const onAbort = () => {
    done = true;
    teardown();
    if (resolveNext) { const r = resolveNext; resolveNext = null; r(); }
  };
  signal?.addEventListener("abort", onAbort, { once: true });

  try {
    while (!done) {
      while (queue.length > 0) {
        const ev = queue.shift()!;
        yield ev;
      }
      if (done) break;
      await new Promise<void>((resolve) => { resolveNext = resolve; });
    }
  } finally {
    done = true;
    signal?.removeEventListener("abort", onAbort);
    teardown();
  }
}
```

`teardown()` closes the top-level watcher, every per-run watcher, and clears
every debounce timer.

## 4. Startup emission order

On first call to `watch()`, immediately emit `added` for each existing run
directory **in alphabetical order by runId** (ascending). Determinism is the
motivating constraint; the TUI will re-sort by its own criteria. Rationale:

- `listRuns()` uses `.sort().reverse()` so the newest runs surface first in
  the CLI. But `watch()` is a stream feeding a UI layer that will sort
  itself — pushing a stable canonical order (ascending) keeps the watch
  contract independent of display concerns.
- Alphabetical works because run ids are ISO-8601 timestamps with `:`/`.`
  replaced by `-` (see `createRun`), which preserves chronological order
  under string comparison.

This ordering is covered by test case §6.9.

## 5. Snapshot reads

Every `added` / `updated` event calls `getRun(runId)` (hoisted to
`getRunLocal` inside the factory closure so both `listRuns` and `watch` share
one reference). `getRun` already does the full `readEventLog` → `replay()`
projection, matching the "projected via replay() into RunInfo" requirement
from the architecture doc.

We deliberately do **not** take a `meta.json`-only fast path. Reasons:

- `RunInfo.steps` is sourced from `snapshot.completedResults`, which comes
  from `events.jsonl`, not `meta.json`. A `meta.json`-only read would
  produce a `RunInfo` with stale `steps`.
- `replay()` is a pure in-memory fold; on a typical run (hundreds of events)
  it is microseconds. The debounce already collapses bursty writes.

Return value of `getRun` is `RunInfo | null`. We treat `null` as "not ready
yet"; the next meta.json write triggers another `scheduleUpdate`, which will
re-project.

Helper (inline in the test file):

```ts
async function collect(
  iter: AsyncIterable<RunEvent>,
  until: (events: RunEvent[]) => boolean,
  timeoutMs = 1000,
): Promise<RunEvent[]> {
  const events: RunEvent[] = [];
  const timeout = new Promise<never>((_, rej) =>
    setTimeout(() => rej(new Error("collect timed out")), timeoutMs),
  );
  const drain = (async () => {
    for await (const ev of iter) {
      events.push(ev);
      if (until(events)) return;
    }
  })();
  await Promise.race([drain, timeout]);
  return events;
}
```

Each test uses a dedicated `AbortController` and aborts it in a `finally`
block to guarantee watcher cleanup even on assertion failure.

**Cases:**

1. **Empty runs dir → no immediate yield.** Create empty runs dir, start
   `watch()` with `AbortSignal`, wait 150 ms, abort. Assert zero events
   collected.
2. **Existing runs → `added` for each on startup.** Pre-create two run
   directories (via `createRun`) before calling `watch()`. Expect two
   `added` events, one per runId, each with a non-null `snapshot.id`
   matching its runId.
3. **New run directory → `added` after startup.** Start `watch()`, then
   invoke `createRun` (which writes the dir, `meta.json`, and
   `events.jsonl`). Expect one `added` event with the new runId within
   500 ms.
4. **`meta.json` update → `updated`.** Start `watch()`, `createRun` (expect
   `added`), then call `completeRun(id, "complete")`. Expect one `updated`
   event with `snapshot.status === "complete"` after the debounce window.
5. **Debounced: three rapid `meta.json` writes within 50 ms → one
   `updated`.** Start `watch()`, `createRun` (expect `added`), then perform
   three `writeFile(meta.json, ...)` calls in a tight loop (no `await`
   between them except the minimum). Assert exactly one `updated` event
   arrives, carrying the final status.
6. **Run directory removed → `removed`.** Start `watch()`, pre-create a run,
   observe `added`, then `rm -rf` the run directory. Expect one `removed`
   event with the matching runId.
7. **AbortSignal cleans up.** Start `watch()`, abort the signal, assert the
   generator returns within 100 ms (loop exits, `for await` completes
   normally). After abort, create another run — assert it is NOT yielded
   (the generator has already returned). Indirect test for "no leaked
   watchers".
8. **`added` before `updated` on same run.** Start `watch()`, `createRun`
   then immediately `completeRun`. Events collected in order must be
   `[added, updated]` for that runId, never the reverse.
9. **Startup order is alphabetical by runId.** Pre-create three runs with
   deliberately non-alphabetical mtimes (create them out of order, then
   rename). Assert the startup `added` events arrive in ascending runId
   order. (Implementation: directly `mkdir` three dirs with ids `aaa`,
   `bbb`, `ccc` in the order `ccc`, `aaa`, `bbb`, and hand-write their
   `meta.json` + empty `events.jsonl` so `getRun` returns a snapshot.)

All cases use `AbortController` and `abort()` in a `finally` block.

## 8. Risks and mitigations

- **`fs.watch` recursion differences across platforms.** Mitigation: do NOT
  use `{ recursive: true }`. Attach exactly one top-level watcher on
  `runsDir` + one watcher per run on `<runsDir>/<runId>/meta.json`. Both are
  shallow. Linux, macOS, and Windows all support this shape.
- **`filename` is sometimes null on Linux.** Mitigation: the
  `onTopLevelChange` handler bails out when `filename` is null. On the next
  real event (file metadata change, next directory mutation) the handler
  re-converges. Alternatives (re-listing the directory on every null
  filename) are strictly worse — O(N) per spurious event.
- **Startup race: a run dir is created between "attach" and "list".**
  Mitigation: attach the top-level watcher **before** enumerating existing
  runs (§3.3). Dedupe by the `seen` set so any overlap is harmless.
- **Debounce timer leaks.** Mitigation: `handleRemoved` clears any pending
  timer for that runId. `teardown()` iterates `debounceTimers.values()` and
  clears all of them. Every `setTimeout` has an explicit `clearTimeout`
  path.
- **meta.json is replaced atomically by some writers; `fs.watch` may lose
  the inode.** Current markflow writers use `writeFile` (not rename), so
  the inode is preserved and the existing watcher continues to fire. If
  future writers switch to atomic-rename, the per-run watcher will go
  silent after the first rename. Documented as a known follow-up; not
  solved in this task because no current writer triggers it.
- **`getRun` returning `null` on a freshly-created dir (between
  `mkdir(runPath)` and `writeFile(meta.json)`).** Mitigation: §3.7 promotes
  the first successful projection to `added` via `emittedAddedFor`.
- **Multiple consumers of the same `RunManager`.** Each `watch()` call
  returns an independent generator with its own queue and watchers. This
  is by design and documented in the JSDoc on `watch`.
