## §0 Starting state

Verified against `feat/TUI` HEAD.

### 0.1 Engine surface is already frozen

- `executeWorkflow(workflow, { runsDir, workspaceDir, resumeFrom, onEvent })` is the resume entrypoint (features.md §2.3). `resumeFrom: ResumeHandle` comes from `RunManager.openExistingRun(runId)` which takes the on-disk exclusive lock (`RunLockedError` surfaces contention).
- The CLI `resume` command (`packages/markflow/src/cli/commands/resume.ts` lines 73–112) shows the exact sequence for pre-resume mutations:
  - For every `--rerun <node>`: look up `tokenId` from `handle.snapshot.tokens` via `{nodeId → tokenId}`, then `handle.runDir.events.append({ type: "token:reset", v: 1, tokenId })`.
  - For every `--input KEY=VALUE`: `handle.runDir.events.append({ type: "global:update", keys: [...], patch: {...} })` in one batch.
- `WorkflowDefinition.inputs: InputDeclaration[]` gives `{name, required, default?, description}` — the wizard reads this to seed the input editor.
- `RunInfo.status` is `"error" | "suspended"` for resumable runs; the CLI rejects any other status (resume.ts lines 40–47). The TUI must mirror this gate.

**No engine edits are required.** Every wizard side-effect reuses existing exports. CLI-purity rule holds.

### 2.1 `src/resume/types.ts`

```ts
/** Minimal run metadata the wizard needs in its header line. */
export interface ResumableRun {
  readonly runId: string;
  readonly workflowName: string;
  readonly status: "error" | "suspended";
  readonly startedAt: string;         // ISO-8601 or pretty-formatted; wizard renders verbatim.
  readonly lastSeq: number;
  readonly lastEventLabel: string;    // e.g. "retry:exhausted at deploy-us" (see derive).
}

/** One row in the "Nodes to re-run" list. */
export interface RerunNode {
  readonly nodeId: string;
  readonly tokenId: string;           // Resolved upfront so `token:reset` is appendable.
  readonly state: "complete" | "error" | "waiting" | "skipped";
  readonly summary: string;           // "failed, 3/3 attempts" / "complete 18s" / "waiting".
  readonly preselected: boolean;      // true for failing/waiting nodes.
}

/** One row in the inputs editor. */
export interface InputRow {
  readonly key: string;
  readonly original: string;          // value from run:start or default.
  readonly draft: string;             // what the user is editing (seeded = original).
  readonly edited: boolean;           // draft !== original.
  readonly required: boolean;
}

/** Local state for the wizard's form + submit FSM. */
export interface ResumeFormState {
  readonly focus: ResumeFocus;
  readonly rerunCursor: number;       // index into nodes[]
  readonly inputsCursor: number;      // index into inputs[]
  readonly rerun: ReadonlySet<string>;
  readonly inputs: Readonly<Record<string, string>>;
  readonly fsm: "idle" | "submitting" | "error";
  readonly error: string | null;
}

export type ResumeFocus = "rerun" | "inputs" | "confirm";

export type ResumeFormAction =
  | { readonly type: "FOCUS_NEXT" }                                     // Tab
  | { readonly type: "FOCUS_PREV" }                                     // Shift+Tab
  | { readonly type: "CURSOR_MOVE"; readonly delta: number }            // ↑/↓ inside focused section
  | { readonly type: "RERUN_TOGGLE" }                                   // Space on current rerun row
  | { readonly type: "INPUT_EDIT"; readonly key: string; readonly value: string }
  | { readonly type: "SUBMIT_START" }
  | { readonly type: "SUBMIT_OK" }
  | { readonly type: "SUBMIT_FAIL"; readonly error: string };

/** Typed outcome of `resumeRun()`. */
export type ResumeSubmitResult =
  | { readonly kind: "ok" }
  | { readonly kind: "locked"; readonly runId: string; readonly lockPath: string }
  | { readonly kind: "notResumable"; readonly status: string }    // race: run became running/complete.
  | { readonly kind: "unknownNode"; readonly nodeId: string }     // rerun target not found in tokens.
  | { readonly kind: "error"; readonly message: string };
```

### 2.2 `src/resume/derive.ts`

```ts
import type {
  EngineEvent,
  InputDeclaration,
  RunInfo,
  Token,
  WorkflowDefinition,
} from "markflow";

/**
 * Snapshot summary for the wizard header. Returns `null` if the run is not
 * resumable (status not in {"error","suspended"}) — the wizard never opens
 * in that case. Pure; deterministic.
 */
export function deriveResumableRun(
  info: RunInfo,
  events: readonly EngineEvent[],
): ResumableRun | null;

/**
 * Build the rerun-node list. `tokens` is optional — when omitted, tokens are
 * reconstructed from `events` (replay). When provided (production path: via
 * `handle.snapshot.tokens`), trusted as source of truth.
 *
 * Order: failing nodes first, then waiting, then complete (by last transition
 * time desc), then skipped. Preselection: the single failing token, if any;
 * else the oldest waiting token; else none.
 */
export function deriveRerunNodes(
  info: RunInfo,
  events: readonly EngineEvent[],
  tokens?: ReadonlyMap<string, Token>,
): readonly RerunNode[];

/**
 * Project `workflow.inputs` + the originating `run:start` event into editable
 * rows. Keys in `run:start.inputs` that are NOT declared in `workflow.inputs`
 * are still surfaced (defensive — users may have added `--input foo=bar`
 * without declaring `foo`). See D8.
 */
export function deriveInputRows(
  workflow: WorkflowDefinition,
  events: readonly EngineEvent[],
): readonly InputRow[];

/**
 * The single "failure site" node to preselect — the nodeId of the most recent
 * `step:complete` with a non-zero exit, OR the step whose token reached
 * `"waiting"` if status is suspended. Returns `null` for happy terminals.
 */
export function findFailingNode(
  info: RunInfo,
  events: readonly EngineEvent[],
): string | null;

/** Boolean gate for `R` binding visibility. */
export function isRunResumable(info: RunInfo | null | undefined): boolean;
```

### 2.4 `src/engine/resume.ts`

```ts
export interface ResumeRunOptions {
  readonly runsDir: string;
  readonly runId: string;
  readonly rerunNodes: readonly string[];       // nodeIds; never tokenIds.
  readonly inputOverrides: Readonly<Record<string, string>>;
  /** Test seams — default to real `markflow` exports. */
  readonly manager?: RunManager;
  readonly execute?: typeof executeWorkflow;
  readonly parse?: typeof parseWorkflow;
  readonly readLog?: typeof readEventLog;
}

export async function resumeRun(
  opts: ResumeRunOptions,
): Promise<ResumeSubmitResult>;
```

**Implementation sketch (mirrors `cli/commands/resume.ts` lines 37–125):**
1. `readLog(runDir)` → `replay(events)` → guard `snapshot.status ∈ {"error","suspended"}`. On mismatch → `{kind:"notResumable", status}`.
2. `manager.openExistingRun(runId)` — on `RunLockedError` → `{kind:"locked"}`.
3. `manager.getRun(runId)` → `parse(meta.sourceFile)`.
4. For each `nodeId` in `rerunNodes`:
   - Look up `tokenId` via `{ for ([id, tok] of handle.snapshot.tokens): map[tok.nodeId] = id }`.
   - Missing → release handle, return `{kind:"unknownNode", nodeId}`.
   - Else `await handle.runDir.events.append({ type:"token:reset", v:1, tokenId })`.
5. If `Object.keys(inputOverrides).length > 0` → `await handle.runDir.events.append({ type:"global:update", keys:[...], patch:inputOverrides })`.
6. `await execute(workflow, { runsDir, workspaceDir: dirname(runsDir), resumeFrom: handle })` — no `onEvent`; the adapter subscription already sees the tail.
7. `finally`: `handle.release()` (idempotent; engine does this too).
8. Any unexpected throw → `{kind:"error", message}`.

## §3 Data flow

### 3.1 Surfacing the wizard (engine → UI)

1. User is viewing a terminal run (`state.mode.kind === "viewing"` and the active run has `status ∈ {"error","suspended"}`).
2. `<App>` computes `runResumable = isRunResumable(activeRun.info)` and threads it via `AppContext` → `GRAPH_KEYBAR` shows `R Re-run`.
3. User presses `R`. App-level `useInput` dispatches:
   ```ts
   dispatch({
     type: "OVERLAY_OPEN",
     overlay: {
       kind: "resumeWizard",
       runId: activeRun.runId,
       rerun: new Set<string>(
         findFailingNode(info, events) ? [findFailingNode(info, events)!] : []
       ),
       inputs: {},
       state: "idle",
     },
   });
   ```
4. Overlay open flips the keybar to `RESUME_KEYBAR` + `[RESUME]` mode pill and renders `<ResumeWizardModal>` centred over the AppShell (same overlay slot as `<ApprovalModal>`).
5. The modal reads `nodes` / `inputs` from `derive*` on mount (memoised on `events` identity).

**Explicit trigger only.** Unlike `<ApprovalModal>`, the wizard does NOT auto-open — terminal runs are a steady state, not an interrupt (features.md §3.8 + mockup §7 wording "R Re-run is the gateway").

### 3.2 Editing (UI → reducer)

- Toggle rerun: modal calls `onToggleRerun(nodeId)` → `<App>` dispatches `RESUME_WIZARD_TOGGLE_RERUN`. Existing reducer helper `toggleResumeRerun` handles it.
- Edit input: modal calls `onSetInput(key, value)` → `<App>` dispatches `RESUME_WIZARD_SET_INPUT`. Existing `setResumeInput` handles it.
- Cursor + focus: fully local to `<ResumeWizardModal>`'s `resumeFormReducer`. Never touches `AppState`.

### 3.3 Confirming (UI → engine)

1. User presses `Enter`.
2. Modal dispatches `RESUME_WIZARD_SUBMIT_START` on `AppState` (gates the keybar pill + guards double-submit) and local `SUBMIT_START`.
3. `<App>`'s `onConfirm` closure calls:
   ```ts
   resumeRun({
     runsDir,
     runId: overlay.runId,
     rerunNodes: Array.from(overlay.rerun),
     inputOverrides: overlay.inputs,
   });
   ```
4. On `{kind:"ok"}`: `<App>` dispatches `RESUME_WIZARD_SUBMIT_DONE` → overlay closes. The `run:resumed` event arrives via the tail and the existing runs-table / step-table update themselves — no special TUI handling.
5. On `{kind:"locked"}`: modal stays open with `"Another resume is in progress — retry"`. No state mutation on engine.
6. On `{kind:"notResumable"}`: modal closes (the run is no longer resumable; `runResumable` flips `false`). No toast (D9).
7. On `{kind:"unknownNode"}`: modal stays open with `"Re-run target '<nodeId>' is no longer in this run"`; user can deselect and retry.
8. On `{kind:"error"}`: modal stays open with the error message; user can retry or Cancel.

### 3.4 `R Re-run` gate

- `AppContext.runResumable` is `true` iff:
  - `state.mode.kind === "viewing"`; AND
  - `effectiveEngineState.activeRun?.runId === state.mode.runId`; AND
  - `isRunResumable(effectiveEngineState.activeRun.info)` — status in `{"error","suspended"}`.
- The binding is hidden-don't-greyed in every other state (rule 5).

## §5 Acceptance criteria (verbatim + derived)

**Verbatim (plan.md).**
> Matches mockups.md §7 layout.

**Derived sub-criteria.** All must hold to close the task.

1. **Layout fidelity.** A golden-frame test at 90×20 reproduces mockups.md §7 lines 274–299: bordered box, title `RESUME · run <id> · <workflow>`, status line, `started:` line, `Nodes to re-run` header + checkbox list, `Inputs (audit-logged)` header + `original → draft` rows with `← edited` annotation, summary line, three buttons `[ ⏎ Resume ]  [ p Preview events ]  [ Esc Cancel ]`, and the two `║` dim-background cells at modal edges.
2. **Trigger.** `R` opens the wizard only when `mode.kind === "viewing"` and the active run's status is `error` or `suspended`. In every other state `R` is hidden (rule 5). Confirmed by the keybar matrix test at the `RUN (graph)` row.
3. **Preselection.** The failing node (from `findFailingNode`) is preselected in the `--rerun` list on open. Waiting nodes (suspended runs) preselect the single waiting tokenId.
4. **Multi-select behaviour.** Space on any row toggles membership in `overlay.rerun` via `RESUME_WIZARD_TOGGLE_RERUN`. The summary line `N re-run · K input changed` updates live.
5. **Input overrides.** Typing on an input row updates `overlay.inputs[key]` via `RESUME_WIZARD_SET_INPUT`. Rows whose draft diverges from the original render `← edited`.
6. **Mode pill.** While `overlay.kind === "resumeWizard"`, the keybar renders `[RESUME]` in inverse video at `full`+`short` tiers (matches mockups §15 RESUME row).
7. **Keybar contents.** Matches mockups §15 RESUME row at all three tiers: `⏎ Resume  Space Toggle  Tab Next  p Preview   Esc   ?`. `?` drops on `keys` tier.
8. **Confirm calls `executeWorkflow` with `resumeFrom`.** Enter triggers `resumeRun(...)` which:
   a. Replays + validates status ∈ `{"error","suspended"}`.
   b. `manager.openExistingRun` acquires the exclusive lock.
   c. Appends one `token:reset` per selected rerun nodeId, resolved via `handle.snapshot.tokens`.
   d. Appends one `global:update` with all input overrides (if any).
   e. Calls `executeWorkflow(workflow, { resumeFrom: handle, runsDir, workspaceDir })`.
   f. Releases the handle in `finally`.
   Test-asserted call order per 4.1 `test/engine/resume.test.ts` case (a).
9. **Double-submit guarded.** Rapid Enter presses produce at most one `RESUME_WIZARD_SUBMIT_START` and one `resumeRun` call (asserted via call-count stub).
10. **Esc Cancel.** Esc dispatches `OVERLAY_CLOSE`; no engine call, no state mutations to the underlying run.
11. **Lock contention.** `RunLockedError` surfaces as an in-modal error line (`"Another resume is in progress — retry"`), not a crash. Cursor + focus + set preserved.
12. **No engine edits.** `git diff` shows zero changes under `packages/markflow/`.
13. **No `.skip`.** `rg '\.skip\(' packages/markflow-tui/test` returns only pre-existing matches unchanged by this task.
14. **Mode transitions preserved.** Existing `test/app/mode-transitions.test.tsx` still passes — opening/closing the wizard does not mutate `state.mode`.
15. **Layout primitive.** Modal uses the same `<Box marginTop={-(stdout.rows)} alignItems="center">` overlay pattern as `<ApprovalModal>` / `<AddWorkflowModal>` (P4-T3 / P7-T1 convention).

## §6 Design decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | **Wizard never auto-opens.** | Terminal runs are a steady state. Auto-popping the wizard every time a run fails would violate the "overlays never hijack an unrelated focus" rule (features.md §5.6 rule 6) and punish users who are intentionally browsing failed runs for diagnosis. Matches mockup §7 wording ("`R Re-run` is the gateway"). |
| D2 | **`rerun` and `inputs` live on `AppState.Overlay`; cursor / focus / FSM live locally.** | The existing `RESUME_WIZARD_TOGGLE_RERUN` / `SET_INPUT` actions already treat the selection as app-state. We keep that (it survives accidental-unmount re-renders and is serialisable for future "draft wizard" persistence), while cursor + focus are ephemeral and don't belong in `AppState`. Mirrors P7-T1 D2. |
| D3 | **Engine bridge in `src/engine/resume.ts`**, not a new `src/worker/`. | Engine-adjacent IO; same envelope as `adapter.ts` and `decide.ts`. Keeps the engine slice coherent (P7-T1 D4). |
| D4 | **Add `resumeWizard.state: "idle" \| "submitting"` + `RESUME_WIZARD_SUBMIT_START/DONE` actions.** | Without a submit-gate the keybar's `[RESUME]` pill cannot dim while in-flight, and double-Enter would race. Symmetric with the already-shipped `approval.state` + `APPROVAL_SUBMIT` pair. Minimal state surface: two variants + two actions. |
| D5 | **Field-focus FSM (`rerun` → `inputs` → `confirm`) via Tab.** | Mockup §7 line 299 literally reads `Tab Next field`. Modal-local focus FSM avoids leaking pane-focus concepts into `AppState.mode`, which is reserved for top-level navigation. |
| D6 | **Token resolution happens in `resumeRun`, not the UI.** | `handle.snapshot.tokens` is only available after `openExistingRun` takes the lock; doing it in the modal would duplicate work and risk staleness. The UI passes nodeIds; the bridge resolves. Mirrors the CLI `resume.ts` flow exactly. |
| D7 | **`p Preview events` is a keybar-visible no-op stub.** | Mockup §7 line 304 explicitly calls it "non-MVP but reserved". Removing the binding entirely would diverge from the mockup keybar; wiring a real preview would balloon scope. No-op stub is the minimal honest path. Tracked as a follow-up. |
| D8 | **`deriveInputRows` surfaces undeclared keys from `run:start.inputs`.** | A prior `--input foo=bar` call with no `foo` declaration is valid per the CLI; the wizard must not hide it or user edits will be lost when the run resumes. Declared-but-unset keys use `declaration.default` (or empty string if none). |
| D9 | **No toast on `notResumable`, just auto-close + keybar re-flip.** | The `R` binding disappears the moment `runResumable` flips `false` — the user gets visible feedback through the missing binding. Toast surface is deferred. Mirrors P7-T1 D6. |
| D10 | **Match checkbox glyphs literally (`[x]` / `[ ]`).** | Mockup §7 lines 283–286 are normative; the approval modal's `◉` / `○` are radio-specific. Introducing a theme slot for a one-off is premature. P7-T1 D10 rationale applies. |
| D11 | **Width / height as props.** | Same contract as `<ApprovalModal>` and `<AddWorkflowModal>`. Deterministic rendering under test; production passes `Math.min(Math.max(40, cols-4), 90)` etc. from `useStdout()`. |
| D12 | **`isRunResumable` uses `RunInfo.status` directly, not `replay()`.** | The status field is authoritative (set by `meta.json` and refreshed from `replay()` by `RunManager.getRun`). Re-replaying in the `R` binding hot-path would be wasteful. |
| D13 | **No shared primitive extracted for the bordered-frame with `<ApprovalModal>`.** | P7-T1 §7 reserved this extraction for the "3rd consumer" moment. With only two consumers (approval + resume), inline + borrow the wrapText helper is enough. Extract in the 3rd task (e.g. confirm-cancel). |
| D14 | **`FOCUS_PREV` wired to `Shift+Tab`** (per Ink `key.shift && key.tab`). | Not in the mockup but a universal convention; omitting it forces users to cycle 3 tabs to go back one field. Cheap addition. |

## §7 Follow-ups / deferred

- `p Preview events` panel — render a diff-style view of the events the resume will append (1 × `run:resumed` + K × `token:reset` + optional `global:update`). Hooks in through the existing adapter once a preview-event renderer exists (features.md §4 lists "diff view" in nice-to-haves).
- Shared bordered-frame primitive (`<Modal>`) when a 3rd modal lands (confirm-cancel, P8-ish).
- Draft persistence: stash `overlay.rerun` + `overlay.inputs` across accidental modal closes (would require moving ephemeral state up — not worth it until users ask).
- `watchRunStatus()` or similar, so `runResumable` updates live without waiting for the adapter's next tail event. Currently the derivation re-reads `activeRun.info` on every render, which is sufficient for the adapter's push frequency.
- VHS tape `resume-wizard.tape` — Phase 8 snapshot harness.
- Cross-run badge `r Resume (N)` on the RUNS keybar (mockups §1 line 36, §15 RUNS row) — requires multi-run status scanning similar to P7-T1 D3's coarse approval counts. Piggyback once a multi-run watcher lands.

## §8 Open questions

None that block implementation. The two potentially ambiguous points — D7 (`p Preview events` stub) and D8 (undeclared input keys) — are resolved above with explicit rationale and are reversible without breaking public APIs. No engine-side changes are required; every mutation the wizard needs is an existing append-only event (`token:reset`, `global:update`) or a public export (`executeWorkflow`, `openExistingRun`, `parseWorkflow`, `readEventLog`).
