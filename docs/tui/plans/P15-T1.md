```ts
/** Strip all ANSI control sequences. Uses Node's `util.stripVTControlCharacters` (18.17+). */
export function stripAnsi(s: string): string;

/**
 * Canonicalise captured screen text for assertion / snapshot comparison.
 *   - Strip ANSI
 *   - Trim trailing whitespace per line (cells are padded to term width)
 *   - Normalise LF-only line endings
 *   - Mask timestamps /\d{2}:\d{2}:\d{2}/            → "HH:MM:SS"
 *   - Mask durations /\d+(?:\.\d+)?(ms|s|m)\b/        → "<dur>"
 *   - Mask short-ids /\b[0-9a-f]{6,8}\b/              → "<id>"
 *   - Mask absolute temp paths /\/tmp\/[A-Za-z0-9\-_.\/]+/ → "<tmp>"
 *   - Mask spinner glyphs /[⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏]/            → "*"
 *     (harness also pins MARKFLOW_TEST=1 to disable spinner animation but the
 *      mask is a belt-and-braces guarantee)
 */
export function canonicalize(s: string): string;

/** Control-char / escape-sequence literals for `session.write(keys.ENTER)` etc. */
export const keys: {
  readonly ENTER: "\r";
  readonly ESC: "\x1b";
  readonly UP: "\x1b[A";
  readonly DOWN: "\x1b[B";
  readonly LEFT: "\x1b[D";
  readonly RIGHT: "\x1b[C";
  readonly TAB: "\t";
  readonly BACKSPACE: "\x7f";
  readonly CTRL_C: "\x03";
};
```

### 2.3 `test/e2e/harness.ts`

```ts
import type { ScratchEnv } from "./tmp.js";

export interface SpawnOpts {
  /** Terminal width. Default 120. */
  readonly cols?: number;
  /** Terminal height. Default 40. */
  readonly rows?: number;
  /** Additional CLI args after the registry flag. Default []. */
  readonly args?: ReadonlyArray<string>;
  /** Scratch env to derive spawn env from. Created if omitted. */
  readonly scratch?: ScratchEnv;
  /** Absolute path to the TUI binary. Defaults to the tsup build output. */
  readonly binaryPath?: string;
}

export interface TuiSession {
  /** Raw screen text via `@xterm/headless Terminal.buffer.active` (NO ANSI; built from cell strings). */
  screen(): string;
  /** `canonicalize(screen())`. Use in assertions + snapshots. */
  snapshot(): string;
  /** Write raw bytes to the PTY. */
  write(bytes: string): void;
  /** Convenience presses. */
  pressEnter(): void;
  pressEsc(): void;
  pressCtrlC(): void;
  /**
   * Poll a predicate against the current `screen()` / `snapshot()` every 50 ms
   * until it returns true or `timeoutMs` elapses. Throws `HarnessTimeoutError`
   * with the final snapshot attached on timeout. Never sleeps blindly.
   */
  waitFor(predicate: (snapshot: string) => boolean, timeoutMs?: number): Promise<void>;
  /** Convenience: `waitFor(snap => snap.includes(needle))`. */
  waitForText(needle: string, timeoutMs?: number): Promise<void>;
  /** Forward SIGWINCH equivalent to the PTY and xterm buffer. */
  resize(cols: number, rows: number): void;
  /** SIGTERM + drain + dispose term. Idempotent. Call in `afterEach`. */
  kill(): Promise<void>;
  /** The scratch env (for writing fixture files into `workspaceDir`, etc.). */
  readonly scratch: ScratchEnv;
}

export class HarnessTimeoutError extends Error {
  readonly snapshot: string;
  constructor(message: string, snapshot: string);
}

/**
 * Spawn the built TUI binary inside a PTY and attach an `@xterm/headless`
 * emulator for screen capture. Always forwards `--list <registryPath>` and
 * `--no-save=false` (persistent registry) so the add-workflow flow can
 * persist across multiple TUI invocations within a single journey if ever
 * needed. Does NOT rely on `process.stdout.isTTY`.
 */
export function spawnTui(opts?: SpawnOpts): Promise<TuiSession>;

/** Default timeouts — one place to tune. */
export const DEFAULT_READY_MS: number;   // 5_000  — initial empty-launch readiness
export const DEFAULT_WAIT_MS: number;    // 15_000 — generic waitFor
export const DEFAULT_RUN_MS: number;     // 30_000 — run-to-completion
```

## §3 Data flow

### 3.1 How node-pty spawns the TUI binary

```
ScratchEnv ──► env (TERM, FORCE_COLOR, NO_COLOR, MARKFLOW_*, HOME, TZ, PATH)
                 │
binaryPath = packages/markflow-tui/dist/cli.js
                 │
                 ▼
       pty.spawn("node", [binaryPath, "--list", registryPath, ...extraArgs], {
         name: "xterm-256color",
         cols, rows,
         cwd: scratch.workspaceDir,
         env,
       })
                 │
                 ▼
       pty.onData(chunk => xtermHeadless.write(chunk))
       pty.onExit(({exitCode, signal}) => session.exitCode = exitCode)
                 │
                 ▼
      TuiSession.screen()  ← for each row in term.buffer.active:
                                line.translateToString(true /* trimRight */)
                                join with "\n"
      TuiSession.snapshot() ← canonicalize(screen())
```

- We spawn `node <binaryPath>` (not the `markflow-tui` bin) so that the test doesn't depend on the workspace symlink being fresh.
- The PTY env **replaces** `process.env` wholesale (not `...process.env`) — only the deterministic pins go through. This matches `testing.md` §4 "Pin TERM, FORCE_COLOR, rows/cols" and anti-pattern §7 "Color profile autodetection in CI".
- `onExit` is captured but assertions don't wait on it; the journeys assert on on-screen readiness, then `kill()` in teardown.

### 3.2 How frames are snapshotted

`@xterm/headless` exposes `term.buffer.active.getLine(y)` → `IBufferLine.translateToString(trimRight)`. The harness reads every row, joins with `\n`, and returns raw plain text. Because `FORCE_COLOR=0` + `NO_COLOR=1` + `MARKFLOW_ASCII=1` are pinned, the emitted output has essentially no styling escape sequences — but `stripAnsi` still runs in `canonicalize` (belt-and-braces for bold/inverse which we keep for focus + selection visibility).

Snapshot strategy (per `testing.md` §5 anti-patterns):
- **No exact-byte snapshots of raw ANSI.** `canonicalize` runs first.
- **No `toMatchInlineSnapshot` with ANSI.** Journey assertions use `expect(snap).toContain(...)` or `toMatch(/regex/)` against the canonicalised frame; one `toMatchSnapshot()` per journey at a "canonical view" moment (per `testing.md` §5 rule 2) — in a file-based snapshot at `test/e2e/__snapshots__/<journey>.e2e.test.ts.snap`.
- **No wall-clock values.** All times/durations/IDs masked.

### 3.3 How canonical user journeys script keystrokes

All three journeys follow the same skeleton:

```ts
let session: TuiSession;

beforeEach(async () => {
  session = await spawnTui({ cols: 120, rows: 40 });
  await session.waitForText("WORKFLOWS", DEFAULT_READY_MS);   // mode tabs visible
});

afterEach(async () => {
  await session.kill();
  await session.scratch.cleanup();
});
```

Each test drives keys then polls:

```ts
session.write("a");                                    // open add-workflow modal
await session.waitForText("Add workflow");
session.write(pathToFixture);
session.pressEnter();                                  // submit
await session.waitForText(path.basename(pathToFixture)); // list row appears
session.write("r");                                    // run
await session.waitForText("▶ running", DEFAULT_WAIT_MS);
await session.waitForText("✓", DEFAULT_RUN_MS);
```

The harness never `setTimeout`-sleeps; every step is a polled predicate. `waitFor` uses 50 ms poll cadence and throws `HarnessTimeoutError` carrying the final snapshot for post-mortem.

### 3.5 How the harness avoids running in the unit-test worker

- `vitest.config.ts` excludes `test/e2e/**`. `npm test -w packages/markflow-tui` (default) stays hermetic — no PTY allocation, no build dependency.
- `vitest.e2e.config.ts` includes only `test/e2e/**/*.e2e.test.ts`. It uses `pool: "forks"` + `singleFork: true` — one worker, sequential — so parallel PTY allocation doesn't hit GH runner limits (testing.md §4) and temp dirs don't race.
- `npm run test:e2e` is the only entry point; `npm test` never loads `node-pty`. Developers without a working `node-pty` install can still run Layers 1 + 2.

### 3.6 Timeout handling

Three tiers, each explicit:
- **Readiness (initial spawn):** `DEFAULT_READY_MS = 5_000`. If the TUI hasn't shown `WORKFLOWS` by 5 s, fail with the final snapshot. Catches broken builds / shebang issues.
- **Interaction (between keystrokes):** `DEFAULT_WAIT_MS = 15_000`. For modal appearance, list updates, etc.
- **Run-to-completion:** `DEFAULT_RUN_MS = 30_000`. For the `✓` assertion.
- **Vitest-level:** `testTimeout: 60_000` gives the whole test a ceiling; individual `waitFor`s have their own tighter bounds so we fail-fast with useful snapshots.

`HarnessTimeoutError` always carries `snapshot` — printed by vitest's default error formatter, giving reviewers the final frame without re-running.

### 3.7 Flaky-frame snapshotting strategy

Per `testing.md` §2 + §5:
1. **Canonicalise before comparing.** Every assertion uses `session.snapshot()` (already canonicalised). Raw `screen()` is debug-only.
2. **Prefer targeted text assertions over full-frame snapshots.** Each journey has 5–10 `toContain` / `toMatch` checks during progress + **one** `toMatchSnapshot` at the terminal state. Lazygit post-mortem pattern (`testing.md` anti-pattern §5.2).
3. **Mask non-determinism.** Timestamps, durations, short IDs, temp paths, spinner glyphs (harness also pins `MARKFLOW_TEST=1` to disable spinner animation entirely — the mask is belt-and-braces).
4. **LF-only + external snapshots.** The `__snapshots__/*.snap` files are LF (ensure `.gitattributes` already covers or add `*.snap text eol=lf` as a sibling — verified not needed since the repo-wide `.gitattributes` already normalises).
5. **Update protocol.** Snapshot updates happen locally via `vitest --config vitest.e2e.config.ts -u`, never in CI (per `testing.md` §4).

| 1 — Add → Run → Complete | `journey-add-run.e2e.test.ts` | Modal opens on `a`; list row appears after `Enter`; `▶`/`[run]` visible; `✓`/`[ok]` on all three steps; `run:complete` or equivalent terminal banner | Final canonicalised frame |
| 2 — Failed → `R` re-run → Success | `journey-rerun.e2e.test.ts` | `[fail]` after first run; wizard opens on `R`; selected node highlighted; `Enter` confirms; re-run reaches `[ok]` | Final canonicalised frame |
| 3 — Approval pending → `a` → Decide → Resume | `journey-approval.e2e.test.ts` | `[wait]` on approval step; modal opens on `a` with prompt text; `Enter` submits; step flips to `[ok]`; downstream runs; terminal state | Final canonicalised frame |

**Negative / robustness sub-cases** (per journey, not separate files):
- **J1:** if `Enter` pressed in the modal with an invalid path, expect error text (not a crash). Then `Esc` cancels. Then retry with a valid path.
- **J2:** pressing `Esc` in the wizard closes it without re-running (single `waitFor(snap => !snap.includes("Resume from"))`).
- **J3:** picking `no` (the rejection option) instead of `yes` routes to the rejection edge and the run ends `[fail]` on that path. This is a second `test("…rejection…")` in the approval journey file.

**Cross-platform:**
- macOS + Linux run under default GH Actions runners. `node-pty` uses openpty — supported.
- Windows is **out of scope** for this task (see §6 D6): the harness throws a clear "e2e tests unsupported on Windows" error if `process.platform === "win32"`, and the journey tests early-return (skipped via `test.skipIf(process.platform === 'win32')` is permitted — `.skipIf` is not `.skip` in the anti-pattern sense; it's a conditional predicate. This is consistent with `instruction.md` global rule "No skipping tests" which targets silent `.skip`, not platform-gated `skipIf`. See §6 D7.).

**Resize:** not tested here (Phase 8 territory). A future task can extend the harness to call `resize(52, 22)` mid-journey.

**Monochrome / ASCII:** enforced by default env — the journey assertions use ASCII glyphs (`[run]`, `[ok]`, etc.) so they're stable across locales.

## §5 Acceptance criteria (verbatim + derived)

**Verbatim (plan.md).**
> All three journeys pass on macOS and Linux. No hard-coded sleeps; use readiness predicates.

**Derived sub-criteria.** All must hold to close the task.

1. **Three journey files exist** at the paths in §1.2, each containing at least one top-level `test(...)` that runs end-to-end the flow in §3.4. Confirmed by file-map + file content.
2. **Harness primitives exist.** `harness.ts` exports `spawnTui`, `TuiSession` interface, `HarnessTimeoutError`, `DEFAULT_READY_MS`, `DEFAULT_WAIT_MS`, `DEFAULT_RUN_MS`. Verified by import in each journey file.
3. **ANSI strip + canonicalisation exist.** `ansi.ts` exports `stripAnsi`, `canonicalize`, `keys`. Journeys use `session.snapshot()` (which calls `canonicalize`) for all text assertions — no raw-ANSI assertion present (`grep 'screen()' test/e2e/*.e2e.test.ts` only matches debug helpers, not assertions).
4. **No hard-coded sleeps.** `grep -E 'setTimeout|sleep\(|await new Promise\(.*setTimeout' test/e2e/**` returns zero matches in journey code. Only the internal poll cadence inside `waitFor` uses `setTimeout(50ms)` — that's the polling implementation, not a guess-sleep.
5. **Readiness predicates on every wait.** Every wait is a `session.waitFor(predicate, timeoutMs)` or `session.waitForText(needle, timeoutMs)` call, never a bare await. Enforced by (4) + code review.
6. **Explicit timeouts.** Every `waitFor*` in journeys either passes a ms value or relies on one of the documented defaults. Enforced by signature: `timeoutMs?: number` defaults only to a declared constant.
7. **Deterministic env.** `createScratchEnv` replaces (not extends) `process.env` except `PATH`. Verified in unit (`test/e2e/tmp.test.ts` — a small Layer-1 test allowed as a sibling since it's pure; see §4 note below).
8. **Per-test isolation.** Each test gets a fresh scratch dir; `afterEach` calls `cleanup()` + `kill()`. No shared registry / runs / workspace.
9. **`npm run test:e2e -w packages/markflow-tui` exists and runs the journeys.** Listed in `package.json#scripts`; command builds the TUI binary first (`npm run build`) then runs vitest with the dedicated config.
10. **`npm test -w packages/markflow-tui` does NOT run journeys.** Exclude list in `vitest.config.ts` covers `test/e2e/**`. Default test run produces the same test count as before + no new `.tsx`/`.ts` under `test/e2e/**`.
11. **devDeps installed.** `node-pty` and `@xterm/headless` appear in `packages/markflow-tui/package.json#devDependencies`.
12. **No engine package churn.** `git diff --stat packages/markflow/` empty after the task. CLI purity preserved.
13. **No `.skip` / `xit`.** `grep -rn '\\.skip(\|xit(' packages/markflow-tui/test/e2e/` returns zero. `skipIf(process.platform === 'win32')` is allowed (conditional gate, not silent skip) — see §6 D7.
14. **Type-first.** Every harness export has an explicit signature; no `any` in `harness.ts` / `ansi.ts` / `tmp.ts`.
15. **macOS + Linux green.** CI runs `npm run test:e2e -w packages/markflow-tui` on both platforms and passes. (CI wiring is orthogonal to this task; the task deliverable is that the script + tests work locally on macOS and a Linux CI container — verifier runs on Linux GH Actions during PR.)
16. **Canonical-view snapshot per journey.** Each `*.e2e.test.ts` has exactly **one** `toMatchSnapshot()` call (the terminal-state canonical view). More would be over-snapshotting (testing.md §5.2).
17. **Timeout errors carry the last snapshot.** `HarnessTimeoutError.snapshot` is non-empty on every raised instance. Verified in harness code review.
18. **Fixtures are self-contained.** Journey fixtures under `test/e2e/fixtures/` do not import engine-internal test utilities; they are plain `.md` workflow files parseable by the shipped engine.
19. **Build dependency.** The `test:e2e` script chains `npm run build` before `vitest`; running `test:e2e` from a clean tree always targets the latest source.

## §6 Design decisions & rationale

| # | Decision | Rationale |
|---|---|---|
| D1 | **Layer 3 harness (node-pty + @xterm/headless), not `@microsoft/tui-test`.** | Testing.md §6 picks Layer 3 for "coverage that needs real PTY + xterm buffer, without the full Playwright-style framework overhead". Adding `tui-test` is a bigger dep + learning curve; node-pty is one dep the TUI ecosystem already understands (it's also what `tui-test` uses internally). Layer 4 (`tui-test`) is explicitly reserved for a future task — not in P9-T1 scope. |
| D2 | **Spawn the built `dist/cli.js` binary, not tsx at source.** | Faithful to Layer 3's "run the built binary under a PTY" directive (testing.md §6 code sketch). Catches shebang/banner regressions that tsx bypasses. Trade-off: every `test:e2e` invocation builds — acceptable because tsup is <2 s. |
| D3 | **Pin `node-pty` at `^1.0.0` (or whichever is current stable on npm at implementation time, ≥1.0, <2.0).** | The 1.x line is stable, ships prebuilt binaries for node 18/20/22 on macOS + Linux. Pinning a minor avoids surprise ABI breaks. Implementation agent verifies the exact minor via `npm view node-pty version` at install time; plan locks only the major. `@xterm/headless` pinned at `^5` (v5 is the current ESM-friendly major). |
| D4 | **Harness under `test/e2e/`, not `test/integration/` nor `e2e/` at workspace root.** | Sits alongside the existing `test/` tree so the file resolver works without new tsconfig path changes. The `.e2e.test.ts` suffix is distinct so the default vitest include doesn't pick them up (double-safety with the `exclude` list). Matches testing.md §6 layer-3 sketch (`test/tui/integration/…`) adapted to this repo's flatter layout. |
| D5 | **Harness avoids running in the unit-test worker via vitest config split, not env gating.** | An env-flag approach (`E2E=1 npm test`) is lossier — devs forget to set it; CI configs drift. A dedicated `vitest.e2e.config.ts` is explicit and self-documenting. Matches the `testing.md` §4 guidance on parallelisation: "Use `test.concurrent.each` carefully" — we use `singleFork` instead. |
| D6 | **Windows (`win32`) is out of scope for journeys.** | `node-pty` uses ConPTY on Windows which has different cursor-motion handling (`testing.md` §7 risk #5). Maintaining dual platform goldens is a follow-up; declaring macOS + Linux primary now matches the verbatim acceptance criterion ("pass on macOS and Linux"). The harness throws a clear error on Windows rather than silently misbehaving. |
| D7 | **`test.skipIf(process.platform === 'win32')` is allowed.** | The global "no `.skip`" rule (instruction.md + testing.md §5) targets silent, unconditional skips that mask broken tests. `skipIf` is a declarative platform gate — equivalent to not building the test on that platform at all — and its state is visible in test output. Verifier should cross-check this interpretation if the rule is tightened in future. |
| D8 | **Harness polls `screen()` every 50 ms; no event-driven push.** | xterm.js emits no high-level "redraw complete" event; the only signal is byte input to the emulator. Polling cadence of 50 ms gives ≤100 ms reaction latency on human-scale interactions and is well under the `DEFAULT_WAIT_MS` budget. Alternative ("subscribe to pty.onData and evaluate predicate after each chunk") races on partial writes mid-frame; polling on a snapshotted buffer is simpler and correct. |
| D9 | **Masked canonicalisation happens in the harness, not the TUI.** | Keeps production rendering honest (real timestamps, real durations). The harness is the right place for test-time masking — it aligns with `testing.md` §2 "Mask non-determinism" in the snapshotting helper, not in the app. |
| D10 | **Env pins override, not extend, `process.env`.** | `testing.md` §4 "Pin `TERM`, `FORCE_COLOR`" + anti-pattern §7 "Color profile autodetection in CI" — leaking host env (e.g. `COLORTERM=truecolor` on a dev laptop) is the #1 CI-only flake source. Overriding explicitly eliminates the class. `PATH` is the only forwarded key (node-pty needs it to find `node`). |
| D11 | **One `toMatchSnapshot()` per journey.** | `testing.md` §5.2 anti-pattern "Over-snapshotting". Every targeted `toContain` already proves the journey; the single snapshot locks the final composed frame for visual-regression-ish coverage without multiplying maintenance. |
| D12 | **`node-pty` + `@xterm/headless` are devDeps, never runtime deps.** | CLI purity + minimal production bundle. Neither is imported from `src/` anywhere. |
| D13 | **Scratch-dir HOME redirection.** | Prevents the TUI's default `~/.markflow-tui.json` registry (see repo-root `.markflow-tui.json` committed in status) from being read or mutated by journeys. Each test starts with a blank slate. |
| D14 | **No dedicated engine mock.** | The engine runs in-process under the TUI as in production. Journeys exercise the real `executeWorkflow`. Fixtures are trivial shell scripts (exit 0/1). This is the whole point of Layer 3. |
| D15 | **The `test:e2e` script always `npm run build`s first.** | Avoids the class of "stale `dist/cli.js`" flakes. `tsup` is fast enough that this is invisible in CI. For watch-mode (`test:e2e:watch`), we don't auto-build — developers are expected to `npm run build -w packages/markflow-tui` in a second shell. |
| D16 | **Readiness predicate for spawn is presence of `WORKFLOWS` mode tab text.** | The mode tabs render immediately after the root component mounts and `detectCapabilities` resolves — no async work. Under ASCII mode the tab still reads `WORKFLOWS` (plain text). Simpler than waiting for a sentinel event. |
| D17 | **Use `@xterm/headless` (not home-rolled row buffer).** | Official, Ink-friendly, handles cursor motion + scroll region + erase-line sequences correctly. Alternatives (`headless-terminal`, custom `chalk + split`) lose fidelity on multiline repaints that Ink does natively. |
| D18 | **`keys` map uses `\r` for Enter (not `\n`).** | Terminal convention; `testing.md` §2 input-injection warns of the classic `\n` pitfall in raw mode. Ink's raw-mode input treats `\r` as Enter. |
| D19 | **No VHS / Layer 5 overlap in this task.** | VHS tapes (Layer 5, testing.md §6) live under `packages/markflow-tui/vhs/` and run separately. P9-T1 is strictly Layer 3. |
| D20 | **Sequential execution (`singleFork: true`).** | PTY allocation on GH Actions is limited and slow; parallelising journeys would barely shave wall-time (each journey is <5 s) while risking temp-dir collisions + PTY throttling. |
| D21 | **Fixtures live under `test/e2e/fixtures/`, not imported from engine tests.** | Engine fixtures evolve under engine tests' rules; depending on them would couple two test suites and risk bleed-through. Three tiny `.md` files are cheap. |
| D22 | **Only one test runner.** | Vitest runs both unit (Layer 1+2) and e2e (Layer 3). No Jest / Mocha / Playwright introduced. Matches the existing repo convention. |
| D23 | **Harness errors surface the final snapshot.** | `HarnessTimeoutError.snapshot` is load-bearing for CI debuggability. Without it, a 30 s timeout gives you nothing; with it, reviewers can see exactly what rendered. Matches testing.md §1 case for `tui-test`'s tracefile affordance (but simpler). |

## §7 Follow-ups / deferred

- **Windows ConPTY support.** Gate behind a separate task; requires per-platform goldens or a loosened canonicaliser.
- **Visual regression (Layer 5).** VHS tapes recording the same journeys — separate task, not needed here.
- **Additional journeys**: narrow-tier (`cols=52`) drill-down (P8-T2 surface), monochrome-only run (already covered indirectly by env pins but no dedicated journey asserts a specific §14 render), multi-region fan-out wait-for-all — all later.
- **Flake detection tooling.** If flake budget is violated in future, consider `--retry=2 --reporter=verbose` in CI plus a flake dashboard — not needed for zero-budget landing.
- **`@microsoft/tui-test` as Layer 4.** Reserved for a follow-up if we need Playwright-like auto-wait + trace replay.
- **`tui-devtools` local debugging.** Not in CI; useful as a `DEV=true` sidecar when a journey fails locally (see testing.md §1).
- **Per-tier harness extension.** `resize()` exists but no journey exercises it; a "responsive journeys" task is a natural sibling.

## §8 Open questions

None that block implementation.

- D3's exact `node-pty` minor is resolved at install time by the implementation agent. The plan pins the major (`^1`) which is sufficient for ABI stability.
- D7's interpretation of the "no `.skip`" rule to permit `skipIf` is defensible per verbatim rule wording (silent skips mask broken tests; conditional platform gates are declarative). If the verifier disagrees, the Windows gate becomes an explicit `throw` in the harness + a test-file-level `if (process.platform === "win32") { return; }` guard — equivalent outcome, trivially adjustable.
- If P7-T2's resume wizard UI diverges from the mockup-assumed key bindings at implementation time (e.g. if it uses `ENTER` twice or a different confirm key), Journey 2 adjusts to the actual keystrokes — no spec change needed. This is a behavioural adaptation, not a redesign.
