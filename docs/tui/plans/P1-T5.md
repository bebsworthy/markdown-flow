## 0. Summary

When `RunManager.openExistingRun(id)` is called, acquire an exclusive on-disk lock at `runs/<id>/.lock`. Expose the lock release via a new `release()` method on the returned `ResumeHandle`. The engine's resume path (`WorkflowEngine.start()` when `resumeFrom` is set) is the only consumer today, and it must release the lock on every terminal path (complete, suspended, error) and on process exit. Contention throws a typed `RunLockedError` (exported from `packages/markflow/src/core/errors.ts` alongside `SidecarNotFoundError`). Both `resume` and `approve` CLI commands — which both call `openExistingRun` — must catch `RunLockedError` and print a clean single-line message (exit code 1) instead of a stack trace.

Out of scope: TUI integration, locking for read-only operations (`listRuns`, `getRun`, `watch`), locking the fresh-run path (`createRun`).

### 1.2 `packages/markflow/src/core/errors.ts` — modify

Append a new exported class `RunLockedError` modelled on `MarkflowError` (gives a `code: "RUN_LOCKED"` string for consistency with other engine errors).

Fields:

- `runId: string` — the run being opened.
- `lockPath: string` — path to `runs/<id>/.lock`, useful for diagnostics.
- `holderPid?: number` — optional, undefined for Phase 1 (see §3.4).

```ts
export class RunLockedError extends MarkflowError {
  readonly runId: string;
  readonly lockPath: string;
  readonly holderPid?: number;
  constructor(runId: string, lockPath: string, holderPid?: number) {
    const who = holderPid ? ` (held by pid ${holderPid})` : "";
    super("RUN_LOCKED", `Run ${runId} is already being resumed${who}`);
    this.name = "RunLockedError";
    this.runId = runId;
    this.lockPath = lockPath;
    this.holderPid = holderPid;
  }
}
```

### 1.3 `packages/markflow/src/core/run-lock.ts` — add (new file)

Thin adapter around `proper-lockfile`, kept separate for testability.

Exports:

- `async acquireRunLock(runPath: string): Promise<() => Promise<void>>`
  - Calls `lockfile.lock(runPath, { lockfilePath: join(runPath, ".lock"), realpath: false, stale: 30_000, retries: 0, onCompromised: () => { /* log no-op */ } })`.
  - Translates `ELOCKED` errors into `RunLockedError(runId, lockPath)`.
  - Returns an idempotent release function (guarded by a `released` flag).
- `lockPathFor(runPath: string): string` — helper returning `join(runPath, ".lock")` for tests and callers.

Implementation notes:

- `realpath: false` avoids symlink-resolution issues.
- `stale: 30_000` — previous-process crash recovery; `proper-lockfile` auto-refreshes every `stale/2` ms by default.
- `retries: 0` — "second fails fast" per acceptance criterion.
- `onCompromised` — log to `stderr` only; never throw (would crash the engine).

### 1.5 `packages/markflow/src/core/engine.ts` — modify

In `WorkflowEngine.start()`, wrap the existing `try`/`catch` with a `finally` block that releases the lock if `resumeFrom` was provided:

```ts
try {
  try {
    // existing body
    return { ... };
  } catch (error) {
    // existing error path
  }
} finally {
  if (resume) await resume.release().catch(() => {});
}
```

Rationale: exactly one release per `openExistingRun`, regardless of terminal status or unexpected throw.

## 3. Lock lifecycle

### 3.1 Acquisition

Synchronous inside `openExistingRun(id)`, before the log is read. Ensures consistency with the exclusive-writer guarantee.

### 3.2 Release — who calls it

`WorkflowEngine.start()` calls `handle.release()` in a `finally` block wrapping its entire `try`/`catch`. Covers:

- Normal completion.
- Suspension.
- Error.
- `ExecutionError("Workflow interrupted")` from abort.
- Unexpected throws.

### 3.3 Release — auto on process exit

Rely on `proper-lockfile`'s built-in SIGINT/SIGTERM/`exit` handlers. If implementation confirms this is disabled by default, fall back to an explicit `process.once("exit", () => lockfile.unlockSync(...))`.

### 3.4 Holder PID

`proper-lockfile` doesn't expose the holder PID. For Phase 1, leave `holderPid` undefined; error message falls back to "Run <id> is already being resumed." If feedback calls for it, add an optional `holder.json` inside the lock dir in a follow-up.

### 3.5 `completeRun` does NOT release the lock

`completeRun` is called by the engine mid-`start()`. If it released the lock, the engine would still need to emit `workflow:complete` after, creating a write-after-release race. Single-release-in-engine-finally keeps the lifecycle tight.

## 6. Tests — `packages/markflow/test/core/run-lock.test.ts`

Uses `vitest`, `mkdtemp` scratch dirs under `tmpdir()`, `beforeEach` cleanup. Style follows existing `resume.test.ts`.

### 6.1 Test matrix

1. **Acquire / release round-trip** — `openExistingRun` returns handle with `release`; `.lock` exists on disk; after `release()` it's gone.
2. **Second acquire throws `RunLockedError`** — with a `Promise.race` vs 500ms timer to prove "fails fast".
3. **Release permits subsequent acquire** — `h1.release() → h2 = openExistingRun(...)` works.
4. **Lock cleaned up after release** — `access(.lock)` → `ENOENT`.
5. **Idempotent release** — double `release()` does not throw.
6. **Stale lock reclaim** — `utimes` the `.lock` dir to 5 min ago; next acquire succeeds.

### 6.2 End-to-end in `resume.test.ts`

Add one case: after `executeWorkflow(def, { runsDir, resumeFrom: handle })` completes, assert `access(.lock)` rejects with `ENOENT`. Proves engine-finally release fires on happy path — the "existing single-process resume path unaffected" criterion.

## 7. Edge cases

1. **Stale lock** — handled by `stale: 30_000`.
2. **Runs dir not existing** — `access(events.jsonl)` already fails first, lock never attempted.
3. **Nested same-process calls** — `proper-lockfile` natively rejects with `ELOCKED`; we translate.
4. **Long-lived TUI host process** — acquire/release per call; no persistent state.
5. **`createRun` is NOT locked** — timestamped IDs; contention impossible.
6. **`release()` throws** — swallowed in engine `finally` (`.catch(() => {})`) so primary error propagates.
7. **Read-only paths (`listRuns`, `getRun`, `watch`) remain unlocked** — they only project, never append.
