### 0.2 Engine data needed

Everything required comes from state already piped to the TUI after P6-T1:

| Field | Source |
|---|---|
| `type` (script vs agent vs approval) | `StepResult.type` on the completed step, or `step:start` event path on a running token. For a running token we infer type from the matching step when it completes; until then we show `"—"` unless P6-T0 snapshots the `StepDefinition` (it does not). For MVP, running-row `type` is derived from a cached `StepResult` of a prior attempt (same node) or shown as `"—"`. |

| `started`, `ended` | `StepResult.started_at` / `completed_at` for completed; `step:start` event timestamp for running. |
| `local`, `global` | `StepResult.local`, `EngineSnapshot.globalContext` (or `GlobalContext`-derived for incremental). `EngineSnapshot.globalContext` is in `packages/markflow/src/core/types.ts` as `Record<string, unknown>`. |
| `last log line` | Most recent `step:output` event in `activeRun.events` whose `nodeId` matches the selected step. Text is `event.chunk` (last non-empty line) + `seq` + `stream`. |
| `stderr tail` (last 3 lines) | Most recent 3 `step:output` events where `stream === "stderr"` and `nodeId === selected`. No file I/O — we use the capped 500-event ring buffer. If fewer than 3 stderr lines exist in the ring, show only what's available. "link to Log tab" is copy only (Phase-6 Log tab is P6-T3). |
| `route-to` | For a terminal step, resolve the outgoing edge via `getOutgoingEdges(graph, nodeId)` filtered by `StepResult.edge`. The graph is NOT in `EngineSnapshot`. **Decision §3.3:** surface `route-to` as `StepResult.edge`-derived label only; display `"fail:max → <nextNode>"` only when the engine emitted a `route` event (`{ from, to, edge }`) in the tail for this token's node. The route event is exactly what mockup §6 shows (`edge fail:max → rollback-us`). No graph re-parse. |
| `ended` / `elapsed` | Computed from `started_at` + (`completed_at` ?? `nowMs`). |

## §2 Data model

All types land in `src/steps/detail-types.ts`.

### 2.1 `StepDetailSelection`

```ts
export interface StepDetailSelection {
  /** StepRow.id from the step table — either a token.id or "batch:<id>". */
  readonly rowId: string;
}
```

The caller (`<StepDetailPanelView>`) receives a `selectedStepId: string | null` prop and wraps it into a `StepDetailSelection` before calling `selectStepDetail`. For a `"batch:*"` row, `selectStepDetail` returns `kind: "aggregate"` (see §2.3) — the detail panel renders a compact aggregate summary instead of a token detail.

### 2.2 `StepDetailModel`

Discriminated union so the component stays dumb:

```ts
export type StepDetailModel =
  | { readonly kind: "empty" }
  | { readonly kind: "not-found"; readonly rowId: string }
  | { readonly kind: "token"; readonly data: StepDetailTokenData }
  | { readonly kind: "aggregate"; readonly data: StepDetailAggregateData };
```

### 2.3 `StepDetailTokenData`

Every field is a pre-formatted string so the component never reaches for formatters. Em-dashes (`"—"`) appear where data is unavailable.

```ts
export interface StepDetailTokenData {
  readonly nodeId: string;          // "deploy-us"
  readonly tokenId: string;
  readonly seq: number | null;      // seq of the latest relevant event (step:start/step:output/step:complete)
  readonly headline: string;        // "deploy-eu · script (bash) · seq=198"
  readonly statusLine: string | null; // "✗ failed (3/3 attempts · exhausted)" for §6; null for §4
  readonly role: ColorRole;         // for the headline + status line color
  readonly glyphKey: GlyphKey;
  readonly fields: ReadonlyArray<StepDetailField>; // ordered: type, attempt, timeout, exit, started, ended, edge, local, global, last log
  readonly stderrTail: ReadonlyArray<StderrTailLine>; // empty for running/ok
  readonly stderrTailNote: string | null; // "(last 3 lines — `2` or Tab for full log)" or null
}

export interface StepDetailField {
  readonly key: string;          // "type", "attempt", "timeout", "exit", "started", "ended", "edge", "local", "global", "last log"
  readonly label: string;        // rendered as the left-col label
  readonly value: string;        // rendered as the right-col value (already em-dashed + truncated)
  readonly layout: "pair" | "full"; // "pair" uses two-column row; "full" spans both columns (local/global get full width when narrow)
}

export interface StderrTailLine {
  readonly seq: number | null;
  readonly text: string;
}
```

### 2.4 `StepDetailAggregateData`

```ts
export interface StepDetailAggregateData {
  readonly batchId: string;
  readonly nodeId: string;
  readonly headline: string;         // "batch [regions] · forEach · 2/3"
```

## §3 Projection logic (`src/steps/detail.ts`)

### 3.1 `selectStepDetail` signature

```ts
export function selectStepDetail(
  snapshot: StepsSnapshot | null,
  info: RunInfo | null,
  events: readonly EngineEvent[],
  selection: StepDetailSelection | null,
  nowMs: number,
): StepDetailModel;
```

- `snapshot` is null when no tokens are in memory yet (fresh `viewing.*` before the tail has produced a `token:created`). We fall back to `info.steps[]` to resolve completed tokens. This is the same fallback `<StepTableView>` uses today.
- `events` is the 500-event ring from `activeRun.events`.
- `selection.rowId` can be either `token.id` or `"batch:<batchId>"`.

### 3.2 Dispatch rules

1. `selection == null` → `{ kind: "empty" }`.
2. `rowId.startsWith("batch:")` → build aggregate variant from `snapshot.batches.get(batchId)`. If missing → `{ kind: "not-found" }`.
3. Otherwise treat `rowId` as `tokenId`:
   - Prefer `snapshot.tokens.get(rowId)` for running/pending/waiting/skipped tokens.
   - Fall back to `info.steps.find(s => matches)`: match on the token's node + index. Since `StepResult` has no `tokenId`, we use the combination of `node` + `started_at` to locate a unique `StepResult` when `snapshot.tokens` has the token in `"complete"` state.
   - Missing in both → `{ kind: "not-found" }`.

### 3.3 Field order and values (matches §4 and §6 mockups)

Rendered in this order (skipping em-dash-only fields when `height` forces trimming — but default rendering keeps every field):

| Key | Value for running token (§4) | Value for failed token (§6) |
|---|---|---|
| `type` | `script (bash)` | `script (bash)` |
| `attempt` | `1 / 3` | `3/3 · exhausted` |
| `timeout` | `90s` | `60s` |
| `exit` | `—` | `1` |
| `started` | `12:04:07 (18s ago)` | `12:03:06` |
| `ended` | — (omitted when null for running) | `12:03:40` |
| `edge` | `—` | `fail:max  →  rollback-us` |
| `local` | `{ region: "eu-west-1", sha: "ab12cd" }` | `{ region: "us-east-1", sha: "ab12cd" }` |
| `global` | `{ sha: "ab12cd", started: "2026-04-15T12:01:00Z" }` | — (omitted in §6 — mockup drops it for stderr room) |
| `last log` | `seq=198  stdout  applying terraform plan (17/32 resources)` | — |

**Decision — field omission rule.** `selectStepDetail` always includes every field; the component (`<StepDetailPanel>`) applies a layout-aware omission (see §5.3): when `height < MIN_DETAIL_ROWS` (derived below) we drop fields in this reverse priority order: `last log` → `global` → `ended` → `started`. Required fields (`type`, `attempt`, `timeout`, `exit`, `edge`, `local`) never drop. This matches mockup §6 which drops `global` + `last log` to make room for stderr tail.

### 3.4 `pickStderrTail`

Scan `events` in reverse; collect up to `max` entries where `type === "step:output"`, `stream === "stderr"`, `nodeId === target`. Split each event's `chunk` on `\n`, keep non-empty trimmed lines. Push tail-lines in display order (oldest→newest of the last 3). `seq` is the owning event's `seq`.

### 3.5 `pickLastLog`

Scan `events` in reverse; first `step:output` for `nodeId === target` (either stream). Value line is the **last** non-empty line in its `chunk`. Annotate with `stream` label and `seq`.

### 3.6 `pickRouteTarget`

Scan `events` in reverse; first `route` event whose `from === nodeId` **and** `edge === stepResult.edge`. Returns `event.to`. When no match: `null`. Display as `<edge>  →  <to>` only when non-null.

### 3.7 `formatJsonOneLine`

- Objects render as `{ k: v, k: v }`; primitives render as JSON scalars.
- String values use double quotes to match mockup notation.
- When the serialized length exceeds `budget`, truncate with `"…"` at the value boundary (not mid-token) — e.g. `{ region: "eu-west-1", sha: "ab12cd" }` → `{ region: "eu-west-1", … }` when budget is tight.
- Null / undefined inputs render as `"—"`.

## §4 Reducer additions (`src/state/*`)

### 4.1 New `AppState` field

```ts
readonly selectedStepId: string | null;
```

Initial value: `null`. Persists across `FOCUS_VIEWING_PANE` transitions; cleared on `MODE_CLOSE_RUN`; reset to `null` on `MODE_OPEN_RUN` (new run, no carry-over).

### 4.2 New actions

```ts
| { readonly type: "SELECT_STEP"; readonly stepId: string | null }
| { readonly type: "STEP_CURSOR_MOVE"; readonly delta: number }
```

`STEP_CURSOR_MOVE` is a placeholder for Phase-6 cursor navigation (already required by mockup §4 `▶ deploy-eu`). Minimum viable impl in this task: **no-op** — the reducer cannot resolve `delta` to a row id without row context. Actual cursor logic ships with a later task (this task only needs to *select* via id, e.g., via keyboard discovery or mouse-ish mechanism). We keep the action in the union so future wiring doesn't bump the Action type again, but the reducer returns `state` unchanged.

**Alternative considered:** omit `STEP_CURSOR_MOVE` entirely until the follow-up task. Rejected — introducing it now keeps the action catalogue symmetric with `RUNS_CURSOR_MOVE` (already in the union from P5-T3), and reducer tests assert it compiles and is a no-op. Documenting this explicitly prevents a later task from thinking it was forgotten.

### 4.3 Reducer changes

- `SELECT_STEP` → `{ ...state, selectedStepId: action.stepId }`, but only when `state.mode.kind === "viewing"`; otherwise no-op.
- `MODE_OPEN_RUN` → also sets `selectedStepId: null` in the same state update.
- `MODE_CLOSE_RUN` → also sets `selectedStepId: null`.
- `STEP_CURSOR_MOVE` → no-op (returns `state`). Reducer guards on `mode.kind === "viewing"` for future-proofing.

### 4.4 Default selection behaviour

The user's cursor lands on the first leaf row automatically once the step table has rendered. Implementation: `<StepTableView>` already computes `rows[]`. We do **not** dispatch a `SELECT_STEP` from the view — that would violate reducer purity (adapter is the only side-effecting feed). Instead, `<StepDetailPanelView>` uses the "first row fallback" rule:

- If `selectedStepId == null` AND `rows.length > 0`, pick `rows[0].id` as the effective selection and build the detail model from it.
- If `selectedStepId != null` but not found in `rows[]`, model is `not-found`.

This keeps the reducer pristine while matching mockup §1 + §4 where the top-row is always in the detail pane by default. The step cursor/keyboard wiring that lets the user *change* the selection is out of scope — that ships with the Phase-6 follow-up task. In the interim, users can still *see* a default detail; the task's acceptance criteria only require rendering parity with the mockups, not keyboard navigation.

## §5 Component layout (`<StepDetailPanel>`)

### 5.1 Structure

```
┌──────────────────────────────────────────────────────────────────────┐
│  deploy-eu · script (bash) · seq=198                                 │  <- headline (1 row)
│                                                                      │  <- spacer
│    type       script (bash)           timeout   90s                  │  <- pair row
│    attempt    1 / 3                   exit      —                    │
│    started    12:04:07 (18s ago)      edge      —                    │
│    local      { region: "eu-west-1", sha: "ab12cd" }                 │  <- full row
│    global     { sha: "ab12cd", started: "…" }                        │  <- full row
│    last log   seq=198  stdout  applying terraform plan (17/32 …)     │  <- full row
└──────────────────────────────────────────────────────────────────────┘
```

### 5.2 Column layout inside the panel

- **Indent gutter** — 4 cols (`"    "`) to match mockup §4/§6 leading whitespace.
- **Label column** — 10 cols left-aligned (`"type      "`, `"attempt   "`, …). Dim.
- **Value column 1** — starts after the label. For pair rows, ends at col `⌊width/2⌋`. For full rows, spans to the right edge.
- **Pair gap** — 3 cols.
- **Label column 2** — 10 cols (`"timeout   "`, `"exit      "`, `"edge      "`). Dim.
- **Value column 2** — remainder.

This matches mockup §4 line 172–176 and §6 line 247–250 within ±1 col.

### 5.3 Height-aware layout

`MIN_DETAIL_ROWS = 9` (headline + spacer + 3 pair rows + local + global + last log = 9 in the full §4 rendering). If `height < MIN_DETAIL_ROWS` we drop in this order: `last log` → `global` → `ended` → `started` (merging started into the existing pair row by swapping it out). Below 5 rows we render the headline + `"detail pane collapsed"` line.

### 5.4 Failed-step layout (§6 parity)

When `model.statusLine` is non-null (terminal failed step), the layout adds a status-line row immediately after the headline:

```
│  deploy-us · script (bash) · seq=214                                  │
│                                                                       │
│    status     ✗ failed (3/3 attempts · exhausted)    exit      1      │
│    started    12:03:06                               ended     12:03:40│
│    timeout    60s                                    edge      fail:max  →  rollback-us│
│    local      { region: "us-east-1", sha: "ab12cd" }                  │
│                                                                       │
│    stderr tail (last 3 lines — `2` or Tab for full log):              │
│      ssh: connect timed out                                           │
│      error: region us-east unreachable                                │
│      retry budget 3/3 exhausted                                       │
```

The `stderrTail` array is rendered as a labelled block with a 2-col indent (`"      "` = 6 cols: 4 gutter + 2 extra). The "(last 3 lines…)" note is emitted from the model's `stderrTailNote` field.

### 5.5 Color rules

- Headline uses `theme.colors[model.role]` for the glyph + node label; rest in default.
- Label column: `theme.colors.dim`.
- `status` value inherits the role (`error` = red, `ok` = green, `running` = brand, etc.).
- `edge` value color: `route` role when non-fail; `error` role when edge starts with `"fail"`.
- stderr tail lines: default fg, dim-label on the header.
- Empty / not-found message: `dim`.

### 5.6 Hide-don't-grey

Empty / not-found panels render a single dim line (no header chrome, no field labels). Matches P6-T1's empty-step-table behaviour.

**Lifted verbatim from plan.md:**
- Fields: type, attempt, timeout, exit, edge, route-to, local, global, last-log-line, stderr tail (last 3 lines, link to Log tab for full).
- Values come from the snapshot projected via `replay()`.
- Rendering matches the detail panes in mockups §1/§4/§6.

**Derived sub-criteria:**

- (A) `<RunDetailPlaceholder mode="zoom">` is no longer mounted in the `viewing.*` bottom slot; `<StepDetailPanelView>` replaces it.
- (B) `<RunDetailPlaceholder mode="follow">` still renders in `browsing.runs` bottom slot (untouched).
- (C) `selectStepDetail` is pure (no React / Ink / I/O) and is added to `test/state/purity.test.ts::files[]`.
- (D) The three mockup scenarios render within ±1 col per row when the fixture and width match.
- (E) Fields in documented order: type, attempt, timeout, exit, started, ended, edge, local, global, last log. `statusLine` appears before fields when the step is terminal-failed (§6 parity).
- (F) Stderr tail renders last-3 lines in chronological order, indented under a labelled header, only when at least one stderr line is present in the 500-event ring.
- (G) `route-to` is rendered as `<edge>  →  <to>` only when a `route` event in the ring matches the terminal step's edge; otherwise just the edge text.
- (H) `selectedStepId` is persisted in `AppState`, cleared on mode transitions, and selectable via `SELECT_STEP` action. Tests assert reducer behaviour.
- (I) When no step is selected **and** the run has rendered rows, the panel defaults to the first row's detail (source: view-layer fallback — reducer stays pure).
- (J) Empty / not-found panels follow hide-don't-grey: single dim message line, no field chrome.
- (K) `npm run lint -w packages/markflow-tui` and `npm test -w packages/markflow-tui -- --run` both green.

## §10 Design decisions & rationale

| # | Decision | Rationale |
|---|---|---|
| D1 | Scope P6-T2 to the zoom-mode (`viewing.*`) bottom slot only. The `browsing.runs` follow bottom slot keeps `<RunDetailPlaceholder mode="follow">`. | §3.4 and the task scope both describe a *step* detail pane. Without a step cursor in `browsing.runs`, there is nothing to show. This also keeps follow-mode's "no run selected" empty-state copy working — P5-T3's behaviour is preserved. |
| D2 | View-layer fallback: when `selectedStepId == null`, default to `rows[0].id`. Do NOT dispatch auto-selection from the view. | Reducer purity (features.md §6.2 "reducer is pure"). Side-effects must come from the adapter, not components. Mockup §1 + §4 both show the first row in the detail pane by default, so the fallback is user-observable, not merely incidental. |
| D3 | `timeout` value shows only when a `step:timeout.limitMs` event has arrived OR the step routed via `timeout`. Otherwise `"—"`. No `StepDefinition.timeout` plumbing. | `StepDefinition` is not on `EngineSnapshot`; plumbing it requires a new engine-API public surface. Out of scope for a TUI-only task (CLAUDE.md rule: engine additions must go through features.md §7 promotion). The mockup shows `90s` / `60s` because the run in that scenario had timeouts emitted. We are faithful to the snapshot. |
| D4 | `route-to` derived from `route` events in the ring, not from the flow graph. | The `FlowGraph` is not carried on `RunInfo` / `EngineSnapshot`. Scanning the tail for `route` events is O(N≤500), deterministic, and uses only data we already have. Mockup §6 shows `fail:max → rollback-us` exactly because the engine emits a `route` event. |
| D5 | `type` for a running token falls back to the most recent completed `StepResult` of the same node, else `"—"`. | `Token` has no `type` field; `StepDefinition` is not snapshotted; adding it would require engine plumbing. For every retried/fan-out step, the same node has a prior `StepResult` whose `type` is identical. This is a best-effort fallback; the em-dash case is rare (first-ever run of a node during its first attempt). |
| D6 | Stderr tail uses the 500-event ring, not sidecar files. | features.md §6.2 + §3.5 reserve sidecar file reads for the log pane (P6-T3 / Phase-6 later task). The detail tab's "last 3 lines" is an at-a-glance preview — the ring buffer is sufficient. The "link to Log tab for full" copy defers deep history to the log pane. |
| D7 | `StepDetailAggregateData` exists even though no mockup shows it. | Defensive: the step-table cursor can land on a `"batch:*"` row. Rendering an empty/placeholder panel would break parity with the "rendering matches the detail pane" criterion once cursor navigation ships. Keeping the aggregate variant small and well-tested costs ~30 LOC. |
| D8 | `formatJsonOneLine` truncates at the key-boundary, not mid-string. | Mockup §4 `global` truncates as `{ sha: "…", started: "…" }` with each string's value shortened. Our implementation prefers dropping whole keys first; when a single key remains too long, its string value is truncated. This is simpler, stable under reorder, and the ±1-col tolerance covers cosmetic drift. |
| D9 | Introduce `SELECT_STEP` + no-op `STEP_CURSOR_MOVE` actions now. | Action-catalog symmetry with `RUNS_CURSOR_*` from P5-T3; later tasks won't need to change the `Action` union type again. `STEP_CURSOR_MOVE` staying a no-op is explicitly documented here so verifiers don't flag it. |
| D10 | Pure projection + view seam + dumb component, mirroring P6-T1's `<StepTableView>`/`<StepTable>` pattern. | Test-injection seam precedent (P5-T1, P6-T1). Lets us unit-test the expensive memo logic (`<StepDetailPanelView>`) with real engine fixtures while keeping `<StepDetailPanel>` tests fast and width-focused. |
| D11 | No new engine-API additions in this task. | CLAUDE.md rule: `packages/markflow/package.json` stays pure. Every field needed is reachable from `EngineSnapshot` (via existing `replay()`) or from the event ring. |
