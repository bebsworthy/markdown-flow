## BLOCKER — signature cannot be fulfilled as written

The proposed signature in the task spec (line 156) is:

```ts
getSidecarStream(runDir: string, seq: number, stream: "stdout" | "stderr"): Promise<ReadableStream<Uint8Array>>
```

The actual on-disk filename in the current codebase (see §1) is
`<seqPadded4>-<nodeId>.{stdout,stderr}.log`. It is **not** derivable from
`(runDir, seq, stream)` alone — `nodeId` is required, and `seq` is
zero-padded to width 4.

Three resolution options; this plan recommends **Option A** (signature change).

- **A. Change the signature** to accept the canonical resolution input —
  either a whole `output:ref` event, or `(runDir, stepSeq, nodeId, stream)`.
  Best-typed; mirrors how the existing CLI `show --output <seq>` resolves
  paths (via the `output:ref` event, not by computing a name). Requires one
  word change to the plan's acceptance criteria: "Resolves the sidecar path
  deterministically from runDir + `output:ref` + stream" (or similar).
- **B. Keep `(runDir, seq, stream)`** and glob `output/NNNN-*.{stream}.log`
  for the matching `seq`. Lossy if two files ever collide on seq (they
  shouldn't — `seq` is monotonic per run) but adds a fs.readdir call per
  lookup and hides what the consumer already knows.
- **C. Keep `(runDir, seq, stream)`** and also change the writer to a
  nodeId-free name (`<seqPadded4>.{stream}.log`). Simpler API but breaks
  on-disk format and loses the human-readable hint. Out of scope for P1.

**Recommended:** Option A. The TUI will have the `output:ref` event in
hand at consumption time (it tails the event log — P1-T3). Making the
resolver accept the event eliminates guessing and matches the existing CLI
semantics.

The rest of this plan assumes **Option A** and pins the signature to:

```ts
import type { OutputRefEvent } from "./types.js";

export function getSidecarStream(
  runDir: string,
  ref: { stepSeq: number; nodeId: string; stream: "stdout" | "stderr" },
): Promise<ReadableStream<Uint8Array>>;
```

The parameter shape is a structural subset of `output:ref`, so callers may
pass the event directly. `runDir` remains a separate argument because the
event carries the absolute `path` already (written by the engine at emit
time), but using it blindly would couple the resolver to whatever cwd the
engine ran in — the TUI may consume runs created elsewhere, so we
reconstruct the path from `runDir` + event fields.

If the maintainer rejects Option A, fall back to **Option B** and document
the glob in the function contract.

## 1. Sidecar file naming — authoritative statement

### From the architecture doc (verbatim)

`/Users/boyd/wip/markdown-flow/docs/arch/event-sourced-run-log.md` lines 65–70:

> ## Sidecar transcripts
>
> Step stdout/stderr is written to `runs/<id>/output/<stepSeq>-<nodeId>.{stdout,stderr}.log`.
>
> - `stepSeq` is the `seq` of the `step:start` event, zero-padded to 4 digits.
> - Keyed on `seq` (not `tokenId` or `nodeId` alone) because tokens traverse
>   multiple nodes and loops/retries re-visit the same node — `seq` is the only
>   identifier that uniquely names a single step execution.

Directory layout (lines 8–16 of the same doc):

> ```
> runs/<id>/
>   meta.json
>   events.jsonl
>   workdir/
>   output/
>     0007-build.stdout.log
>     0007-build.stderr.log
> ```

### Confirmed against the writer

`/Users/boyd/wip/markdown-flow/packages/markflow/src/core/engine.ts` lines 385–406
(located via grep for `SidecarPaths|sidecar|output:ref`):

```ts
const outputDir = join(this.runDir.path, "output");
await mkdir(outputDir, { recursive: true });
const seqStr = String(startEvent.seq).padStart(4, "0");
const stdoutPath = join(outputDir, `${seqStr}-${token.nodeId}.stdout.log`);
const stderrPath = join(outputDir, `${seqStr}-${token.nodeId}.stderr.log`);
await this.emit({
  type: "output:ref",
  stepSeq: startEvent.seq,
  tokenId: token.id,
  nodeId: token.nodeId,
  stream: "stdout",
  path: stdoutPath,
});
// … mirrored for stderr …
const sidecar = { stdoutPath, stderrPath };
```

**Authoritative pattern (code-level, canonical):**

```
<runDir>/output/<stepSeq zero-padded to 4>-<nodeId>.<"stdout"|"stderr">.log
```

Notes:
- Extension is **`.stdout.log` / `.stderr.log`** — NOT `.stdout` / `.stderr`
  as the task spec hints.
- `stepSeq` is zero-padded to **4 digits** (`String(seq).padStart(4, "0")`).
- `nodeId` is the graph node id of the executing step (e.g. `build`,
  `report`). The pair `(stepSeq, nodeId)` is always unique per run because
  `stepSeq` already is.
- The `output/` directory is created lazily by the engine on first step —
  runs with no step execution yet won't have it.

### Run-directory convention (`runDir` parameter)

All code that deals with a single run uses the **run directory** path
(`runs/<id>/`), not the parent `runs/` directory:

- `RunDirectory.path` (`run-manager.ts:17-21`) is `join(runsDir, id)`.
- `readEventLog(runDir)` in `src/core/replay.ts` joins `events.jsonl`
  directly onto the argument.
- `ResumeHandle.runDir` follows the same convention.

**Decision:** `getSidecarStream` takes the run directory (e.g.
`/abs/path/to/runs/20260416-…`), matching every other sibling API. The
plan will call it `runDir` consistently.

## 2. Function signature

```ts
// src/core/sidecar.ts

export interface SidecarRef {
  stepSeq: number;
  nodeId: string;
  stream: "stdout" | "stderr";
}

/**
 * Resolve the sidecar transcript file for a single step execution and
 * return a Web ReadableStream of its bytes.
 *
 * The sidecar file is written by the engine at step start; its canonical
 * location is
 *   <runDir>/output/<stepSeq-padded-to-4>-<nodeId>.<stream>.log
 *
 * @param runDir absolute path to a run directory (e.g. `runs/20260416-…/`),
 *   NOT the parent `runs/` directory.
 * @param ref the identifying triple; in practice the caller will have this
 *   as an `output:ref` event payload.
 * @returns a Web `ReadableStream<Uint8Array>` over the file contents. The
 *   stream is backed by a Node `fs.ReadStream`; cancelling the returned
 *   stream via `stream.cancel()` destroys the underlying Node stream
 *   (handled by `Readable.toWeb`).
 * @throws SidecarNotFoundError when either `<runDir>/output/` or the
 *   specific sidecar file is missing. All other fs errors propagate.
 */
export function getSidecarStream(
  runDir: string,
  ref: SidecarRef,
): Promise<ReadableStream<Uint8Array>>;
```

### New error class

```ts
// src/core/errors.ts  (addition)

/**
 * Thrown by `getSidecarStream` when the requested sidecar transcript file
 * cannot be located on disk. Distinguishes a "no such transcript" case
 * (expected for steps that haven't run yet, or after transcripts have been
 * pruned) from a generic Node `ENOENT`, so consumers can render a clean
 * "no output yet" state without inspecting `err.code`.
 */
export class SidecarNotFoundError extends MarkflowError {
  readonly runDir: string;
  readonly stepSeq: number;
  readonly nodeId: string;
  readonly stream: "stdout" | "stderr";

  constructor(
    runDir: string,
    ref: { stepSeq: number; nodeId: string; stream: "stdout" | "stderr" },
    resolvedPath: string,
    reason: "file-missing" | "dir-missing",
  ) {
    const detail = reason === "dir-missing"
      ? `output directory does not exist`
      : `sidecar file does not exist`;
    super(
      "SIDECAR_NOT_FOUND",
      `Sidecar ${detail}: ${resolvedPath} (runDir=${runDir}, stepSeq=${ref.stepSeq}, nodeId=${ref.nodeId}, stream=${ref.stream})`,
    );
    this.name = "SidecarNotFoundError";
    this.runDir = runDir;
    this.stepSeq = ref.stepSeq;
    this.nodeId = ref.nodeId;
    this.stream = ref.stream;
  }
}
```

Design note: both "output directory missing" and "file missing" map to
**the same error class** with distinct `reason` strings in the message.
Rationale — from a consumer's standpoint, "sidecar does not exist" is one
concept; they shouldn't need two catch arms. The `reason` payload is
informational. If the `runDir` itself doesn't exist, we also map to
`SidecarNotFoundError` (dir-missing variant) — same reason: one class, one
catch.

All other errors (permission denied, I/O error mid-stream, etc.) are
allowed to propagate as raw Node errors; callers can branch on
`err.code === "EACCES"` etc. as needed.


## 3. Implementation location and sketch

**New file:** `/Users/boyd/wip/markdown-flow/packages/markflow/src/core/sidecar.ts`

Small, pure, focused — single function plus the `SidecarRef` type.

```ts
import { createReadStream } from "node:fs";
import { stat } from "node:fs/promises";
import { Readable } from "node:stream";
import { join } from "node:path";
import { SidecarNotFoundError } from "./errors.js";

export interface SidecarRef {
  stepSeq: number;
  nodeId: string;
  stream: "stdout" | "stderr";
}

export async function getSidecarStream(
  runDir: string,
  ref: SidecarRef,
): Promise<ReadableStream<Uint8Array>> {
  const seqStr = String(ref.stepSeq).padStart(4, "0");
  const fileName = `${seqStr}-${ref.nodeId}.${ref.stream}.log`;
  const sidecarPath = join(runDir, "output", fileName);

  try {
    const s = await stat(sidecarPath);
    if (!s.isFile()) {
      throw new SidecarNotFoundError(runDir, ref, sidecarPath, "file-missing");
    }
  } catch (err) {
    const code = (err as NodeJS.ErrnoException).code;
    if (code === "ENOENT" || code === "ENOTDIR") {
      // ENOTDIR covers the "output is missing entirely" case when an
      // intermediate component of the path isn't a directory; ENOENT covers
      // both "output/ missing" and "file missing". We don't need to
      // distinguish at the fs layer — the resolved path is the same.
      throw new SidecarNotFoundError(runDir, ref, sidecarPath, "file-missing");
    }
    throw err;
  }

  const nodeStream = createReadStream(sidecarPath);
  // Readable.toWeb bridges Node → Web Streams. Stable in Node 18+. When the
  // consumer calls `stream.cancel()` on the returned ReadableStream, the
  // underlying Node ReadStream is destroyed — no manual wiring needed.
  return Readable.toWeb(nodeStream) as ReadableStream<Uint8Array>;
}
```

### Bridge choice: why `Readable.toWeb`

Three options considered:

1. `Readable.toWeb(nodeReadable)` — one line; stable in Node ≥18; backpressure
   and cancellation handled internally.
2. `new ReadableStream({ start(controller) { … }, cancel() { … } })` —
   manual bridge; must wire `data`/`end`/`error` and teardown on cancel.
3. `ReadableStream.from(asyncIterable)` — Node ≥20 only, and requires
   iterating the Node stream as an async iterator (works but is
   round-trip-y).

**Choice: option 1.** Shortest, most idiomatic for a Node 18+ project,
and cancellation propagates correctly per the Node docs
("Destroys the `Readable` on which `.toWeb()` was called when the
returned `ReadableStream` is canceled.").

### `Readable.toWeb` availability — Node version check

`/Users/boyd/wip/markdown-flow/packages/markflow/package.json` currently
has **no `engines.node` field**. `Readable.toWeb` is stable from Node
18.0.0 (experimental 16.5+, unflagged 17+, stable 18.0). The repo already
uses `@types/node@^22` and the CI likely runs on a modern Node.

Recommend: **add `"engines": { "node": ">=18" }`** to the markflow package
as a side task. If the maintainer prefers to keep engines unpinned, leave
it out and document the requirement in the TSDoc on `getSidecarStream`.
Listed in §7 as a minor risk, not a blocker.

## 4. Error handling algorithm (precise)

```
Input: runDir (string), ref = { stepSeq, nodeId, stream }

1. Compute seqStr = String(stepSeq).padStart(4, "0")
2. Compute fileName = `${seqStr}-${nodeId}.${stream}.log`
3. Compute sidecarPath = join(runDir, "output", fileName)
4. Call stat(sidecarPath)
   - if ENOENT or ENOTDIR:
       throw new SidecarNotFoundError(runDir, ref, sidecarPath, "file-missing")
   - if stat succeeds but !isFile():
       throw new SidecarNotFoundError(runDir, ref, sidecarPath, "file-missing")
   - any other fs error: propagate raw
5. Open nodeStream = createReadStream(sidecarPath)
6. Return Readable.toWeb(nodeStream)

Runtime stream errors (e.g. disk I/O mid-read, file truncated during read)
propagate via the returned ReadableStream's error channel — consumer sees
them as a rejection on the reader. Not wrapped in SidecarNotFoundError
(different failure mode, different recovery strategy).
```

Confirmation on the `seq` basis: the spec says "the `seq` of the owning
`step:start` event". This matches the writer code (uses `startEvent.seq`
from the `step:start` envelope). No ambiguity.

## 5. Test plan

**File:** `/Users/boyd/wip/markdown-flow/packages/markflow/test/core/sidecar.test.ts`

**Setup approach:** real filesystem via `os.tmpdir()` + `fs.mkdtempSync`.
No mocks. Matches the run-persistence testing style (engine.events.test.ts
also writes real files under a temp runs dir). Each test gets its own
tmpdir created in `beforeEach` and torn down in `afterEach` via
`rm(tmpDir, { recursive: true, force: true })`.

Helper (inline in the test file):

```ts
async function makeRunDir() {
  const tmp = await fs.mkdtemp(join(os.tmpdir(), "markflow-sidecar-"));
  await fs.mkdir(join(tmp, "output"), { recursive: true });
  return tmp;
}

async function writeSidecar(runDir: string, stepSeq: number, nodeId: string, stream: "stdout"|"stderr", body: Buffer | string) {
  const fileName = `${String(stepSeq).padStart(4, "0")}-${nodeId}.${stream}.log`;
  await fs.writeFile(join(runDir, "output", fileName), body);
}

async function drain(stream: ReadableStream<Uint8Array>): Promise<Uint8Array> {
  const chunks: Uint8Array[] = [];
  const reader = stream.getReader();
  for (;;) {
    const { done, value } = await reader.read();
    if (done) break;
    if (value) chunks.push(value);
  }
  // concat
  const total = chunks.reduce((n, c) => n + c.byteLength, 0);
  const out = new Uint8Array(total);
  let off = 0;
  for (const c of chunks) { out.set(c, off); off += c.byteLength; }
  return out;
}
```

### Test cases

1. **Happy path — stdout.**
   - `writeSidecar(dir, 42, "build", "stdout", "hello world")`
   - `const s = await getSidecarStream(dir, { stepSeq: 42, nodeId: "build", stream: "stdout" });`
   - Assert `new TextDecoder().decode(await drain(s))` equals `"hello world"`.
   - Also verify the resolved file on disk is at
     `output/0042-build.stdout.log` (sanity check on the padding).

2. **Happy path — stderr.** Same shape with `stream: "stderr"` and a
   distinct payload. Asserts `.stderr.log` extension is honoured.

3. **Zero-padding at small seqs.**
   `writeSidecar(dir, 7, "a", "stdout", "x")` →
   file must be at `output/0007-a.stdout.log` (matches the arch doc's
   example `0007-build.stdout.log`). Resolver finds it.

4. **Large seqs (> 4 digits) still work.**
   `writeSidecar(dir, 12345, "a", "stdout", "x")` →
   file at `output/12345-a.stdout.log` (padStart is a no-op past width 4).
   Resolver finds it.

5. **Missing file throws `SidecarNotFoundError`** with populated fields.
   - `output/` exists; no sidecar written.
   - `await expect(getSidecarStream(dir, { stepSeq: 99, nodeId: "x", stream: "stdout" })).rejects.toBeInstanceOf(SidecarNotFoundError);`
   - Catch the rejection and assert `.stepSeq === 99`, `.nodeId === "x"`, `.stream === "stdout"`, `.runDir === dir`, `.code === "SIDECAR_NOT_FOUND"`.

6. **Missing `output/` directory throws `SidecarNotFoundError`.**
   Create a tmpdir but do **not** `mkdir output`. Resolver must throw the
   same class (not ENOENT). Consumers get one unified exception.

7. **Missing `runDir` itself throws `SidecarNotFoundError`.**
   Pass a path that doesn't exist at all. Resolver still maps to
   `SidecarNotFoundError` — see §2 design note.

8. **Binary content preserved.**
   Write `Buffer.from([0x00, 0xFF, 0x7F, 0x01, 0xAB])`. Drain the stream
   as `Uint8Array`. Assert byte-for-byte equality (do NOT decode as
   UTF-8 — the input isn't valid UTF-8).

9. **1 MB file streams successfully.**
   Write a 1 MB buffer (e.g. `Buffer.alloc(1024 * 1024, 0x61)`). Drain.
   Assert `result.byteLength === 1024 * 1024`. Per the task spec's note,
   we don't try to prove laziness — only that the stream completes for a
   sizeable file.

10. **Cancellation destroys the underlying Node stream.**
    Write a medium file. Call `getSidecarStream`, get a reader, read one
    chunk, then call `reader.cancel()`. Assert no unhandled errors on the
    process (catch via `process.once("uncaughtException")` guard scoped to
    the test, or rely on vitest's default fail-on-unhandled). This test
    documents the cancellation contract — if it flakes in CI, downgrade
    to a skipped/xfail documenting the expected behaviour.

Not practically enforceable in Jest/Vitest without AST scanning; rely on

## 7. Risks & unknowns

| # | Risk | Mitigation |
|---|------|------------|
| 1 | Signature as written in the spec cannot encode the full filename (§BLOCKER). | Resolve up-front; see Options A/B/C. Plan is written against Option A. |
| 2 | Filename pattern mismatch — guessing `.stdout` when reality is `.stdout.log`. | Resolved by reading `engine.ts:385-406` + arch doc §"Sidecar transcripts". Pattern confirmed: `<seqPadded4>-<nodeId>.<stream>.log`. |
| 3 | `Readable.toWeb` availability on ancient Node. | Node 18+ has it stable. Add `engines.node >= 18` (recommended) or TSDoc note. Repo builds with `@types/node@^22` — clearly not targeting old Node. |
| 4 | Cancellation semantics subtle across the Node↔Web stream bridge. | `Readable.toWeb` documents cancel → destroy. Test case #10 exercises it. If flaky, downgrade to an xfail with a comment. |
| 5 | `output/` directory might be absent for a freshly created run (engine creates it lazily at first step). | Mapped to the same `SidecarNotFoundError` as missing file — one class, one catch (§2 design note). Test #6 covers. |
| 6 | `nodeId` may contain characters that appear in filenames (hyphens, underscores). Looking at the fixtures (`a`, `build`, `setup`, `report`) nothing exotic. | Validator already restricts node ids syntactically via Mermaid. No sanitisation needed at the sidecar layer. |
| 7 | Consumer double-reads the stream. | Documented: each call to `getSidecarStream` returns a fresh stream over a fresh `fs.ReadStream`. Callers that need to re-read must call again. |
