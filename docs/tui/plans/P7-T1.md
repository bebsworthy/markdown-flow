### 0.1 Engine surface is already frozen
- `EngineEvent` includes `step:waiting` `{v:1, nodeId, tokenId, prompt, options: string[]}` and `approval:decided` `{v:1, nodeId, tokenId, choice, decidedAt, decidedBy?}` — `packages/markflow/src/core/types.ts` lines 308–324. - `StepApprovalConfig` and `StepType = "approval"` already exist on `StepDefinition` (lines 103–116), but the waiting modal DOES NOT need them: the `step:waiting` event carries `prompt` and `options` directly. This is load-bearing — the TUI never has to parse the markdown source.

## §2 Exported names + signatures

### 2.1 `src/approval/types.ts`

```ts
/** A single in-flight approval gate, extracted from the event tail. */
export interface PendingApproval {
  readonly runId: string;
  readonly nodeId: string;
  readonly tokenId: string;
  readonly prompt: string;
  readonly options: readonly string[];
  /** seq of the `step:waiting` event that surfaced this gate. */
  readonly waitingSeq: number;
}

/** Local state for the modal's option cursor + submit FSM. */
export interface ApprovalFormState {
  readonly cursor: number;   // index into options
  readonly fsm: "idle" | "submitting" | "error";
  readonly error: string | null;
}

export type ApprovalFormAction =
  | { readonly type: "CURSOR_MOVE"; readonly delta: number }
  | { readonly type: "SUBMIT_START" }
  | { readonly type: "SUBMIT_OK" }
  | { readonly type: "SUBMIT_FAIL"; readonly error: string };

/** Typed outcome of a `decideApproval()` call. */
export type ApprovalSubmitResult =
  | { readonly kind: "ok" }
  | { readonly kind: "locked"; readonly runId: string; readonly lockPath: string }
  | { readonly kind: "notWaiting" }         // race: gate already decided
  | { readonly kind: "invalidChoice" }       // race: options changed
  | { readonly kind: "error"; readonly message: string };
```

### 2.2 `src/approval/derive.ts`

```ts
/**
 * Scan an event tail (newest-last) for open approval gates. A gate is "open"
 * if a `step:waiting` for (nodeId, tokenId) is NOT followed by an
 * `approval:decided` for the same (nodeId, tokenId). Deterministic, O(n).
 */
export function derivePendingApprovals(
  events: readonly EngineEvent[],
  info?: RunInfo | null,
): readonly PendingApproval[];

/**
 * Pending-count per known run. Uses `EngineState.runs.get(id).status` as a
 * cheap first-pass filter (only "suspended" or "running" can have gates);
 * only the currently-tailed `activeRun.events` are scanned in depth. For
 * non-active runs, a coarse "1 if status==='suspended' else 0" is used —
 * the TUI cannot see per-run prompts without a file read, and the badge
 * only needs counts. See §4 D3.
 */
export function countPendingApprovalsByRun(
  runs: ReadonlyMap<string, RunInfo>,
  activeRun: LiveRunSnapshot | null,
): ReadonlyMap<string, number>;

/** Pick the next pending approval to surface (by lowest waitingSeq, stable). */
export function findPendingApproval(
  events: readonly EngineEvent[],
  nodeId?: string,
): PendingApproval | null;
```

## §3 Data flow

### 3.1 Surfacing an approval (engine → UI)

1. Engine emits `step:waiting` (persisted), followed by the paired `token:state running→waiting`.
2. Adapter (`engine/adapter.ts`) yields `{kind:"run", runId, event}` into the engine-reducer queue.
3. `engineReducer` appends the event to `activeRun.events` (ring, cap 500).
4. `<App>` memoises `derivePendingApprovals(activeRun.events, activeRun.info)` — a pure projection executed every render. When its output is non-empty and `state.overlay === null`, `<App>` dispatches `OVERLAY_OPEN` for the first pending gate.
5. Overlay open flips the keybar to `APPROVAL_KEYBAR` and renders `<ApprovalModal>` centered over the AppShell (same overlay slot as `<AddWorkflowModal>`; see `app.tsx` lines 426–448).

**Auto-open guard:** only when `state.mode.kind === "viewing"` and `state.mode.runId === activeRun.runId`. Browsing mode shows only the badge; `a` is the explicit-open trigger from list-view. See §5 D1.

### 3.2 Deciding (UI → engine)

1. User presses Enter in the modal.
2. `<ApprovalModal>` dispatches `SUBMIT_START` locally and also dispatches `APPROVAL_SUBMIT` on the app reducer (keybar pill visually gates on `overlay.state === "submitting"`, matching §5.6 rule 8 + the existing FSM field).
3. `<App>` supplies `onDecide` as `(choice) => decideApproval({ runsDir, runId: overlay.runId, nodeId: overlay.nodeId, choice, decidedBy: process.env.USER })`. Awaited.
4. On `{kind:"ok"}`: `<App>` dispatches `OVERLAY_CLOSE`. The `approval:decided` event flows in via the tail and `derivePendingApprovals` re-reads empty, so no re-open races.
5. On `{kind:"locked"}`: modal stays open, shows `"Another approve is in progress — retry"`. No state mutation on engine.
6. On `{kind:"notWaiting"}` or `{kind:"invalidChoice"}`: modal closes automatically (the gate is gone); pending-counts derive → 0. A toast is out-of-scope for P7-T1 (follow-up — see §7 D6).

### 3.3 Pending-approvals indicator (features §3.7)

- `a Approve (N)` binding uses `countPendingApprovalsByRun` summed via `ctx.toggleState.pendingApprovalsCount`. Per features.md §3.7, for cross-run badging we use the `activeRun`-scoped count only in Phase 7; a global across-all-runs count would require `runManager.watch()` per-run tails, deferred to P8.
- The `[APPROVAL]` mode pill is rendered only while the modal is open. Mode pill is keybar-only (rule 8); the header chrome is untouched.

## §6 Design decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | **Auto-open only in `viewing.<sameRun>`**, not globally. | Matches features §5.8 walkthrough (user is already viewing the run). Auto-popping a modal while the user is in `browsing.workflows` violates "overlays never hijack an unrelated focus" (features.md rule 6). From other modes, `a` is the explicit trigger. |
| D2 | **Component-local `approvalFormReducer`**, NOT global `AppState`. | Cursor + submit FSM are ephemeral, only live during overlay open. Matches P6-T3 / P6-T4 pane-local precedent and keeps `AppState` minimal. The `APPROVAL_SUBMIT` action on `AppState` stays — it is the single boolean the keybar needs for mode pill dimming. |
| D3 | **Per-run pending count is coarse for non-active runs** (`status==='suspended' ? 1 : 0`). | Without reading `events.jsonl` we cannot know the true count. The TUI's `activeRun` ring gives exact counts for the one run the user is tailing; other runs only need a "has at least one" signal for the runs-table badge. Precise multi-run counts ship with P7-T2 or a future `runManager.watchApprovals()`. |
| D4 | **`decideApproval` lives in `src/engine/`**, not a new `src/worker/`. | It is engine-adjacent IO (same envelope as `adapter.ts`: `node:path` + `markflow` imports). Adding another top-level folder dilutes the organisation; a single `decide.ts` alongside `adapter.ts` keeps the engine slice coherent. Purity probe excludes this file explicitly. |
| D5 | **`s` and `Esc` are equivalent in P7-T1** (both close, run stays suspended). | Mockup §5 caption says "Suspend exits the modal without deciding"; no semantic distinction is drawn for `Esc`. Collapsing them avoids a "what did I do?" confusion. If a later iteration wants Esc to mean "keep deciding later in-session" we can re-split without breaking existing tests (they assert behaviour per key, not equivalence). |
| D6 | **No toast on `notWaiting` / `invalidChoice` races**, just auto-close. | The modal closes because `derivePendingApprovals` re-reads empty once the decided/cancelled event reaches the tail. Adding a toast before a toast surface lands (§7 follow-up) would mean inventing one just for this corner. The keybar's pending-count badge returning to 0 is user-visible feedback enough. |
| D7 | **`RunLockedError` surfaces as a modal error, not a crash**. | The CLI `approve` handler prints and exits 1; in a TUI we keep the overlay open with a `"locked — retry"` line. The user's Enter key becomes the retry trigger. Matches the CLI UX intent (the fix is always "wait and retry"). |
| D8 | **Re-derive on every render** (no memoisation on `derivePendingApprovals`). | The 500-cap ring makes the scan O(500) worst case; negligible per render. Memoising on `events` identity is tempting but `activeRun.events` is freshly-allocated on every append (immutable slice), so `useMemo([events])` would re-fire anyway. |
| D9 | **`decidedBy` defaults to `process.env.USER`**. | Mirrors `packages/markflow/src/cli/commands/approve.ts` line 115. `process.env` access lives in the impure `decide.ts`, not in any pure module. |
| D10 | **Match radio glyphs to mockup literally (`◉` / `○`)**, not theme glyphs. | The mockup is normative; introducing a theme slot for "radio" when only this one modal uses them adds surface without payoff. If a second radio-input overlay emerges (resume wizard P7-T2), we theme-ify in that task. |
| D11 | **No use of `StepDefinition.approvalConfig` in the TUI.** | The `step:waiting` event already carries `prompt` + `options`. Reading the markdown source duplicates work the engine already did and risks divergence when inputs re-template the prompt. |
| D12 | **Width/height-as-prop for `<ApprovalModal>`**, same contract as `<AddWorkflowModal>`. | Matches the monolayout convention; keeps rendering deterministic in tests (inject size) while production passes `Math.min(Math.max(40, cols-4), 90)` etc. from `useStdout()` as `app.tsx` already does. |
