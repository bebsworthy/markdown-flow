### 3.2 `EngineAdapterOptions`

```ts
// src/engine/types.ts
export interface EngineAdapterOptions {
  /** Path to `runs/` parent. Required when `runManager` is omitted. */
  readonly runsDir?: string;

  /**
   * Pre-constructed RunManager. If omitted, the adapter builds one from
   * `runsDir`. Provided primarily for tests that want to wrap a custom
   * RunManager impl.
   */
  readonly runManager?: RunManager;

  /**
   * Optional initial run id to tail. If set, the adapter starts a
   * `tailEventLog` subscription immediately. If unset, the adapter only
   * yields list events (from `runManager.watch`) until
   * `controller.selectRun(runId)` is called (see §3.5 for the control
   * surface — the `AsyncIterable` value itself carries a method, see §3.6).
   */
  readonly runId?: string;

  /**
   * Inclusive starting seq for the per-run tail. Defaults to 0 (all
   * events). When replaying an existing run that was previously hydrated
   * from `readEventLog`, the caller passes `snapshot.lastSeq + 1` to skip
   * what's already folded.
   */
  readonly fromSeq?: number;

  /**
   * Abort signal. When fired, both the list watcher and any active
   * per-run tail unsubscribe cleanly; the adapter async iterable then
   * returns.
   */
  readonly signal: AbortSignal;
}
```

### 3.3 `EngineAdapterEvent` — the merged yield shape

Discriminated union so the reducer's switch stays exhaustive:

```ts
// src/engine/types.ts
export type EngineAdapterEvent =
  | {
      readonly kind: "list";
      /** Forwarded 1:1 from `runManager.watch`. */
      readonly event: MarkflowRunEvent;
    }
  | {
      readonly kind: "run";
      /** Which run this engine event belongs to (the active tail). */
      readonly runId: string;
      readonly event: EngineEvent;
    }
  | {
      readonly kind: "run:detached";
      /** Fired when the per-run tail ends because the run reached a terminal
       *  workflow event (workflow:complete / workflow:error) or the active
       *  runId was swapped. Gives the reducer a hook to freeze its live
       *  snapshot. */
      readonly runId: string;
      readonly reason: "terminal" | "swapped" | "aborted";
    };
```

Where `MarkflowRunEvent` is imported from `markflow` and re-exported by the adapter module so consumers have a single import point.

### 3.4 Merging the two sources — bounded queue / fan-in

Internal implementation (not exported):

```ts
//  1. createEngineAdapter(opts) returns `async function* generator()`.
//  2. Inside the generator we maintain:
//       - a FIFO `queue: EngineAdapterEvent[]`
//       - a `waiter: { resolve: () => void } | null`
//       - a `listController = new AbortController()` — linked to opts.signal
//       - a `tailController: AbortController | null` — per active runId
//       - `activeRunId: string | null`
//  3. push(event) pushes to queue + resolves any pending waiter.
//  4. A background pump for the list: `for await (const e of rm.watch({signal: listController.signal})) push({kind:"list", event:e})`.
//     When the pump returns (abort / error), push is idempotent: queue drains
//     then generator ends.
//  5. A per-run pump is spawned on demand (see §3.5):
//     `for await (const e of tailEventLog(runDir, fromSeq, {signal: tailController.signal})) push({kind:"run", runId, event:e})`.
//     When it ends naturally, push a `{kind:"run:detached", reason:"terminal"}`.
//     When swapped out, push `{kind:"run:detached", reason:"swapped"}`.
//  6. The generator's main `while` loop:
//       while (!done) {
//         while (queue.length) { yield queue.shift()!; if (done) break; }
//         if (done) break;
//         if (opts.signal.aborted) { done = true; break; }
//         await new Promise<void>((r) => { waiter = { resolve: r }; });
//       }
//  7. `finally` aborts both controllers and awaits both pumps' completions
//     to ensure clean teardown before returning.
```

This is the same pattern used by `run-manager.ts::watchImpl` (see lines 252–432) — reusing the proven queue/waiter idiom rather than reinventing.

### 3.5 Changing `runId` mid-stream (swap without restarting list)

Per the task brief: *"When a new runId is selected (user navigates into viewing mode), how the adapter swaps the per-run stream without dropping list updates."*

**Decision.** The adapter does **not** expose a mutable control surface. Instead, the pattern is:

- The React hook owns the `AbortController` for the whole adapter.
- When `runId` changes, the hook tears down the **entire** adapter (aborts the controller, awaits the `for await` loop to exit) and creates a fresh one with the new `runId`.
- Rationale: mirrors how `tailEventLog` and `runManager.watch` are individually designed (single-use iterators). Keeps the adapter pure and immutable — no "subscribe()/unsubscribe()" surface, no internal mutable state machine to reason about.
- Cost: the list watcher is briefly torn down and re-hydrated. Because `runManager.watch` emits one `added` event per existing run on first subscribe (see run-manager.ts line 407–410), the reducer simply re-accepts them; the state slice is idempotent on `{kind: "list", event: {kind: "added", …}}` (§5.2 below spells this out).

This matches features.md §6.2 — the event log is the single source of truth; the reducer folds idempotently.

**Not exposed (but documented as a future extension, §10 risks):** a `SwappableEngineAdapter` variant that exposes `.selectRun(runId)` for callers that demonstrably can't tolerate the re-hydration tick. Not needed for MVP.

## 4. Adapter implementation sketch

Not final code — enough to prove the pattern works and to pin down the subtle race points so the implementer doesn't get surprised.

```ts
// src/engine/adapter.ts

import {
  createRunManager,
  tailEventLog,
  type RunManager,
} from "markflow";
import type { EngineAdapterEvent, EngineAdapterOptions } from "./types.js";
import { join } from "node:path";
// NOTE: only `node:path` allowed here — deterministic, zero-side-effect
// path join. No fs / no ink / no react.

export function createEngineAdapter(
  options: EngineAdapterOptions,
): AsyncIterable<EngineAdapterEvent> {
  return { [Symbol.asyncIterator]: () => run(options) };
}

async function* run(
  options: EngineAdapterOptions,
): AsyncIterator<EngineAdapterEvent> {
  const rm: RunManager =
    options.runManager ??
    createRunManager(requireRunsDir(options));

  const queue: EngineAdapterEvent[] = [];
  let waiter: { resolve: () => void } | null = null;
  let done = false;

  const wake = () => {
    const w = waiter;
    waiter = null;
    w?.resolve();
  };

  const push = (event: EngineAdapterEvent) => {
    if (done) return;
    queue.push(event);
    wake();
  };

  const listController = new AbortController();
  const tailController: AbortController | null = options.runId
    ? new AbortController()
    : null;

  // Link the caller's signal → both controllers.
  const onExternalAbort = () => {
    listController.abort();
    tailController?.abort();
  };
  if (options.signal.aborted) onExternalAbort();
  else options.signal.addEventListener("abort", onExternalAbort, { once: true });

  // --- list pump ---------------------------------------------------------
  const listPump = (async () => {
    try {
      for await (const ev of rm.watch({ signal: listController.signal })) {
        push({ kind: "list", event: ev });
      }
    } catch {
      // fs.watch errors surface here; swallow (matches engine convention).
    }
  })();

  // --- per-run tail pump (optional) --------------------------------------
  let tailPump: Promise<void> = Promise.resolve();
  if (options.runId && tailController) {
    const runId = options.runId;
    const runDir = join(requireRunsDir(options), runId);
    const fromSeq = options.fromSeq ?? 0;
    tailPump = (async () => {
      try {
        for await (const ev of tailEventLog(runDir, fromSeq, {
          signal: tailController.signal,
        })) {
          push({ kind: "run", runId, event: ev });
        }
        push({
          kind: "run:detached",
          runId,
          reason: tailController.signal.aborted ? "aborted" : "terminal",
        });
      } catch {
        push({ kind: "run:detached", runId, reason: "aborted" });
      }
    })();
  }

  try {
    while (!done) {
      while (queue.length > 0) {
        const ev = queue.shift() as EngineAdapterEvent;
        yield ev;
        if (done) break;
      }
      if (done) break;
      if (options.signal.aborted) {
        done = true;
        break;
      }
      await new Promise<void>((resolve) => {
        waiter = { resolve };
      });
    }
  } finally {
    done = true;
    options.signal.removeEventListener?.("abort", onExternalAbort);
    listController.abort();
    tailController?.abort();
    await Promise.allSettled([listPump, tailPump]);
  }
}

function requireRunsDir(o: EngineAdapterOptions): string {
  if (o.runsDir) return o.runsDir;
  throw new Error(
    "createEngineAdapter: runsDir is required when runManager is not provided",
  );
}
```

### Subtle points flagged for implementer

1. **Order guarantee within a single source.** Events from `rm.watch` preserve their internal ordering; events from `tailEventLog` preserve theirs. We make **no** cross-source ordering guarantee — list events and run events are two independent streams (as the task describes). The reducer treats them independently.
2. **Terminal detachment before abort.** If the tail completes naturally (`workflow:complete`), we emit `run:detached{reason:"terminal"}`, then the tail pump resolves. The list pump continues until `opts.signal` fires. This matches the brief: *"unsubscribes cleanly on unmount"*.
3. **`allSettled` on teardown.** We must await both pumps in `finally` or the event loop would leak. Using `allSettled` (not `all`) ensures one pump's rejection doesn't skip the other's cleanup.
4. **`node:path` exception.** The adapter joins `runsDir + runId` to derive the per-run directory. `node:path` is deterministic and side-effect-free; the purity test allows it explicitly for this file (see §9 for the updated forbidden-regex list).

## 5. Reducer / state slice decision

Per the task brief: *"Convert engine events into `Action`s that extend the existing union (requires extending `types.ts`) **or** produce a separate `EngineAction` union that flows to a separate engine slice."*

**Decision: (b) — separate engine slice.**

### 5.1 Justification

- P3-T1's `src/state/types.ts` explicitly reserves this: *"Engine-derived state (tokens, snapshot, batches) is not stored here; that lands in P3-T2 via an engine adapter that keeps snapshots in a separate slice."*
- The existing UI reducer is pure UI-only (modes, overlays, filters). Folding ~25 engine event types into it would bloat the switch and couple UI-only concerns to engine-event schema drift.
- Two separate reducers, each pure, each tested independently, matches the "single fold" discipline the engine itself uses (`replay.ts`).
- React composition is trivial: two `useReducer` hooks in the root `<App>`, passed down via context. No Redux-style combineReducers ceremony.

### 5.2 `EngineState` shape

export interface LiveRunSnapshot {
  readonly runId: string;
  readonly info: RunInfo;            // from list events (added / updated)
  readonly events: readonly EngineEvent[]; // bounded tail (see §5.4)
  readonly lastSeq: number;
  readonly terminal: boolean;

export interface EngineState {
  /** Ordered list of known runs (from `rm.watch`). Newest first, matching
   *  the spec's run-list ordering. */
  readonly runs: ReadonlyMap<string, RunInfo>;

  readonly activeRun: LiveRunSnapshot | null;

  /** True if `rm.watch` has emitted its initial burst of `added` events. */

  runs: new Map(),
  activeRun: null,
  listHydrated: false,

### 5.3 Action variants (the adapter's dispatch surface)

Per the brief — *"If you add new Action variants, list them precisely."*

```ts
// src/engine/types.ts
export type EngineAction =
  // --- List-watcher-driven -------------------------------------------------
  | { readonly type: "RUN_ADDED"; readonly runId: string; readonly info: RunInfo }
  | { readonly type: "RUN_UPDATED"; readonly runId: string; readonly info: RunInfo }
  | { readonly type: "RUN_REMOVED"; readonly runId: string }
  | { readonly type: "LIST_HYDRATED" }                // (*) see §5.5
  // --- Per-run tail-driven -------------------------------------------------
  | { readonly type: "RUN_TAIL_EVENT"; readonly runId: string; readonly event: EngineEvent }
  | { readonly type: "RUN_TAIL_DETACHED"; readonly runId: string; readonly reason: "terminal" | "swapped" | "aborted" };
```

### 5.4 Event-buffer bounding

Keeping every `EngineEvent` in `activeRun.events` unbounded would grow without bound on long runs. For MVP we keep at most the last **500** events in a ring; `snapshot` projection from the snapshot library (engine's `replay`) will be layered in later phases. The reducer simply slices when over-limit. Consumers that need full history use `readEventLog` on demand.

### 5.5 Translating `EngineAdapterEvent` → `EngineAction`

The adapter does **not** dispatch directly — it just yields `EngineAdapterEvent`s. The hook (or a helper `toEngineAction(evt): EngineAction`) translates:

```ts
// src/engine/reducer.ts (or adapter.ts — helper only, pure)
export function toEngineAction(e: EngineAdapterEvent): EngineAction | null {
  switch (e.kind) {
    case "list":
      switch (e.event.kind) {
        case "added":   return { type: "RUN_ADDED",   runId: e.event.runId, info: e.event.snapshot };
        case "updated": return { type: "RUN_UPDATED", runId: e.event.runId, info: e.event.snapshot };
        case "removed": return { type: "RUN_REMOVED", runId: e.event.runId };
      }
    case "run":
      return { type: "RUN_TAIL_EVENT", runId: e.runId, event: e.event };
    case "run:detached":
      return { type: "RUN_TAIL_DETACHED", runId: e.runId, reason: e.reason };
  }
}
```

Then `reducer(engineState, action)` folds it. Pure, testable.

*(*) LIST_HYDRATED is optional and not directly produced by the adapter; the current design does not emit a hydration marker. Keep it in the type for future extension (e.g. adding `{kind:"list:hydrated"}` to `EngineAdapterEvent`). We'll wire it if a UI spinner needs it.*

## 6. React integration pattern — `useEngineAdapter`

The only file that imports from `react` in this task.

```tsx
// src/hooks/useEngineAdapter.ts
import { useEffect, useReducer, useRef } from "react";
import { createEngineAdapter } from "../engine/adapter.js";
import { engineReducer, initialEngineState, toEngineAction } from "../engine/reducer.js";
import type { EngineAdapterEvent, EngineState } from "../engine/types.js";

interface UseEngineAdapterOptions {
  runsDir: string;
  runId?: string;
  fromSeq?: number;
}

export function useEngineAdapter(opts: UseEngineAdapterOptions): EngineState {
  const [state, dispatch] = useReducer(engineReducer, initialEngineState);

  useEffect(() => {
    const controller = new AbortController();
    const adapter = createEngineAdapter({
      runsDir: opts.runsDir,
      runId: opts.runId,
      fromSeq: opts.fromSeq,
      signal: controller.signal,
    });

    let cancelled = false;
    (async () => {
      for await (const event of adapter) {
        if (cancelled) break;
        const action = toEngineAction(event);
        if (action) dispatch(action);
      }
    })().catch(() => {
      /* adapter errors are non-fatal; teardown handled below */
    });

    return () => {
      cancelled = true;
      controller.abort();
    };
    // runId change tears down and re-mounts — matches §3.5.
  }, [opts.runsDir, opts.runId, opts.fromSeq]);

  return state;
}
```

**Lifecycle guarantees checked off:**

- Mount → creates `AbortController`, spawns async loop.
- `runId` prop change → React cleanup runs → `controller.abort()` → adapter `finally` runs → both pumps' signals abort → `fs.watch` handles closed → adapter returns → `for await` exits. **Then** the new effect runs with the new `runId`.
- Unmount → same cleanup path; no lingering FS watchers.
- Strict mode double-invoke → cleanup between invocations prevents watcher leaks.

**Tested indirectly** via the adapter unit tests (component tests for this hook are out of scope — a P3-T2 acceptance criterion is adapter-level only). A later task can add `ink-testing-library` renders of a host component once one exists.

## 8. Test plan

Vitest convention: explicit `import { describe, it, expect, beforeEach, afterEach } from "vitest"` at the top of each file — matching the engine's `tail-event-log.test.ts` and `run-manager-watch.test.ts`.

### 8.1 Fixture strategy — build tiny run dirs inline

The engine's own tests (`test/core/run-manager-watch.test.ts` and `test/core/tail-event-log.test.ts`) build run dirs via `mkdtemp` + helpers that write `meta.json` and append to `events.jsonl`. We reuse the exact same pattern — **no committed run-dir fixtures**. Reasons:

- Stays consistent with the engine's own idiom; reviewers recognise it instantly.
- Forces tests to state their preconditions inline, which doubles as documentation.
- No cleanup mess across workspaces.
- Timestamps / ids are generated per-test, avoiding global state collisions.

## 9. Purity enforcement

### 9.1 Extend the existing purity test

Replace the `files` array in `test/state/purity.test.ts` with:

```ts
// test/state/purity.test.ts (edited)
const files = [
  "../../src/state/reducer.ts",
  "../../src/state/types.ts",
  // Engine adapter + reducer — same no-React rule, but node:path is allowed
  // for adapter.ts only. See exception list below.
  "../../src/engine/types.ts",
  "../../src/engine/reducer.ts",
  "../../src/engine/adapter.ts",
  "../../src/engine/index.ts",
];

const FORBIDDEN = [
  /\bfrom\s+["']ink["']/,
  /\bfrom\s+["']ink\//,
  /\bfrom\s+["']react["']/,
  /\bfrom\s+["']react\//,
  /\bimport\(\s*["']ink["']/,
  /\bimport\(\s*["']react["']/,
];

// Files that are allowed to touch node:path (deterministic, side-effect-free)
// but NOTHING ELSE from node:*. Every other node: import is still forbidden
// for these modules.
const NODE_PATH_ONLY: ReadonlySet<string> = new Set([
  "../../src/engine/adapter.ts",
]);

const NODE_ANY = /\bfrom\s+["']node:([a-z_/-]+)["']/g;

describe("pure-module purity", () => {
  for (const rel of files) {
    it(`${rel} has no forbidden imports`, () => {
      const source = readFileSync(resolve(__dirname, rel), "utf8");
      for (const re of FORBIDDEN) expect(source).not.toMatch(re);
      // Scan every `node:*` import: only `node:path` allowed, only in the
      // adapter file.
      const matches = [...source.matchAll(NODE_ANY)];
      for (const m of matches) {
        const spec = m[1];
        const allowed = NODE_PATH_ONLY.has(rel) && spec === "path";
        expect(
          allowed,
          `${rel} imports node:${spec} which is not in the allowlist`,
        ).toBe(true);
      }
    });
  }

  it("reducer module loads in a Node-only context", async () => {
    const mod = await import("../../src/state/reducer.js");
    expect(typeof mod.reducer).toBe("function");
  });

  it("engine adapter loads without Ink/React", async () => {
    const mod = await import("../../src/engine/adapter.js");
    expect(typeof mod.createEngineAdapter).toBe("function");
  });
});
```

This extends — not replaces — the P3-T1 test. The existing three state-reducer assertions still run; four new engine assertions are added.

### 9.2 Why not a separate `test/engine/purity.test.ts`?

Could do. Chose to extend because:
- One purity file = one source of truth for the allowlist.
- The forbidden list is identical for both state and engine; splitting duplicates.
- Rename the top-level `describe` so the report reads sensibly.

## 11. Risks / unknowns

1. **Cross-source ordering.** The adapter gives no ordering guarantee between `list` and `run` events. If a future UI flow depends on "the list `updated` event for run X must arrive after the last `run` tail event for X", we'd need a join buffer. Not in MVP — flagged here.
2. **`fs.watch` reliability on macOS & Linux.** Already the engine's problem, not the adapter's. We inherit whatever `tailEventLog` and `rm.watch` ship — both have existing test coverage on macOS/Linux.
3. **Ring buffer size (500) is a magic number.** Callers that need replay scroll will fetch `readEventLog` on demand. If the ring buffer is too small for the step-detail view, bump it or swap for an `EngineSnapshot` projection in a later task (not MVP-blocking).
4. **No ordering of per-run tails across swaps.** If adapter A for `runId=r1` is aborting while adapter B for `runId=r2` is mounting, the React `useEffect` cleanup ensures A is fully torn down before B mounts (cancellation flag + await on `for await`). We rely on React's effect contract; the adapter is stateless wrt its neighbours, so there is no cross-instance state to corrupt.
5. **`process.getActiveResourcesInfo?.()` in test 6** requires Node ≥18.10. A feature-detect fallback that silently skips the test on older Node is fine — matches the engine's own conservative approach.
6. **Importing `markflow` inside tests.** Vitest/tsx resolves the workspace dep through `node_modules/markflow` → the built `dist/`. If `markflow` hasn't been built, the adapter imports will fail. **Mitigation:** the validation plan in §10 runs `npm test -w packages/markflow -- --run` first (a soft dependency that the engine package has been built for its own tests). CI already builds both workspaces before testing. For local dev, the `packages/markflow/package.json` `main` points at `dist/core/index.js` — if the engine is not built, `import from "markflow"` will 404, which is a clear error. Not a blocker; document in the test file's header comment.
7. **`EngineAdapterEvent.kind: "run:detached"` naming.** Colon in a discriminant string is legal but slightly unusual. Alternatives: `runDetached`, `run_detached`. Picked `"run:detached"` to stay consistent with the engine's event type naming (`step:start`, `workflow:complete`). Flagging for reviewer preference.
8. **Optional `LIST_HYDRATED` action is unused in MVP.** Keep the slot or drop? Recommend **dropping** it (YAGNI) and adding a hydrate marker only when a real UI needs it. Final call with user — marked as a minor API decision.
