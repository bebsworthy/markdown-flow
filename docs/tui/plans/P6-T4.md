### 0.3 Engine data available

The Graph tab consumes exactly what P6-T1 consumes: `EngineState.runs.get(runId).info` + `EngineState.activeRun.events` (ring, cap 500). No new engine surface needed.

The Events tab consumes `EngineState.activeRun.events` (ring, cap 500). That ring is the exact "feed" required — every persisted + non-persisted event flows through it (`step:output` included; see `NON_PERSISTED_EVENT_TYPES` in `packages/markflow/src/core/types.ts` line 335). For a **terminal** run (no live tail), the ring is empty but the table still needs to show the historical events. **Decision §7 D3:** for terminal runs, read `runs/<id>/events.jsonl` lazily via `readEventLog(runDir)` (already re-exported from `markflow` per features.md line 62) on tab open, merge-by-`seq` with the ring (ring wins for in-flight runs). This keeps the live path zero-IO and the post-run path exactly one file read.

### 0.4 Pane-local state precedent

P6-T3 § "Why local reducer, not global `AppState`" (lines 259–264) established the pattern: pane-local `useReducer` inside the pane's View component. The Events tab follows the same pattern; the Graph tab is read-only (no pane-local state beyond what P6-T1 already supplies via `StepTableView` memos + `selectedStepId` from `AppState`) — so only the Events tab needs a reducer.

### 0.5 Acceptance criterion "scroll position per tab"

`AppState` currently has no per-tab scroll offsets. The criterion requires that switching `1→2→3→4→1` preserves scroll position in each tab. Scroll offsets live in:

- Graph tab — reuses step-table cursor. No cursor state persisted in `AppState` today (P6-T1 renders read-only; P6-T2 `selectedStepId` is the only "cursor-ish" bit). **Decision §7 D5:** the Graph tab treats `selectedStepId` as its "cursor" (cursor-follows-selection); it is already persisted in `AppState`, so swapping tabs preserves it trivially.
- Detail tab — no scroll state (single-screen key/value, height-aware field dropping).
- Log tab — reducer is pane-local (P6-T3 D2). Unmounting the pane loses scroll. **Decision §7 D6:** the Log-tab pane-local reducer state **is preserved across tab switches via a ref** in `viewing-panes.tsx` (see §4), not remounted. This matches the "scroll position per tab" criterion literally.
- Events tab — same treatment as Log: pane-local reducer, preserved across tab switches.

## §2 Data model

### 2.1 Events tab

```ts
// src/events/types.ts
import type { EngineEvent, EngineEventType } from "markflow";

export type EventKindGroup = "run" | "token" | "step" | "route" | "retry" | "batch";

export interface EventsFilter {
  readonly kinds: "all" | ReadonlySet<EventKindGroup>;
  readonly nodeId: string | null;
  readonly search: string;   // substring, case-insensitive, matched against summary+kind+nodeId
}

export interface EventsPanelSettings {
  readonly follow: boolean;
}

export interface EventsPanelState {
  readonly events: readonly EngineEvent[];   // merged ring + disk, seq-ordered
  readonly filter: EventsFilter;
  readonly settings: EventsPanelSettings;
  readonly cursor: number;           // index into FILTERED view
  readonly pausedAtHeadFilteredIdx: number | null;
  readonly searchOpen: boolean;
  readonly searchDraft: string;
}

export interface EventsPanelRow {
  readonly seq: number;
  readonly ts: string;           // "12:03:06.148"
  readonly kindLabel: string;    // padded 14 cols
  readonly group: EventKindGroup;
  readonly nodeId: string | null;
  readonly summary: string;
  readonly role: ColorRole;      // dispatch for color
}

export interface EventsPanelModel {
  readonly header: string;
  readonly banner: { readonly kind: "paused"; readonly newSincePause: number } | null;
  readonly rows: readonly EventsPanelRow[];
  readonly footer: { readonly kind: "more-below"; readonly hidden: number }
                 | { readonly kind: "live-tail" }
                 | null;
  readonly empty: EventsPanelEmptyReason | null;
  readonly isFollowing: boolean;
}

export type EventsPanelEmptyReason =
  | { readonly kind: "no-run" }
  | { readonly kind: "no-events" }
  | { readonly kind: "filtered-out" };

export type EventsReducerAction =
  | { readonly type: "SET_EVENTS"; readonly events: readonly EngineEvent[] }
  | { readonly type: "SET_FILTER_KINDS"; readonly kinds: "all" | ReadonlySet<EventKindGroup> }
  | { readonly type: "SET_NODE_FILTER"; readonly nodeId: string | null }
  | { readonly type: "SET_SEARCH"; readonly search: string }
  | { readonly type: "OPEN_SEARCH" }
  | { readonly type: "CLOSE_SEARCH"; readonly commit: boolean }
  | { readonly type: "TOGGLE_FOLLOW" }
  | { readonly type: "SCROLL_DELTA"; readonly delta: number }
  | { readonly type: "SCROLL_JUMP_HEAD" }
  | { readonly type: "SCROLL_JUMP_TOP" }
  | { readonly type: "SCROLL_PAGE"; readonly direction: "up" | "down"; readonly pageSize: number }
  | { readonly type: "RESET" };
```

### 3.2 Events tab derivation

```ts
// src/events/derive.ts
export function deriveEventsModel(input: DeriveInput): EventsPanelModel {
  const { events, state, viewport, nowMs } = input;
  const filtered = events.filter((e) => matchesFilter(e, state.filter));
  const rows = filtered.map((e) => formatEventRow(e, viewport.width));
  const window = state.settings.follow
    ? rows.slice(Math.max(0, rows.length - viewport.height + headerRows))
    : centerWindow(rows, state.cursor, viewport.height - headerRows);
  const header = `Events · ${shortRunId} · ${filtered.length} / ${events.length}`;
  // … banner + footer computation
  return { header, banner, rows: window, footer, empty, isFollowing: state.settings.follow };
}
```

### 3.3 Merge (`src/events/merge.ts`)

```ts
export function mergeEventSources(
  ring: readonly EngineEvent[],
  persisted: readonly EngineEvent[],
): EngineEvent[] {
  const bySeq = new Map<number, EngineEvent>();
  for (const e of persisted) bySeq.set(e.seq, e);
  for (const e of ring) bySeq.set(e.seq, e);   // ring wins — includes non-persisted (step:output)
  return Array.from(bySeq.values()).sort((a, b) => a.seq - b.seq);
}
```

Non-persisted events (`step:output`) exist only in the ring — they will appear in the Events tab for live runs and drop out for terminal runs read from disk. Documented in code comments and tested.

## §5 Mount strategy — preserve state across tab switches

### 5.1 `<ViewingBottomSlot>` design

```tsx
export function ViewingBottomSlot(props: Props): React.ReactElement {
  const { focus } = props;
  return (
    <>
      <PaneVisibility visible={focus === "detail"}><StepDetailPanelView {...detailProps}/></PaneVisibility>
      <PaneVisibility visible={focus === "log"}><LogPanelView {...logProps}/></PaneVisibility>
      <PaneVisibility visible={focus === "events"}><EventsPanelView {...eventsProps}/></PaneVisibility>
    </>
  );
}
```

`<PaneVisibility>` renders its children inside a `<Box height={visible ? height : 0} overflow="hidden">` wrapper — React keeps the component mounted (preserving `useReducer` state, refs, and in-flight effects) while Ink simply allocates zero rows when hidden.

### 5.2 Why not conditional render

Conditional (`focus === "log" ? <LogPanelView/> : null`) would unmount & remount the log pane on every tab swap. `<LogPanelView>`'s `useSidecarStream` would close + re-open the file on every switch — wasteful and drops the follow/paused state mid-run. Keeping them mounted:

- Preserves `useReducer` state (scroll, cursor, filter, search, follow/paused) → satisfies acceptance criterion.
- Keeps file handles open — no re-read thrash.
- Effects should not perform visible work while hidden: both `<LogPanelView>` and `<EventsPanelView>` already derive their model from props; hidden render simply emits an empty-height `<Box>` and the inner render cost is negligible. We add a guard in `deriveLogModel`/`deriveEventsModel` short-circuit when `height <= 0`.

### 5.3 Graph tab top-slot swap

`focus === "graph"` is a special case because mockup §1 + §4 show the step table *and* the graph as peers — however the authoritative mapping is:

- **RUN mode top slot** = zoomed canvas.
- With `focus === "graph"`, the **step table** is no longer the most informative thing (the tree already supersedes it per features.md §3.3 "same tree as step table"). We collapse both slots into one full-height `<GraphPanelView>`.
- For `focus !== "graph"`, the step table stays on top (where the user interacts) and the focused pane occupies the bottom.

The graph pane is read-only; no state-preservation concern (no `useReducer`).

### 5.5 Keystrokes (viewing mode)

Register in `app.tsx` at the same level as existing global bindings (inside a `viewing.*`-only `useInput`):

| Key | Dispatch |
|---|---|
| `1` | `FOCUS_VIEWING_PANE: "graph"` |
| `2` | `FOCUS_VIEWING_PANE: "detail"` |
| `3` | `FOCUS_VIEWING_PANE: "log"` |
| `4` | `FOCUS_VIEWING_PANE: "events"` |

Because `viewing.*` tab-switch takes priority (numeric keys are handled at the app level, not inside the pane), **we move the log-pane stream filter to `s` (stdout/stderr/both cycle)**. Documented in P6-T4 §7 D1. `<LogPanelView>::useInput` loses its `1/2/3` handlers; adds an `s` cycle. Existing log-panel-view tests updated accordingly (covered in §6 tests).

## §7 Design decisions & rationale

| # | Decision | Rationale |
|---|---|---|
| D1 | Re-key tabs to `1/2/3/4 = graph/detail/log/events`; move log-pane stream filter to `s`. | Plan.md verbatim: "`1/2/3/4` switches tabs". P6-T3 used `2` for log because graph didn't exist yet (D7 there is explicit). Tab-switch is global in viewing mode; pane-internal filter gets a letter key. Destructive-capital rule (§5.6 rule 4) is not violated — stream filter is non-destructive. |
| D2 | `focus === "graph"` consumes the full pane (top + bottom); all other focuses keep the split layout (table on top, focused pane on bottom). | Mockup §1 line 27 shows the tree in the bottom pane (follow mode), and mockup §4/§6 show the step table + bottom pane layout. In RUN mode (zoom), features.md §3.3 says "indented DAG tree". When the user explicitly picks the Graph tab they want the tree, not the table — showing both is redundant. Simpler height math, cleaner mockup parity. |
| D3 | Terminal runs read `events.jsonl` via `readEventLog` once on Events-tab open; live runs use only the ring. | Ring is 500-event capped; terminal runs need full history for "useful for debugging" (task scope). One read on tab open satisfies the "scroll per tab" criterion without any polling. Ring-vs-disk merge is pure and deterministic. |
| D4 | Graph tab has no internal keyboard navigation in P6-T4; `↑/↓` are no-ops. | Plan.md acceptance criterion is tab-switch scroll preservation only. Graph cursor movement depends on `STEP_CURSOR_MOVE` (P6-T2 §4.2 D9, still a no-op) — deferred to a dedicated task. |
| D5 | `selectedStepId` doubles as Graph-tab cursor. | Already in `AppState` (P6-T2); zero new state needed; auto-preserved across tab switches. |
| D6 | `<ViewingBottomSlot>` keeps Detail, Log, Events panes all mounted; Graph-pane has no pane-local state, so it's mounted only when active. | Pane-local `useReducer` state (scroll, follow, filter, search) must survive tab swaps (acceptance criterion). Unmount-on-focus-change would lose it. Visibility gating via `Box height={0}` is Ink-native, does not require refs or portals. |
| D7 | Events-tab search is pane-local inline input, not global FIND. | Global FIND (`Ctrl-P`) scope is workflow discovery, not event stream. An inline one-line prompt inside the Events pane keeps keyboard focus clear and matches mockup §15 row `LOG (follow)` convention where `/ Search` is pane-local. |
| D8 | `readEventLog` is injected via an `eventLogReader: (runDir) => Promise<EngineEvent[]>` prop seam on `<EventsPanelView>`. | CLI purity (no direct Node `fs`) — `readEventLog` is already re-exported from `markflow`. Injection seam keeps component tests fs-free. |
| D9 | `matchesFilter` search is case-insensitive substring against `kind + nodeId + summary` concatenated. | Covers debugging needs without a query language. Fast and predictable. |
| D10 | `EventKindGroup` collapses ~30 event types into 6 user-level groups. | Per-type filter UX would require a multi-select modal; groups map to the mental model in features.md §2.3 event-stream table. `k` cycles through group subsets (`all → step → retry → step+retry → route → batch → all`). |
| D11 | Graph tab disables forEach collapse (`collapseThreshold: Infinity`). | Features.md §3.3: fan-out children are indented under parent; batch aggregation is a step-table concern for dense rows. The Graph tab is the "show me everything" view. |
| D12 | No new engine API. | CLI purity rule; everything needed is already exported (`readEventLog`, `EngineEvent`, `EngineEventPayload`). |
| D13 | Keybar: add `GRAPH_KEYBAR`, `EVENTS_{FOLLOWING,PAUSED}_KEYBAR`; widen `KeybarCtx` with `eventsIsFollowing`. | Matches mockup §15 row `RUN (graph)` and the LOG paused/follow precedent (P6-T3 §6.4). Zero context drift with existing panes. |
| D14 | Purity: `src/events/*` all pure (types, format, filter, reducer, derive, merge). Only `<EventsPanelView>` touches effects. | Mirrors P6-T3 envelope: heavy logic testable with hand-built fixtures, view-layer owns effect seam. |

## §9 Acceptance criteria

**Lifted verbatim from plan.md:**

- **Graph tab** — the indented DAG tree rendered full-size (same tree as step table but with graph-specific framing; reuse P6-T1 rendering).
- **Events tab** — textual stream of raw `EngineEvent` records for the selected run, filterable by kind. Useful for debugging.
- `1/2/3/4` switches tabs without losing scroll position per tab.

**Derived sub-criteria:**

- (A) `ViewingFocus` widens to include `"events"`; existing reducer path untouched.
- (B) `1` → graph, `2` → detail, `3` → log, `4` → events (global in `viewing.*`). `Esc` returns to `graph` from any non-graph tab; `Esc` from graph exits to runs.
- (C) `<GraphPanelView>` reuses `buildStepRows` from P6-T1 with `collapseThreshold: Infinity`; renders full-height when `focus === "graph"`; delegates row rendering to `<StepTable>`.
- (D) `<EventsPanelView>` merges ring + disk events (`mergeEventSources`) and renders via `deriveEventsModel` → `<EventsPanel>`; filterable by kind group, node id, and substring search.
- (E) All four tab panes (Detail, Log, Events persistently; Graph only while active) preserve pane-local state across tab switches — implemented via `<ViewingBottomSlot>` visibility gating.
- (F) Pure modules (`src/events/*`) pass `purity.test.ts`: no `react`/`ink`/`node:*` imports.
- (G) Keybar matches mockup §15 row `RUN (graph)` when `focus === "graph"`; switches to Events-following/paused variants when `focus === "events"`.
- (H) Non-persisted `step:output` events flow through the Events tab from the ring; terminal-run Events tab shows only persisted kinds (documented limitation).
- (I) Log-pane stream filter keybinding migrates from `1/2/3` to `s` cycle; tests updated.
- (J) `npm run lint -w packages/markflow-tui` + `npm test -w packages/markflow-tui -- --run` both green.
- (K) No engine-side change; `packages/markflow/package.json` untouched (CLI purity rule).
