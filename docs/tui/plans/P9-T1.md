```ts
/** One row in the input-prompt modal. */
export interface RunInputRow {
  readonly key: string;
  readonly description: string;
  readonly required: boolean;
  /** Default from `InputDeclaration.default` (empty string if none). */
  readonly placeholder: string;
  /** User-entered override; empty string means "use placeholder". */
  readonly draft: string;
}

/** Overlay-local form FSM — mirrors ResumeFormState. */
export type RunInputFsm = "idle" | "submitting" | "error";

export interface RunInputFormState {
  readonly rows: readonly RunInputRow[];
  readonly cursor: number;            // index into rows
  readonly fsm: RunInputFsm;
  readonly error: string | null;
}

export type RunInputFormAction =
  | { readonly type: "CURSOR_MOVE"; readonly delta: number }
  | { readonly type: "CURSOR_SET"; readonly index: number }
  | { readonly type: "SET_DRAFT"; readonly key: string; readonly value: string }
  | { readonly type: "SUBMIT_START" }
  | { readonly type: "SUBMIT_OK" }
  | { readonly type: "SUBMIT_FAIL"; readonly error: string };

/** Typed outcome of `runWorkflow` (engine/run.ts). */
export type RunWorkflowResult =
  | { readonly kind: "ok"; readonly runId: string }
  | { readonly kind: "invalidInputs"; readonly missing: readonly string[] }
  | { readonly kind: "parseError"; readonly message: string }
  | { readonly kind: "locked"; readonly runId: string; readonly lockPath: string }
  | { readonly kind: "error"; readonly message: string };
```

## §6 Design decisions & rationale

**D1 — When inputs are declared but all are optional, should the modal open?**
*Decision:* Yes. Features.md §3.1 says "opens input prompt if required inputs are declared". Reading strictly, this means the trigger is "required inputs declared". We follow that: `inputs.length > 0 && inputs.some(required)` opens the modal; purely optional inputs auto-launch. This matches walkthrough §5.7 ("only if required inputs missing"). **Users can still invoke `:run` and later edit via resume wizard** — this is not the only entry.

**D2 — Where does the row-draft state live: app reducer or modal-local reducer?**
*Decision:* Modal-local, same as `<ResumeWizardModal>`. The app reducer only carries snapshot `seedRows` + the submit FSM. Rationale:
- Matches P7-T2 precedent.
- Keeps the reducer thin — per-keystroke SET_DRAFT actions add no value at the app level and bloat the action union.
- Modal unmount on Esc drops the drafts, which is the desired semantics (no orphan state).

**D3 — Keybar fixture for the input modal.**
*Decision:* Do **not** add a new `[RUN-INPUT]` keybar fixture. Reuse the existing modal-contextual keybar the way the add-workflow modal does (no dedicated fixture — the modal paints its own footer row `[ ⏎ Run ] [ Esc Cancel ]` and the keybar beneath stays on the current mode's fixture). Rationale: features.md §5.6 only requires a keybar fixture for `[APPROVAL]` and `[RESUME]`; `[RUN-INPUT]` is never named in mockups §15. Adding one would be gold-plating.

**D4 — Enter key behaviour on non-terminal rows.**
*Decision:* `Enter` on any row submits when `canSubmit` is true. On non-terminal rows when submit is blocked, `Enter` advances the cursor (`CURSOR_MOVE delta:+1`). Rationale: matches standard web form UX and keeps the `<ResumeWizardModal>` precedent where `Enter` on the confirm button is the canonical submit but `Enter` from any input is also accepted.

**D5 — `:run <arg>` matching strategy.**
*Decision:* Exact → unique-prefix → fail. Case-sensitive on `workflow.name`; fall back to `entry.id`. Rationale: smallest footprint, mirrors `gh` CLI conventions, avoids fuzzy scoring (no need here — palette users typically know the name).

**D6 — `workspaceDir` derivation.**
*Decision:* `dirname(sourceFile)` — same as `engine/resume.ts:137`. For `.markflow.json`-rooted workspaces the resolver normalises `absolutePath` to the workflow file inside the workspace; `dirname` yields the workspace root. Rationale: one convention, one bug surface.

**D7 — Error-message format for `invalidInputs`.**
*Decision:* Parse the engine's `Missing required workflow inputs: K1, K2.` message (prefix-matched) and surface each key in the modal row's inline banner. Fallback to the raw message. Rationale: the engine owns the canonical list; reproducing it in the TUI would drift.

**D8 — Hide-don't-grey for `r` in runs table.**
*Decision:* The binding is a **silent no-op** on non-terminal rows. The keybar does *not* advertise `r Run` in that context. Rationale: aligns with instruction.md §Global rule "hide-don't-grey". See task body: *"From runs table: only when selection has no active run (hide-don't-grey)."*

**D9 — How does the modal close on successful start?**
*Decision:* The **host** (`App`) closes it, via `dispatch(OVERLAY_CLOSE)` inside `onRunStart`. The modal never self-closes. Rationale: the modal should not know about mode transitions; keeping close control in `App` keeps the FSM diagram clean (OVERLAY_CLOSE → MODE_OPEN_RUN is a two-action sequence, both app-owned).

**D10 — Production `runsDir` still unthreaded.**
*Decision:* Leave threading to the broader P6/P9 integration; in production the modal surfaces `"runsDir is not configured"` identical to the resume wizard today. Rationale: expanding `cli.tsx` here exceeds task scope ("No engine package changes" + implied TUI-only scope).

**D11 — Are we allowed to add a new Overlay variant?**
*Decision:* Yes. Features.md §6.3 FSM sketch is explicitly labeled a *sketch* and the codebase has already extended `Overlay` beyond it (`addWorkflow`). A new `runInput` variant is additive and conforms to the task scope: *"open an input-prompt modal (new `src/components/input-prompt-modal.tsx`)"*. Task body says "no new overlay needed — inputs reuse a short-lived modal"; we interpret this as "no new *Mode*" — the Mode FSM is untouched. Adding a ninth Overlay variant is necessary because the modal needs runId-free, workflowId-scoped state that no existing variant carries.

## §7 Acceptance criteria

**Verbatim from `plan.md` §P9-T1 "Acceptance criteria":**

- [ ] From a fresh `markflow-tui`, a user can register a workflow and start a run without leaving the TUI.
- [ ] `:run <workflow>` matches the browser-`r` behavior.
- [ ] Workflow with required inputs shows the modal; submit is blocked until required fields are non-empty; `Esc` cancels cleanly with no orphan run directory.
- [ ] `RunLockedError` surfaces in-modal, modal stays open (mirrors P7-T1).
- [ ] No engine package changes.

**Derived sub-criteria (enforce in tests):**
- [ ] `packages/markflow/**` is byte-identical to pre-change (assert via CI changed-files view).
- [ ] `engine/run.ts` imports only `markflow` + `node:path` — no `ink`/`react`/`fs` (mirrors `engine/resume.ts`).
- [ ] `runStart/*` passes the purity probe (`test/state/purity.test.ts`).
- [ ] On `r` in browser for workflow with zero inputs, `runWorkflow` is called exactly once and no modal is mounted.
- [ ] On `Esc` in modal, `runWorkflow` is **not** called; overlay clears.
- [ ] `<RunsTable>` never fires `r` on a row whose `info.status` is `running` or `suspended` (i.e. active).
- [ ] `:run foo` resolves via exact-then-unique-prefix match; ambiguous arg returns a palette `usage` result.
- [ ] Lint & tests: `npm run lint -w packages/markflow-tui` and `npm test -w packages/markflow-tui -- --run` both green.
