## §3 Filter grammar

The `/`-bar input is a space-separated term list, case-insensitive for term keys, lazy-matched: the first matching grammar rule wins, no backtracking. Everything unrecognised becomes either a malformed term or a free-text id-prefix per §3.2.5.

### 3.1 Lexing

```
input     := WS? term (WS+ term)* WS?
WS        := [ \t]+
term      := QUOTED | ATOM
ATOM      := [^ \t]+   (no shell-escape rules — users paste)
QUOTED    := '"' [^"]* '"'       (allows embedded spaces in workflow names)
```

Quoted atoms are **decoded** (strip surrounding quotes) before being classified. `workflow:"multi region"` matches on the literal `multi region`. Unterminated quotes fall back to treating the whole remainder as a single atom (defensive — never throws).

### 3.2 Term classification

For each atom, try rules in order:

#### 3.2.1 `status:<value>`

- Case-insensitive `status:` prefix.
- Valid values: `running`, `complete`, `error`, `suspended`.
- Aliases (applied before validation): `ok` → `complete`, `failed` → `error`.
- Any other tail: `{kind: "malformed", raw}`.

#### 3.2.2 `workflow:<substring>`

- Case-insensitive `workflow:` prefix.
- Tail may be empty (e.g. `workflow:`) — that's **malformed** because an empty substring would match everything (confusing UX).
- Tail with content: `{kind: "workflow", value: tailLowerCased}`. The predicate does a case-insensitive `indexOf` on `RunInfo.workflowName`. **We match on `workflowName`, not the display title** — the display title is `workflowName` today (no registry override applies in runs mode); if a future task adds title overrides, update this term's predicate in lockstep.

#### 3.2.3 `since:<duration>`

- Case-insensitive `since:` prefix.
- Tail is fed to `tryParseDurationMs`. `null` → `{kind: "malformed", raw}`.
- Success → `{kind: "since", durationMs}`. Predicate: `info.startedAt >= now - durationMs` (we use `startedAt` not `completedAt` because "show me runs since 30 minutes ago" naturally means "started in the last 30 minutes").

#### 3.2.4 `tryParseDurationMs` grammar (TUI-local)

Subset of the engine's `parseDuration`:

- Pattern: `^(\d+[smhd])+$` after trimming + lowercasing.
- Units: `s` = 1000, `m` = 60_000, `h` = 3_600_000, `d` = 86_400_000.
- `d` (days) is TUI-only — the engine doesn't recognise it, but users will type `since:3d` for archive-range queries. If we promoted `parseDuration` from the engine instead, we'd need to extend it there, which is a wider change than this task wants. See §12 for the full rationale.
- `0` is treated as invalid (`null`) — `since:0s` is nonsense; users who want "everything" clear the filter.
- Never throws. Returns `null` on any parse failure.

#### 3.2.5 Free-text id-prefix

- Any remaining atom becomes `{kind: "idPrefix", value: atomLowerCased}`.
- Predicate: `info.id.toLowerCase().startsWith(value)`. Case-insensitive.
- Empty-string idPrefix is treated as matching nothing (guard at the predicate). An empty bar (no atoms) matches everything (see §3.3).

### 3.3 AND-combining + the empty-filter identity

- Parsed `terms` list with zero non-malformed entries → identity filter (every row passes). The UI still shows malformed terms so the user can correct them.
- Multiple non-malformed terms → logical AND. Each term's predicate must return `true` for the row to pass.
- Two `status:` terms AND together — effectively always empty unless the same value is listed twice. Documented in the filter test but deliberately not special-cased (consistent AND semantics).

### 3.4 Re-parse cadence

- `RUNS_FILTER_INPUT` stores the draft string but does NOT reparse. This keeps the reducer cheap (mid-typing states are common) and keeps the bar from underlining partially-typed terms like `status:r` as malformed.
- `RUNS_FILTER_APPLY` (Enter) triggers the parse. The parsed `RunsFilterInput` is stored in `runsFilter.applied` and the bar closes.
- `RUNS_FILTER_CLEAR` zeroes both draft and applied, and closes the bar.

## §4 Archive policy

### 4.1 Predicate (pure, deterministic)

```ts
// src/runs/filter.ts (conceptual)

export function isArchived(
  info: RunInfo,
  policy: RunsArchivePolicy,
  nowMs: number,
): boolean {
  if (info.status === "running") return false;
  if (info.status === "suspended") return false;
  if (info.completedAt == null) return false; // defensive
  const completedMs = Date.parse(info.completedAt);
  if (!Number.isFinite(completedMs)) return false;
  const age = nowMs - completedMs;
  if (info.status === "complete") return age > policy.completeMaxAgeMs;
  if (info.status === "error")    return age > policy.errorMaxAgeMs;
  return false; // exhaustiveness fallback
}
```

### 4.2 Rules

- **`running` / `suspended` never archived.** They are always shown regardless of archive toggle.
- **`complete` archived when `(now - completedAt) > 24h`.** Strict greater-than — a completion at exactly 24h is still visible.
- **`error` archived when `(now - completedAt) > 7d`.** Same strict boundary.
- **Thresholds live in `RUNS_ARCHIVE_DEFAULTS`.** Configurable on `RunsArchivePolicy`; no env-var override in this task.
- **Source of `now`.** A `nowMs: number` prop on `<RunsTable>` (already threaded for elapsed-time in P5-T1 — we extend its usage). `nowMs` defaulting to `Date.now()` at the call site happens in P5-T3; tests freeze it.
- **Toggle `a` flips `policy.shown`.** Two states only — included vs. excluded from the displayed list. Never a third "only show archived" state.
- **Footer counts both sides.** `shown` = rows passing filter + not archived (or all passing filter when `shown=true`); `archived` = rows passing filter but currently hidden by archive rule. Sum = total rows passing filter.

### 4.3 `applyArchive` signature

```ts
export function applyArchive(
  rows: ReadonlyArray<RunsTableRow>,
  policy: RunsArchivePolicy,
  nowMs: number,
): {
  readonly shown: ReadonlyArray<RunsTableRow>;
  readonly archived: ReadonlyArray<RunsTableRow>;
} {
  const shown: RunsTableRow[] = [];
  const archived: RunsTableRow[] = [];
  for (const row of rows) {
    const a = isArchived(row.info, policy, nowMs);
    if (policy.shown || !a) shown.push(row);
    if (a) archived.push(row);
  }
  return { shown, archived };
}
```

The shape returns both lists so the footer can use `archived.length` directly without re-scanning.

### 4.4 Performance

`applyArchive` runs in one pass over the input list. For 10 000 rows it's ~10k integer compares — trivially under the perf budget. Tests measure it explicitly in §9.

### 6.1 Visual layout

Per mockup §1 line 15 (the `filter: all · last 24h` footer text hints that a filter is *applied*, not that the bar is open). When the bar is open, it sits **between the header row and the first data row** as a single-line overlay:

- Draft is rendered in `theme.colors.fg` (normal weight).
- After the draft, we render a space + dim-colored caret `_`.

### 6.3 Key handling (bar only)

| Key | Effect |
|---|---|
| Printable char | Append to local echo, dispatch `RUNS_FILTER_INPUT(newDraft)`. |

## §7 Virtualisation strategy

### 7.1 Window math (pure)

```ts
// src/runs/window.ts (conceptual)

export function computeWindow(args: {
  rowCount: number;
  cursor: number;
  offset: number;
  visibleRows: number;
}): RunsWindowState {
  if (args.rowCount <= 0 || args.visibleRows <= 0) {
    return { offset: 0, visibleRows: Math.max(0, args.visibleRows), cursor: 0 };
  }
  const clampedCursor = Math.max(0, Math.min(args.rowCount - 1, args.cursor));
  let offset = args.offset;
  offset = Math.max(0, Math.min(offset, Math.max(0, args.rowCount - args.visibleRows)));
  if (clampedCursor < offset) offset = clampedCursor;
  if (clampedCursor >= offset + args.visibleRows) {
    offset = clampedCursor - args.visibleRows + 1;
  }
  return { offset, visibleRows: args.visibleRows, cursor: clampedCursor };
}

export function sliceWindow<T>(
  rows: ReadonlyArray<T>,
  window: { offset: number; visibleRows: number },
): ReadonlyArray<T> {
  return rows.slice(window.offset, window.offset + window.visibleRows);
}
```

### 7.2 `visibleRows` derivation

```ts
export function deriveVisibleRows(
  height: number,
  headerRows: number,
  footerRows: number,
): number {
  return Math.max(0, height - headerRows - footerRows);
}
```

- `headerRows = 2` (column header + optional filter bar line).
- `footerRows = 1` (footer status line).
- At `height = 24`, `visibleRows = 21`. At `height = 10` (default), `visibleRows = 7`.

### 7.3 Scroll model

- Cursor and viewport are independent state. The cursor lives in `RunsTable`'s props (eventually from `AppState.runsCursor` in P5-T3); the viewport offset is local `useState` inside `<RunsTable>` reset on major input changes (filter apply, archive toggle, mount).
- Cursor-follow rule:
  - If cursor < offset → offset = cursor.
  - If cursor >= offset + visibleRows → offset = cursor - visibleRows + 1.
  - Otherwise offset unchanged.
- Offset never goes negative and never exceeds `rowCount - visibleRows`.

### 7.4 Deterministic memoisation

Inside `<RunsTable>`:

```tsx
const filteredRows = useMemo(
  () => applyFilter(rows, runsFilter.applied),
  [rows, runsFilter.applied],
);
const { shown: visibleRows, archived } = useMemo(
  () => applyArchive(filteredRows, runsArchive, nowMs),
  [filteredRows, runsArchive, nowMs],
);
const sortedRows = useMemo(
  () => sortRows(visibleRows, sort),
  [visibleRows, sort],
);
const window = useMemo(
  () => computeWindow({
    rowCount: sortedRows.length,
    cursor,
    offset: viewportOffset,
    visibleRows: deriveVisibleRows(height, 2, 1),
  }),
  [sortedRows.length, cursor, viewportOffset, height],
);
const windowSlice = useMemo(
  () => sliceWindow(sortedRows, window),
  [sortedRows, window.offset, window.visibleRows],
);
```

`nowMs` is passed as a prop; the component never reads `Date.now()`. Tests freeze it.

### 7.5 Complexity guarantees

- `applyFilter`: O(n × t) where t = number of terms (≤ ~10 in practice).
- `applyArchive`: O(n).
- `sortRows`: O(n log n) via native sort.
- `computeWindow` + `sliceWindow`: O(window) for the slice plus O(1) math.
- Render path: O(window) — we only mount `<RunsTableRow>` for rows in the slice.

For the 10k fixture, the filter → archive → sort → slice pipeline runs in well under 25ms on the perf harness (§9) on a commodity laptop; render touches only `visibleRows` (≤ ~30) `<RunsTableRow>` mounts.

### 7.6 Why not React virtualized libraries?

- Additional runtime dependency adds install-time weight for a CLI tool (the TUI already has Ink + React + @inkjs/ui).
- Ink's flexbox layout + monospace-cell model doesn't expose the scroll events such libs rely on.
- The windowing math here is ~90 LOC pure TypeScript; unit-testing it is trivial.

## §9 Perf test harness

### 9.1 Discipline (per testing.md §2 + §4)

- **Never sleep.** All timings are `performance.now()` deltas, no `setTimeout`.
- **Freeze time.** `nowMs = new Date("2026-04-16T12:00:00Z").getTime()`. Fixture generator uses seeded pseudo-random so rows are stable across runs.
- **Pin `FORCE_COLOR=0`** in perf test env — ANSI escape encoding adds overhead we don't want measured.
- **Warmup.** Run each stage once before measuring (V8 JIT warmup). Reported time is median of 5 measured runs.
- **No Ink rendering in L1 perf.** The 10k-row render test lives in L2 with a looser budget; the tight budget is on the pure pipeline only.


### 9.3 Budget multiplier

```ts
function budgetMs(base: number): number {
  const mult = Number(process.env.MARKFLOW_PERF_MULT ?? "1");
  return base * (Number.isFinite(mult) && mult > 0 ? mult : 1);
}
```

CI can set `MARKFLOW_PERF_MULT=3` on under-powered runners. Local dev keeps `1`. This matches the testing.md §4 "CI considerations" spirit of deterministic-yet-tunable budgets.

### 9.4 L2 render budget

In `test/components/runs-table.test.tsx`:

```ts
test("1000-row render < 32ms", () => {
  const rows = generateRows(1000, NOW);
  const t0 = performance.now();
  const { unmount } = render(<RunsTable rows={rows} …/>);
  const elapsed = performance.now() - t0;
  unmount();
  expect(elapsed).toBeLessThan(budgetMs(32));
});
```

The L2 budget is deliberately looser than the plan.md "16ms / frame" target because **`ink-testing-library`'s `render()` cost is not the same as Ink's production per-frame render cost** — it includes layout + string composition up-front. The plan.md criterion is satisfied by the two-layer proof:

1. L1 perf proves pure pipeline < 25ms on 10k rows.
2. L2 proves windowed render is O(visible) — 1000-row fixture renders ≤ 32ms because only ~30 `<RunsTableRow>` components mount; 10k-row render is the same cost because the window is the same size.

We do NOT run a 10k-row render test because ink-testing-library's layout pass over 10k rows (even un-mounted) allocates ~O(n) and pollutes the measurement. The task brief's "10 000-entry fixture" is satisfied by the pure perf file (L1) + the guarantee that the render path is O(window).

### 9.5 Fallback plan

If CI renders 32ms is too tight: (a) narrow the L2 test to 500 rows with 16ms budget, (b) keep the L1 10k perf test as the authoritative budget, (c) document in plan §9 (this section). The acceptance criterion is satisfied as long as (L1 pipeline) + (render path is O(window)) together prove "10k fixture renders without jank".

## §12 Risks + unknowns

1. **`parseDuration` export decision.**
   - Option A: Promote `parseDuration` from `packages/markflow/src/core/duration.ts` into the public API (`src/core/index.ts`).
   - Option B: Mirror the subset we need in `packages/markflow-tui/src/runs/duration.ts` as `tryParseDurationMs`.
   - **Decision: Option B (mirror).** Rationale:
     - The engine's `parseDuration` throws `ConfigError` on malformed input — wrong semantics for a UI filter where malformed is a valid user state (underline, don't crash).
     - The TUI needs `d` (days) which the engine doesn't; promoting + extending would couple the CLI release cycle to a UI feature.
     - Mirror size is ~60 LOC — small enough to not pay the abstraction cost of a public-API change.
     - Pure-mirror module sits under `src/runs/` which is already a scanned purity module — the mirror fits the pattern.
   - **Revisit trigger.** If a third consumer (web UI? another TUI?) needs duration parsing, promote the engine version and have the TUI wrap it in a `tryX` non-throwing adapter.
2. **`nowMs` ticker frequency.**
   - Archive boundaries are 24h and 7d — re-deriving the archived set every second is wasteful (~10k rows × 1 Hz = 10k compares/sec for a feature that only matters at tick-of-hour). Meanwhile, the elapsed column (P5-T1) does want 1 Hz refresh.
   - **Decision.** Two `nowMs` values:
     - `nowMs` for elapsed column — updates at 1 Hz via P5-T3's ticker.
     - Archive recomputation is triggered by: mount, `RUNS_ARCHIVE_TOGGLE`, and a 60-second throttled refresh. Since archived rows only flip at hour/day boundaries, 60s granularity is user-imperceptible.
   - Implementation: `<RunsTable>`'s `applyArchive` memo uses `floorToMinute(nowMs)` as its dep, not `nowMs` directly. One-liner inside the component; no new reducer state.
3. **Filter-bar caret + editing.**
   - We don't model insertion-point editing (only append + backspace + clear). Mid-string edit would need a cursor-position component. Documented in §6.2 as a deliberate simplification; P5-T3 or later can promote to a full `<TextInput>`-style primitive.
4. **Underline for malformed terms.**
   - ANSI underline exists but renders inconsistently in test terminals (the raw `\x1B[4m` may or may not survive ink-testing-library's string composition). We ship color-based annotation (dim background / failed-color text) instead; tests assert on the color attribute, not an underline escape. Reviewers may push back — document in §13 as a potential follow-up.
5. **`RUNS_FILTER_INPUT` dispatching per keystroke.**
   - Each char typed produces one reducer dispatch. For a 50-char filter string, that's 50 dispatches — trivial cost, but visible in redux-style devtools. Alternative: debounce via `useDeferredValue`. Decision: ship the simple version; measure if it becomes a problem.
6. **Interaction with `RUNS_SORT_CYCLE` while filter bar is open.**
   - If the user hits `s` while typing a filter, is it "append s to draft" or "cycle sort"? The `useInput` gate in the table is disabled when the bar is open, and the bar owns `s` as a printable char. So `s` while bar open → appends to draft. Tested in §10.6.
7. **10k perf test flakiness on CI.**
   - GitHub Actions x86 runners occasionally GC mid-test. `MARKFLOW_PERF_MULT=3` gives 3× headroom without invalidating the assertion. Fallback in §9.5 exists if the failure rate is still non-zero.
8. **Memo dep-array correctness.**
   - If a dep changes reference every render (e.g. new `rows` array from parent), the entire pipeline recomputes each frame. `<RunsTable>`'s parent (P5-T3 wiring) MUST memoise its `rows` prop; we add a dev-mode warning in `<RunsTable>` if it detects a new identity more than 10 times/sec. Deferred to P5-T3 as a parent-side concern.
9. **Filter behaviour when runs list is empty.**
   - If the filter matches zero rows, we render "no runs match" as the empty-state text (distinct from "no runs yet"). Add a small branch in `<RunsTable>` based on whether the unfiltered list was empty or filtering made it empty. Tested in `runs-table.test.tsx`.
10. **Narrow-tier filter bar placement.**
    - At <60 cols, the filter bar either takes the full width (per mockup §13) or disappears into a breadcrumb. Mockup §13 shows "breadcrumb + one pane at a time" — we don't ship narrow-tier explicitly in this task (keybar + mode primitives for narrow are still P5-T3 concerns). Filter bar tested at wide + medium only.
