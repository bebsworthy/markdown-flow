## §0 Starting state

Verified against the live tree on `main`.

- **Reducer + state model (P3-T1).** `src/state/types.ts` defines `Mode` as the discriminated union `{ kind: "browsing"; pane: "workflows" | "runs" } | { kind: "viewing"; runId: string; focus: "graph" | "detail" | "log" }`. `src/state/reducer.ts` exports the pure `reducer(state, action)` and `initialAppState`. The four mode-tab transitions we need are **already actions**:

  - `MODE_SHOW_RUNS` → `browsing.runs`
  - `MODE_OPEN_RUN { runId, focus? }` → `viewing(runId)`
  - `MODE_CLOSE_RUN` → `browsing.runs`
  These cover what the mode-tab row needs to dispatch (F1/F2/F3 → workflows / runs / current run). **No new reducer action is required for this task.** The optional pre-selection of a `runId` for the F3 case is handled by reading `state.selectedRunId` in the component — the reducer already preserves it across mode switches.
- **Engine adapter (P3-T2).** `src/engine/adapter.ts` and `src/hooks/useEngineAdapter.ts` produce engine snapshots. The app shell does not consume them directly in this task — the slots simply forward `children`. P5/P6 wires real content into those slots.

- **Keybar (P3-T4).** `src/components/keybar.tsx` is merged and exports `Keybar` + `KeybarProps`. The barrel `src/components/index.ts` re-exports it. The shell **composes** but does not modify keybar.
- **App entry.** `src/cli.tsx` mounts `<App onQuit={...} />`. `src/app.tsx` is still the scaffold stub (no `ThemeProvider`, no shell). Wiring the shell into `App` is part of this task.
- **Test harness.** `vitest.config.ts` picks up everything under `test/**/*.{test,spec}.{ts,tsx}`. `test/state/purity.test.ts` already scans an explicit `files` list for forbidden imports; we add the new pure helper module to that list.
- **No `app-shell.tsx` / `mode-tabs.tsx` / `app-shell-layout.ts` exists yet** (verified via `Glob`). All four are net-new files.

## §2 Component design

### 2.1 `AppShell` (`src/components/app-shell.tsx`)

```tsx
export interface AppShellProps {
  /** Top half of the frame — typically a runs table or workflow browser. */
  readonly top: React.ReactNode;
  /** Bottom half — tabbed pane (Graph / Detail / Log / Events) or workflow preview. */
  readonly bottom: React.ReactNode;
  /** Bottom-of-frame keybar (already-rendered <Keybar/> element). Optional;
   *  callers without a keybar pass `null`. */
  readonly keybar?: React.ReactNode;
  /** Mode-tabs row to render in the top frame line. Typically the
   *  `<ModeTabs/>` component, but accepted as a prop so the shell stays
   *  decoupled from key dispatch in tests. */
  readonly modeTabs: React.ReactNode;
  /** Width override for tests. Production wraps in `useStdout`. */
  readonly width?: number;
  /** Height override for tests. Production reads `useStdout().stdout.rows`
   *  with a sane default of 30 when unset. */
  readonly height?: number;
}
```

**Responsibilities.**

1. Read `theme.frame` (the new glyph table) — never hardcode `═ ║ ╔`.
2. Read terminal `cols × rows` via `useStdout()` unless an override prop is set (tests always pass an override; ink-testing-library can't resize).
3. Compose three frame rows: top (with mode-tabs embedded into the top edge), middle splitter (`╠═══╣`), and bottom edge (`╚═══╝`).
4. Render two slots:
   - **Top half**: `top` content, vertically clamped to `Math.floor((rows - 4) / 2)` lines (4 = 2 frame edges + 1 splitter + 1 keybar).
   - **Bottom half**: `bottom` content, occupying the remaining rows.
5. Render `keybar` outside the frame on the very last terminal row (matching mockups.md §1 line 36).

**What it does NOT do.**

- Does not own `useInput`. All key dispatch lives in `ModeTabs` (or higher up in `App`).
- Does not know about the active mode. The mode-tabs row is just a `ReactNode` prop.

### 2.2 `ModeTabs` (`src/components/mode-tabs.tsx`)

```tsx
export interface ModeTabsProps {
  readonly mode: AppState["mode"];
  readonly selectedRunId: string | null;
  readonly dispatch: (action: Action) => void;
}
```

**Responsibilities.**

1. Read `theme.colors.accent` and `theme.colors.dim` for the inactive vs active labels.
2. Render the literal label row `WORKFLOWS  RUNS  RUN` with the active label wrapped in a reverse-video pill (`[ RUNS ]`). Choice of inverse vs theme color follows the keybar precedent — the mode pill in `keybar.tsx` uses `<Text inverse>` (mockups.md §1 line 14 / §4 line 157 explicitly bracket the active mode). Per features.md §5.6 rule 8 the pill is "reverse video" — encoded as the structural Ink prop `<Text inverse>`, not a theme literal.
3. Wire `useInput((input, key) => …)` to dispatch:
   - `key.f1` or `input === "1"` → `MODE_SHOW_WORKFLOWS`
   - `key.f2` or `input === "2"` → `MODE_SHOW_RUNS`
   - `key.f3` or `input === "3"`:
     - if `mode.kind === "viewing"` → no-op (already in RUN mode)
     - else if `selectedRunId !== null` → `MODE_OPEN_RUN { runId: selectedRunId }`
     - else → no-op (hide-don't-grey: RUN tab is rendered dim and the key is unhandled). Per features.md §5.6 rule 5 the affordance is hidden, not greyed — see §9 risk 4.
4. Compute the active-tab key from `mode`:
   - `mode.kind === "browsing" && mode.pane === "workflows"` → `"WORKFLOWS"`
   - `mode.kind === "browsing" && mode.pane === "runs"` → `"RUNS"`
   - `mode.kind === "viewing"` → `"RUN"`

**What it does NOT do.**

- Does not draw frame characters; it produces only the centred label row text. The shell embeds the row inside the top frame edge. (The frame-with-tabs composition is a pure function — see §4.)

### 2.3 State pieces / reducer

**No new reducer action.** The four existing mode actions (`MODE_SHOW_WORKFLOWS`, `MODE_SHOW_RUNS`, `MODE_OPEN_RUN`, `MODE_CLOSE_RUN`) cover every transition the tab row dispatches.

A pure helper `keyToMode(input, key, ctx) → Action | null` (see §4) encodes the mapping above; `mode-tabs.tsx`'s `useInput` is a thin wrapper that calls `dispatch(keyToMode(...))` when non-null.

## §3 Rendering algorithm

### 3.1 Width source — prop, not hook (test parity)

Same precedent as the keybar (P3-T4 §3.1). `<AppShell width={N} height={M}>` accepts overrides; production usage in `app.tsx` wraps in `useStdout()` to read live values:

```tsx
const { stdout } = useStdout();
return (
  <AppShell
    width={stdout.columns}
    height={stdout.rows}
    top={<RunsTable/>}
    bottom={<TabbedDetailPane/>}
    modeTabs={<ModeTabs mode={state.mode} selectedRunId={state.selectedRunId} dispatch={dispatch}/>}
    keybar={<Keybar bindings={...} ctx={...} width={stdout.columns}/>}
  />
);
```

Tests always pass explicit `width={140}` / `height={20}` to pin layout deterministically.

### 3.2 Step-by-step render at width=140

1. **Read theme.** `const theme = useTheme();` — throws if no provider (matches keybar contract).
2. **Compose top edge** via pure helper `composeTopRow(width, modeTabsLabel, frameGlyphs)`:
   - Open with `theme.frame.tl` (`╔` or `+`).
   - Append a single `theme.frame.h` (`═` or `-`) as the gap before the mode-tab text.
   - Append the mode-tab labels: `WORKFLOWS  RUNS  [ RUN ]` (active wrapped). The `[ ` and ` ]` characters are part of the rendered pill.
   - Pad with `theme.frame.h` to fill to `width - 1` columns.
   - Close with `theme.frame.tr` (`╗` or `+`).
3. **Render top slot** as a `<Box flexDirection="column" height={topHeight}>` with `theme.frame.v` (`║` or `|`) as left/right borders on each row. Children are passed through and naturally wrap inside the inner content area `(width - 2) × topHeight`.
4. **Compose middle splitter** `╠ + h*(width-2) + ╣` (or ASCII `+ + - + +`).
5. **Render bottom slot** identically to top with `bottomHeight` rows.
6. **Compose bottom edge** `╚ + h*(width-2) + ╝`.
7. **Append keybar** below the frame as a separate `<Box>` row.

Inner content area dims (140 cols, 30 rows total terminal):
- Frame chrome consumes 4 rows (top edge + splitter + bottom edge + keybar row) and 2 columns (left + right vertical bars).
- `topHeight = Math.floor((rows - 4) / 2) = 13`.
- `bottomHeight = (rows - 4) - topHeight = 13`.
- Inner content area per slot: `138 cols × 13 rows`.

### 3.4 Behavior at degraded widths

- **width < 60**: per features.md §5.4 the frame collapses to a breadcrumb-style header. **Out of scope for this task** (handled by P8-T2). The shell still renders something reasonable: it falls back to a single-frame line with no top/bottom split, drawing only the active-mode label without `[ ]` brackets. Defer the polished narrow-width layout to P8-T2; document in §11.
- **width 60–139**: same stacked layout as 140; no column drops in the shell itself. The keybar handles its own three-tier fallback.

## §4 Pure helpers

All in `src/components/app-shell-layout.ts`. **Zero React/Ink/node imports.** Imports `type` from `./types.js`, `../state/types.js`, and `../theme/glyphs.js` (for `FrameGlyphs` type only).

```ts
import type { Action, AppState } from "../state/types.js";
import type { FrameGlyphs } from "../theme/glyphs.js";

/** Symbolic name for one of the three top-level mode tabs. */
export type ModeTabKey = "WORKFLOWS" | "RUNS" | "RUN";

/** Maps the current `AppState.mode` onto the active tab key. */
export function activeTabFromMode(mode: AppState["mode"]): ModeTabKey;

/**
 * Pure key-to-action mapper. Given a single keystroke (Ink's `input`
 * + `key` parameters projected into a flat shape), returns the action
 * to dispatch, or `null` if the key is not bound.
 *
 * Caller is responsible for short-circuiting `useInput` with the
 * returned value. The function deliberately accepts the canonical run
 * id from state so the F3 binding can pre-fill `MODE_OPEN_RUN`.
 */
export interface KeyEvent {
  readonly input: string;
  readonly f1?: boolean;
  readonly f2?: boolean;
  readonly f3?: boolean;
}
export function keyToMode(
  ev: KeyEvent,
  ctx: { readonly mode: AppState["mode"]; readonly selectedRunId: string | null },
): Action | null;

/**
 * Returns the frame title row text given the active tab and the available
 * width. The active tab is wrapped in `[ ... ]`. Inactive tabs render as
 * plain uppercase labels separated by 2 spaces. This is the centerpiece
 * helper called by `composeTopRow`.
 */
export function frameTitle(
  active: ModeTabKey,
  opts?: { readonly hideRun?: boolean },
): string;

/**
 * Visual style descriptor for a mode tab. Returned by `pickActiveTabStyle`
 * and consumed by the React component to set the `inverse`/`bold` props
 * (still no theme literals). The `inverse` field is structural per
 * features.md §5.6 rule 8.
 */
export interface TabStyle {
  readonly inverse: boolean;
  readonly bold: boolean;
}
export function pickActiveTabStyle(
  tab: ModeTabKey,
  active: ModeTabKey,
): TabStyle;

/**
 * Composes the entire top row of the frame given width, frame glyphs,
 * and the title text from `frameTitle`. Returns the full string ready
 * for rendering as a single `<Text>` element. Pure / no React.
 */
export function composeTopRow(
  width: number,
  title: string,
  glyphs: FrameGlyphs,
): string;

/**
 * Computes top-half and bottom-half row counts given total terminal rows.
 * Reserves 4 rows for chrome (top edge + splitter + bottom edge + keybar).
 */
export function pickFrameSlots(rows: number): {
  readonly topRows: number;
  readonly bottomRows: number;
};
```

**Why split out `composeTopRow` etc as pure functions.** Direct snapshot comparison (`expect(composeTopRow(140, ..., UNICODE_FRAME)).toBe("╔ WORKFLOWS  RUNS  [ RUN ] ════…╗")`) gives crisper diffs than a full Ink render. The component test then asserts only the React-level concerns (slot composition, theme-prop wiring).

## §9 Risks / pitfalls

1. **ink-testing-library terminal width.** Re-confirmed: `render()` does not accept a `cols` option; `Instance.stdout.columns` is set by Ink internally. **Mitigation: width as prop**, identical to the keybar precedent. App-shell tests always pass `width={140}`. Production wraps in `useStdout` (see §3.1).
2. **Frame glyph hardcoding.** Easy to slip `═`/`║`/`╔` into the component. **Mitigation: extend `theme/glyphs.ts` with a `FrameGlyphs` table and `theme.frame.*` access.** ASCII fallback (`+ - |`) inherits from existing `MARKFLOW_ASCII=1` capability detection. The pure-helper `composeTopRow(width, title, glyphs)` makes hardcoding impossible by signature.
3. **`useStdout` and SIGWINCH.** Ink v5 hooks `stdout.columns`/`rows` and triggers a re-render on `resize` events. The shell does not handle `resize` itself — it just consumes the (live) prop. No special wiring needed for this task. (Per features.md §6.2 the `SIGWINCH` discipline is engine-level — the keybar/shell read live cols from props.)
4. **Active-pill style.** Per features.md §5.6 rule 8 the pill is "reverse video"; the keybar precedent uses `<Text inverse>` (structural Ink prop, not a theme literal). **Mitigation: use the same idiom.** Bold + accent for non-active labels comes via `theme.colors.accent`/`theme.colors.dim`. **No `inverse={true}` literal sprinkled in component code beyond the pill itself** — the value comes from `pickActiveTabStyle(tab, active).inverse`.
5. **F-key handling in tests.** Ink translates ANSI escape `\x1bOP/Q/R` into `key.f1/f2/f3`. Tests must `await new Promise(r => setImmediate(r))` before writing to allow Ink's raw-mode listener to attach (see `scaffold.test.tsx`).
6. **`useInput` mounted in tests = raw-mode toggle.** `ModeTabs` calls `useInput`, which puts stdin into raw mode under the hood. ink-testing-library handles this in-process; no real TTY is needed. Verified by P3-T4's similar tests.
7. **Hide-don't-grey for RUN tab.** When no run is selected and `mode.kind === "browsing"`, F3 must do nothing AND the RUN label must be omitted (not greyed). The pure helper `frameTitle(active, { hideRun: true })` handles the omission; `keyToMode` returns `null` for the dispatch side. Both paths are tested.
8. **Mode-tabs row text width vs frame width.** The composed `WORKFLOWS  RUNS  [ RUN ]` text is ~26 chars including brackets and 2-space gaps. At width=140 that leaves 114 chars of `═` padding, comfortably fitting. At width <40 the title overflows; the helper truncates with `…` rather than wrapping (defensive — narrow tier is P8-T2's domain).
9. **Inner-content area dims contract.** `topRows`/`bottomRows` are computed from `(rows - 4) / 2`. Children render via Ink's flex; the shell does not clip explicitly — it relies on Ink's `<Box height={N}>` to constrain the slot. If a child overflows, Ink truncates rather than expanding (verified in keybar tests with `<Box>` constrained heights).
10. **`AppShell.width` defaulting.** When neither `width` nor `useStdout` provides a value (test forgot to pass it; `process.stdout.isTTY === false`), default to 80. Documented in the component header.
11. **Reducer dependency direction.** `keyToMode` returns an `Action` from `state/types.ts`. Importing `type Action` is a pure type-only import — no runtime coupling. Verified by purity scan.
12. **`mockups.md §1 line 14`.** The literal text `WORKFLOWS  RUNS  RUN` in §1's top frame edge does not show explicit `[ ]` brackets around the active mode, but §4 line 157 does (`WORKFLOWS  RUNS  [ RUN ]`). Both depict the same visual concept (reverse-video pill); the brackets in §4 are the textual convention used to indicate inverse video in ASCII-only docs. **Default pick:** the implementation always emits brackets around the active mode for unambiguous text rendering. See §11 question 1.

## §11 Open questions / clarifications

None are hard blockers — each has a default answer above — but reviewer sign-off avoids rework.

1. **Active-pill bracketing in §1 vs §4.** mockups.md §1 line 14 reads `WORKFLOWS  RUNS  RUN`; §4 line 157 reads `WORKFLOWS  RUNS  [ RUN ]`. **Default pick:** always render brackets around the active label (matches §4 / §6 / §7 / §11 / §12 — every other place the active mode is shown). The §1 omission is read as a typesetting artefact in a doc that already uses inverse-video for the keybar pill (line 36 keybar implies but doesn't textually mark active state). Confirm.
2. **Frame at width <60.** Spec defers narrow-tier to P8-T2 (single-pane breadcrumb layout). **Default pick:** in this task the shell still renders a degraded but visible frame: single-row top edge, no top/bottom split, no `[ ]` brackets. P8-T2 will replace this with the proper breadcrumb layout. Confirm acceptable as a temporary state.
3. **Mode-tab row as part of the keybar or a separate row?** mockups.md §1 line 14 places the labels INSIDE the top frame edge (between `╔` and `╗`). **Default pick:** treated as part of the frame's top edge composition (`composeTopRow`), not as a separate above-frame row. Matches every mockup (§1, §2, §4, §6, §7, §10, §11, §12). Confirm.
4. **F-key vs digit-key precedence.** Both F1/F2/F3 and `1`/`2`/`3` are wired. The digit fallback is necessary for terminals where F-keys are intercepted by the emulator (features.md §5.6 anti-pattern note). **Default pick:** both bindings always active; if a user types `1` while focus is in a text input, the input swallows it first (Ink semantics — `useInput` only fires when no editable component has focus). Confirm.
5. **Tab labels uppercase.** Mockups consistently render `WORKFLOWS  RUNS  RUN` in uppercase. **Default pick:** literal uppercase strings in `frameTitle`. Confirm.
6. **`AppShell.height` from `useStdout()`.** Ink exposes `stdout.rows` (not `stdout.lines`). **Default pick:** read `rows`, default 30 if unset. Confirm.
7. **Theme frame glyph location.** Recommended path is `theme/glyphs.ts` (centralised). Alternative: a separate `theme/frame.ts` file. **Default pick:** extend `glyphs.ts` per the task brief's recommendation. Confirm.
8. **Mode-tabs receiving `dispatch` directly vs reading from a context.** No `Dispatch` context exists yet; the keybar precedent passes data via props. **Default pick:** prop-injection. A future `AppStateContext` (likely landing with P4-T1) can wrap this without breaking the prop. Confirm.
9. **Should `App` be wired in this task?** Yes — minimal wiring only (placeholder slot content). Real top/bottom content lands in P4–P6. Without this wiring, the shell is dead code at runtime. Confirm.

None require spec changes.
