## §2 Exported names + signatures

### 2.1 `src/components/viewing-pane-tabs-layout.ts`

```ts
/** Stable keys for the four viewing-pane tabs (matches ViewingFocus sum). */
export type ViewingTabKey = "graph" | "detail" | "log" | "events";

/** Three width tiers for the tab header, independent of keybar tier. */
export type ViewingTabTier = "wide" | "medium" | "narrow";

/**
 * Tier thresholds:
 *   width >= 120 → "wide"     (full words:       "[ Graph ]  Detail  Log  Events")
 *   width >= 70  → "medium"   (letter-bracket:   "[G]raph  [D]etail  [L]og  [E]vents")
 *   else         → "narrow"   (single letters:   "G  D  L  E" with active inverted)
 *
 * The 70-col medium/narrow split is chosen so the full-letter-bracket row
 * (29 glyphs incl. separators) fits comfortably inside a <90-col pane after
 * its chrome borders (~4 cols) and the run-header right-alignment (~28 cols
 * for "abcd12 · deploy · build · seq=142"). At widths below 70, even the
 * letter-bracket row begins to compete with that right-side annotation, so
 * we step down to bare letters and omit the right-side annotation.
 */
export function pickViewingTabTier(width: number): ViewingTabTier;

/**
 * Formats a single tab label for a given tier.
 *   wide   + active  → "[ Graph ]"       (wrapped in brackets → inverse pill)
 *   wide   + !active → "Graph"
 *   medium + active  → "[G]raph"         (opening bracket around initial, then the rest)
 *   medium + !active → "[G]raph"         (same text; the active tab receives inverse styling around the whole token — see §3.3)
 *   narrow + active  → "G"               (single letter, inverted)
 *   narrow + !active → "G"
 * The component (not this helper) decides inverse vs. plain styling. This
 * function returns structural text only.
 */
export function formatViewingTabLabel(
  key: ViewingTabKey,
  tier: ViewingTabTier,
): string;

/**
 * Composes the full tab-header row text (no ANSI). The active tab's token
 * is included plainly; the React layer wraps it in `<Text inverse>`.
 * Separator is two spaces at wide/medium, two spaces at narrow.
 * The `suffix` argument is rendered right-aligned at the same row; passed
 * through to the component which decides styling. At narrow tier the
 * suffix is omitted to preserve the letter row.
 */
export function composeViewingTabRow(
  active: ViewingTabKey,
  width: number,
  suffix?: string,
): { readonly tokens: ReadonlyArray<{ readonly text: string; readonly active: boolean }>;
     readonly suffix: string | null;
     readonly tier: ViewingTabTier };
```

## §3 Data flow

### 3.1 How terminal width reaches the derivers

Unchanged from P5-T1 / P6-T1 / P3-T4 / P7-T3. Verified trace:

```
Ink useStdout() → stdout.columns
  → <App> passes to <AppShell width=stdout?.columns>
    → <AppShell> forwards innerWidth to top/bottom slots
      → <RunsTable width=innerWidth>   → pickColumnSet(width)   → COLUMNS_{140|100|80}
      → <StepTable width=innerWidth>   → pickStepColumnSet(w)  → STEP_COLUMNS_{WIDE|MEDIUM|NARROW}
      → <ViewingPanes width=innerWidth> → (NEW) composeViewingTabRow(focus, width)
      → <Keybar ctx={width: stdout?.columns}>  → pickTier(w, catCount) → {full|short|keys}
```

No new plumbing needed. The `<ViewingPanes>` component already receives a width prop via its parent (`packages/markflow-tui/src/app.tsx` passes `innerWidth`).

### 3.2 Tier detection — width matrix summary

| Surface | Threshold → tier |
|---|---|
| Runs columns | `≥120 → wide`, `≥90 → medium (AGE, drop STARTED)`, `<90 → narrow (drop ELAPSED too)` |
| Step columns | `≥120 → wide`, `≥90 → medium (drop ATTEMPT)`, `<90 → narrow (drop ATTEMPT + ELAPSED)` |
| Keybar | `≥100 (with ≤2 categories) → full`, `≥60 → short`, `<60 → keys` |
| Viewing tabs (NEW) | `≥120 → wide ([ Graph ])`, `≥70 → medium ([G]raph)`, `<70 → narrow (G)` |

At `width = 90` the composite is: runs=medium, steps=medium, keybar=short, tabs=medium. Matches §12 verbatim.

### 3.3 Viewing-pane tab component render

1. `<ViewingPanes>` receives `focus` + `width`. It calls `composeViewingTabRow(focus, width)` → `{ tokens, suffix, tier }`.
2. The active token is the one with `active === true`. The component wraps it in `<Text inverse bold>` at every tier (same pill convention as top-frame ModeTabs).
3. At medium tier, the active token is e.g. `"[G]raph"` — the whole 7-char token inverts. This matches mockup §12 line 460 where `[G]raph` is the active tab: reverse-video covers the bracketed-letter + suffix combined, giving the user a single-glance highlight at one letter cost.
4. At narrow tier, token is `"G"`; the `<Text inverse>` covers only the letter; the suffix is omitted.
5. `suffix` (e.g., `abcd12 · deploy · build · seq=142`) renders right-aligned at wide and medium tiers. Dropped at narrow.

### 3.4 Column-drop order for runs at 90 cols (consolidated rule)

**Authority.** mockups.md §12 line 475 verbatim, read against the P5-T1 implementation:

1. Drop `STARTED` absolute-time column (done — `COLUMNS_100` has no `STARTED`).
2. Shorten `ELAPSED` header → `AGE` (done — `COLUMNS_100` substitutes `COL_AGE`).
3. Fold attempt info into `STEP` text (done by the step-row projector — attempts live in `row.step` string like `"build 2/3"`).
4. Truncate `NOTE` with `…` (done — `fitCell` handles growBudget overflow).

Never drop `ID`, `STATUS`, or `STEP`.

### 3.5 Column-drop order for steps at 90 cols

1. Drop the `ATTEMPT` column (done — `STEP_COLUMNS_MEDIUM` omits it).
2. `ELAPSED` is retained as-is (no rename at step tier — mockup §12 shows `12s` / `8s` / `—` under no header; our table keeps an `ELAPSED` column header per mockup §4 precedent).
3. Attempt info surfaces via the existing `NOTE` column (retry text etc.) and via the inline `"attempt 2/3"` that the step-derive writes to `row.note`.

### 3.6 Keybar tier behaviour at 90 cols

- `pickTier(90, catCount)` returns `"short"` for any `catCount`. No category-overflow edge case at 90.
- Fixtures define `shortLabel` + `hideOnTier` to trim content. The matrix test (§4.1) verifies every fixture's 90-col render fits within a 90-character budget.

## §6 Design decisions & rationale

| # | Decision | Rationale |
|---|---|---|
| D1 | **Do not change existing column tier thresholds or drop orders.** | `COLUMNS_100` / `STEP_COLUMNS_MEDIUM` already match mockup §12's drop order when read correctly. Editing derivers when the test gap is the real issue would risk regressing P5-T1 / P6-T1. This task's core value is acceptance-test anchoring at width=90. |
| D2 | **Re-interpret "fold `ATTEMPT` into `STEP`" differently for runs vs steps tables.** | The runs-table has no `ATTEMPT` column at any tier; its "fold" is already materialised in the `r.step` projector (`"build 2/3"`). The step-table has an `ATTEMPT` column only at wide tier; its "fold" is accomplished by dropping the column and surfacing `attempt N/M` through the NOTE column's retry string. Interpreting the mockup literally for each table preserves existing invariants and avoids a new responsibility for the step-row label projector (which would conflict with the indented label pattern). |
| D3 | **Introduce a dedicated `viewing-pane-tabs-layout.ts` helper instead of inlining label logic in `viewing-panes.tsx`.** | The helper is a pure, well-defined function with three tier branches and four keys — 12 discrete cases. Keeping it out of the React file makes the tier logic independently testable (purity probe) and matches the established precedent (`runs/columns.ts`, `steps/columns.ts`, `keybar-layout.ts`). |
| D4 | **Medium tab tier threshold at 70 cols, not 90.** | §12's "~90 col" tier shows letter-bracketed tabs (`[G]raph …`); the narrow mockup §13 at 52 cols doesn't show these tabs at all (it uses a breadcrumb). The crossover is therefore somewhere in the 60-70 range. Picking 70 ensures medium (90-col) renders letter-brackets and the below-70 edge case degrades cleanly to bare letters without wrap. The 70-col floor also aligns with mockups.md §12 line 477 ("at `<70` cols they become single letters `G D L E`"). |
| D5 | **Narrow tier is a tab-row concern only in this task.** | The full narrow mockup (§13) is scoped for a future task (P8 subtask or later). P8-T1 just ensures the tab row degrades gracefully at <70. The breadcrumb / single-pane collapse is out of scope per §4.5. |
| D6 | **Active tab at medium wraps the whole `[G]raph` token in inverse video.** | Alternative: invert only the bracketed letter. Rejected — the inverse-video "pill" convention is consistent at wide tier where the whole `[ Graph ]` token is inverted; downshifting the pill's visual area at medium would be jarring and weaken the "selected"-affordance. The 7-char pill at medium covers one letter of letter-branded content and one word tail; the eye still lands on the letter first. |
| D7 | **No edits to keybar-layout or fixtures.** | `pickTier(90, …)` already returns `short`, and every fixture was authored against mockups §15's `short` column. The work of "switching to short tier" is already done; what's missing is the verification. Adding a matrix test is the right guard rail; editing fixtures would risk breaking existing wide/full-tier tests. |
| D8 | **Matrix test asserts ≤ 90 chars, not exact length.** | Mockup ASCII art has trailing whitespace which a terminal trims. Exact-length assertions are brittle; a ≤ bound with a verbatim short-tier string comparison (after right-trim) is both achievable and diff-stable. |
| D9 | **Top-frame `ModeTabs` are NOT subject to medium compression.** | §12 line 449 shows `WORKFLOWS  RUNS  RUN` full-spelled at 90 cols; only the bottom-pane tabs compress. Compressing the top tabs would change `frameTitle` which is already used by `<AppShell>` and would cascade into the narrow-tier work that isn't this task's scope. An import-boundary assertion (criterion 16) prevents a future drive-by from accidentally applying the compression upwards. |
| D10 | **Suffix (e.g. `abcd12 · deploy · build · seq=142`) survives medium tier but is dropped at narrow.** | Mockup §12 line 460 shows the suffix `abcd12 · deploy · build · seq=142` present at medium. Narrow (<70) needs the full row for the four letters. `composeViewingTabRow` returns `suffix: null` at narrow. |
| D11 | **Implement via test-first anchoring.** | The task is largely verifying existing behaviour. The natural implementation order is: write the tier tests (4.1), then write the component test (4.2), then the integration test (4.3). The sole source change — `viewing-pane-tabs-layout.ts` + `viewing-panes.tsx` wiring — lands last once the test shape is clear. |
| D12 | **No new types file.** `viewing-pane-tabs-layout.ts` declares its own ~2 type aliases inline. | Module size (≤ 40 LoC total) doesn't justify a separate types file. Matches the style of `app-shell-layout.ts` which colocates `ModeTabKey`, `KeyEvent`, `TabStyle` with the helpers. |
| D13 | **Do not extend the keybar registry for a "medium" mode.** | The keybar already decides tier internally via width; no `short`-specific fixture exists (all fixtures declare `shortLabel` / `hideOnTier`). Adding a medium-specific registry entry would fork a surface that was consolidated in P7-T3. |
| D14 | **Runs-table `AGE` column width = 8 (pre-existing).** | At width=90 the growBudget for NOTE is 27 chars which accommodates the longest sample `"exit 1 · exhausted"` (18 chars). Shrinking `AGE` to 6 would push `NOTE` to 29 — a negligible gain. Leave as-is. |
| D15 | **`hideOnTier`/`shortLabel` audit uses the matrix test rather than a fixture-level audit script.** | A runtime assertion across every fixture is self-maintaining — when a new fixture is added, the matrix test automatically covers it. A script-based audit requires maintenance and adds an unrelated target to `package.json`. |
