## 2. New function: `tokensByBatch`

### Signature (to be implemented)

```ts
// packages/markflow/src/core/queries.ts
import type { EngineSnapshot, Token } from "./types.js";

/**
 * Return all child tokens belonging to the given forEach batch, in
 * first-seen (insertion) order. Returns [] (never undefined) when the
 * batch id is unknown or the snapshot has no tokens for it.
 *
 * O(n) over snapshot.tokens; n = total tokens in the run. Callers that
 * render per-frame should memoize on (batchId, tokens.size) if hot.
 */
export function tokensByBatch(
  snapshot: EngineSnapshot,
  batchId: string,
): Token[] {
  const out: Token[] = [];
  for (const token of snapshot.tokens.values()) {
    if (token.batchId === batchId) out.push(token);
  }
  return out;
}
```

### Behavior

- Scans `snapshot.tokens.values()` (confirmed: `snapshot.tokens` is a `Map<string, Token>` per `types.ts` line 381, not an array).
- Filters by `token.batchId === batchId`. Field name confirmed at `types.ts` line 156: `batchId?: string`.
- Unknown `batchId` → returns `[]`, MUST NOT return `undefined`. The function is total.
- Empty snapshot → returns `[]`.
- Preserves first-seen order: `Map#values()` iterates in insertion order, which matches the order tokens were `token:created` during replay. No sorting.
- No mutation of `snapshot`. Pure function.

### File placement — new `src/core/queries.ts` (justified)

**Decision:** create a new file `/Users/boyd/wip/markdown-flow/packages/markflow/src/core/queries.ts`, not append to `graph.ts`.

**Justification:**

1. `graph.ts` is currently a pure **static-topology** module — every function takes `FlowGraph` and returns node IDs / edges / scopes. None of its functions touches `EngineSnapshot`, `Token`, or any runtime state.
2. `tokensByBatch` is a **runtime-state query**: it operates on `EngineSnapshot` (live token map), which is semantically distinct — it's closer to `replay.ts` / snapshot-reading code than to Mermaid-graph code.
3. Putting it in `graph.ts` would mix two concerns and force `graph.ts` to import `Token`/`EngineSnapshot` for one helper, mildly polluting what is currently a tight topology-only module.
4. Future snapshot queries (e.g. `tokensByNode`, `runningTokens`, `tokensForRetry`) would naturally accrete into the same `queries.ts` — creating the file now establishes a home.

**Reviewer override note:** if the reviewer prefers to keep surface area small and put `tokensByBatch` inside `graph.ts` alongside graph helpers, the impact is cosmetic — update the `export { tokensByBatch } from "./queries.js"` line in `index.ts` to `from "./graph.js"`, and adjust the test file path accordingly. The behavior is unchanged.

### Type import rule

`Token` and `EngineSnapshot` are imported from `./types.js` via `import type { ... }` (verbatim — no duplication of type definitions). The function signature uses the types as published.

### 3.1 New test file — `tokensByBatch`

Create `/Users/boyd/wip/markdown-flow/packages/markflow/test/core/queries.test.ts`.

Fixture strategy: **inline-construct `EngineSnapshot` objects**. Do not call `executeWorkflow` or `parseWorkflow`. The existing `graph.test.ts` uses workflow fixtures because `graph.ts` functions operate on the parser output; for `tokensByBatch` we need direct control over `snapshot.tokens`, so inline `Map` construction is cleaner.

Helper (local to the test file):

```ts
function makeSnapshot(tokens: Token[]): EngineSnapshot {
  return {
    tokens: new Map(tokens.map((t) => [t.id, t])),
    retryBudgets: new Map(),
    globalContext: {},
    completedResults: [],
    status: "running",
    batches: new Map(),
  };
}
```

Test cases (all in one `describe("tokensByBatch", () => { ... })`):

1. **Returns children of a known batch id.** Build a snapshot with three tokens where two share `batchId: "B1"` and one has `batchId: "B2"`. Assert `tokensByBatch(snap, "B1").length === 2` and IDs match the two `B1` tokens.
2. **Returns `[]` for an unknown batch id (not `undefined`).** Same snapshot. Assert `tokensByBatch(snap, "does-not-exist")` is an array, `.length === 0`, and explicitly `expect(result).not.toBeUndefined()` (spec-level guard).
3. **Returns `[]` for an empty snapshot.** `makeSnapshot([])`, assert `.length === 0`.
4. **Does not include tokens from a different batch.** Snapshot with `B1` and `B2` tokens plus a token with `batchId: undefined`. Result for `"B1"` must be exactly the `B1` tokens — no `B2`, no un-batched.
5. **Preserves first-seen (insertion) order.** Insert tokens into the `Map` in order `[t3, t1, t2]` (all same `batchId`). Assert the returned array's `.map((t) => t.id)` equals `["t3", "t1", "t2"]`. This locks in the ordering contract: the function does not sort by `itemIndex` or `id`.

All tokens are built inline with the minimal required `Token` shape — `{ id, nodeId, generation, state, batchId? }` — no fake step results needed.

The acceptance criterion names the `"markflow"` package-name import. Inside the workspace this resolves via `workspace:*` linkage, but vitest's module resolver may or may not pick up the `package.json#main`/`exports` field depending on tsconfig paths.
