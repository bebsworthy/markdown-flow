## §2 Module design

### 2.1 `RegistryEntry` — verbatim from features.md §3.1 lines 167–173

> ```json
> [
>   { "source": "./flows/deploy.md",     "addedAt": "2026-04-15T10:22:00Z" },
>   { "source": "../ops-repo/smoke.md",  "addedAt": "..." },
>   { "source": "./workspaces/ingest",   "addedAt": "..." }
> ]
> ```

And from features.md §3.1 line 176:

> `source` is what the user typed (or the workspace path produced by a URL materialisation). Resolution happens at display time.

**Type definition in `src/registry/types.ts`:**

```ts
/**
 * A single registry entry.
 *
 * Shape is authoritative per features.md §3.1 (Persistence).
 *   - source:  the string the user added (verbatim — path, URL, workspace
 *              dir). Resolution (absolute path, URL materialisation, parse
 *              check) happens at display time in P4-T2, NOT here.
 *   - addedAt: ISO-8601 timestamp in UTC. Format matches the spec example
 *              ("2026-04-15T10:22:00Z"); we serialise with `.toISOString()`
 *              at write time and parse with `Date.parse()` at read time.
 */
export interface RegistryEntry {
  readonly source: string;
  readonly addedAt: string; // ISO-8601 UTC, e.g. "2026-04-15T10:22:00Z"
}

/** Current registry slice. Separate from `AppState` (P4-T2 may project). */
export interface RegistryState {
  readonly entries: ReadonlyArray<RegistryEntry>;
}

/** Result of `loadRegistry`. */
export interface LoadResult {
  readonly state: RegistryState;
  /** True when the file was present but malformed (and a `.bak` was made). */
  readonly corruptionDetected: boolean;
  /** Absolute path of the backup written, iff `corruptionDetected`. */
  readonly backupPath: string | null;
}

/** Typed errors surfaced by the registry API. Structural, no Error subclass. */
export type RegistryError =
  | { readonly kind: "io"; readonly cause: unknown; readonly path: string }
  | { readonly kind: "invalid-path"; readonly path: string; readonly reason: string };

/** CLI-flag-derived config, consumed by the (future) React layer. */
export interface RegistryConfig {
  /** Resolved absolute path; `null` means "in-memory only" (`--no-save`). */
  readonly path: string | null;
  /** Mirror of `path !== null` for call-site readability. */
  readonly persist: boolean;
}
```

### 2.2 Public API — `src/registry/store.ts`

```ts
import * as fs from "node:fs/promises";
import * as path from "node:path";
import { writeFileAtomic } from "./atomic-write.js";
import {
  parseRegistryJson,
  serializeRegistry,
} from "./helpers.js";
import type {
  LoadResult,
  RegistryEntry,
  RegistryState,
} from "./types.js";

/**
 * Resolve the registry path given the raw `--list` value (or `null` for the
 * default) and a working directory (typically `process.cwd()`). Returns an
 * absolute path. When `override === null`, returns
 * `path.resolve(cwd, ".markflow-tui.json")`.
 *
 * Pure in the sense that it performs no I/O; it only normalises paths.
 * Kept here (not in helpers.ts) because it imports `node:path`.
 */
export function resolveRegistryPath(
  override: string | null,
  cwd: string,
): string;

/**
 * Load the registry from disk.
 *
 * Behaviour (per features.md §3.1 + plan.md acceptance):
 *   - File missing (ENOENT)      → returns `{ entries: [] }`, no backup.
 *   - File present + valid JSON + shape → returns parsed entries.
 *   - File present + invalid JSON OR wrong shape:
 *        1. Read the raw bytes.
 *        2. Write them to `<registryPath>.bak` via `writeFileAtomic`
 *           (overwrites any previous `.bak`).
 *        3. Return `{ entries: [] }` with `corruptionDetected: true` and
 *           `backupPath` set.
 *        4. Do NOT delete or overwrite the original file here — the next
 *           `saveRegistry` call will naturally replace it. This keeps the
 *           invariant "`load` is read-only on the happy path" and means
 *           that crashing between `load` and the first `save` still leaves
 *           the original corrupt file intact alongside the backup.
 *   - Any other I/O error (EACCES, EISDIR, …) → rethrow as a typed
 *     `RegistryError { kind: "io" }`. Caller chooses whether to prompt.
 *
 * **Ordering justification.** Doing the backup inside `load` (not lazily at
 * `save`) is safer: a user who quits immediately after launch still ends up
 * with `.bak` preserved, and the UI can tell them about the corruption
 * immediately via `corruptionDetected`. The alternative "backup on next
 * save" loses the evidence if the user never saves.
 */
export async function loadRegistry(registryPath: string): Promise<LoadResult>;

/**
 * Save the registry atomically.
 *
 * 1. `serializeRegistry(state)` → pretty-printed JSON string, 2-space indent,
 *    trailing newline.
 * 2. `writeFileAtomic(registryPath, contents)` (see §3).
 *
 * Never mutates `state`. Never retries on failure — caller decides.
 */
export async function saveRegistry(
  registryPath: string,
  state: RegistryState,
): Promise<void>;
```

### 2.3 Pure helpers — `src/registry/helpers.ts`

```ts
import type { RegistryEntry, RegistryState } from "./types.js";

/**
 * Parse raw file bytes into a validated `RegistryState`. Returns `null` if
 * the JSON is syntactically invalid OR if the structural shape fails
 * (must be an array of `{source: string, addedAt: string}` objects). The
 * caller (store.ts) treats `null` as "corrupt — back up + start empty".
 *
 * Wrong-shape inputs caught:
 *   - top-level value is not an array (`{}`, `"foo"`, `null`, `42`)
 *   - any entry is not a plain object
 *   - any entry has `source` that is not a string
 *   - any entry has `addedAt` that is not a string or not a parseable date
 *   - entries with extra fields are tolerated (forward-compat).
 */
export function parseRegistryJson(raw: string): RegistryState | null;

/**
 * Serialise `state.entries` with stable 2-space indentation and a trailing
 * newline. Entry order is preserved (the caller handles sort semantics).
 */
export function serializeRegistry(state: RegistryState): string;

/**
 * Pure — returns a new `RegistryState` with the entry added. De-dupe
 * semantics: if an entry with `isSameSource(existing.source, entry.source)`
 * already exists, the existing entry is **replaced** with the new one
 * (preserves the new `addedAt`; this mirrors "re-adding a workflow should
 * move it to the top of the MRU"). See §10 for reviewer sign-off.
 *
 * The resulting array places the added/replaced entry at **the end** of the
 * list (append on new; for replace, we also move it to the end). This keeps
 * insertion order as the on-disk source of truth; UI sort is applied at
 * display time.
 */
export function addEntry(
  state: RegistryState,
  entry: RegistryEntry,
): RegistryState;

/**
 * Pure — returns a new `RegistryState` with all entries satisfying
 * `predicate` removed. `removeEntry(state, e => e.source === "./foo")`.
 * Removals are hard (entry dropped from the array — NOT marked). No
 * soft-delete; the file is the source of truth.
 */
export function removeEntry(
  state: RegistryState,
  predicate: (e: RegistryEntry) => boolean,
): RegistryState;

/**
 * Case- and whitespace-sensitive source equality. Sources are compared
 * verbatim — we do NOT normalise paths (features.md §3.1: "source is what
 * the user typed"). Two entries with the same resolved absolute path but
 * different spelling (`./foo` vs `foo`) are considered distinct. This is
 * the safe default; a later task can add path-normalisation opt-in if
 * needed. See §10 clarification 1.
 */
export function isSameSource(a: string, b: string): boolean;

/** Utility for UI display — does not mutate. Stable sort by `addedAt` desc. */
export function sortByAddedAt(
  entries: ReadonlyArray<RegistryEntry>,
): ReadonlyArray<RegistryEntry>;

/** Returns true iff `e` is a plain object with valid `source` + `addedAt`. */
export function validateEntry(e: unknown): e is RegistryEntry;
```

**`addEntry` and `removeEntry` are pure** — they take and return `RegistryState`, never touch disk. The React layer will call them inside a reducer or a `useState` updater in P4-T2/T3. `saveRegistry` is the only thing that writes.

### 2.4 File format — verbatim from features.md §3.1

> File: `./.markflow-tui.json` in the directory `markflow-tui` was launched from. Plain JSON, checkable into the repo for a shared curated list. No global user-level registry (deliberately out of scope for now).
>
> ```json
> [
>   { "source": "./flows/deploy.md",     "addedAt": "2026-04-15T10:22:00Z" },
>   { "source": "../ops-repo/smoke.md",  "addedAt": "..." },
>   { "source": "./workspaces/ingest",   "addedAt": "..." }
> ]
> ```
>
> - Auto-saved on every add/remove.
> - `source` is what the user typed (or the workspace path produced by a URL materialisation). Resolution happens at display time.

**Implementation-level pins:**

- **Indentation:** 2 spaces, via `JSON.stringify(entries, null, 2)`.
- **Trailing newline:** yes (POSIX convention; diff-friendly).
- **Encoding:** UTF-8 (Node default).
- **Filename:** `.markflow-tui.json` relative to the launch directory (i.e. `process.cwd()` when `markflow-tui` is invoked). `--list <path>` overrides the filename to **exactly** the given path, not joined with cwd if absolute.
- **No BOM.** `fs.writeFile` with a string defaults to none; assertions pin this.

### 2.5 Backup behaviour — decision

Per §2.2 `loadRegistry` spec: **backup-during-load**, backup written next to the registry file.

- **Location:** `<registryPath>.bak` — same directory as the registry, filename with `.bak` appended. Concretely, if `registryPath === "/abs/.markflow-tui.json"`, backup goes to `/abs/.markflow-tui.json.bak`. If `--list /custom/path.json`, backup goes to `/custom/path.json.bak`. This keeps the backup co-located with the failing file regardless of the override — simplest for users debugging.
- **Fresh empty file: NOT initialised by `load`.** `load` returns the empty state in-memory but does not write the empty file to disk. The first `saveRegistry` call (P4-T2/T3 after the first add/remove) atomically replaces the corrupt original. Until then, the corrupt file sits alongside `.bak`. This preserves evidence and avoids a surprising write during what the user perceives as a read-only "launch".

Reviewer sign-off on backup location requested in §12 question 4.

## §3 Atomic write algorithm

Implemented in `src/registry/atomic-write.ts`.

```ts
import * as fs from "node:fs/promises";
import * as path from "node:path";
import { randomBytes } from "node:crypto";

/**
 * Atomically write `contents` to `targetPath`.
 *
 * Algorithm (POSIX-first):
 *
 *   1. Compute a temp path in the *same directory* as `targetPath`:
 *        `<dir>/.<basename>.tmp-<pid>-<hex>`
 *      Same-directory is required so `fs.rename` is atomic (same filesystem).
 *      The leading dot hides the temp from `ls` and helps tools that skip
 *      dotfiles. `randomBytes(6).toString("hex")` gives 2^48 temp collisions
 *      per pid — more than enough to avoid clashes under concurrent saves.
 *
 *   2. `const fh = await fs.open(tempPath, "wx", 0o644);`
 *      `"wx"` fails if the temp somehow already exists, which guarantees we
 *      never truncate another writer's temp file by accident.
 *
 *   3. `await fh.writeFile(contents, "utf8");`
 *
 *   4. `await fh.sync();`  // fsync the file's data + metadata
 *
 *   5. `await fh.close();`
 *
 *   6. `await fs.rename(tempPath, targetPath);` // POSIX atomic within FS
 *
 *   7. (OPTIONAL, behind a future flag) `fsync` the containing directory
 *      for true durability against power loss. Current tradeoff: we skip
 *      this to keep the write fast (~1ms). If the host crashes between
 *      rename and dir-fsync, the rename may be lost even though the inode
 *      persists. Acceptable for a registry file — a dropped save here
 *      simply means the user re-adds the workflow. Re-evaluate if
 *      users report lost entries.
 *
 * Error handling:
 *   - Any step 1–5 failure: best-effort `fs.unlink(tempPath)` then rethrow
 *     the original error. The target file is untouched.
 *   - Step 6 failure: target file is untouched (rename is atomic). Temp
 *     file may leak — best-effort unlink before rethrowing.
 *
 * Platform notes:
 *   - **POSIX (macOS, Linux):** `fs.rename` replaces any existing target
 *     atomically. This is the supported platform.
 *   - **Windows:** `fs.rename` may fail with `EPERM` / `EACCES` if the
 *     destination file is held open by another process, and `EXDEV` is
 *     not an issue within a single directory. We do not special-case
 *     Windows in this task — the TUI target platforms per `plan.md` §8
 *     (Phase 9 E2E) are macOS and Linux. Track as §12 open question 2
 *     for future hardening; in practice a Windows-only quirk that loses a
 *     single save is strictly less bad than corruption.
 *
 * Concurrent writes:
 *   - Two simultaneous `writeFileAtomic(path, A)` and `writeFileAtomic(path, B)`
 *     calls pick distinct temp paths (randomised), each completes its temp
 *     write, and then both race to rename. POSIX guarantees the rename is
 *     atomic, so the final target contains **one of {A, B}** byte-exact,
 *     never a mix. Last-writer-wins.
 *   - We deliberately do NOT add advisory locking. Rationale: the TUI is
 *     a single-process interactive program. Concurrent writes only occur
 *     when two `markflow-tui` instances run in the same cwd, which is
 *     already an operational footgun (the later-rendered UI will simply
 *     reflect its own copy). Future task may add `proper-lockfile` (same
 *     dep as `runs/<id>/.lock` from P1-T5) if users report issues.
 */
export async function writeFileAtomic(
  targetPath: string,
  contents: string,
): Promise<void>;
```

## §4 Pure helpers

All pure logic lives in `src/registry/helpers.ts` (per §2.3) plus `src/registry/types.ts` (types only). Both are scanned by the purity probe.

**Zero-import contract:** `helpers.ts` imports ONLY `type` from `./types.js`. No `fs`, no `path`, no `os`, no `crypto`, no `react`, no `ink`, no `node:*` runtime imports. `types.ts` imports nothing.

**Design rationale.** Splitting parse / serialise / add / remove / dedupe / sort into `helpers.ts` means every non-trivial decision in the registry logic is reachable from tests that need no fs fixture — dramatically reduces flake surface and keeps test runtime under 10ms per case. The test file `test/registry/helpers.test.ts` contains ~40 `it` blocks; each runs in sub-millisecond and uses only JS data structures.

### 5.2 Parser — `parseRegistryFlags(argv: string[])`

Exported from `src/cli.tsx` (or split into a small `src/cli/args.ts` if reviewer prefers — either is fine; default is to keep it in `cli.tsx` alongside the existing render code for discoverability, with an `export` so `cli-flags.test.ts` can import it). The function is **pure** in the narrow sense that it consumes an argv array and returns `{ config: RegistryConfig; rest: string[] }`; it does not read files or `process.cwd()` — cwd is applied later by the consumer via `resolveRegistryPath`.

```ts
export function parseRegistryFlags(
  argv: ReadonlyArray<string>,
): {
  readonly config: {
    readonly listPath: string | null;
    readonly persist: boolean;
  };
  readonly rest: ReadonlyArray<string>;
};
```

Behaviour:

| Argv | `listPath` | `persist` | `rest` |
|---|---|---|---|
| `[]` | `null` | `true` | `[]` |
| `["--no-save"]` | `null` | `false` | `[]` |
| `["--list", "/x.json"]` | `"/x.json"` | `true` | `[]` |
| `["--list=/x.json"]` | `"/x.json"` | `true` | `[]` |
| `["--list", "/x.json", "--no-save"]` | `"/x.json"` | `false` | `[]` |
| `["foo.md"]` | `null` | `true` | `["foo.md"]` |
| `["--list", "/x.json", "foo.md", "bar.md"]` | `"/x.json"` | `true` | `["foo.md", "bar.md"]` |
| `["--list"]` (no value) | `null` | `true` | `[]` (emit a typed error in §12 clarification 5) |

**Defaults explicitly:**
- `--no-save` absent → persist = true, path = default `"./.markflow-tui.json"` (resolved to absolute via `resolveRegistryPath` at call site).
- `--list <p>` absent → path = default.
- Both present → `persist: false`, `path: null` (spec: `--no-save` wins). This is a deliberate choice: when the user explicitly disables saving, they never want to touch the file on disk, so the `--list` value becomes purely informational. Tested.
- Unknown flags (e.g. `--foo`) are treated as positional (pushed into `rest`). Downstream code can reject them if desired; this task doesn't own arg validation for file paths.

## §10 Risks / pitfalls

1. **Per-test tmpdir discipline (BLOCKING).** Every test that touches disk MUST create a fresh tmpdir under `os.tmpdir()` in `beforeEach` and `rm -rf` it in `afterEach`. Use `mkdtempSync(join(tmpdir(), "markflow-tui-"))`. **Never** write `.markflow-tui.json` to the repo root or cwd — a test that does will leak state across CI runs and poison future test runs. Pattern:
   ```ts
   let dir: string;
   beforeEach(() => { dir = mkdtempSync(join(tmpdir(), "markflow-tui-")); });
   afterEach(() => { rmSync(dir, { recursive: true, force: true }); });
   ```
2. **`fs.rename` atomicity requires same-filesystem.** `os.tmpdir()` on macOS returns a path on the same filesystem as the user's home dir by default, so `writeFileAtomic` inside a tmpdir test is atomic. If `TMPDIR` is set to a separate mount, the test still works because our temp-file and target live in the same tmpdir — we always write the temp in the *target's directory*, not a system-wide tmp. This is why step 1 of §3 enforces same-directory temps.
3. **JSON.parse error messages differ across Node versions.** Do NOT assert on the error string; assert on `corruptionDetected: boolean`. Tests that need to trigger malformed input use `writeFileSync(path, "{not json")` or `writeFileSync(path, '{"source":"x"}')` (wrong shape). Node 20 vs 22 emit different `SyntaxError.message` prefixes.
4. **Do NOT depend on engine internals.** This task writes its own atomic-write helper rather than importing anything from `packages/markflow/`. The engine's `event-logger.ts` is append-only (no rename-based replace), and `plan.md` §"CLI purity" forbids new runtime deps in `packages/markflow/`. Verify via `grep -r "from \"markflow\"" packages/markflow-tui/src/registry/` → zero matches expected.
5. **`source` semantics — "what the user typed".** Per features.md §3.1 line 176: "source is what the user typed (or the workspace path produced by a URL materialisation)". Interpretations:
   - Absolute path: as-typed (e.g. `"/Users/boyd/flows/deploy.md"`).
   - Relative path: as-typed (e.g. `"./flows/deploy.md"`).
   - URL: as-typed until materialisation; P4-T3 replaces it with the workspace path (this task does not do the swap — we just store the string).
   - We do NOT normalise, resolve, or canonicalise the path at write time. De-dupe is byte-identity via `isSameSource`. See §12 Q1 for the reviewer sign-off: a stricter dedupe that normalises paths before comparison would catch `./foo` vs `foo` duplicates but would also silently modify user input, which violates the "as typed" spec. Default picked: byte-identity.
6. **`addedAt` format.** Default: ISO-8601 UTC string via `new Date().toISOString()` — matches the spec example `"2026-04-15T10:22:00Z"` byte-for-byte (Node emits the same format). Alternative considered: epoch milliseconds. Rejected — spec is unambiguous. See §12 Q2.
7. **Concurrent-write test flake.** Vitest's `restoreMocks: true` + the real-fs concurrent test can run in parallel within a file; do not share `dir` across two concurrent test cases. Each `it` gets a fresh tmpdir via `beforeEach`. This is standard vitest behaviour (sequential within a file by default).
8. **ESM + `.js` suffix imports.** The package is ESM with `moduleResolution: "bundler"` (per `tsconfig.base.json`). All relative imports use explicit `.js` suffixes (e.g. `import { parseRegistryJson } from "./helpers.js"`). Verified precedent in every P3 plan + component.
9. **Strict TS, no `any`.** `parseRegistryJson` returns `RegistryState | null` (not `any`). Validation uses type guards (`validateEntry: (e: unknown) => e is RegistryEntry`). The I/O error path returns a discriminated union (`RegistryError`), not `Error`.
10. **Backup chain risk.** If the user's `.markflow-tui.json` becomes corrupt twice in succession, the second `.bak` overwrites the first — we do NOT keep rotating backups. Tested explicitly (§6.3). Rationale: a single most-recent backup is the sufficient recovery point; rotation invites unbounded disk use.
11. **Windows rename-while-open.** Noted in §3. This task does NOT handle Windows; if the verifier runs on Windows, skip the atomic-write tests via `it.skipIf(process.platform === "win32")` (per the plan.md "No skipping tests" rule — wait, re-reading: the rule forbids `.skip` / `xit` / commented-out assertions. `it.skipIf` is a conditional execution primitive, not a skipped test; verify with reviewer. Default: keep tests unconditional; if CI on Windows ever fires, we'll add the conditional. Currently CI is Linux/macOS per `plan.md` §P2-T3.)
12. **Flag parser — `--list` with no value.** `parseRegistryFlags(["--list"])` is malformed. Default: throw a typed `Error` with a clear message (`"--list requires a path argument"`). Tested. Alternative: silently treat as no `--list`; rejected — silent failures on launch are unfriendly. See §12 Q5.

## §12 Open questions / clarifications

Each has a default pick; none are hard blockers. Reviewer sign-off avoids rework.

1. **Path normalisation for `source`.** features.md §3.1 says "source is what the user typed". **Default pick:** verbatim / byte-identity. A future P4-T3 polish task could opt-in to `path.resolve()`-based dedupe if users report duplicate entries from `./foo` vs `foo`. Confirm.
2. **`addedAt` format.** Spec example `"2026-04-15T10:22:00Z"` is ISO-8601 UTC. **Default pick:** `new Date().toISOString()` at write time; `Date.parse()` tolerant parse at read time (as long as it returns a finite number). Alternative (rejected): epoch milliseconds. Confirm.
3. **`removeEntry` soft vs hard.** No tombstones; entry vanishes from the array. **Default pick:** hard delete. Reviewer may prefer soft (mark-and-filter) to preserve `addedAt` history; rejected here because the file is the source of truth and the UI shows "last run" from the engine, not the registry. Confirm.
4. **`.bak` location.** Co-located with the registry file: `<registryPath>.bak`. **Default pick:** yes, same directory. Alternative: under `os.tmpdir()` — rejected because the user wants to recover the file from the project, not from a system-tmp they'll never look at. Confirm.
5. **`--list` with no following value.** **Default pick:** throw a typed `Error` with message `"--list requires a path argument"`; propagate so that `waitUntilExit().catch` emits to stderr with exit code 1. Alternative: silently ignore. Rejected — silent is unfriendly. Confirm.
6. **CLI flag consumption scope.** **Default pick:** parse + propagate here; consume in P4-T2/T3. Reviewer could require wiring `saveRegistry` inside `<App/>` now behind a `registryConfig.persist` guard; rejected because there is no registry UI in this task to generate add/remove actions. Confirm.
7. **`addEntry` dedupe — replace vs no-op.** **Default pick:** replace + move-to-end (MRU semantics). Alternative: no-op (keep the original `addedAt`). Rejected because users expect "re-adding bubbles up recency" — the spec is silent; the mockups show no explicit ordering for duplicates. Confirm.
8. **`serializeRegistry` — compact vs pretty.** **Default pick:** pretty (`JSON.stringify(x, null, 2)` + trailing `\n`). The spec example in features.md §3.1 is pretty-printed; diff-friendly; users may hand-edit. Confirm.
9. **`parseRegistryFlags` home.** **Default pick:** export it from `src/cli.tsx` (current cli file is tiny). Alternative: split to `src/cli-args.ts`. Either is fine; `cli.tsx` picked for discoverability. Confirm if reviewer prefers the split.
10. **Empty / zero-byte file treatment.** An empty file IS syntactically invalid JSON. **Default pick:** treat as malformed → write `.bak` → return empty state. Alternative: treat as empty list silently. Rejected — the file's presence with zero bytes is a signal the user touched it; evidence-preserving backup is safer. Confirm.

None of these require a spec change in `features.md` — they're all clarifications within the latitude the spec already gives.
