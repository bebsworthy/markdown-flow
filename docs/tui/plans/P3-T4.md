### Rules for markflow's keybar ‚Äî verbatim from features.md ¬ß5.6 lines 436‚Äì450

> 1. **Always visible**: `?` help, `q` quit (top level) or `Esc` back (inside detail).
> 2. **Category headers** in muted bold: max two categories per line (e.g., `RUN` + `VIEW`).
> 3. **Ordering within a line**: local verbs (open / approve / cancel) ‚Üí view toggles (follow, wrap, filter) ‚Üí globals (help, quit).
> 4. **Destructive actions in red** (`X Cancel`, `D Deny`). Everything else uses the neutral theme.
> 5. **Hide, don't grey.** Omit hints whose `when(ctx)` returns false.
> 6. **Toggle label flips** ‚Äî `Follow` ‚Üî `Unfollow`, `Wrap` ‚Üî `NoWrap`.
> 7. **Width tiers**:
>    - ‚â•100 cols: full labels.
>    - 60‚Äì100 cols: `shortLabel` if defined, else truncate label with `‚Ä¶`.
>    - <60 cols: keys only + a one-line tip "`?` for labels".
> 8. **Mode pill** (reverse video) when inside a modal overlay: `[APPROVAL]`, `[RESUME]`.
> 9. **Single keymap array.** `defineBinding({ keys, label, shortLabel, category, when, action })` feeds both the handler (`useInput`) and the renderer (`<KeyBar>`).
> 10. **Theme-driven colors** via a `useTheme()` hook; no literals in the component.

### Spec ambiguities surfaced (resolutions in ¬ß11)

1. **Rule 7 (60‚Äì100 tier) wording** ‚Äî "**`shortLabel` if defined, else truncate label with `‚Ä¶`**". The ¬ß15 matrix shows that in practice at 60‚Äì100 cols almost every row drops to *keys only with occasional single-letter label heads* (`‚Üë‚Üì ‚èé r e`, `‚Üë‚Üì ‚èé a R X`, etc.). This is consistent with "no `shortLabel` defined ‚Üí truncate to zero label, keep key". **Default pick:** render `shortLabel` when defined; when absent, render key-only (label elided entirely ‚Äî which is a form of truncation to empty + no `‚Ä¶` because nothing ambiguous remains). Reviewer sign-off requested in ¬ß11.
2. **Rule 7 (<60 tier) right-side hint** ‚Äî ¬ß5.6 line 447 says "keys only + a one-line tip `?` for labels"; ¬ß5.6 line 486 ASCII mock shows `press ? for labels` right-aligned; mockups.md ¬ß15 row footnote says the tip is "omitted here for density". **Default pick:** the hint is rendered by `<Keybar>` as part of the frame, right-aligned, and ¬ß15's keys-only strings (e.g. `‚Üë‚Üì ‚èé r e ?`) are the **left-aligned** portion; the right-aligned tip appears only when terminal width leaves room for it.
3. **Categories beyond two** ‚Äî rule 2 caps at two categories per line but ¬ß5.6 does not say what to do when three appear. **Default pick:** if more than two categories are present in the filtered binding list, the keybar drops to the `shortLabel` tier early (as if width were 60‚Äì100 even if ‚â•100) to reclaim space. Alternative considered: wrap to two rows ‚Äî rejected because the keybar is defined as a single-line element in features.md ¬ß5.6 ("the bottom status/command line").
4. **Arrow-group formatting** ‚Äî features.md line 415 says "`<‚Üê‚Üì‚Üë‚Üí> Change focus` as one hint, not four"; line 416 says "Angle-bracket key format". Yet mockups.md ¬ß15 shows `‚Üë‚Üì` unbracketed. **Default pick:** when a single `Binding.keys` array contains only arrow-key names (subset of `["Left","Down","Up","Right"]`), `formatKeys` emits the compact grouped form `‚Üë‚Üì` / `‚Üê‚Üí` / `‚Üë‚Üì‚Üê‚Üí` **without angle brackets**; the angle-bracket style is reserved for printable letters and named keys (`<r>`, `<Enter>`, `Ctrl + <r>`). Matches mockups.md ¬ß15.
5. **ASCII-only arrow rendering** ‚Äî under `unicode=false`, `‚Üë‚Üì` becomes `^v` and `‚Üê‚Üí` becomes `<-><-` (ugly). **Default pick:** the keybar renders arrows as unicode always. The ASCII fallback guaranteed by P3-T3 is for *status glyphs*; arrow direction cues in the keybar have no ASCII substitute that preserves meaning. This is a deliberate scope boundary; revisit when a user reports a garbled keybar. See ¬ß11.
6. **Snapshot format** ‚Äî `ink-testing-library`'s `lastFrame()` returns a string that typically includes ANSI escape codes when color is on. **Default pick:** all snapshot tests run under `buildTheme({color: false, unicode: true})` so `lastFrame()` is plain text; color rules (R4, R10) are exercised by separate non-snapshot assertions that inspect the `<Text>` element props via a custom render (see ¬ß5.4).

## 2. `Binding` + `AppContext` type design

**Source quote** (features.md lines 452‚Äì465) reproduced in ¬ß0.

### 2.1 `Binding` ‚Äî pins per spec quirk

```ts
// src/components/types.ts
import type { AppState } from "../state/types.js";

/**
 * A key descriptor. Matches the spec examples verbatim (features.md line 456):
 *   ["f"]                         ‚Äî single letter
 *   ["Ctrl", "r"]                 ‚Äî modifier + letter (last element is the key)
 *   ["Left","Down","Up","Right"]  ‚Äî grouped directional family
 *
 * The array is interpreted by `formatKeys()` in keybar-layout.ts. We do NOT
 * constrain the element type beyond `string` here ‚Äî a union type (e.g.
 * `"Ctrl" | "Shift" | "Enter" | ‚Ä¶`) would be nicer but over-tightening it
 * before the `useInput` wiring (P4-T1) risks churn when we discover a key
 * the union missed. `formatKeys` treats unknown tokens as literal text.
 */
export type KeySpec = ReadonlyArray<string>;

/**
 * Category label ‚Äî free-form per features.md ¬ß5.6 examples (`RUN`, `VIEW`,
 * `LOGS`). Kept as `string` because the spec shows no fixed closed set;
 * tightening to a union would require updating every feature task that
 * introduces a new category. Convention: uppercase ASCII, ‚â§6 chars.
 */
export type Category = string;

export interface Binding {
  readonly keys: KeySpec;
  readonly label: string;
  readonly shortLabel?: string;
  /**
   * `state` is typed as `unknown` (matches features.md line 459). Each
   * binding that uses `toggleLabel` knows its own state shape ‚Äî typically
   * a boolean, but the spec deliberately leaves it open. The binding
   * author is responsible for narrowing.
   *
   * Declared `toggleState` on `AppContext` (¬ß2.2) is the canonical source
   * the renderer passes. See ¬ß3 step 2.
   */
  readonly toggleLabel?: (state: unknown) => string;
  readonly category?: Category;
  readonly destructive?: boolean;
  readonly when: (ctx: AppContext) => boolean;
  /**
   * Stored for future `useInput` wiring (P4-T1). The renderer never
   * invokes `action`; we keep the field on the type so a single keymap
   * array (rule 9) drives both the visible hint and the handler.
   */
  readonly action: (ctx: AppContext) => void | Promise<void>;
}
```

**Pinned decisions:**
- `keys` elements are literal modifier / key names: `"Ctrl"`, `"Shift"`, `"Alt"`, `"Meta"`, `"Enter"`, `"Esc"`, `"Space"`, `"Tab"`, `"Left"`, `"Right"`, `"Up"`, `"Down"`, plus single-character strings for printable keys. `formatKeys` knows a small substitution table; anything else is rendered literally.
- `category` is `string` ‚Äî convention, not type-enforced.
- `toggleLabel` signature is left at `(state: unknown) => string` per the spec.
- `action` is declared but NEVER called by `<Keybar>`; this is the single-keymap-array contract (rule 9). The implementer must add a comment at the call site in P4-T1 that reads `action` from the same binding list.

### 2.2 `AppContext` ‚Äî minimal subset the keybar actually reads

```ts
// src/components/types.ts (continued)

/**
 * Read-only view the keybar passes to every `when(ctx)` and (later) every
 * `action(ctx)`. Deliberately NOT a full `AppState` ‚Äî the keybar's purity
 * boundary is narrower: it should be swappable with a different source of
 * mode/overlay info without touching the component.
 *
 * Fields are additive. Any new predicate that needs a new field must add
 * it here, not reach into `AppState` ad-hoc.
 */
export interface AppContext {
  /** Mirrors `AppState.mode` verbatim. See state/types.ts. */
  readonly mode: AppState["mode"];
  /** Mirrors `AppState.overlay`. `null` means "no modal". */
  readonly overlay: AppState["overlay"];
  /** True while a run has an approval waiting. Derived upstream by the
   *  engine adapter (P3-T2). */
  readonly approvalsPending: boolean;
  /** True while the log panel is in tail-follow mode. */
  readonly isFollowing: boolean;
  /** True while the log panel wraps long lines. */
  readonly isWrapped: boolean;
  /**
   * Generic payload threaded to `toggleLabel`. For the log `f` binding
   * this is `isFollowing`; for the `w` binding it is `isWrapped`. The
   * binding author picks which field to narrow, so the type is `unknown`.
   */
  readonly toggleState: Readonly<Record<string, unknown>>;
}
```

**Design notes.** `AppContext` is not `AppState`; it's a projection. An adapter (later task) builds it from `AppState` + derived engine data. The keybar never imports from `packages/markflow` ‚Äî it only reads the projected context.

## 3. Rendering algorithm (`keybar.tsx`)

The component's public API:

```tsx
// src/components/keybar.tsx
import React from "react";
import { Box, Text } from "ink";
import { useTheme } from "../theme/context.js";
import type { Binding, AppContext } from "./types.js";
import {
  pickTier, filterBindings, groupByCategory,
  formatKeys, renderableLabel,
} from "./keybar-layout.js";

export interface KeybarProps {
  readonly bindings: ReadonlyArray<Binding>;
  readonly ctx: AppContext;
  /**
   * Terminal width in columns. Tests pass an explicit value to pin a
   * tier. Integration (P3-T5) wires `useStdout().stdout.columns`.
   */
  readonly width: number;
  /**
   * Optional mode pill shown with reverse video (rule 8). Typical values:
   * `"APPROVAL"`, `"RESUME"`, `"COMMAND"`, `"FIND"`, `"HELP"`. The pill
   * is rendered without surrounding square brackets by the caller; the
   * keybar wraps it in `[ ... ]` itself.
   */
  readonly modePill?: string;
}

export function Keybar({ bindings, ctx, width, modePill }: KeybarProps): React.ReactElement { /* ‚Ä¶ */ }
```

### 3.1 Width source ‚Äî prop, not hook

`ink-testing-library@4`'s `render(tree)` signature does not accept a `cols` option (verified in `node_modules/ink-testing-library/build/index.d.ts`). `useStdout().stdout.columns` is not test-controllable. Therefore:

- **`<Keybar width={N}>` takes width as a prop.**
- The future app-shell component (P3-T5) wraps it: `const {stdout} = useStdout(); return <Keybar width={stdout.columns} ‚Ä¶/>;`.
- Tests always pass an explicit width (`100`, `80`, `40`).

### 3.2 Step-by-step render

1. **Read theme.** `const theme = useTheme();` ‚Äî throws if no provider (expected).
2. **Resolve toggle labels + filter.** For each binding, compute a resolved `Binding` with `label` replaced by `toggleLabel(ctx.toggleState)` if `toggleLabel` is set. Then filter via `when(ctx)` (rule 5 ‚Äî hide, don't grey).
3. **Sort by ordering class (rule 3).** Classify each binding:
   - **local** (no `category` OR category is `"RUN"` or other mode-specific verb: `"RUN"`, `"LOGS"`, `"APPROVAL"`, `"RESUME"`, `"COMMAND"`, `"FIND"`) ‚Äî 0
   - **toggle** (category is `"VIEW"`) ‚Äî 1
   - **global** (key is exactly `["?"]` or `["q"]` or `["Esc"]`) ‚Äî 2
   Sort stable by this class; within a class keep insertion order. Convention documented in fixtures.
4. **Count categories.** If `>2` distinct categories survive filtering (rule 2), force tier down to `"short"` or `"keys"` as needed (see step 5).
5. **Pick tier** via `pickTier(width, measuredFullWidth, categoryCount)`:
   - `width >= 100 && categoryCount <= 2` ‚Üí `"full"`.
   - `width >= 60` (inclusive; covers 60..99) ‚Üí `"short"`.
   - `width < 60` ‚Üí `"keys"`.
   Boundary justification: "‚â•100" and "60‚Äì100" and "<60" from ¬ß5.6 rule 7. The 60‚Äì100 range is read as **closed at 60, open at 100** (the 100-cutoff belongs to "full"). Documented in code.
6. **Format per binding (`renderableLabel`).**
   - **full** tier: `formatKeys(b.keys) + " " + b.label` (after toggleLabel resolution).
   - **short** tier: if `b.shortLabel` is defined, render `formatKeys(b.keys) + " " + b.shortLabel`; else render just `formatKeys(b.keys)` (label elided ‚Äî see ¬ß0 ambiguity 1). The ¬ß15 matrix confirms this (`‚Üë‚Üì ‚èé r e` has no labels).
   - **keys** tier: render just `formatKeys(b.keys)`.
7. **Format keys via `formatKeys(keys: ReadonlyArray<string>): string`.** Pure helper with the following rules:
   - **Single printable key** (`["r"]`) ‚Üí `"r"` (unbracketed in the matrix; angle brackets only when features.md ¬ß5.6 shows them in the "wide" mock ‚Äî which for `r` it does not). Check mockups.md ¬ß15 row WORKFLOWS: `r Run` ‚Äî unbracketed `r`. **Default pick: unbracketed single-key.** Angle brackets only appear when multiple tokens are joined, per the zellij precedent reference lines 413‚Äì416.
   - **Modifier + key** (`["Ctrl","r"]`) ‚Üí `"Ctrl + <r>"` (per features.md line 416 explicit example).
   - **Arrow family** (subset of `["Left","Down","Up","Right"]`, in any order):
     - `["Up","Down"]` or `["Down","Up"]` ‚Üí `"‚Üë‚Üì"`
     - `["Left","Right"]` or `["Right","Left"]` ‚Üí `"‚Üê‚Üí"`
     - 3 or 4 arrow members ‚Üí `"‚Üë‚Üì‚Üê‚Üí"` (order fixed in helper: up, down, left, right)
   - **Named keys**: `"Enter"` ‚Üí `"‚èé"`, `"Esc"` ‚Üí `"Esc"`, `"Space"` ‚Üí `"Space"`, `"Tab"` ‚Üí `"Tab"`. (Mockups.md ¬ß15 shows `‚èé` for Enter, and the others literal.)
   - **Unknown token**: pass through literally.
8. **Group by category (rule 2).** Walk the sorted bindings; emit category headers (`"RUN:"`, `"VIEW:"`) once per category transition, in muted bold via `theme.colors.dim` + `<Text bold dimColor>`.
9. **Color assignment (rule 10, all via theme):**
   - **Key token** (`formatKeys(...)` output): `theme.colors.accent` with `bold`.
   - **Label**: no color / no dim ‚Äî default text.
   - **Category header**: `theme.colors.dim` + `bold`.
   - **Destructive binding** (`b.destructive === true`): entire `formatKeys + " " + label` rendered in `theme.colors.danger`. The accent-on-key rule is suppressed for destructive bindings ‚Äî they render uniformly red.
   - **Mode pill**: `<Text inverse>` wrapped text `[APPROVAL]` etc. Inverse is structural (Ink prop), not a theme color ‚Äî reasonable because rule 8 says "reverse video" explicitly.
10. **Layout.** Single `<Box flexDirection="row">`. Categories and bindings flow left-to-right. The `<60` keys tier additionally appends a right-aligned hint `"press ? for labels"` using `<Box marginLeft="auto">` + `<Text dimColor>`. In `>=60` tiers the hint is omitted.
11. **Memoization.** The component wraps its body in `React.memo(Keybar)` keyed on prop shallow-equality. Additionally, the expensive `filterBindings ‚Üí pickTier ‚Üí ‚Ä¶` chain is wrapped in `useMemo` keyed on `[bindings, ctx, width, modePill]`.

### 3.3 Forbidden in `keybar.tsx`

- No hardcoded color strings (`color="red"`). Every color flows through `theme.colors[role]`. (Rule 10.)
- No import from `packages/markflow` (the engine). The keybar is pure presentational.

## 4. Pure helpers module ‚Äî `src/components/keybar-layout.ts`

Deliberately split so layout logic is (a) unit-testable without rendering and (b) purity-scannable. Exported API:

```ts
// src/components/keybar-layout.ts
import type { Binding, AppContext } from "./types.js";

export type Tier = "full" | "short" | "keys";

/** Applies `toggleLabel(ctx.toggleState)` where set, then filters by `when(ctx)`. */
export function filterBindings(
  bindings: ReadonlyArray<Binding>,
  ctx: AppContext,
): ReadonlyArray<Binding>;

/**
 * Returns the tier given a terminal width. Does NOT account for measured
 * content width ‚Äî the caller is expected to re-measure if full-tier output
 * overflows and retry at the next tier down. (This prevents the layout
 * module from needing to know key glyph widths.)
 */
export function pickTier(width: number, categoryCount: number): Tier;

/** Formats a KeySpec per ¬ß3 step 7. */
export function formatKeys(keys: ReadonlyArray<string>): string;

/** Resolves the rendered label for a binding at a given tier. Returns
 *  "" (empty) if the tier shows no label (keys tier, or short tier with
 *  no shortLabel). */
export function renderableLabel(b: Binding, tier: Tier): string;

/**
 * Groups the already-filtered-and-sorted bindings by category, preserving
 * order. Returns a list of [category|null, Binding[]] pairs, where `null`
 * means "no category ‚Äî render with no header".
 */
export function groupByCategory(
  bindings: ReadonlyArray<Binding>,
): ReadonlyArray<readonly [string | null, ReadonlyArray<Binding>]>;

/** Sort by ordering class per rule 3; stable within a class. */
export function sortByOrder(
  bindings: ReadonlyArray<Binding>,
): ReadonlyArray<Binding>;
```

**Purity.** No `react`, no `ink`, no `node:*`. Imports `type` only from `./types.js`. Registered in `test/state/purity.test.ts`.

## 9. Risks / unknowns

1. **Ink `<Text>` string-length vs cell width.** Arrow glyphs `‚Üê‚Üì‚Üë‚Üí` are narrow (1 cell each) on all modern terminals ‚Äî not CJK-wide. Mockups.md ¬ß15 assumes single-cell arrows. Emoji are excluded from the keybar vocabulary by design. Risk: low. Mitigation: the matrix tests assert exact strings; if a terminal's width differs, the test fails loudly.
2. **`useStdout().stdout.columns` on SIGWINCH.** Ink v5 hooks stdout and emits `resize` that Ink's own reconciler acts on. Width-as-prop makes this moot ‚Äî when the app shell (P3-T5) later reads `columns`, a re-render re-passes the prop. No risk in P3-T4.
3. **`ink-testing-library` terminal width.** Verified: `render()` does not accept a `cols` option; `Instance.stdout.columns` is not a configurable input. **Resolution: width is a `Keybar` prop.** The app-shell task is responsible for wiring the live value.
4. **Arrow-group formatting under ASCII.** ¬ß0 ambiguity 5. Default pick: always render arrows as unicode regardless of `capabilities.unicode`. Reviewer sanity-check requested in ¬ß11. If reviewer rejects, the fallback is `^v` / `<->` with `‚Üë‚Üì‚Üê‚Üí` rendered only when unicode is on, which would force the ¬ß15 matrix strings to differ by theme capability ‚Äî expensive test explosion.
5. **Category overflow.** ¬ß0 ambiguity 3. Default: >2 categories forces tier-drop. Reviewer sanity-check requested.
6. **Color-split key vs label.** Features.md line 423 is explicit: key bold-cyan, label default. Encoded as `accent` role for key + no-color for label. See ¬ß3 step 9.
7. **Snapshot stability + ANSI escapes.** `lastFrame()` under `color:true` includes SGR codes. The matrix tests strip ANSI via a local helper; the rule-coverage tests that check color inspect ANSI explicitly. No `strip-ansi` package dep.
8. **Width boundary inclusivity.** "‚â•100" and "60‚Äì100" and "<60" ‚Äî we read these as `[100,‚àû)`, `[60,99]`, `[0,59]`. Documented in `pickTier` and in ¬ß3 step 5.
9. **`Binding.action` unused in this task.** The type carries `action` but `<Keybar>` never invokes it. A lint reviewer may flag this as dead code; a comment block at the top of `types.ts` documents rule 9 (single keymap array). The first invoker is P4-T1.
10. **`toggleLabel` param type `unknown`.** This forces each binding author to narrow their toggle state. A generic `Binding<T>` would be nicer but would propagate into the `Binding[]` heterogeneous array, requiring `Binding<any>[]`. Keeping `unknown` matches the spec (features.md line 459) and accepts the ergonomic cost.
11. **Memoization correctness.** `React.memo` does shallow prop equality. `ctx` is rebuilt every app render; if callers rebuild fresh object identities every time, memo is ineffective. This is a P3-T5 concern; for this task we document the contract and the adapter wires a stable `ctx` reference.
12. **`components/index.ts` pulling React transitively.** A purity-scanned module must not import from `components/index.ts` ‚Äî it would drag the keybar component (and React) in. The only importer of the barrel should be `app.tsx` / future app-shell. Fixtures import the narrower `./types.js` and `./keybar.js` directly to keep the purity test honest.
