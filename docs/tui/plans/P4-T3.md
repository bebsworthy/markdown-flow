### 3.3 State placement decision: component-local vs reducer

**Recommendation: component-local for everything except the tab.** The modal's ephemeral state — query text, root path, selected result index, walker candidates, ranked results, ingestion in-flight flag, ingestion error — lives in `useState` inside `<AddWorkflowModal>`. Only the tab sits in the reducer.

**Rationale.**
1. **Reducer purity + test cost.** The reducer is scanned by `test/state/purity.test.ts`. Every piece of state we push into the reducer needs a new Action and a new case branch, multiplying test surface. Keeping 6 ephemeral fields local saves ~12 reducer tests for zero product benefit.
2. **Undo/redo is out of scope.** None of these fields need time-travel or be persisted across modal-close/reopen. `commandPalette` precedent has exactly the opposite trade-off (query text *does* persist across minor re-renders of sibling components, so it lives in the reducer).
3. **Why tab is different.** The tab is visible in the overlay's mode pill ("FIND" vs "URL") and *could* be driven externally (e.g. a future command-palette command "addWorkflow url"). Having it in the reducer future-proofs that entry point for cost ~1 action, ~4 tests. Worth it.
4. **Concrete list of component-local fields:**
   - `query: string` — fuzzy search input.
   - `root: string` — current root directory.
   - `rootPickerOpen: boolean` + `rootPickerDraft: string` — root picker input state.
   - `candidates: ReadonlyArray<Candidate>` — streamed in from walker.
   - `walkerTruncated: boolean` — sentinel from walker.
   - `selectedRowIndex: number` — cursor in fuzzy result list.
   - `urlInput: string` — URL tab text input.
   - `ingesting: boolean` — in-flight network call.
   - `ingestError: string | null` — last ingestion error, cleared on new attempt.

**Considered-and-rejected alternative:** put the whole modal state in `overlay`. This would let the reducer test tab-changes-without-rerender and support programmatic opening pre-filled ("open add modal with query=foo"). Neither is a requirement of P4-T3. Defer.

### 5.1 Pure types — `src/add-modal/types.ts`

```ts
import type { ValidationDiagnostic } from "markflow";  // type-only

export type AddModalTab = "fuzzy" | "url";

export type CandidateKind = "file" | "workspace" | "truncated";

/** One row the walker has discovered. Classification happens lazily per row. */
export interface Candidate {
  readonly kind: "file" | "workspace";
  /** Absolute path on disk. */
  readonly absolutePath: string;
  /** Display path: if inside root, a root-relative form; else absolute. */
  readonly displayPath: string;
  /** Depth from root (for debug / future width-tier heuristics). */
  readonly depth: number;
}

/** Sentinel yielded when the walker hits `maxCandidates`. */
export interface TruncatedSentinel {
  readonly kind: "truncated";
  readonly scannedCount: number;
}

/** A candidate with a fuzzy-score. Ordering is score desc, then displayPath asc. */
export interface RankedCandidate {
  readonly candidate: Candidate;
  readonly score: number;
  /** Index positions in `candidate.displayPath` where each query char matched. */
  readonly matchPositions: ReadonlyArray<number>;
}

/** Options accepted by `walkCandidates`. */
export interface WalkerOptions {
  /** Cap on total candidates yielded. Default 500. */
  readonly maxCandidates?: number;
  /** Cap on directories descended. Default unbounded. */
  readonly maxDepth?: number;
  /** Directory names to skip verbatim. Default: [".git", "node_modules", ".markflow-tui"]. */
  readonly skipDirs?: ReadonlyArray<string>;
  /** AbortSignal that stops the walk. */
  readonly signal?: AbortSignal;
}

/** Result of `validateCandidate`. Never throws — always returns this. */
export type ValidationResult =
  | { readonly kind: "file-valid" }
  | { readonly kind: "file-parse-error"; readonly message: string; readonly diagnostics?: ReadonlyArray<ValidationDiagnostic> }
  | { readonly kind: "workspace" }
  | { readonly kind: "workspace-invalid"; readonly message: string };

/** Result of `ingestUrl`. */
export type UrlIngestResult =
  | { readonly ok: true; readonly workspaceDir: string; readonly workflowPath: string }
  | { readonly ok: false; readonly reason: string };
```

### 5.2 Fuzzy ranker — `src/add-modal/fuzzy.ts`

```ts
import type { Candidate, RankedCandidate } from "./types.js";

/**
 * Rank candidates by fuzzy match. Deterministic — same input order + query
 * always produces the same output order.
 *
 * Scoring (all values integer, higher = better):
 *   base           = 0
 *   match any char = +1
 *   consecutive    = +4 per run-of-matched-chars beyond the first
 *   prefix match   = +10 if query[0] matches displayPath[0]
 *   basename hit   = +20 if ALL query chars land inside basename(displayPath)
 *   case-sensitive = +2 if query char matches path char verbatim (not folded)
 *
 * Non-matches (missing any query char in order) are dropped.
 * Empty query returns candidates in input order with score 0 (no sort).
 *
 * Ties break by `displayPath` lexicographically asc (stable).
 *
 * Returns the top `limit` entries (default 50) to keep render cheap.
 */
export function rankCandidates(
  query: string,
  candidates: ReadonlyArray<Candidate>,
  limit?: number,
): ReadonlyArray<RankedCandidate>;

/**
 * Pure subsequence scorer. Exposed for unit tests.
 * Returns null if `query` is not a subsequence of `text` (case-folded).
 */
export function scoreSubsequence(
  query: string,
  text: string,
): { score: number; positions: ReadonlyArray<number> } | null;
```

Implementation sketch (for the implementer):

```ts
export function scoreSubsequence(query, text) {
  if (query.length === 0) return { score: 0, positions: [] };
  const q = query.toLowerCase();
  const t = text.toLowerCase();
  const positions: number[] = [];
  let ti = 0;
  for (const qc of q) {
    while (ti < t.length && t[ti] !== qc) ti++;
    if (ti >= t.length) return null;
    positions.push(ti);
    ti++;
  }
  // score: +1 per match, +4 per consecutive run beyond first, +10 prefix, +20 basename-hit, +2 per exact-case char.
  let score = positions.length;
  for (let i = 1; i < positions.length; i++) {
    if (positions[i] === positions[i - 1]! + 1) score += 4;
  }
  if (positions[0] === 0) score += 10;
  const lastSlash = Math.max(text.lastIndexOf("/"), text.lastIndexOf("\\"));
  if (positions.every((p) => p > lastSlash)) score += 20;
  for (let i = 0; i < positions.length; i++) {
    if (text[positions[i]!] === query[i]) score += 2;
  }
  return { score, positions };
}
```

### 5.3 Walker — `src/add-modal/walker.ts`

```ts
import { readdir, stat } from "node:fs/promises";
import { join, relative, resolve as resolvePath } from "node:path";
import type { Candidate, TruncatedSentinel, WalkerOptions } from "./types.js";

const DEFAULT_SKIP = [".git", "node_modules", ".markflow-tui"] as const;

/**
 * BFS walk starting at `root`. Yields `Candidate` for:
 *   - any *.md file (extension case-insensitive)
 *   - any directory that contains a readable `.markflow.json`
 *
 * Skips directories whose basename starts with "." OR matches opts.skipDirs.
 * Yields a final `TruncatedSentinel` when maxCandidates is reached — consumer
 * uses this to render the "showing N/N+" footer.
 *
 * Never throws on fs errors — silently skips unreadable dirs.
 * Honors opts.signal: after every yield, checks signal.aborted and returns.
 *
 * Deterministic ordering: siblings sorted by readdir output then case-folded
 * name ASC; BFS level-by-level. Tests rely on this.
 */
export async function* walkCandidates(
  root: string,
  opts?: WalkerOptions,
): AsyncGenerator<Candidate | TruncatedSentinel, void, void>;
```

Algorithm:

```
Queue Q = [{dir: root, depth: 0}]
while Q not empty and yielded < cap and not aborted:
  { dir, depth } = Q.shift()
  if depth > maxDepth: continue
  try entries = await readdir(dir, {withFileTypes: true})
  catch: continue
  sort entries by name (case-fold)

  // First pass: is this dir a workspace? (contains .markflow.json)
  const hasConfig = entries.some(e => e.isFile() && e.name === ".markflow.json")
  if hasConfig:
    yield {kind:"workspace", absolutePath: dir, displayPath: relative(root, dir) || ".", depth}
    if ++yielded >= cap: emit TruncatedSentinel; return
    // Do NOT descend into workspaces — their contents aren't candidates
    continue

  // Second pass: .md files + descend.
  for entry of entries:
    const full = join(dir, entry.name)
    if entry.isFile() && /\.md$/i.test(entry.name):
      yield {kind:"file", absolutePath: full, displayPath: relative(root, full), depth+1}
      if ++yielded >= cap: emit TruncatedSentinel; return
    else if entry.isDirectory():
      if entry.name.startsWith(".")) continue
      if skipDirs.includes(entry.name)) continue
      Q.push({dir: full, depth: depth+1})
```

Workspaces are terminal for the walker (we don't descend into them to avoid doubling them up via the inner `.md`).

## §6 URL-ingest engine-reuse decision

**Decision: inline a minimal fetch helper in `src/add-modal/url-ingest.ts`. Do NOT spawn the CLI as a subprocess. Do NOT modify the engine.**

### 6.1 Options considered

1. **Spawn `npx markflow` as a subprocess.** Rejected. `npx` lookup adds ~500 ms start cost; subprocess I/O between Ink and a child is brittle (stdout redirection, TTY detection); doesn't eliminate our fetch code (we still need to read the fetched workspace to pass back to the caller); introduces `execa` or `child_process` as a surface this module must own.
2. **Import `materializeRemoteTarget` directly from the engine.** Rejected. The engine's `package.json` `exports` field only publishes `dist/core/index.js`; `materializeRemoteTarget` lives in `dist/cli/workspace.js`. Adding an export would require touching the frozen engine package.
3. **Inline a minimal copy.** Chosen. ~50 LOC of `fetch` + `mkdir` + `writeFile` + `.markflow.json` serialization. The disk layout is identical to engine's, so if the engine exports `materializeRemoteTarget` in a later task, we delete this file and dispatch the call.

### 6.2 Identical-to-engine layout (invariant)

To make the future engine swap trivial, `ingestUrl` MUST write:

- `<workspaceDir>/flow.md`            (UTF-8 body of the fetched URL)
- `<workspaceDir>/.markflow.json`     (exact shape below)

```json
{
  "workflow": "flow.md",
  "origin": { "type": "url", "url": "https://...", "fetchedAt": "2026-04-16T10:22:00.000Z" }
}
```

Matches engine's `MarkflowJson` verbatim (workspace.ts:11–14).

### 6.3 Where the workspace lives

`<baseDir>/.markflow-tui/workspaces/<slug>/`. The `.markflow-tui/` prefix keeps TUI-generated artefacts grouped (sibling to `.markflow-tui.json`) so repository-level `.gitignore` can target them with one pattern. Engine-managed workspaces live anywhere the user runs `markflow run <url>`; we don't trample those.

### 6.4 Conflict handling

If `<workspaceDir>/.markflow.json` already exists and has `origin.url === url`, overwrite (re-fetch, bump `fetchedAt`). If it exists with a *different* URL (collision on slug), append a numeric suffix: `slug-2`, `slug-3`, …

### 7.1 Modal layout (mirrors mockups.md §2 lines 97–113)

- Frame glyphs come from `theme.frame`. No literal box-drawing chars in TSX.
- Tab header: active tab wrapped in `[ ... ]` with Ink `inverse bold` — same pattern as `ModeTabs`.
- Result list: 10 rows max visible. Rows render `cursor-glyph + displayPath + badge` columns, padded to widths computed like `composeListRows` (reuse `pickBadgeColumnWidth` from `browser/list-layout.ts`).
- Footer row: three bindings in keybar-lite style (plain text; no `Keybar` primitive here — the modal owns its own small footer).

### 7.2 Walker lifecycle

Walker starts on mount (root = `baseDir`) and streams candidates via `useEffect` + the async generator:

```ts
useEffect(() => {
  const abort = new AbortController();
  const collected: Candidate[] = [];
  let truncated = false;

  (async () => {
    for await (const out of walker(root, { signal: abort.signal })) {
      if (abort.signal.aborted) return;
      if (out.kind === "truncated") { truncated = true; continue; }
      collected.push(out);
      // Batch updates: flush every 50 candidates OR on walker completion.
      if (collected.length % 50 === 0) {
        setCandidates([...collected]);
      }
    }
    setCandidates([...collected]);
    setWalkerTruncated(truncated);
  })();

  return () => abort.abort();
}, [root, walker]);
```

### 7.3 Ranking + display

Ranking runs inside a `useMemo` on every keystroke:

```ts
const ranked = useMemo(() => rankCandidates(query, candidates, 20), [query, candidates]);
```

Top 20 kept; render first 10 visible.

### 7.4 Validation flag on visible rows

Per-row `parseWorkflow` is expensive. Defer to visible rows only via a map `validationByPath: Map<string, ValidationResult>`:

```ts
useEffect(() => {
  const visible = ranked.slice(0, 10);
  for (const r of visible) {
    if (!validationByPath.has(r.candidate.absolutePath)) {
      validateCandidate(r.candidate).then((res) => {
        setValidationByPath((m) => new Map(m).set(r.candidate.absolutePath, res));
      });
    }
  }
}, [ranked]);
```

Display: `[file]`, `[workspace]`, or `[file · ✗ parse]` / `[workspace · ✗ invalid]`. Mirrors mockup §2 line 109.

### 7.5 Root picker

`Ctrl+Up` toggles `rootPickerOpen` to `true`. The input bar above the result list swaps to a free-text input backed by `rootPickerDraft`. `Enter` validates the draft path exists via `stat`; on success, setState `root=draft; rootPickerOpen=false; candidates=[]`; the walker `useEffect` re-runs. On stat failure, shows inline error "path not found". `Esc` while picker is open → closes picker without changing root. **No `~` expansion** — the user types an absolute path. (A future polish task may add `os.homedir()` expansion; out of scope.)

### 7.6 Bounded memory

`maxCandidates=500`. Walker yields `TruncatedSentinel` when hit. The modal displays a footer line `showing 500/500+ — refine` below the result list. Empirically confirmed OK for repos up to ~10 k files; larger trees need the user to narrow the root.

### 7.7 Submit flow

Fuzzy-tab Enter:
```
const row = ranked[selectedRowIndex];
if (!row) return;
const source = row.candidate.absolutePath;
await onSubmit(source);
```

URL-tab Enter:
```
if (!/^https?:\/\//i.test(urlInput)) {
  setIngestError("expected http:// or https://");
  return;
}
setIngesting(true);
const res = await ingestor(urlInput, baseDir);
setIngesting(false);
if (!res.ok) { setIngestError(res.reason); return; }
await onSubmit(res.workspaceDir);
```

`source` passed to `onSubmit` is ALWAYS an absolute path (fuzzy) or an absolute workspace path (URL). This matches features.md §3.1: "source is what the user typed (or the workspace path produced by a URL materialisation)".

## §8 Key routing

Modal owns all input while open — no bubbling to the browser. Ink's `useInput` runs every mounted handler; the modal's handler returns early for keys it doesn't own (q, mode-switch keys) so the app-level `q` quit still works. **Trade-off: the user can still quit the app from inside the modal with `q`.** This matches existing overlay conventions (the `q` binding in `app.tsx` is global). Acceptable.

| Key | Behavior |
|---|---|
| `Tab` | Dispatch `ADD_MODAL_SET_TAB` flipping between `"fuzzy"` and `"url"`. |
| `Esc` | `onCancel()` — dispatches `OVERLAY_CLOSE`. |
| `↑` / `↓` / `k` / `j` | Move `selectedRowIndex` (fuzzy tab only). Clamp `[0, ranked.length - 1]`. URL tab ignores. |
| `Enter` | Submit (see §7.7). |
| `Ctrl+Up` | Open root picker (fuzzy tab only). |
| printable chars | Append to `query` (fuzzy) or `urlInput` (URL) or `rootPickerDraft` (root picker open). |
| `Backspace` | Delete last char of current input. |

For keyboard primitives: Ink's `useInput` exposes `key.ctrl` + `key.upArrow`, so Ctrl+Up is `if (key.ctrl && key.upArrow) ...`. Arrow keys arrive as `key.upArrow`/`key.downArrow`. Printable chars arrive via `input` parameter.

## §10 Theme + hide-don't-grey discipline

- All glyphs (`▶`, `╔`, `═`, `┌`, `│`, `└`, `→`) come from `theme.frame` or `theme.glyphs`. No literal box-drawing chars in any `.tsx`.
- Active tab (`[ Fuzzy find ]`) uses Ink's structural `inverse bold` — matches `mode-tabs.tsx` pattern.
- Inactive tab (`Path or URL`) uses `theme.colors.accent` — matches `ModeTabs` inactive-tab color.
- Error lines (invalid URL prefix, ingestion failure, root-picker stat failure, workspace-invalid candidate badge suffix `· ✗ parse`) use `theme.colors.danger`.
- "Fetching…" in-flight indicator: `theme.colors.dim`.
- **Hide-don't-grey enforcement:**
  - The Enter-to-submit hint is not rendered when the query is empty AND no row is selected (rather than greyed). The footer text is static (`Tab switch  ⏎ Add  Esc Cancel`) because all three actions are always available — Tab flips tabs, Enter is a no-op when nothing is selectable, Esc always cancels. No conditional greying needed.
  - Result list is simply empty when `ranked.length === 0` — no greyed "nothing to show" placeholder; we render `(no matches)` in default dim instead.
  - Empty-state keybar: exactly three bindings, no `r`/`d`/`↑`/`↓`/`⏎` — none of those actions exist to be hidden.

## §14 Risks + open questions

### 14.1 Risks

1. **Modal overlay rendering strategy.** Ink does NOT have true z-ordering; the "overlay" is a subtree rendered after the shell with a large negative `marginTop`. This trick already works for mode-tabs overlay in `app-shell.tsx:152`. Risk: on narrow widths the modal's content wraps. **Mitigation:** cap modal width at `min(stdout.cols - 4, 90)` and height at `min(stdout.rows - 4, 18)`; document the min-width threshold (80 cols); below that, content wraps — acceptable for MVP.
2. **Walker performance on huge trees.** A monorepo with 200 k files will hit `maxCandidates=500` quickly and render the truncated footer. The walk itself may still take seconds. **Mitigation:** Walker yields 50 candidates at a time (batch flush) so initial results appear within ~100 ms. Tests assert cap is hit without crash.
3. **Async generator cancellation in tests.** ink-testing-library doesn't run the event loop to exhaustion between keystrokes. Use `await new Promise(r => setImmediate(r))` after each key dispatch, and wrap walker in a `vi.fn()` that returns a finite generator synchronously for component tests.
4. **`useEffect` + late `setState` after modal unmount.** Same pattern as `workflow-browser.tsx:79–106`: `let cancelled = false; return () => { cancelled = true; };` for walker, validator, and ingestor effects.
5. **`dispatch` identity in callbacks.** `useCallback(..., [])` with `latestStateRef` pattern prevents stale closures. Tested indirectly by `add-remove-roundtrip.test.tsx`.
6. **Saving a corrupt registry.** `saveRegistry` writes atomically; if the user's disk is full mid-write, the `.tmp` file is left. Deferred to a later robustness pass; this task swallows `saveRegistry` errors.
7. **Test isolation around global fetch stub.** `vi.stubGlobal("fetch", ...)` + `vi.unstubAllGlobals()` in `afterEach`. Standard Vitest pattern.
8. **Reducer purity after `types.ts` imports `AddModalTab` from add-modal.** The type-only import does not trigger the `FORBIDDEN` regex; verified by running the purity test against a synthetic `types.ts` that includes the new import before wiring is complete.
9. **Keybar fixture must re-render when registry changes from empty → non-empty.** `app.tsx` already re-renders on `setRegistryState`; the keybar slot conditional picks up the change automatically.
10. **`initialLaunchArgs` ingestion ordering.** Awaited sequentially (for-of with await). If one URL fetch hangs, subsequent args block. **Mitigation:** `Promise.allSettled` alternative considered but rejected — sequential is easier to reason about for P4-T3; a follow-up task may parallelize.

### 14.2 Open questions

Each has a default pick; none are hard blockers.

1. **Confirm: no engine modification.** This task inlines a URL fetcher. A follow-up task should expose `materializeRemoteTarget` from `packages/markflow` and replace our inline copy. Register this as a backlog item. **Default: defer.**
2. **Should the restricted empty-state keybar also render in `--no-save` mode?** **Default: yes.** Add/help/quit are all still valid actions; persistence is orthogonal.
3. **Should `d` on a selected entry fire a transient status message ("removed N")?** **Default: no.** Hide-don't-grey + the list visibly shrinks — self-evident feedback.
4. **Root picker: expand `~/code` → `os.homedir() + "/code"`?** **Default: no expansion.** Absolute paths only; deferred.
5. **Glob expansion for positional launch args (features.md §3.1 line 149 mentions `flows/*.md`).** **Default: pass verbatim.** Bad paths visibly flagged by resolver. Glob expansion is a follow-up task.
6. **Fuzzy result ordering when query is empty.** **Default: walker insertion order.** Consistent with "lazy discovery" mental model.
7. **URL workspace slug collision behavior.** **Default: append numeric suffix.** Alternatives: overwrite (destructive), error (friction). Suffix is the least surprising.
8. **Do we filter fuzzy results to "valid-only" pre-render, or show all + flag invalid?** mockups.md §2 line 109 shows an invalid workflow with `[file · ✗ parse]` badge (so user can add broken ones to fix them). **Default: show all, flag invalid.** Matches mockup.
9. **`?` in the empty-state keybar — what does it do?** The help overlay is deferred. **Default: `?` is declared in the keybar fixture but the handler is a no-op comment.** Implementation date: future "help overlay" task.
10. **Modal's own keybar line (mockup §2 line 111: "Tab switch input mode  ⏎ Add  Esc Cancel") — render via the `<Keybar>` primitive or inline?** **Default: inline.** Only three static bindings, all always visible; using the full Keybar for this would add tier-picking overhead for no benefit.
