### 0.2 Engine data available

| Need | Source | Notes |
|---|---|---|
| The selected step's `stepSeq` (= `seq` of its owning `step:start` event) | Walk `engineState.activeRun.events` for the `step:start` whose `tokenId === selectedStepId` (or `nodeId === rowNode` fallback for `batch:*` rows). P6-T2's `selectStepDetail` already resolves token→node; we factor out a `resolveStepSeq()` helper. | If no `step:start` has been observed (running step has not been tailed yet), the stream is not available — show empty state. |
| `nodeId` for the selected token | Same `step:start` event. | Needed to build the `output:ref` payload accepted by `getSidecarStream`. |
| `runDir` | `runsDir + "/" + runId` — same pattern the engine adapter uses (`adapter.ts::requireRunsDir`). `runsDir` threads through from `useEngineAdapter`. We add a `runsDir` prop to `<LogPanelView>` so the pane can build the run directory path without touching the adapter. | Must be passed down from `app.tsx` — see §6. |
| Live `step:output` chunks | `engineState.activeRun.events` — chunks are **non-persisted** (features.md event table line 88). Available only while the tail is active. | For terminal runs the ring is empty; sidecar file read is the only source. |
| Persisted bytes | `getSidecarStream(runDir, { stepSeq, nodeId, stream })` returns a Web `ReadableStream<Uint8Array>`. | Already exported from `markflow` (P1-T2). |

### 0.3 Why we need **two** sources

Features.md §3.5 — "seek on demand; do not buffer entire history in memory (ring buffer of last ~2k lines + lazy file read)". The 500-event `activeRun.events` ring is too coarse (contains non-log events) and is cleared across `viewing.*` mode re-entries. The canonical persistent store is the sidecar file. For live-follow updating within 100ms, the ring gives us event-append latency; for paused-mode scroll and post-run replay, the file read gives us the full history. The log panel reconciles the two: **sidecar file is the source of truth; `step:output` events are a low-latency preview prepended to the tail between file polls.**

## §2 Data model (`src/log/types.ts`)

```ts
export type LogStream = "stdout" | "stderr";

/** A single parsed line ready to render. */
export interface LogLine {
  readonly seq: number;           // stepSeq of the owning step:start (same for all)
  readonly lineIndex: number;     // monotonic per (stream) within this step
  readonly stream: LogStream;
  readonly ts: string | null;     // ISO-8601 when known; null for sidecar-file lines (no per-line ts)
  readonly segments: readonly LogLineSegment[];
  readonly rawLength: number;     // length without ANSI, used for truncation math
}

/** ANSI parser output — one run of text with resolved attributes. */
export interface LogLineSegment {
  readonly text: string;
  readonly color?: AnsiColor;
  readonly bgColor?: AnsiColor;
  readonly bold?: true;
  readonly dim?: true;
  readonly italic?: true;
  readonly underline?: true;
}

export type AnsiColor =
  | "black" | "red" | "green" | "yellow" | "blue"
  | "magenta" | "cyan" | "white" | "gray"
  | "brightRed" | "brightGreen" | "brightYellow"
  | "brightBlue" | "brightMagenta" | "brightCyan" | "brightWhite"
  | { readonly kind: "256"; readonly index: number }      // pass-through to Ink
  | { readonly kind: "rgb"; readonly r: number; readonly g: number; readonly b: number };

export interface LogPanelSettings {
  readonly streamFilter: "stdout" | "stderr" | "both";
  readonly wrap: boolean;
  readonly timestamps: boolean;     // `t` toggle — accepted in action union, default false
}

/** Mutable-looking but immutable state carried in the reducer. */
export interface LogPanelState {
  readonly lines: readonly LogLine[];  // ring-buffer, cap = 2000 most recent
  readonly dropped: number;            // lines evicted from the front of the ring (so seq is still faithful)
  readonly follow: boolean;
  readonly cursor: number;             // index into lines[] of the highlighted row
  readonly pausedAtHeadSeq: number | null; // head seq at pause time → powers "N new since pause"
  readonly settings: LogPanelSettings;
  readonly partialByStream: Readonly<Record<LogStream, string>>; // unterminated tail buffered across chunks
}

/** Presentation model produced by `deriveLogModel`. */
export interface LogPanelModel {
  readonly header: string;
  readonly banner: { readonly kind: "paused"; readonly linesSincePause: number } | null;
  readonly rows: readonly LogPanelRow[];
  readonly footer: { readonly kind: "more-below"; readonly hidden: number } | { readonly kind: "live-tail" } | null;
  readonly empty: LogPanelEmptyReason | null;   // when non-null, rows/header are ignored
  readonly isFollowing: boolean;                // proxied out for keybar ctx
}

export type LogPanelEmptyReason =
  | { readonly kind: "no-selection" }                 // no step selected
  | { readonly kind: "pending" }                      // step:start not yet observed
  | { readonly kind: "not-found"; readonly id: string }
  | { readonly kind: "aggregate" };                   // "batch:*" row — ask user to pick a leaf

export type LogReducerAction =
  | { readonly type: "APPEND_LINES"; readonly lines: readonly LogLine[] }
  | { readonly type: "SET_FOLLOW"; readonly follow: boolean }
  | { readonly type: "SET_WRAP"; readonly wrap: boolean }
  | { readonly type: "SET_STREAM_FILTER"; readonly filter: LogPanelSettings["streamFilter"] }
  | { readonly type: "SCROLL_DELTA"; readonly delta: number }
  | { readonly type: "SCROLL_JUMP_HEAD" }
  | { readonly type: "SCROLL_JUMP_TOP" }
  | { readonly type: "SCROLL_PAGE"; readonly direction: "up" | "down"; readonly pageSize: number }
  | { readonly type: "RESET" };
```

**Size cap — 2000 lines.** Features.md §3.5 calls out "ring buffer of last ~2k lines + lazy file read" explicitly. On eviction, `dropped` increments so the header can still show the correct `head` seq. When paused and the cursor would fall off the front of the ring, the oldest visible seq is clamped to `lines[0].seq`; the "▲ more above — press g to jump to top" affordance is reserved for a later pass (mockup §9 only shows `▼ more below`).

## §3 ANSI handling (`src/log/ansi.ts`)

### 3.1 What we parse

| Sequence | Action |
|---|---|
| SGR CSI `\x1b[ … m` | Interpret: `0` reset; `1` bold; `2` dim; `3` italic; `4` underline; `22/23/24` clear those; `30–37`, `90–97` FG; `40–47`, `100–107` BG; `38;5;N` / `48;5;N` → 256-color; `38;2;r;g;b` / `48;2;r;g;b` → truecolor; `39`/`49` reset FG/BG. Unknown codes ignored (keep current state, drop token). |
| OSC `\x1b] … (\x07 \| \x1b\\)` | Strip entirely (window-title, hyperlinks — Ink can't render them reliably). |
| CSI non-m (`\x1b[…A/B/C/D/H/J/K/f/s/u/r`, `\x1b[?25l`, etc.) | Strip — "incompatible escape sequences" per task body. |
| ESC-only (`\x1b` followed by a single letter other than `[` / `]`) | Strip. |
| C1 controls (`\x9b`, `\x9d`, etc.) | Strip — rare in shell output. |
| `\t` `\r` `\b` | Left as literal characters. We do **not** interpret carriage-return-overwrite or tab-expand. Documented in code comments — parity with what Ink will actually render. |

### 3.2 Signature

```ts
export function parseAnsi(text: string, initial?: SgrState): {
  readonly segments: readonly LogLineSegment[];
  readonly final: SgrState;
};

export function stripAnsi(text: string): string;

export const ANSI_PATTERN: RegExp;
```

`parseAnsi` returns the trailing SGR state so multi-chunk lines can pick up mid-stream without a reset (relevant when a single logical line spans two `step:output` events). `ingest.ts` threads the `final` state forward per-stream, alongside `partialByStream`.

### 3.3 Bespoke vs npm

We do **not** pull in `ansi-regex` / `strip-ansi` / `ansi-parser`. Rationale:
1. The three popular packages bundle ~1k lines and overlap this parser's job by < 40%.
2. The TUI's Ink version (v6 under the hood) exposes a fixed palette; we must *map* ANSI colors to Ink props anyway — a full 3rd-party parser emits objects we'd translate immediately.
3. Keeping `ansi.ts` pure + dependency-free simplifies the `purity.test.ts` envelope (no need to allow-list packages that might themselves reach for `process.stdout.getColorDepth()` etc.).
4. One file, ~80 LoC of parser + ~100 LoC of tests is faster to reason about than vendor drift.

## §4 Reducer behaviour (`src/log/reducer.ts`)

### 4.1 Follow / pause state machine

```
┌────────────┐     APPEND_LINES                   ┌──────────┐
│  follow    │─────── cursor := head ─────────────▶│ follow   │
│  cursor=head│                                    │ cursor=head│
└────┬───────┘                                     └────┬─────┘
     │ SCROLL_DELTA (up) OR SCROLL_PAGE(up)              ▲
     ▼                                                   │
┌────────────┐                                           │
│  paused    │                                           │ SCROLL_JUMP_HEAD
│  cursor<head│◀───── APPEND_LINES: increments          │    OR
│            │       linesSincePause,                    │ SET_FOLLOW(true)
│            │       cursor unchanged (frozen)           │
└────┬───────┘───────────────────────────────────────────┘
     │ SCROLL_DELTA (down, cursor==head)
     ▼  auto-resume follow
```

Key rules (from mockups §9):
- Any upward scroll while `follow: true` auto-pauses — mirror lazydocker behaviour called out in features.md §3.5.
- `APPEND_LINES` during pause **never** moves `cursor`; it only grows `lines` and bumps `pausedAtHeadSeq`-delta.
- `SCROLL_JUMP_HEAD` (= `G` or `F`) resumes follow + sets `cursor = lines.length - 1`.
- `RESET` fires on target change — clears lines, re-enables follow.
- `APPEND_LINES` only processes lines whose `stream` matches the current `streamFilter`; filtered-out lines are still stored (so switching filter back shows history) — filter is applied later in `derive.ts`.

### 4.2 Why local reducer, not global `AppState`

Global `AppState` already carries step-table / runs-table cursors. Log-pane state (follow, cursor, wrap, stream filter) is **pane-local**: closing the log pane and reopening it (e.g., moving to another step) legitimately resets state; carrying it globally would require tear-down actions in three transitions (`FOCUS_VIEWING_PANE`, `MODE_OPEN_RUN`, `MODE_CLOSE_RUN`). Precedent: the add-modal keeps its query / URL / walker state in `useState` inside `<AddWorkflowModal>` (P4-T3, called out in `state/types.ts` line ~71). Log-pane state follows the same pattern.

### 4.3 No new `AppState` actions

The only `state/reducer.ts` touch is documentary — an inline comment stating "log-pane state is pane-local; see `src/log/reducer.ts`". Tab switching re-uses the existing `FOCUS_VIEWING_PANE` action.

## §5 Data flow

```
          runsDir, runId                engineState.activeRun.events
                │                                     │
                ▼                                     ▼
          ┌──────────────┐                    ┌────────────────┐
          │ <LogPanelView>│  resolveLogTarget │ step:output    │
          │               │◀─────────────────│ events filter   │
          │               │                   └───────┬────────┘
          │  useReducer(  │                           │
          │   logReducer) │◀── APPEND_LINES ──────────┘
          │               │                           ▲
          │               │                           │
          │               │◀── useSidecarStream ──────┤  sidecar file
          │               │        (stdout + stderr)  │  via getSidecarStream
          │               │                           │
          │               │── deriveLogModel ─▶ <LogPanel>
          └──────────────┘                 (stateless Ink)
```

- **Two feeders, one reducer.** `useSidecarStream` (x2, one per stream) feeds `APPEND_LINES` from the sidecar file. A second `useEffect` watches `engineState.activeRun.events` identity and slices everything past the last-ingested `seq` into `APPEND_LINES`. `ingest.ts::mergeSidecarTail` dedupes by `(stream, lineIndex)` so if the file read catches up to the live tail we never double-print.
- **Mount ordering.** On mount / target change, we dispatch `RESET`, then kick off `useSidecarStream` (reads bytes already on disk for historical lines), then subscribe to ring-buffer slicing. For an in-flight step whose sidecar hasn't closed yet, `getSidecarStream` returns a `ReadableStream` of bytes written so far — the stream ends at EOF without blocking; the live tail continues via ring events. We do **not** re-poll the sidecar file; the 100-ms acceptance criterion is served purely by the ring.
- **Latency budget.** Engine emits event → TailEventLog delivers → adapter dispatches `RUN_TAIL_EVENT` → engine reducer appends to `activeRun.events` → React re-renders `<LogPanelView>` with a new `events` identity → effect dispatches `APPEND_LINES` → `<LogPanel>` re-renders within the same frame. Target: < 100 ms from emit to visible line (§acceptance criterion). No debouncing / batching is added that would push beyond that.

## §8 Acceptance criteria

**Lifted verbatim from plan.md:**

1. Consumes `getSidecarStream` (P1-T2) for the selected step's `seq`.
2. `f` toggles follow. Scrolling past last line enters **paused** mode with a banner (mockups §9).
3. Interpret ANSI color; strip incompatible escape sequences.
4. `w` toggles wrap vs truncate.
5. Following a live stream updates within 100ms of an event append; paused mode freezes output.

**Derived sub-criteria:**

- (A) `<LogPanelView>` mounts in the `viewing.*` bottom slot **iff** `state.mode.focus === "log"`; otherwise `<StepDetailPanelView>` continues to render (P6-T2 regression).
- (B) Pressing `2` in viewing mode dispatches `FOCUS_VIEWING_PANE: "log"`; `Esc` returns to `"graph"`.
- (C) `<LogPanel>` is stateless — all state lives in `logReducer` (tested by injected fixtures).
- (D) `logReducer`, `parseAnsi`, `deriveLogModel`, `appendEventLines`, `resolveLogTarget`, and `log/types.ts` pass `purity.test.ts` — no `react` / `ink` / `node:*` imports.
- (E) Live-append latency ≤ 100 ms measured via fake-timer integration test.
- (F) Upward scroll auto-pauses; `F` (uppercase) + `G` both resume+jump-to-head; `g` jumps to top while staying paused.
- (G) ANSI: `parseAnsi` emits segments for SGR colors + bold/dim/italic/underline; all other CSI / OSC / C1 escape bytes are stripped (`stripAnsi` parity); mapped to Ink `<Text>` props.
- (H) `w` toggles wrap; wrap-off truncates with trailing `…` at `width - gutter`; wrap-on hard-wraps.
- (I) Empty / pending / not-found render a single dim line — hide-don't-grey.
- (J) Keybar shows `LOG_FOLLOWING_KEYBAR` when following and `LOG_PAUSED_KEYBAR` when paused; strings match mockups §8 / §9 verbatim.
- (K) `npm run lint -w packages/markflow-tui` + `npm test -w packages/markflow-tui -- --run` both green.
- (L) No engine-side change; `packages/markflow/package.json` untouched (CLI purity rule).

## §9 Design decisions & rationale

| # | Decision | Rationale |
|---|---|---|
| D1 | Two feeders (sidecar file + event ring) merged via `(stream, lineIndex)` dedup key. | Sidecar is the persistent source-of-truth; ring gives the ≤100 ms live-tail latency. Single-source would either require re-reading the file on every event (wasteful + latent) or miss historical lines for runs the TUI joined mid-flight. |
| D2 | Log-pane state is pane-local (`useReducer` inside `<LogPanelView>`), not `AppState`. | Mirrors the add-modal precedent. Prevents contaminating the pure `state/reducer.ts` action catalogue with seven pane-only actions and avoids tear-down coupling to `MODE_*` transitions. |
| D3 | Bespoke ANSI parser (`src/log/ansi.ts`) rather than `ansi-regex` / `strip-ansi`. | Pure + dep-free keeps `purity.test.ts` envelope clean; third-party parsers overlap our job by < 40% and we'd translate their output to Ink props immediately anyway. |
| D4 | Ring buffer cap = 2000 lines (features.md §3.5 verbatim). | Prevents unbounded memory; satisfies combray TUI Development anti-pattern call-out. `dropped` counter preserves seq semantics. |
| D5 | `getSidecarStream` resolves via `{ stepSeq, nodeId, stream }` — exactly as landed by P1-T2. No new engine API. | CLI purity rule. |
| D6 | Upward scroll auto-pauses; `F` uppercase resumes. | Features.md §3.5 + mockup §9 verbatim. Destructive-unambiguity rule (§5.6 rule 4): Capital `F` never collides with the lowercase `f` toggle that exists on other panes. |
| D7 | Bind Log tab to `2`, Detail to `d`, no binding for `3`/`4` yet. | Mockup tab-bar header shows Graph/Detail/Log/Events but there is no Graph/Events renderer. `1` is reserved for intra-table row jumps; `3` is overloaded inside the log pane for stream filter per §3.5. `d` is a mnemonic fallback until tab bar UI lands. |
| D8 | Do not interpret `\r` / `\b` / `\t` — render literally. | Interpreting them requires a second render pass + screen buffer model; Ink's diff renderer does not support it cleanly. Documented limit; covered by two tests. |
| D9 | ANSI truecolor + 256-color carry through as structured `AnsiColor` objects rather than pre-mapped Ink palette. | Ink v6 supports both forms via `<Text color="#rrggbb">` and `<Text color={n}>`. Keep the parser neutral; the component renders verbatim. Terminals without truecolor degrade gracefully via Ink. |
| D10 | No timestamps shown by default; `t` toggle acknowledged but deferred. | `step:output` events don't currently carry per-line `ts` in the chunk; only the event envelope has `ts`. We *could* stamp every line inside a chunk with the event `ts`, but that would lie about the timing of later lines in the same chunk. Better to gate the feature behind an explicit follow-up task; the toggle exists in settings but renders nothing until the engine emits per-line timestamps or the pane settles on a "chunk-time" approximation. Keybar shows `t Timestamps` because mockup §8 does; pressing it is a no-op in this task. Documented in component code + tests. |
| D11 | `runsDir` passed as a prop, not context. | One consumer today; a full context is premature; prop-injection keeps the view unit-testable without fixture wrappers. |
| D12 | `LogPaneStatusContext` (the follow/paused bit surfaced to the keybar) is created inside `log-panel-view.tsx` and consumed in `keybar.tsx`; default value is `null`. | Smaller surface than a second reducer action; keybar already consumes other pane-specific ctx (filter-bar open state). |
| D13 | `RESET` on target change, not on every `selectedStepId` tick. | We RESET only when the resolved `{ runDir, stepSeq, nodeId }` triple changes — re-selecting the same step (e.g., via keyboard noise) must not wipe the pane. |
| D14 | Mount ordering: `RESET → sidecar read → ring subscription`. | Ensures the historical backlog is absorbed before live lines merge in; dedup still protects if they arrive interleaved. |
| D15 | `useSidecarStream` is the ONLY hook in this task that touches Node I/O (via `getSidecarStream`). | Keeps the I/O envelope tight and test-overridable. All other log-pane code is pure. |
