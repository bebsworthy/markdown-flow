```ts
// reducer.ts

export const initialAppState: AppState = {
  mode: { kind: "browsing", pane: "workflows" },

  selectedWorkflowId: null,
  selectedRunId: null,
  runsSort: { key: "attention", direction: "desc" },

// ... inside the switch:
case "RUNS_SORT_CYCLE":
  return cycleRunsSort(state);

function cycleRunsSort(state: AppState): AppState {
  const next = cycleSortKey(state.runsSort.key);
  if (next === state.runsSort.key) return state;   // defensive — never fires today
  return { ...state, runsSort: { key: next, direction: "desc" } };
}
```

Direction stays `"desc"` for every cycle step — the columns we cycle through all make sense "biggest/newest first" (attention puts actionables first; started/ended/elapsed want newest first; status ordering puts terminal errors above ok; workflow/id are incidental alphabetical fallbacks users reach for last). ## §4 Attention-first sort ordering

This is the only non-trivial ordering in the module; every other sort key is a lexicographic or numeric compare.

### 4.1 Buckets

```ts
export type AttentionBucket = "active" | "terminal";

function attentionBucket(info: RunInfo): AttentionBucket {
  if (info.status === "running" || info.status === "suspended") return "active";
  return "terminal"; // "complete" | "error"
}
```

Bucket `"active"` always sorts before `"terminal"` regardless of direction. This matches mockups §1 where `▶ running` and `⏸ waiting` runs occupy the top of the table and `✗ failed` / `✓ ok` sit below.

### 4.2 Within-bucket ordering

```
within "active"   : startedAt desc                    (newest active run at top)
within "terminal" : (completedAt ?? startedAt) desc   (most-recently-finished first)
tie-break for both: id asc                            (stable, deterministic)
```

Note: `terminal` uses `completedAt ?? startedAt` because although a terminal run should always have `completedAt`, the fallback keeps the sort total-ordered under engine/replay races.

### 4.3 Pseudocode

```ts
export function attentionCompare(a: RunsTableRow, b: RunsTableRow): number {
  const ab = attentionBucket(a.info);
  const bb = attentionBucket(b.info);
  if (ab !== bb) return ab === "active" ? -1 : 1;

  const aKey = ab === "active"
    ? Date.parse(a.info.startedAt)
    : Date.parse(a.info.completedAt ?? a.info.startedAt);
  const bKey = bb === "active"
    ? Date.parse(b.info.startedAt)
    : Date.parse(b.info.completedAt ?? b.info.startedAt);
  if (aKey !== bKey) return bKey - aKey;   // desc

  return a.info.id < b.info.id ? -1 : a.info.id > b.info.id ? 1 : 0;
}
```

### 4.4 Other sort keys

| Key | Compare |
|---|---|
| `started` | `Date.parse(info.startedAt)` desc, tie-break id asc |
| `ended` | `Date.parse(info.completedAt ?? info.startedAt)` desc, tie-break id asc |
| `elapsed` | `deriveElapsedMs(info, nowMs)` desc, tie-break id asc |
| `status` | ordinal `{running:0, suspended:1, error:2, complete:3}` asc (actionables first), tie-break started desc, then id asc |
| `workflow` | `info.workflowName` asc (case-fold), tie-break id asc |
| `id` | `info.id` asc |

`nowMs` is passed through the sort callsite so every call is deterministic (the component threads `props.nowMs ?? Date.now()` into `sortRows`; tests freeze it).

### 4.5 `cycleSortKey`

```ts
const ORDER: ReadonlyArray<SortKey> = [
  "attention", "started", "ended", "elapsed", "status", "workflow", "id",
];

export function cycleSortKey(current: SortKey): SortKey {
  const i = ORDER.indexOf(current);
  return ORDER[(i + 1) % ORDER.length]!;
}
```

Stateless, total, unit-testable.

### 5.5 `StatusCell` as a structured value

Status is the only column that carries color + glyph, so the column returns a `StatusCell` (not just a string) via `projectStatus`:

```ts
export interface StatusCell {
  readonly glyph: string;      // from theme.glyphs
  readonly label: string;      // e.g. "running", "waiting", "failed", "ok"
  readonly role: ColorRole;    // so the Ink layer picks the color
}
```

The Ink layer then renders:
```tsx
<Text color={theme.colors[role].color}>{cell.glyph} {cell.label}</Text>
```

This keeps the pure column module theme-agnostic (it doesn't import from `theme/`). It only exports the `role` name; the component resolves color via `useTheme()`.

### 5.6 Status label canonicalisation

Engine statuses `"complete" | "error"` map to user-facing labels `"ok" | "failed"` per mockups.md §1 line 20 (`✗ failed`, `✓ ok`). The mapping lives in `derive.ts::runStatusToLabel` so it stays a pure table:

```ts
export function runStatusToLabel(s: RunStatus): string {
  switch (s) {
    case "running":   return "running";
    case "suspended": return "waiting";
    case "complete":  return "ok";
    case "error":     return "failed";
  }
}
```

### 6.3 STEP derivation (`deriveStepLabel`)

The engine's `RunInfo.steps[]` holds `StepResult` entries for each *completed* step. There is no "current step" field. Algorithm:

```ts
export function deriveStepLabel(info: RunInfo): string {
  const steps = info.steps;
  if (steps.length === 0) return "—";

  // Last completed step is the most recent action.
  const last = steps[steps.length - 1]!;

  // If the run is still running, the "current" step is the one we're about
  // to execute — but we don't know that without live engine snapshot state.
  // Fall back to naming the last completed step with a "next" hint.
  if (info.status === "running") {
    // Detect batch progression: run through last two steps' nodes; if they
    // belong to the same batch (prefix "batchId:"), report "N/M".
    const batch = extractBatchProgress(steps);
    if (batch) return `${last.node} ${batch.current}/${batch.total}`;
    return last.node;
  }

  if (info.status === "suspended") return last.node;
  return last.node;   // ok / failed — show the step that finalised the run
}
```

`extractBatchProgress` is a small helper that scans `steps[].node` for batch-item naming (`step#N`). If the engine doesn't produce that naming, it returns `null` and we fall back to a bare node name.

**This is "best-effort projection" and is acceptable because:**
- The mockup shows `build 2/3` — we match that shape when we can.
- When we can't, we render just `build` (no suffix). The column is narrow enough that missing a progress count is invisible at a glance.
- Engine changes that would give us a first-class "currentStep" field are out of scope.

### 6.4 NOTE derivation (`deriveNote`)

```ts
export function deriveNote(info: RunInfo): string {
  if (info.status === "error") {
    const last = info.steps[info.steps.length - 1];
    if (last?.summary) return last.summary;
    if (last?.exit_code && last.exit_code !== 0) {
      return `exit ${last.exit_code} · retries exhausted`;
    }
  }
  if (info.status === "suspended") {
    const last = info.steps[info.steps.length - 1];
    if (last?.summary) return `"${last.summary}"`;
    return "waiting";
  }
  if (info.status === "running") {
    // Retry context — the mockup shows "↻ retrying · delay 4s" but the
    // engine's retry state isn't plumbed through RunInfo. Best-effort: if
    // the last step's exit_code is non-zero but status is "running", we're
    // mid-retry. Without an explicit delay field, render just "↻ retrying".
    const last = info.steps[info.steps.length - 1];
    if (last && last.exit_code != null && last.exit_code !== 0) {
      return "↻ retrying";
    }
    return "";
  }
  return "";   // ok — blank
}
```

Same "best-effort" framing. The mockup's `↻ retrying · delay 4s` is aspirational; we deliver `↻ retrying` without the delay field. **Deferred:** full retry/timeout surface when the engine exposes retry state per-run.

## §10 Theme + hide-don't-grey discipline

- Every glyph (`▶`, `✓`, `✗`, `⏸`, `↻`, `—`, `…`) comes from `theme.glyphs`. No literal glyph chars in any `.tsx`.
- The "horizontal separator" between the footer and the data rows (if any) uses `theme.frame.h`.
- Status colors resolve via `theme.colors[role].color` and `theme.colors[role].dim`. `role` is the value returned by `runStatusToRole(info.status)`.
- **Hide-don't-grey enforcement:**
  - Empty `steps[]` → `STEP` column renders `—`, not greyed.
  - No retry → `NOTE` column is blank, not `"no retry"`.
  - `formatStartedHMS` on a malformed ISO returns `—`, not a greyed warning.
  - Empty table → single `<Text dimColor>no runs yet</Text>` message, no table chrome.
  - Column-drop when narrow: columns are **removed**, not rendered in grey. `pickColumnSet` returns a narrower array; nothing is hidden-but-present.
- **Task brief vs. mockup color conflict.** Plan.md line 376 reads "red for `failed`, green for `ok`, yellow for `running`". mockups.md §1 lines 18–21 show running in blue (default running-role color). **Resolution:** follow mockups + theme tokens (running = blue per `COLOR_TABLE`). The task-brief line uses "yellow" loosely to mean "not red and not green"; flipping `running` to yellow would also change `waiting`'s visual (both become yellow). Documented here so reviewers don't flag it.

## §14 Risks

1. **STEP derivation is best-effort.** Engine's `RunInfo.steps[]` does not expose "current step" or "batch progress" as first-class fields. `deriveStepLabel` infers from the last completed step; for batches it scans node names. If the engine changes node-naming conventions, the `"N/M"` suffix silently stops rendering. **Mitigation:** an explicit unit test pins the current behaviour; a future engine task may expose a `currentStep` field.
2. **NOTE derivation similarly best-effort.** `↻ retrying · delay 4s` from the mockup requires engine retry state that isn't on `RunInfo`. We render `↻ retrying` without the delay. **Mitigation:** documented in §12; a follow-up engine task can extend `RunInfo`.
3. **`nowMs` in elapsed sort + formatting is not live.** Tests freeze it. In production the consumer must thread a refreshing value (P5-T3 ticker). If P5-T3 doesn't refresh, elapsed times will freeze at mount. **Mitigation:** `<RunsTable>` takes `nowMs?: number` so live-updating is an external concern; this keeps the component pure/testable and defers the ticker responsibility.
4. **Column-width drift under monospace fonts.** Widths are char-based; a double-width glyph (e.g. `⏱`, `▶`) theoretically breaks alignment. **Mitigation:** `UNICODE_GLYPHS` were chosen for single-col-width in `theme/glyphs.ts`; `ASCII_GLYPHS` fall back to multi-char bracketed labels which will overflow narrower columns. Tests cover Unicode only; ASCII fallback rendering at narrow tiers is tracked as a follow-up in `test/components/runs-table.test.tsx` with a `it.skip(...)` marker.
5. **Purity-safe cross-slice import.** `reducer.ts` importing `cycleSortKey` from `../runs/sort.js` extends the purity graph slightly. Mitigation: `runs/sort.ts` is scanned by `purity.test.ts`, so the reducer's envelope is preserved. The pattern matches P4-T3's `AddModalTab` type-only import from `../add-modal/types.js`; this extends it to a runtime value import (still pure). No new FORBIDDEN patterns triggered.
6. **Narrow-tier test coverage.** We only snapshot at 140/100/80. Intermediate widths (e.g. 115) might produce unusual layouts. **Mitigation:** `pickColumnSet` threshold tests exhaust the bucket boundaries; intermediate widths re-use the 100-col column set without further surprises because `grow: true` on `note` absorbs the difference.
7. **Task-brief color wording vs. theme.** As called out in §10, the brief says "yellow for running" but theme + mockup say blue. If reviewers insist on the brief, changing `running` to `waiting`'s yellow role would make running visually indistinguishable from suspended. **Mitigation:** documented decision in §10; defer to code review to confirm.
