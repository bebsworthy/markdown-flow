## 2. Design decisions

### 2.1 Two jobs, not a matrix

Per spec: "Separate jobs per workspace so a TUI failure doesn't block CLI releases." Explicit `engine` and `tui` jobs with no `needs:` between them so they run in parallel on independent runners. A matrix (`strategy.matrix.workspace`) would collapse their logs into one job view and — more importantly — share a failure status, which contradicts the "doesn't block" requirement.

### 2.2 One `npm ci` per job, not once shared

GitHub Actions jobs do not share a filesystem. Each job runs its own checkout + `npm ci`. The cost is ~15–30 s of redundant install per run; the benefit is that jobs stay independent and cacheable individually. Caching via `actions/setup-node@v4`'s `cache: 'npm'` keys off `package-lock.json` at the repo root — both jobs hit the same cache entry on warm runs.

### 2.3 Workspace-scoped commands

Each job runs only the scripts for its workspace (`-w packages/<name>`) so a TUI failure produces a red TUI check and a green engine check. Running the root fan-out (`npm run lint`) in either job would couple them.

### 2.4 Node 22.x

Matches local dev (per task brief). `setup-node@v4` with `node-version: '22.x'` picks up the latest 22-line patch on each run — acceptable for a monorepo this size and consistent with Node 22 being the active LTS through 2027.

### 2.5 Concurrency

`concurrency.group: ${{ github.workflow }}-${{ github.ref }}` + `cancel-in-progress: true` cancels a stale run when a new commit lands on the same branch/PR. Standard, saves CI minutes, avoids merge-queue-style pile-ups.

### 2.6 VHS gated behind `workflow_dispatch`

Spec: "Not run in CI yet (gated behind a workflow dispatch)." Interpretation: the `ci.yml` triggers are `push`, `pull_request`, and `workflow_dispatch`, but VHS is **not invoked** inside it. The gated dispatch refers to a future visual-regression workflow. For P2-T3 we only add the tape + README — no VHS workflow file. Document the future dispatch path in `vhs/README.md`.

## 3. `.github/workflows/ci.yml` (new)

```yaml
name: CI

on:
  push:
  pull_request:
  workflow_dispatch:

concurrency:
  group: ${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  engine:
    name: engine (packages/markflow)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22.x'
          cache: 'npm'

      - name: Install dependencies (monorepo, hoisted)
        run: npm ci

      - name: Lint
        run: npm run lint -w packages/markflow

      - name: Test
        run: npm test -w packages/markflow -- --run

      - name: Build
        run: npm run build -w packages/markflow

  tui:
    name: tui (packages/markflow-tui)
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '22.x'
          cache: 'npm'

      - name: Install dependencies (monorepo, hoisted)
        run: npm ci

      - name: Lint
        run: npm run lint -w packages/markflow-tui

      - name: Test
        run: npm test -w packages/markflow-tui -- --run

      - name: Build
        run: npm run build -w packages/markflow-tui
```

**Notes.**

- No `needs:` between jobs → parallel.
- `npm ci` is run at the repo root even though the subsequent commands are workspace-scoped. This is required because the monorepo lockfile + workspaces layout needs the top-level install to link `markflow` into `markflow-tui/node_modules`. Running `npm ci -w packages/<x>` against a hoisted monorepo is **not** equivalent and can leave `node_modules` in an unlinked state.
- `npm test -- --run` forwards `--run` to vitest (one-shot, no watch). This is how the engine tests are invoked throughout plan.md.
- No `FORCE_COLOR`, no `MARKFLOW_TEST` — neither is needed for lint/test/build at this phase. Colour-sensitive tests arrive in Phase 9 (node-pty/tui-test) and will set these at that point.
- No cache key override — default `setup-node` npm cache keys on `**/package-lock.json`, which is exactly what we want.

## 5. `packages/markflow-tui/vhs/scaffold.tape` (new)

```tape
# scaffold.tape — records the empty markflow-tui scaffold.
# Run locally (not in CI yet): vhs scaffold.tape
# Prereq: `npm run build -w packages/markflow-tui` (produces ../dist/cli.js).

Output out/scaffold.gif

Set Shell "bash"
Set FontSize 14
Set Width 1200
Set Height 600
Set TypingSpeed 50ms
Set Theme "Dracula"

Type "node ../dist/cli.js"
Enter
Sleep 1s
Type "q"
Sleep 500ms
```

**Notes.**

- `Output out/scaffold.gif` — path is resolved relative to the tape file's directory (per VHS docs), so the GIF lands in `packages/markflow-tui/vhs/out/scaffold.gif`.
- `Type "node ../dist/cli.js"` — also relative to the tape file, so it runs the tsup-built binary in `packages/markflow-tui/dist/cli.js`. Using `node <path>` rather than `./dist/cli.js` avoids depending on the file's executable bit being set after clone.
- `Sleep 1s` after `Enter` — gives Ink time to mount and render the scaffold frame.
- `Type "q"` + `Sleep 500ms` — triggers the `onQuit` handler (P2-T1 §7) and holds the last frame long enough to end up in the final GIF frame.
- `Set Shell "bash"` — VHS default on Linux is `/bin/sh`; forcing bash matches developer local envs. Inert on macOS where bash is also available.
- No `Env FORCE_COLOR "0"` — the scaffold is monochrome text; colour settings don't affect output.
- Comments at the top document the prereq so someone opening the file cold knows to `npm run build` first.

## 9. Risks / unknowns

1. **`actions/setup-node@v4` + monorepo npm cache.** The default cache key (`cache: 'npm'`) hashes `package-lock.json` at the repo root — correct for a hoisted workspace layout. Verified against setup-node docs. If caching misbehaves (cold key on warm runs), add an explicit `cache-dependency-path: package-lock.json`.
2. **`npm ci` in monorepo.** `npm ci` at the root installs all workspace deps and links `markflow` into `markflow-tui/node_modules`. Running `npm ci -w packages/<x>` would **not** link the sibling — confirmed by npm docs. We stay on the root install.
3. **VHS not installed locally in CI runner.** Not an issue this task — VHS is not invoked from CI. Phase 9 will install it via `charmbracelet/vhs-action`.
4. **Pre-existing `dist/cli.js` in test runs.** `npm run build -w packages/markflow-tui` uses `tsup` with `clean: true` (P2-T1 §5), so any stale file is overwritten. Safe.
5. **`workflow_dispatch` trigger on the CI workflow.** Including it means someone can re-run CI on demand via the Actions UI — harmless and occasionally useful. Does **not** mean VHS runs manually; that's a separate workflow for Phase 9.
6. **`concurrency.cancel-in-progress` vs. protected branches.** Cancelling an in-flight run on `main` is fine — the next push triggers a fresh run. If branch protection requires a green CI status, ensure the replacement run completes before merging. Standard behaviour.
7. **Dracula theme availability.** Built-in VHS theme. Confirmed in VHS repo; no font-family concerns at this phase because the output is GIF (not pixel-diffed).
8. **`Sleep` durations.** 1s mount / 500ms post-quit are generous; if the scaffold ever becomes async-heavy, these may need tuning. Flag for Phase 9 tape work.
