### Spec conflicts surfaced

1. **Plan.md line 294 lists two different `[wait]` labels** — `[wait]` for pending and `[wait]` for waiting. Features.md §5.10 distinguishes them with different glyphs (`⊙` pending vs. `⏸` waiting/approval), so the ASCII table must distinguish them too. See §12 for the resolution (`[wait]` for approval/suspended, `[pend]` for pending; documented explicitly so reviewers don't think it's a typo).
2. Features.md §5.10 prose gives only 5 ASCII samples (`[run] [ok] [fail] [wait] [retry]`); mockups.md §14 adds `[batch]`; plan.md lists 10. We ship the full 10-way ASCII table keyed by `GlyphKey` (§4). Extras beyond the quoted samples are spec-extrapolations for the `timeout`, `skipped`, and `route` states which §5.10 enumerates without giving ASCII variants. Picks are justified per-key in §4.

### Ink API contract (the only consumer)

`<Text>` props (from `ink ^5.0.1`): `color?: string`, `dimColor?: boolean`, `bold?: boolean`, `inverse?: boolean`. `color` accepts chalk-style named colors (`"green"`, `"red"`, `"yellow"`, `"blue"`, `"magenta"`, `"cyan"`, `"gray"`, `"white"`) and hex strings (`"#rrggbb"`). Named colors are preferred — they degrade sanely on 8-color terminals. See §3.4 for the hex-vs-named risk analysis.

## 2. `src/theme/tokens.ts` — color roles

Role names are pinned to the spec keys in §5.10 plus the three non-state roles the task brief calls out (`accent`, `dim`, `danger`). The role keys are the **public API** of this module; the table may evolve (themes / NO_COLOR variant), the keys may not.

```ts
// src/theme/tokens.ts

/**
 * Status colors — paired 1:1 with the glyphs in src/theme/glyphs.ts.
 * Values cite docs/tui/features.md §5.10 verbatim.
 */
export type StatusRole =
  | "pending"   // §5.10: "dim"
  | "running"   // §5.10: "blue"
  | "complete"  // §5.10: "green"
  | "failed"    // §5.10: "red"
  | "skipped"   // §5.10: "dim grey"
  | "waiting"   // §5.10: "yellow"       (approval-pending)
  | "retrying"  // §5.10: "yellow"
  | "timeout"   // §5.10: "red"
  | "batch"     // §5.10: "magenta"
  | "route";    // §5.10: "dim cyan"

/**
 * Non-status color roles. `accent` is for category headers / active
 * selection chrome; `dim` is for de-emphasised chrome; `danger` is for
 * destructive keybar hints ("X Cancel"). These three roles are named in
 * the P3-T3 task brief; see plan.md line 293.
 */
export type ChromeRole = "accent" | "dim" | "danger";

export type ColorRole = StatusRole | ChromeRole;

/**
 * The rendered color value. `undefined` means "use terminal default /
 * inherit" — the correct signal to Ink's <Text> when NO_COLOR is set.
 * `dim` is a structural property, not a hue, but Ink exposes it via
 * `dimColor`; we encode it as a tuple so renderers know to apply both.
 */
export interface ColorSpec {
  readonly color?: string;   // named color, e.g. "green", or undefined
  readonly dim?: boolean;    // applies <Text dimColor>
}

export type ColorTable = Readonly<Record<ColorRole, ColorSpec>>;

/** Full-color table for capable terminals. Named colors only — they
 *  degrade gracefully on 8-color terminals. */
export const COLOR_TABLE: ColorTable = {
  // status
  pending:  { dim: true },                  // "dim"
  running:  { color: "blue" },
  complete: { color: "green" },
  failed:   { color: "red" },
  skipped:  { color: "gray", dim: true },   // "dim grey"
  waiting:  { color: "yellow" },
  retrying: { color: "yellow" },
  timeout:  { color: "red" },
  batch:    { color: "magenta" },
  route:    { color: "cyan", dim: true },   // "dim cyan"
  // chrome
  accent:   { color: "cyan" },
  dim:      { dim: true },
  danger:   { color: "red" },
};

/**
 * Monochrome table — used when NO_COLOR or TERM=dumb is detected.
 * Every role maps to "inherit" (empty ColorSpec). Consumers must still
 * pair the role with a glyph and a text label, per features.md §5.10:
 * "Every state-encoding color is paired with a glyph and a text label
 *  in the detail pane."
 */
export const MONOCHROME_COLOR_TABLE: ColorTable = Object.freeze({
  pending:  {},
  running:  {},
  complete: {},
  failed:   {},
  skipped:  {},
  waiting:  {},
  retrying: {},
  timeout:  {},
  batch:    {},
  route:    {},
  accent:   {},
  dim:      {},
  danger:   {},
});
```

**Purity notes.** No imports from `ink`, `react`, or `node:*`. The file exports *only* types and `const` tables. `Object.freeze` on `MONOCHROME_COLOR_TABLE` is a belt-and-braces guard against accidental mutation.

**Rendering integration (documented, not code in this module).** A consumer renders a state glyph like this:

```tsx

return <Text color={spec.color} dimColor={spec.dim}>{theme.glyphs[key]}</Text>;
```

### 3.4 Role-to-glyph mapping helper

```ts
/** Maps a semantic `ColorRole` to its corresponding `GlyphKey`. */
export function glyphKeyForRole(role: import("./tokens.js").ColorRole): GlyphKey {
  switch (role) {
    case "pending":  return "pending";
    case "running":  return "running";
    case "complete": return "ok";
    case "failed":   return "fail";
    case "skipped":  return "skipped";
    case "waiting":  return "waiting";
    case "retrying": return "retry";
    case "timeout":  return "timeout";
    case "batch":    return "batch";
    case "route":    return "arrow";
    // Chrome roles (accent, dim, danger) have no glyph. Consumers that
    // call this with a chrome role get a compile-error via the `never`
    // exhaustiveness check below.
    case "accent":
    case "dim":
    case "danger":
      throw new Error(`glyphKeyForRole: no glyph for chrome role "${role}"`);
    default: {
      const _exhaustive: never = role;
      return _exhaustive;
    }
  }
}
```

## 4. `src/theme/capabilities.ts` — pure detection

The core decision: one pure function, fully input-driven, so tests can pass a fake env.

```ts
// src/theme/capabilities.ts

/** Runtime capability hints. Kept minimal and immutable. */
export interface Capabilities {
  /** Whether colored output is supported / desired. */
  readonly color: boolean;
  /** Whether unicode glyphs are supported / desired. */
  readonly unicode: boolean;
}

export interface DetectOptions {
  /**
   * Whether stdout is a TTY. Default caller passes `process.stdout.isTTY`.
   * Tests pass an explicit boolean. Defaults to `false` when omitted
   * (the conservative choice — CI / piped output / non-TTY harnesses).
   */
  readonly stdoutIsTTY?: boolean;
}

/**
 * Capability detection precedence (highest to lowest):
 *   1. `MARKFLOW_ASCII=1`            → unicode = false
 *   2. `NO_COLOR` (any non-empty)    → color   = false (https://no-color.org)
 *   3. `TERM=dumb`                   → color   = false
 *   4. Locale heuristic              → unicode = /UTF-?8/i.test(LC_ALL || LANG || LC_CTYPE)
 *                                              && stdoutIsTTY
 *   5. Default                       → color = stdoutIsTTY, unicode = false (conservative)
 *
 * All inputs are read from the `env` argument — never from `process.env`
 * directly — so tests can construct synthetic environments.
 */
export function detectCapabilities(
  env: NodeJS.ProcessEnv,
  opts: DetectOptions = {},
): Capabilities {
  const stdoutIsTTY = opts.stdoutIsTTY ?? false;

  // 1. Hard override for ASCII forcing.
  const forceAscii = env.MARKFLOW_ASCII === "1";

  // 2. NO_COLOR — de facto standard: presence of the var (any non-empty
  //    string) disables color. The spec explicitly says empty string is
  //    treated as "not set" (https://no-color.org/ §"No Color").
  const noColor = typeof env.NO_COLOR === "string" && env.NO_COLOR.length > 0;

  // 3. TERM=dumb — ancient-but-live signal for "no ANSI".
  const dumbTerm = env.TERM === "dumb";

  // 4. Locale heuristic. Pick the first non-empty of LC_ALL > LC_CTYPE >
  //    LANG (POSIX precedence). UTF-8 match is case-insensitive, allows
  //    "utf8", "UTF-8", and "UTF8".
  const locale = env.LC_ALL || env.LC_CTYPE || env.LANG || "";
  const utf8Locale = /UTF-?8/i.test(locale);

  const color = !(noColor || dumbTerm) && stdoutIsTTY;
  const unicode = !forceAscii && utf8Locale && stdoutIsTTY;

  return Object.freeze({ color, unicode });
}
```

**Design notes.**

- **No `process.*` access.** The function is injected with `env` and `stdoutIsTTY`. The single public bootstrap call site (`context.tsx`) does the one unavoidable `process.env` / `process.stdout.isTTY` read and passes them in. This keeps the module test-pure.
- **`stdoutIsTTY` defaults to `false`.** Non-TTY contexts (CI, piped output, test runners) must not get color or unicode by accident. Features.md §5.10 Turborepo #10726 is cited as the cautionary case.
- **`NO_COLOR` treats empty string as "unset"** per the spec. A common mistake is `env.NO_COLOR !== undefined` — we use `length > 0`.
- **UTF-8 locale check is lenient by design.** Matches `en_US.UTF-8`, `en_US.utf8`, `C.UTF-8`, `C.utf8`. Fails on `C`, `POSIX`, or an unset locale. That is correct: we want the conservative default.
- **`MARKFLOW_ASCII=1`** only forces ASCII — it does not disable color. A user who wants both sets `NO_COLOR=1 MARKFLOW_ASCII=1`. This matches the task brief (plan.md line 298 mentions them as independent switches).
- **Return value is frozen** so downstream reducers/memoisation can treat it as value-equal.

**Purity notes.** Only imports `NodeJS.ProcessEnv` as an ambient type (no runtime `node:*` module import). No `react`, no `ink`.

## 5. `src/theme/theme.ts` — `buildTheme`

Composes the three pure modules into a single immutable `Theme` value.

```ts
// src/theme/theme.ts
import type { Capabilities } from "./capabilities.js";
import type { ColorTable } from "./tokens.js";
import type { GlyphTable } from "./glyphs.js";
import { COLOR_TABLE, MONOCHROME_COLOR_TABLE } from "./tokens.js";
import { UNICODE_GLYPHS, ASCII_GLYPHS } from "./glyphs.js";

export interface Theme {
  readonly colors: ColorTable;
  readonly glyphs: GlyphTable;
  readonly capabilities: Capabilities;
}

/**
 * Builds an immutable Theme for the given capabilities. Pure — calling
 * twice with equal caps returns structurally equal Themes (same table
 * references; no per-call allocation for the tables themselves).
 */
export function buildTheme(capabilities: Capabilities): Theme {
  return Object.freeze({
    colors: capabilities.color ? COLOR_TABLE : MONOCHROME_COLOR_TABLE,
    glyphs: capabilities.unicode ? UNICODE_GLYPHS : ASCII_GLYPHS,
    capabilities,
  });
}
```

**Four-quadrant matrix** produced by `buildTheme`:

| `color` | `unicode` | `colors` table          | `glyphs` table   | Typical env                                   |
|---|---|---|---|---|
| true | true  | `COLOR_TABLE`            | `UNICODE_GLYPHS` | local iTerm2 / Ghostty / gnome-terminal        |
| true | false | `COLOR_TABLE`            | `ASCII_GLYPHS`   | `MARKFLOW_ASCII=1` on a normal terminal        |
| false| true  | `MONOCHROME_COLOR_TABLE` | `UNICODE_GLYPHS` | `NO_COLOR=1` on a UTF-8 terminal               |
| false| false | `MONOCHROME_COLOR_TABLE` | `ASCII_GLYPHS`   | CI boxes, `TERM=dumb`, piped output, non-TTY    |

**Purity notes.** Only module-scoped imports from sibling theme files (all pure). No `react`, no `ink`, no `node:*`.

## 6. `src/theme/context.tsx` — React context + hook

The only file in `src/theme/` that imports React/Ink.

```tsx
// src/theme/context.tsx
import React, { createContext, useContext, useMemo } from "react";
import { buildTheme, type Theme } from "./theme.js";
import { detectCapabilities } from "./capabilities.js";

const ThemeContext = createContext<Theme | null>(null);

export interface ThemeProviderProps {
  readonly children: React.ReactNode;
  /**
   * Explicit theme override (primarily for tests and for `:theme` command-
   * palette future use). If omitted, the provider detects capabilities
   * from `process.env` + `process.stdout.isTTY` once at mount and
   * memoises the result.
   */
  readonly value?: Theme;
}

export function ThemeProvider(
  { children, value }: ThemeProviderProps,
): React.ReactElement {
  const theme = useMemo<Theme>(() => {
    if (value) return value;
    // The single unavoidable process-global read. All other theme code
    // is pure / input-driven.
    const caps = detectCapabilities(process.env, {
      stdoutIsTTY: Boolean(process.stdout.isTTY),
    });
    return buildTheme(caps);
  }, [value]);

  return (
    <ThemeContext.Provider value={theme}>{children}</ThemeContext.Provider>
  );
}

/**
 * Hook: returns the active Theme. Throws if used outside <ThemeProvider>
 * — defensive because silent fallback to a default theme would mask bugs
 * and break the NO_COLOR guarantee.
 */
export function useTheme(): Theme {
  const theme = useContext(ThemeContext);
  if (!theme) {
    throw new Error("useTheme: no <ThemeProvider> in the component tree");
  }
  return theme;
}
```

**Why a context + hook instead of a free-standing `getTheme()`?**

- Consistent with the `useEngineAdapter` pattern (P3-T2) — hooks are the established way TUI components read "global" state.
- Test overrides: `<ThemeProvider value={buildTheme({color:false, unicode:false})}>` lets a component test pin a tier without touching `process.env`.
- Future `:theme` command (features.md line 279) has a natural insertion point — swap the provider's `value` prop.

**Purity notes.** `context.tsx` imports from `react`. `src/theme/index.ts` (the barrel) **does not** re-export from `context.tsx` — it exports only the pure surface — so any module that wants React-free access to theme primitives imports from `markflow-tui/src/theme` and gets no React transitively. Components that want the hook import `markflow-tui/src/theme/context.js` directly.

## 12. Risks / unknowns

1. **Duplicate `[wait]` in plan.md line 294.** Surfaced in §0 and §3.3. Resolution: `[pend]` for `pending`, `[wait]` for `waiting`. This is a **proposed spec clarification** — reviewer sign-off requested before code lands. If reviewer wants `[wait]` on both, they will collide visually; a workable compromise would be `[wait·pend]` / `[wait·appr]` but those are ugly, so the `[pend]` resolution is preferred.
2. **§5.10 ASCII samples are a subset.** Features.md §5.10 ASCII prose lists only 5 ASCII labels; mockups.md §14 lists 6. Plan.md line 294 lists 10. The 4 spec-extrapolations are `[skip]`, `[time]`, `[pend]`, `->`. Reviewer sanity-check requested — none of these are controversial, but they are extrapolations.
3. **Hex vs named colors.** All 10 status colors in §5.10 (`blue`, `green`, `red`, `yellow`, `magenta`, `cyan`, `gray`) have Ink-recognised names. No hex needed for MVP. A future high-contrast theme (features.md §5.10 paragraph) may need hex; we leave the `ColorSpec` shape open (`color?: string`) so hex values can be introduced without a type change.
4. **Named color vs "dim grey".** §5.10 uses the phrase "dim grey" for `skipped`. Ink accepts `gray` (or `grey` — both aliases). We pick `gray` (US spelling, matches Ink's canonical form) plus `dim: true`. Documented in the token table.
5. **No-TTY CI defaulting to `color=false, unicode=false`.** Correct but may surprise first-run contributors. `features.md §6.2` already sets expectations; the test suite makes it explicit. If a CI environment *does* want color (e.g., Buildkite with `CI=true && FORCE_COLOR=3`), we can add a `FORCE_COLOR` override later — out of scope for P3-T3 per the task brief.
6. **Windows consoles.** `cmd.exe` historically has no UTF-8 locale variable; `LANG`/`LC_ALL` are usually unset. Result: `unicode=false`, which is the conservative correct answer. Windows Terminal sets `WT_SESSION` and uses UTF-8 but won't set `LANG` — users on Windows Terminal may opt in with `LANG=C.UTF-8` or we document `MARKFLOW_ASCII=0` as a future "force unicode" knob (not in this task).
7. **`process.stdout.isTTY` in the ThemeProvider.** Ink's `render()` sometimes wraps stdout; we read `process.stdout.isTTY` directly. For Ink's `ink-testing-library` the value is typically `undefined`, which coerces to `false` — tests thus get the non-TTY tier by default. Tests that want a specific tier pass `value={buildTheme(…)}` explicitly.
8. **`useMemo` deps in `ThemeProvider`.** Only `[value]` is a dep — we intentionally do *not* re-detect on every render if `value` is `undefined`. A one-time capability read at mount is correct: users do not change locale mid-session. If future work adds a `:theme` reload command, it flips the `value` prop; the memo invalidates naturally.
9. **`glyphKeyForRole` throw vs silent fallback.** Throwing on chrome roles (accent/dim/danger) is defensive — callers that want a generic "render this role" helper would prefer a silent `undefined`. Flagged for review; current choice prioritises finding bugs at dev time over graceful prod behaviour.
10. **`Object.freeze` cost.** Negligible (once at module load) and gives us runtime immutability in addition to `Readonly<...>` types. Keep.
11. **Barrel exclusion of `context.tsx` is a deliberate constraint.** Some teams re-export everything from `index.ts` as a matter of style. Doing that here would pull React into any consumer of theme data, violating purity. A comment at the top of `index.ts` documents this. If the barrel rule is later relaxed, we must re-verify the P3-T2 purity boundary for the hook (`useEngineAdapter`), which currently imports from `../engine/adapter.js` — a sibling that could become React-dirtied by accident.
