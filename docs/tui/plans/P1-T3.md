## 0. Summary

Add `tailEventLog(runDir, fromSeq, options?)` to the public API. It returns an
`AsyncIterable<EngineEvent>` that first drains any events already on disk with
`seq >= fromSeq`, then follows `events.jsonl` via `fs.watch`, yielding each
newly-appended event until either a terminal event is observed or the consumer
aborts (via `AbortSignal`) / `break`s out of the `for await`. No writer-side
changes.

The TUI log pane consumes this in Phase 6; nothing in this task wires it up.

## 1. Grounding — what the writer actually does

From `/Users/boyd/wip/markdown-flow/packages/markflow/src/core/event-logger.ts`
lines 44–63:

```ts
append(payload: EngineEventPayload): Promise<EngineEvent> {
  const stamped = { ...payload, seq: ++seq, ts: now() } as EngineEvent;
  if (NON_PERSISTED_EVENT_TYPES.has(payload.type)) {
    return Promise.resolve(stamped);
  }
  const line = JSON.stringify(stamped) + "\n";
  tail = tail.then(() => appendFile(filePath, line, "utf-8"));
  return tail.then(() => stamped);
}
```

**Key properties this plan relies on:**

- **Atomic append unit = one whole line.** `JSON.stringify(stamped) + "\n"` is a
  single `appendFile` call. Node's `fs.appendFile` opens, `write`s, closes — on
  a local filesystem the line body and its trailing `\n` are passed to the
  kernel together. We treat "line ends in `\n`" as the completeness signal;
  nothing else is safe.
- **Serialization.** All appends are chained through a single `tail` promise,
  so on-disk order matches `seq` order (confirmed by
  `packages/markflow/test/core/event-logger.test.ts` lines 34–56).
- **Gaps are legal.** `step:output` claims a `seq` but is not persisted
  (`NON_PERSISTED_EVENT_TYPES`); the tail MUST tolerate a missing `seq` and
  not wait for it. We treat `seq` as "monotonically increasing within the
  persisted stream" — same contract `replay()` uses
  (`packages/markflow/src/core/replay.ts` lines 36–43).
- **File existence.** `RunManager.createRun` touches `events.jsonl` to empty
  string before `run:start` is ever appended
  (`packages/markflow/src/core/run-manager.ts` line 76). So inside an active
  run directory the file always exists. For cold start before `createRun`
  (or against a bare directory) the tail must still handle absence.

Reference reader we mirror for line-splitting semantics:
`packages/markflow/src/core/replay.ts` lines 295–326 (`readEventLog`). Key idea
reused: split on `\n`, drop the trailing empty entry if the buffer ended with
`\n`, and treat a non-terminated tail as "not yet complete" — in our streaming
case we keep it in the buffer instead of discarding.

## 2. Signature and exports

### 2.1 Signature (exact)

```ts
export function tailEventLog(
  runDir: string,
  fromSeq: number,
  options?: { signal?: AbortSignal },
): AsyncIterable<EngineEvent>;
```

Implemented as an `async function*` — the returned `AsyncIterableIterator` is a
structural supertype of `AsyncIterable`, so the declared return type is
satisfied without an explicit wrapper. Consumers use:

```ts
for await (const evt of tailEventLog(runDir, lastSeq + 1, { signal })) { ... }
```

### 2.2 Semantics pinned

- **`fromSeq` is inclusive.** The first event yielded has `seq >= fromSeq`.
  (Task spec wording "from `fromSeq` onward" is ambiguous; inclusive matches
  the natural consumer call site `tailEventLog(runDir, lastSeq + 1)` and the
  existing `seq` semantics in `replay()`.)
- **Order preserved.** Yielded `seq` values are strictly increasing. Gaps are
  possible (non-persisted types) and are passed through untouched.
- **No duplicates.** Driven by a single "last-yielded byte offset" watermark
  inside the generator; we never re-open from the beginning after the initial
  drain.
- **Terminal events yielded.** The terminal event itself is yielded, then the
  generator returns. See §4.
- **Abort is clean.** On `options.signal.aborted` (or `break` in `for await`)
  the generator's `finally` block closes the open file descriptor, closes the
  `fs.watch` handle, and returns — no throw, no leaked handles.

## 3. File handling

### 3.1 Path

```ts
import { join } from "node:path";
const filePath = join(runDir, "events.jsonl");
```

### 3.2 File-may-not-exist path

Two scenarios:

1. Caller is attaching to an already-running run → file exists (the
   `RunManager.createRun` touch guarantees this).
2. Caller is attaching **before** the run has started → file does not yet
   exist, but `runDir` probably does.

Strategy:

- **Subscribe to the parent directory first** via `fs.watch(runDir, { recursive: false })`,
  **then** probe for `events.jsonl` with `fs.access`. If present, proceed to
  the drain phase. If absent, wait for a `rename`/`change` event on the watch
  whose filename is `events.jsonl`, then proceed.
- Ordering (watch-then-probe) closes the race where the file would be created
  between our existence check and our watch subscription and we'd miss the
  only create event.
- **`runDir` missing.** If `runDir` itself does not exist, `fs.watch` on it
  throws `ENOENT`. We surface this as a thrown `Error` — it's a programmer
  error (caller passed a bad run id). Not polled.
- **No timeout.** The function blocks on the file appearing indefinitely; the
  consumer controls timeout by wiring an `AbortSignal` to their own timer.
  Documented in the JSDoc.

### 3.3 Partial-line safety

Buffer-and-split, one pass per wake-up:

```ts
let fd = await fs.promises.open(filePath, "r");
let offset = 0;              // bytes consumed from file
let carry = "";              // incomplete line carried across reads

async function drain(): Promise<EngineEvent[]> {
  const stat = await fd.stat();
  if (stat.size <= offset) return [];
  const buf = Buffer.alloc(stat.size - offset);
  const { bytesRead } = await fd.read(buf, 0, buf.length, offset);
  offset += bytesRead;
  const text = carry + buf.toString("utf-8", 0, bytesRead);
  const parts = text.split("\n");
  carry = parts.pop() ?? "";  // last element is either "" (line ended in \n)
                              // or a partial line we'll resume with.
  const out: EngineEvent[] = [];
  for (const line of parts) {
    if (line.length === 0) continue; // defensive
    out.push(JSON.parse(line) as EngineEvent);
  }
  return out;
}
```

- `carry` holds any trailing bytes that have no terminating `\n` yet; we do
  **not** attempt to `JSON.parse` them until the next `drain()` appends more
  bytes and the `\n` arrives.
- `offset` is the watermark. Next read always starts where the previous one
  ended — this is what prevents duplicates even if `fs.watch` fires twice.
- A `JSON.parse` failure on a non-final line (i.e. we already have a trailing
  `\n` but the line does not parse) is genuine corruption; re-throw as-is.
  Matches `readEventLog`'s `TruncatedLogError` semantics.

### 3.4 Single-pass drain per watch event

On every `fs.watch` notification we call `drain()` once and emit the events
produced. No debouncing. We also call `drain()` once at startup after opening
the fd, to pick up everything already written.

## 4. Termination criteria

### 4.1 Terminal event kinds

From `/Users/boyd/wip/markdown-flow/packages/markflow/src/core/types.ts` lines
304–305 — the `EngineEventPayload` union has exactly two terminal types:

```ts
| { type: "workflow:complete"; results: StepResult[] }
| { type: "workflow:error"; error: string }
```

There is no `run:cancelled` kind. Cancellation surfaces as `workflow:error`
with a cancellation message; suspended/waiting runs remain non-terminal (they
end on a `step:waiting` event but the run is still live and can be resumed).

The tail treats `workflow:complete` and `workflow:error` as terminal. The
terminal event is **yielded first**, then the generator returns on the next
iteration. A suspended run (last event is `step:waiting`) is **not** treated
as terminal: the tail continues watching, which is the right behaviour if the
run is later resumed into the same directory.

### 4.2 Exit points

The generator exits (clean `return`, no throw) when **any** of:

- The last yielded event's type is `workflow:complete` or `workflow:error`.
- `options.signal?.aborted === true` is observed between yields.
- The consumer `break`s out of the `for await`, which triggers the
  generator's `finally`.

On exit, the `finally` block must:

1. Abort the in-flight `fs.watch` iterator (its `AbortController.abort()`).
2. `close()` the open file descriptor.
3. Resolve any pending "file appeared" promise so the initial-wait code path
   unblocks.

### 4.3 Meta-based fast path (optional, documented but not required for MVP)

`meta.json` (see `run-manager.ts`) caches terminal status. We could shortcut:
if at call time `meta.json` shows status `complete`/`error`, read the file
to EOF synchronously and return without ever installing a watcher. Recommended
as a follow-up optimization but not in scope for this task — the watch-based
path is correct in all cases.

## 5. Watch vs polling

Use **`fs.watch`** with documented mitigations — no new dependencies.

- `fs.watch(runDir, { recursive: false })` is the sole subscription. It
  surfaces both the `events.jsonl` create (if we attached before the run
  started) and subsequent modifications. Using the parent directory avoids
  the macOS-specific problem of `fs.watch` on a not-yet-existing file path.
- **Always re-stat before reading.** macOS FSEvents occasionally coalesces
  or drops events. Mitigation: our drain loop keys off `stat.size > offset`,
  not the number of watch events. If two writes coalesce into one event, we
  still read both; if one is dropped but the next one fires, we catch up.
- **No debounce.** Latency matters; the cost of a no-op drain (one `fstat`
  call) is negligible.
- Use a Node >=18 async iterator bridge:
  ```ts
  const ac = new AbortController();
  const watcher = fs.promises.watch(runDir, { signal: ac.signal });
  for await (const evt of watcher) { ... }
  ```
  Aborting `ac` (in the `finally`) releases the underlying libuv handle.
- **Filter.** Ignore events whose `filename` is not `events.jsonl`. (macOS
  sometimes reports `null` filenames; in that case do a defensive drain
  rather than ignore.)

### 5.1 AbortSignal wiring

Two internal `AbortController`s:

- `watchAC` — passed to `fs.promises.watch`; aborted in `finally`.
- `externalAC` — a listener on `options.signal` that, on abort, aborts
  `watchAC` and wakes any pending "file appeared" promise.

If `options.signal` is already aborted at entry, return immediately (no yields,
no fd open).

2. **Attach before any events exist.** `events.jsonl` absent but `runDir`
   exists. Start the tail in the background. After a 20ms delay touch

3. **Writer appends while consumer is awaiting.** Start consuming (suspended
   at the first `await`), write an event, read it, write another, read it.
   Assert both arrive and each arrives within a small timeout
   (e.g. 200ms) of its write.
4. **Skips events below `fromSeq`.** Write events with `seq` 1..10; call
   `tailEventLog(runDir, 5)`. Collect up to the first event and assert its
   `seq === 5`. Then write `workflow:complete` to terminate.
5. **Never duplicates.** Write 20 events + terminal. Fully consume. Assert the
   set of observed `seq` values has size 21 (20 + terminal) and is strictly
   increasing.
6. **Abort cleanly.** `new AbortController()`, pass `signal`, write two
   events, consume them, call `ac.abort()`. Assert the `for await` finishes
   within 100ms and no error is thrown. Assert the run directory's file
   descriptor count (via a `setImmediate` + manual `fd` count heuristic, or
   a sentinel spy on `fs.watch`) returns to baseline.
7. **Break exits cleanly.** Same as #6 but the consumer `break`s after the
   first event instead of aborting. Assert no uncaught error, assert
   `watchAC` was aborted (spy via `jest.fn`-equivalent on
   `AbortController.prototype.abort` or a wrapper).
8. **Partial-line handling.** Using raw `fs.writeFileSync(..., { flag: "a" })`
   write the full JSON body of one event **without** the trailing `\n`, wait
   20ms, assert no event has been yielded yet, then append `\n` plus a full
   second event. Assert both events now arrive.
9. **Run already terminated when attached.** Pre-write 3 events ending in
   `workflow:complete`. Call `tailEventLog(runDir, 1)`. Assert all 3 events
   yielded, generator returns, no watcher left installed (check by writing
   a stray event after and confirming the iterator has already completed).
10. **`fromSeq` past the end.** Pre-write events 1..3 (non-terminal). Call
    `tailEventLog(runDir, 10)`. Assert no events yielded until a new event
    with `seq >= 10` arrives; then assert it does arrive.
11. **Already-aborted signal.** Pre-write events. Pass an already-aborted
    `signal`. Assert zero events yielded and the generator returns
    immediately.

in §5 is documented behaviour, not tested explicitly. The "never duplicates"
and "always drain on size delta" invariants protect against it regardless.

## 8. Risks & open questions

- **`fs.watch` on macOS.** Known to occasionally drop/coalesce events. Fully
  mitigated by the "drain to EOF on every wake-up + size watermark" loop.
- **Watch-vs-create race.** Mitigated by subscribing to `runDir` before
  probing for `events.jsonl` existence (§3.2).
- **Concurrent resume.** `docs/arch/event-sourced-run-log.md` §"Concurrent
  resume" notes that two processes resuming the same run can race on append
  order. The tail is a **reader only** — it doesn't exacerbate the problem,
  but in such a race the tail may yield events with non-sequential `seq`. We
  do not validate `seq` ordering; we simply pass it through. Downstream
  consumers (replay) catch inconsistencies.
- **Symlinked `runDir`.** `fs.watch` with recursive behaviour on symlinks is
  flaky. Not in scope — document as unsupported.
- **No terminal event ever appended.** If the process dies mid-run, the tail
  will never see `workflow:complete`/`workflow:error` and will wait forever.
  The consumer is expected to bound the wait via `AbortSignal` (doc'd in
  JSDoc). This is the same invariant the TUI will rely on.
- **Very large existing log.** On cold start we read `stat.size` bytes in one
  `Buffer.alloc` + `fd.read`. For multi-MB logs this is fine; for hundreds of
  MB it's a memory blip. Acceptable at current scale — a chunked read loop
  is a trivial follow-up if ever needed.

No blockers. All required semantics (path, atomicity unit, terminal kinds)
were derivable from the current code.
