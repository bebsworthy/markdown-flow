## §0 Starting state

### 0.1 Engine surface is already frozen

- **`RunInfo`** (`packages/markflow/src/core/types.ts` lines 175–183) holds `{id, workflowName, sourceFile, status, startedAt, completedAt?, steps: StepResult[]}`. `steps` is an ordered array of **completed** `StepResult`s only — no "currently running" field.
- **`StepResult`** (same file, lines 162–171) is `{node, type, edge, summary, local?, started_at, completed_at, exit_code}`. No `attempt`, no `parent`, no `batch` columns. Children of a fan-out only show up here as independent entries; fan-out parentage lives in `Token.parentTokenId` on the *snapshot*, not on the RunInfo.
- **`Token`** (lines 149–160) is the key type for this task: `{id, nodeId, generation, state: TokenState, edge?, result?, batchId?, itemIndex?, parentTokenId?, itemContext?}`. **`TokenState`** (line 147) is `"pending" | "running" | "complete" | "skipped" | "waiting"` — matches the five glyph slots mockups §4/§6 demand minus `retrying` (which is a NOTE-column annotation, not a token state).
- **`EngineSnapshot`** (lines 380–387) is `{tokens: Map<string,Token>, retryBudgets: Map<string,{count,max}>, globalContext, completedResults, status, batches: Map<string,BatchState>}`. `retryBudgets` is keyed `"nodeId:label"`; `batches` is keyed by `batchId`.
- **`BatchState`** (lines 348–370) is `{nodeId, expected, completed, succeeded, failed, onItemError, itemContexts, results, done, status?}`. The fields `expected`/`completed`/`succeeded`/`failed` are exactly what the §4 / §6 aggregate row needs (`2 / 3` + `1 retry · 0 failed` suffix).
- **`step:retry` event** (lines 269–276) is `{type, nodeId, tokenId, attempt (1-indexed), delayMs, reason}`. `attempt` is the 1-indexed attempt that will run *after* the delay; `delayMs` is the absolute pause length, not a countdown. `replay()` treats `step:retry` as a pure notification (no snapshot mutation — see `packages/markflow/src/core/replay.ts` line 103). **This means the snapshot alone does NOT carry retry-countdown state** — we need to track it either (a) from the latest `step:retry` event in the ring buffer (`EngineState.activeRun.events`) or (b) from the raw event stream as it flows through. See §6 and §12.
- **`tokensByBatch(snapshot, batchId)`** (`packages/markflow/src/core/queries.ts`) is exported from `markflow` (re-export on `src/core/index.ts` line 21). Good for aggregate-row derivation though `BatchState.succeeded/failed/completed` already cover what we need without scanning tokens.
- **`getFanOutTargets(graph, nodeId)`, `getOutgoingEdges`, `getUpstreamNodes`, `isMergeNode`, `getTerminalNodes`** all live in `packages/markflow/src/core/graph.ts`. `markflow`'s public re-export (`src/core/index.ts` lines 14–21) exposes `getTerminalNodes`, `getUpstreamNodes`, `isMergeNode`, `getIncomingEdges`, `getOutgoingEdges` — **not `getFanOutTargets`**. If we need `getFanOutTargets` we either (i) compute it locally (trivial — `Array.from(new Set(getOutgoingEdges(g,n).map(e=>e.to)))`), or (ii) accept the public subset and lean on `getOutgoingEdges` directly. **Decision (§3.3):** compute locally; no engine edit.
- **Mermaid `FlowGraph`** is NOT on `RunInfo` or `EngineSnapshot`. It lives on `WorkflowDefinition`, which the engine loads from the sourceFile at execute time. The TUI has no current path from `RunInfo.sourceFile` to a parsed `FlowGraph` in the active-run slice. **Implication:** parent/child inference for P6-T1 must come from **`Token.parentTokenId` + `Token.batchId`**, not from the graph. Mockup §4 shows `fan-out` as an explicit node that has children indented under it — the engine marks those children with `parentTokenId` pointing at the fan-out's token id. See §3.

### 2.1 `StepStatus`

```ts
/**
 * User-facing status label for a step row. Derived from `TokenState` with
 * an added `"retrying"` overlay (not a real token state — fed by the
 * latest `step:retry` event for the row's token).
 *
 * Maps 1:1 onto `ColorRole` via `tokenStateToRole` and a glyph via
 * `tokenStateToGlyphKey` — both in `src/steps/derive.ts`.
 */
export type StepStatus =
  | "pending"
  | "running"
  | "complete"
  | "skipped"
  | "waiting"
  | "retrying"
  | "failed";
```

**Why `"failed"` is separate from the `TokenState` union.** The engine's `TokenState` doesn't have `failed` — a failure is a `complete` token whose `result.edge === "fail"` (or `"fail:max"`). Mockups §6 shows `✗ failed` explicitly for `deploy-us`. So `StepStatus` synthesises `"failed"` from `(state === "complete") && (result.edge.startsWith("fail"))`. **Derivation rule** is centralised in `tokenStateToStatus(token)` — see §5.3.

## §3 Tree construction

Location: `src/steps/tree.ts`. Pure over a minimal `StepsSnapshot` shape so tests don't need a full engine snapshot:

```ts
export interface StepsSnapshot {
  readonly tokens: ReadonlyMap<string, Token>;
  readonly retryBudgets: ReadonlyMap<string, { count: number; max: number }>;
  readonly completedResults: ReadonlyArray<StepResult>;
  readonly batches: ReadonlyMap<string, BatchState>;
}
```

Adapter in the view component transforms `{info, events}` → this shape (we don't fold the full engine `replay()` here — we project just what the step table needs; see §8).

### 3.1 Input

`buildStepRows(snapshot, info, nowMs, retryHints, opts): StepRow[]`

`opts = { collapseThreshold: number }` — defaults to `BATCH_COLLAPSE_THRESHOLD` (see §4).

### 3.2 Algorithm

1. **Filter tokens into aggregatable vs renderable.** A token participates in an aggregate iff `token.batchId != null` AND that batch's `expected >= collapseThreshold` AND `batches.get(batchId).done !== true` OR (done && aggregate-mode preserved even when done — mockup §6 shows the aggregate row after terminal). **Decision:** always aggregate when `expected >= threshold`, regardless of done-state. Sub-threshold batches (e.g. 2 children under a fan-out) render flat. This keeps §4 / §6 parity: `⟳ batch [regions]` appears in both.
2. **Identify roots vs children.** A "root" is any token with no `parentTokenId`. A "child" has a `parentTokenId`. Fan-out children also have `parentTokenId`. This is pure graph-free: `parentTokenId` is authoritative.
3. **Group children by parent.** `indexByParent(tokens)` builds `Map<parentId, Token[]>`. Children within the same parent are ordered by earliest `started_at` (read from matching `completedResults` entry when present; else fall back to `token:created` order, which we approximate by insertion order in the `Map<tokenId, Token>`).
4. **Depth assignment.** `depth = 0` for root; `depth = parent.depth + 1` for each child. Tokens deeper than 2 continue indenting; mockups don't show depth > 2 but the algorithm is not specialised for 2 levels — arbitrary depth supported.
5. **Ordering within a depth.**
   - Roots: **topological + started-asc.** Topological order means a root's index cannot precede any of its upstream ancestors' indices. Without a `FlowGraph` in the TUI we don't have a canonical topo sort; we approximate via the order `completedResults[]` appears in `info.steps` PLUS any remaining non-complete tokens in insertion order (they are strictly downstream of anything already complete). Pending tokens that share no upstream ordering with running tokens fall in token-insertion order.
   - Children: ordered by earliest `started_at`, ties broken by `itemIndex` (for batch siblings), then `tokenId` asc.
6. **Emit rows.** Depth-first traversal. For each token:
   - If it participates in an aggregated batch AND the current parent has NOT yet emitted an aggregate row for `batchId`, emit the aggregate row (kind `"batch-aggregate"`) and mark `batchId` seen. Skip every subsequent token in the same batch (they live on the aggregate row).
   - Otherwise, emit a `"leaf"` row via `toLeafRow(token, depth, info, retryHints, nowMs)` (see §5).
7. **"Seen batches" set** is passed through the recursion so each batch is aggregated exactly once even if its tokens span multiple parents (unlikely for `forEach` but safe to guard).

### 3.3 Fan-out parent/child example

Mockup §4 lines 160–167:

```
fan-out       ▶ running  —   30s  2 children active
  deploy-us   ▶ running  2/3 21s  ↻ retrying in 4s
  deploy-eu   ▶ running  1/3 18s
⟳ batch [regions] ▶ running — 25s 2 / 3 ...
```

- `fan-out` is a normal token with `parentTokenId` unset — root. `deploy-us` and `deploy-eu` both have `parentTokenId = <fan-out's token id>` — children, depth 1.
- Neither `deploy-us` nor `deploy-eu` has `batchId` set (they're the fan-out branches, not a `forEach` batch). The `regions` batch is a separate token set whose parent is the forEach source (not shown as a row itself because it's a source node above the collector).
- Aggregate row for `batch [regions]` sits at depth 0 because the `forEach` source's tokens aren't children of `fan-out` — they're roots generated by the `forEach` edge. That's consistent with mockup §4 which places `⟳ batch [regions]` at parent depth.

### 3.4 Pseudocode

```ts
export function buildStepRows(
  snapshot: StepsSnapshot,
  info: RunInfo,
  nowMs: number,
  retryHints: ReadonlyMap<string, RetryHint>,
  opts?: { collapseThreshold?: number },
): ReadonlyArray<StepRow> {
  const threshold = opts?.collapseThreshold ?? BATCH_COLLAPSE_THRESHOLD;
  const rows: StepRow[] = [];
  const seenBatches = new Set<string>();
  const childrenByParent = indexByParent(snapshot.tokens);
  const roots = orderRoots(snapshot, info);

  function visit(token: Token, depth: number) {
    // Batch aggregation check
    if (token.batchId != null) {
      const b = snapshot.batches.get(token.batchId);
      if (b && b.expected >= threshold) {
        if (seenBatches.has(token.batchId)) return;
        seenBatches.add(token.batchId);
        rows.push(
          aggregateBatchRow(b, token.batchId, depth, snapshot, retryHints, nowMs),
        );
        return;
      }
    }
    rows.push(toLeafRow(token, depth, info, snapshot, retryHints, nowMs));
    const children = childrenByParent.get(token.id) ?? [];
    for (const c of children) visit(c, depth + 1);
  }

  for (const root of roots) visit(root, 0);
  return rows;
}
```

### 3.5 Determinism

The tree is a **pure function of `(snapshot, info, nowMs, retryHints)`**. No global clock reads, no iteration-order nondeterminism (tokens Map iteration order is insertion order in V8 — deterministic given the same event stream). Tests freeze `nowMs` and use fixture maps; the purity probe scans `tree.ts`.

## §4 forEach batch aggregation

Location: `src/steps/aggregate.ts`.

### 4.1 Threshold

```ts
export const BATCH_COLLAPSE_THRESHOLD = 6;
```

Rationale: mockup §4 aggregates `batch [regions]` which has `expected = 3` — **yet the mockup still shows it as an aggregate row.** This tells us the threshold in the mockup is effectively 0 (always aggregate). But the task brief reads *"children count > collapse threshold"* which implies a threshold exists. **Decision:** set the default threshold to `0` (aggregate every batch) AND expose the threshold as an option so tests and future UX tweaks can raise it. Sub-threshold batches would render their children inline as individual tokens.

Wait — re-read. The task body says "when child count > collapse threshold" — so there IS a threshold. Mockup §4's `2 / 3` for 3 regions means 3 total items. If threshold is 0, all batches aggregate. If threshold is 5 and items = 3, they would render inline. **Mockup §4 shows the aggregate anyway for expected = 3** — which suggests `threshold <= 2`. **Final decision:** `BATCH_COLLAPSE_THRESHOLD = 2` (aggregate any batch with ≥ 2 items — i.e. every real forEach since single-item batches are degenerate). This aligns with §4 and leaves the threshold semantics future-proof. Document in the plan comment.

### 4.2 Aggregate-row projection

```ts
export function aggregateBatchRow(
  batch: BatchState,
  batchId: string,
  depth: number,
  snapshot: StepsSnapshot,
  retryHints: ReadonlyMap<string, RetryHint>,
  nowMs: number,
): StepRow {
  const aggregate = toBatchAggregate(batch, batchId, snapshot, retryHints);
  const elapsedMs = deriveAggregateElapsedMs(batch, snapshot.tokens, nowMs);
  const status = deriveAggregateStatus(batch);
  return {
    id: `batch:${batchId}`,
    kind: "batch-aggregate",
    depth,
    label: `⟳ batch [${batch.nodeId}]`,    // glyph swapped by Ink layer for ASCII fallback
    status,
    attempt: "—",
    elapsed: formatStepElapsed(elapsedMs),
    elapsedMs,
    note: formatAggregateNote(aggregate),
    aggregate,
    role: aggregateRole(status),
    glyphKey: "batch",
    nodeId: batch.nodeId,
  };
}
```

### 4.3 Columns for aggregate rows

| Column | Aggregate value |
|---|---|
| STEP | `⟳ batch [<nodeId>]` (label field on row; STEP column's `projectText` just returns `row.label`) |
| STATUS | glyph/label per `deriveAggregateStatus(batch)`:<br/>• `!done && failed === 0` → `"running"`<br/>• `!done && failed > 0` → `"running"` (mockup §4 shows still-running even with a retry; fail latches only after `done`)<br/>• `done && status === "ok"` → `"complete"`<br/>• `done && status === "error"` → `"failed"` |

### 4.4 Progress bar + NOTE composition

Mockup §4 NOTE for the `batch` row: `"2 / 3   ██████░░░   1 retry · 0 failed"`.
Mockup §6 NOTE for the `batch` row: `"1 / 3   ███░░░░░░   1 ✗ · 0 ⏸"`.

Spec the NOTE as:

```
<completed> / <expected>   <bar>   <retry-summary>
```

Where:
- `<bar>` is `aggregate.ts::formatProgressBar(completed, expected, barWidth)`. Default `barWidth = 9` matches mockup; exposed as an option on the column so narrow tiers can shrink.
- `<retry-summary>` is:
  - running: `"${retries} retry · ${failed} failed"` (pluralise `retry`/`retries` — keep lowercase per mockup).
  - terminal ok: `""` (mockup §6 omits the retry suffix when batch terminal is shown plainly — actually §6 uses `1 ✗ · 0 ⏸`, a failed/waiting glyph summary when terminal with errors). **Decision:** running → retries+failed count; terminal → `"<failed-glyph> failed · <waiting-glyph> waiting"` style. Both drawn from `theme.glyphs`.

`formatProgressBar`:

```ts
export function formatProgressBar(completed: number, total: number, width: number): string {
  if (width <= 0 || total <= 0) return "";
  const ratio = Math.max(0, Math.min(1, completed / total));
  const filled = Math.round(ratio * width);
  return FILLED.repeat(filled) + EMPTY.repeat(width - filled);
}

const FILLED = "█";     // UNICODE
const EMPTY  = "░";
```

ASCII fallback glyphs: `#` / `.` (picked from existing `theme.glyphs`-adjacent conventions; we DO NOT add new glyph keys — the progress bar is a block element, not a status). **Decision:** add a new glyph pair `progressFilled` / `progressEmpty` to `theme/glyphs.ts` — this IS a new glyph addition (no status-glyph addition, but block chars). See §12.

### 4.5 `retries` count

We don't have per-batch retry budget data. `retries` is computed from the retry-hint map as "count of distinct tokenIds whose most-recent `step:retry` hint is still live AND whose token has `batchId === this.batchId`". Plus any tokens that completed after a retry (which we can only detect if they had a `step:retry` hint before their `step:complete` — the hint is cleared on complete; so running-mid-retry is the only live count). **Best-effort:** `retries` in the `BatchAggregate` reflects *in-flight* retries, not historical ones. Mockup §4 shows `1 retry` — plausibly one in-flight retry on one item. Good enough for MVP; documented in §13.

### 4.6 Risk: `completed` includes retried attempts?

Per the engine's `batch:item:complete` event, `completed++` only fires once per item (when the item terminates on any edge — see `packages/markflow/src/core/replay.ts` line 190). So `completed` is "items that are done," retries included only if the retry eventually resolved. Consistent with mockup.

## §6 Retry countdown

Location: `src/steps/retry.ts`.

### 6.1 Data source

`step:retry` events (`EngineEventPayload` lines 269–276):

```
{ type: "step:retry", nodeId, tokenId, attempt, delayMs, reason, seq, ts }
```

`ts` is ISO-8601 from the engine's wall clock at emit time. `delayMs` is the intended pause; `attempt` is the 1-indexed attempt that will run after the pause.

**Critical gap.** `replay()` ignores `step:retry` for snapshot state (see §0.1) — the `EngineSnapshot` handed to the view does NOT include retry countdown data. To show `↻ retrying in 4s` the view must observe the **raw event stream**. The TUI slice already preserves up to 500 events on `activeRun.events` (P3-T2) — we fold those into a `RetryHintMap`.

### 6.2 `applyRetryEvent`

Pure reducer over events. Called by `step-table-view.tsx` in a `useMemo` keyed on `activeRun.events` identity:

```ts
export type RetryHintMap = ReadonlyMap<string, RetryHint>;

export function applyRetryEvent(
  hints: RetryHintMap,
  event: EngineEvent,
): RetryHintMap {
  switch (event.type) {
    case "step:retry": {
      const next = new Map(hints);
      next.set(event.tokenId, {
        tokenId: event.tokenId,
        nodeId: event.nodeId,
        attempt: event.attempt,
        scheduledAtMs: Date.parse(event.ts),
        delayMs: event.delayMs,
        reason: event.reason,
      });
      return next;
    }
    case "step:start":
    case "step:complete": {
      if (!hints.has(event.tokenId)) return hints;
      const next = new Map(hints);
      next.delete(event.tokenId);
      return next;
    }
    default:
      return hints;
  }
}

export function buildRetryHints(
  events: ReadonlyArray<EngineEvent>,
): RetryHintMap {
  let hints: RetryHintMap = new Map();
  for (const e of events) hints = applyRetryEvent(hints, e);
  return hints;
}
```

### 6.3 Countdown formatter

```ts
export function formatRetryCountdown(
  hint: RetryHint,
  nowMs: number,
): string | null {
  const endsAt = hint.scheduledAtMs + hint.delayMs;
  const remainingMs = endsAt - nowMs;
  if (remainingMs <= 0) return "retrying in 0.0s";
  const seconds = remainingMs / 1000;
  return `retrying in ${seconds.toFixed(1)}s`;
}
```

- Returns `"retrying in 4.0s"` → `"retrying in 3.2s"` → `"retrying in 0.0s"` → `null` once the matching `step:start` / `step:complete` event arrives and `applyRetryEvent` deletes the hint.
- Composable with the `↻` glyph prepended by the column derivation: final NOTE text is `` `${glyphs.retry} ${formatRetryCountdown(hint, nowMs)}` `` → `"↻ retrying in 3.2s"`. Matches mockups §4 line 164 (`↻ retrying in 4s` — decimal-seconds variant; mockup is rounded down for readability, our `.toFixed(1)` carries the precision).
- **`nowMs` reaches here from the view component's prop.** The component's caller in P6-T0 wires a 1 Hz ticker (pre-existing per P5-T3 §12); P6-T1 inherits the ticker — no new effect added here.

### 6.4 Dependency note

If the 500-event ring buffer has evicted the `step:retry` event (run has produced >500 events since the retry), the hint will be missing. **Mitigation:** the ring buffer is 500 events — at typical step-granularity that's dozens of steps of history, far more than any in-flight retry window. Documented risk in §13.

## §7 `upstream: failed` for skipped rows

Location: `src/steps/upstream.ts`.

### 7.1 When to show

Mockup §6 line 241: `deploy-ap · ○ skipped · — · — · upstream failed`.

Shown iff:
1. `token.state === "skipped"`, AND
2. At least one of the token's upstream nodes is in a failed terminal state. Without a `FlowGraph`, "upstream" must come from engine state. Two plausible proxies:
   - **`token.parentTokenId`** is set → check the parent's status. Good for fan-out children (mockup §6 shows `deploy-ap`'s parent is `fan-out` which is `failed`). ✅ matches.
   - **Any token sharing `batchId`** with this token has `result.edge === "fail"`. Good for `forEach`-style skips. Rare in MVP mockups.

### 7.2 Predicate

```ts
export function isUpstreamFailed(
  token: Token,
  tokensById: ReadonlyMap<string, Token>,
): boolean {
  if (token.state !== "skipped") return false;
  if (!token.parentTokenId) return false;
  const parent = tokensById.get(token.parentTokenId);
  if (!parent) return false;
  return parentFailed(parent);
}

function parentFailed(parent: Token): boolean {
  if (parent.state !== "complete") return false;
  if (!parent.result) return false;
  return parent.result.edge.startsWith("fail");
}
```

**Verified terms via engine types:** `StepResult.edge` is a plain string (types.ts line 165). The engine emits `"fail"` or `"fail:max"` for failure routes (see `packages/markflow/src/core/router.ts` conventions). We match on `.startsWith("fail")` to cover both.

### 7.3 Note label

```ts
export function upstreamNoteLabel(
  token: Token,
  tokensById: ReadonlyMap<string, Token>,
): string | null {
  if (!isUpstreamFailed(token, tokensById)) return null;
  return "upstream failed";
}
```

Mockup reads `"upstream failed"` without a colon (line 241: `upstream failed`). Leaving out the colon matches verbatim. **Note:** the task body §7 lists `upstream: failed` with a colon — but mockup §6 does not use a colon. **Decision:** render without colon (follow mockup); documented as a decision in §13.

### 7.4 Wiring into `toLeafRow`

```ts
function noteForLeaf(token, tokensById, retryHints, nowMs, info): string {
  const retryHint = retryHints.get(token.id);
  if (retryHint) {
    return `${UNICODE_GLYPHS.retry} ${formatRetryCountdown(retryHint, nowMs)}`;
  }
  const upstream = upstreamNoteLabel(token, tokensById);
  if (upstream) return upstream;
  if (token.state === "complete" && token.result) {
    return formatEdgeNote(token.result);   // "→ next" / "exit 1 · retries exhausted"
  }
  if (token.state === "waiting") {
    const r = matchingStepResult(token, info);
    return r?.summary ? `"${r.summary}"` : "waiting";
  }
  // pending / running default — empty
  return "";
}
```

Where `formatEdgeNote` mirrors the runs-derive `deriveNote` approach for terminal nodes, producing `"→ next"` for success (edge === ok / unlabeled), `"exit N · retries exhausted"` for `"fail:max"`, etc.

### 8.2 `<StepTableView>` responsibility

```tsx
export interface StepTableViewProps {
  readonly runId: string;
  readonly engineState: EngineState;
  readonly width: number;
  readonly height: number;
  readonly nowMs: number;
}

function StepTableViewImpl({ runId, engineState, width, height, nowMs }) {
  const info = engineState.runs.get(runId);
  const events = engineState.activeRun?.runId === runId
    ? engineState.activeRun.events
    : EMPTY_EVENTS;

  const retryHints = useMemo(() => buildRetryHints(events), [events]);
  const snapshot = useMemo(() => projectStepsSnapshot(events, info), [events, info]);
  const rows = useMemo(
    () => info ? buildStepRows(snapshot, info, nowMs, retryHints) : EMPTY_ROWS,
    [snapshot, info, nowMs, retryHints],
  );
  const columns = useMemo(() => pickStepColumnSet(width), [width]);

  if (!info) return <StepTable rows={EMPTY_ROWS} columns={columns} width={width} height={height} nowMs={nowMs} />;
  return <StepTable rows={rows} columns={columns} width={width} height={height} nowMs={nowMs} />;
}
```

`projectStepsSnapshot(events, info)` runs a **scoped replay** over the ring-buffered events. It uses the **existing** `replay()` from `markflow`? **No** — `replay` requires a strict-monotonic log starting from `run:start`. Our 500-event ring may start mid-run. **Decision:** write a local, permissive `projectStepsSnapshot` that folds only the events this task needs (`token:created`, `token:state`, `step:complete`, `batch:*`). It's ~80 LOC of very similar logic and lives in **`src/steps/tree.ts::projectStepsSnapshot`** (still pure). Tests cover the subset.

### 8.3 Test-injection seam

Component tests render `<StepTable>` directly with handcrafted `StepRow[]` fixtures — no engine touches. `<StepTableView>` tests render it with handcrafted `EngineState` fixtures — no adapter touches. Integration tests in `test/app/run-mode-step-table.test.tsx` render `<App engineState={fixture} />` and assert the top slot wiring.

### 8.4 Memoisation

`buildRetryHints` and `projectStepsSnapshot` both iterate `events` — O(N) per render. With `events` capped at 500 and a 1 Hz render cadence, that's negligible. `useMemo` keyed on `events` identity avoids recomputing when only `nowMs` changes; `buildStepRows` is re-run on every `nowMs` tick (needed for elapsed + retry countdown). **Acceptable CPU cost:** ≲1ms per tick at 500 events.


### 9.2 Layout

Top region of RUN-mode zoom viewport (`viewing.*`). From `pickFrameSlots(rows)`: `topRows` = floor(content/2). At 30 rows terminal → top = 13. Step table fills those 13 rows: 1 row header + up to 12 data rows. Overflow: drop data rows from the bottom (no windowing/scrolling in P6-T1 — cursor navigation + virtualisation is P6-T2).

### 9.3 Header row

```
  STEP                        STATUS        ATTEMPT    ELAPSED    NOTE
```

Rendered with `theme.colors.dim` + bold via `<Text bold dimColor>`.

### 9.4 Data rows

Each row:

```
<cursorGlyph> <indent-spaces><label…>   <status-glyph> <status-label> <attempt> <elapsed> <note>
```

- `cursorGlyph`: `"▶"` when `cursorRowIndex === i`, else `" "` — occupies a 2-col leading gutter. Mockup §4 uses 2 cols including a trailing space.
- `<indent-spaces>`: `row.depth * 2` spaces, PREPENDED to `row.label` inside the STEP column's `projectText`.
- STEP cell: `fitStepCell(prefix + label, width["step"], "left")`.
- STATUS cell: rendered as `<Text color={theme.colors[row.role].color} dimColor={theme.colors[row.role].dim}>{theme.glyphs[row.glyphKey]} {statusLabel}</Text>`.
- Aggregate row STEP label: `⟳ batch [<nodeId>]` — the leading glyph is part of `row.label` not a column-level adornment. No indentation prefix for aggregates (they sit at `depth` they inherit from the parent scope — usually 0).
- ATTEMPT cell: `row.attempt` — plain text, no color.
- ELAPSED cell: `row.elapsed` — plain text.
- NOTE cell: `row.note` — plain text, truncated with `…`. For aggregate rows the NOTE is pre-composed with the progress bar; the Ink layer reads it verbatim.

### 9.5 No cursor/keyboard for now

- `<StepTable>` does NOT own `useInput`.
- `selectedStepId` / `cursorRowIndex` are props to let future tasks thread cursor state without a component rewrite.
- The P5-T3 precedent applies: read-only components take the cursor as a prop, future tasks add keyboard routing.

### 9.6 Empty state

`rows.length === 0` → render `<Text dimColor>no steps yet</Text>` on the first row, blank padding rows below. Hide-don't-grey: no header, no column chrome.

### 9.7 Theme usage

- Every glyph via `theme.glyphs[row.glyphKey]` (not string literals).
- Every color via `theme.colors[row.role].color` + `.dim`.
- Progress bar via `theme.glyphs.progressFilled` / `theme.glyphs.progressEmpty` (new keys — see §12).
- Aggregate-row `⟳` glyph via `theme.glyphs.batch` (already exists).

## §12 Risks + unknowns

1. **`step:retry.delayMs` not in the engine snapshot.** Confirmed: `replay()` treats `step:retry` as a no-op (see §0.1). Mitigation: fold a separate `RetryHintMap` from the raw event tail (§6). **Risk:** the 500-event ring buffer may evict a `step:retry` for a long-delay retry over a very-chatty run — the countdown would then silently disappear. Documented. A future engine enhancement could surface the latest retry per node on `EngineSnapshot`, removing this indirection.
2. **`BatchState` doesn't surface a retry count directly.** The per-item retry info lives on individual tokens / events. We derive `retries` from the retry-hint map as in-flight-only (§4.5). Mockup §4 shows `"1 retry · 0 failed"` which plausibly means "1 in-flight retry". Best-effort; a future engine enhancement could attach per-batch retry totals.
3. **Token ordering heuristics lack a true topological sort.** Without `FlowGraph` in the TUI slice, root ordering relies on `info.steps[]` order for completed roots + token-insertion order for pending roots. This is deterministic but not guaranteed to match a strict graph topo sort. **Mitigation:** unit tests lock in the ordering; P6-T0 could later pipe `WorkflowDefinition.graph` into the step-table view for a proper topo sort.
4. **`batch-aggregate` row depth.** Mockup §4 places `⟳ batch [regions]` at parent depth (0). But a `forEach` source emits tokens whose `parentTokenId` is the source itself. If the batch source renders as its own row (e.g. `fan-out`), the aggregate would visually appear at `depth = source.depth + 1`. **Mitigation:** aggregate row inherits `depth = source.depth` (not `+1`) so it visually sits beside the source. Alternative — always render at depth 0 — is inconsistent for nested forEach. Documented decision.
5. **`"upstream failed"` rendering without a colon.** Task body says `upstream: failed`; mockup §6 renders without colon. We follow mockup (§7.3). Reviewers should confirm.
6. **Progress-bar new glyph keys.** `theme.glyphs.progressFilled` / `progressEmpty` are NEW (`█` / `░`). This is the first status-NOT glyph we add to the theme. Rationale: the alternative (inline literals in the component) breaks the "every glyph via theme" discipline. **Alternative considered:** put the bar glyphs in a module-private const on `src/steps/aggregate.ts`; rejected because ASCII fallback needs a capability-driven choice which ONLY `theme/glyphs.ts` handles. Plan adds the new keys to both `UNICODE_GLYPHS` and `ASCII_GLYPHS` (ASCII: `"#"` / `"."`). This is a ONE-line change per table and is the minimal engine-agnostic addition.
7. **`pickStepColumnSet` thresholds differ from `pickColumnSet` tiers.** We pick 120/90 as breakpoints matching `runs/columns.ts`. Phase-8 polish may tune; documented out-of-scope.
8. **Memoisation invalidation.** `buildStepRows` depends on `nowMs` (elapsed + retry countdown), so it re-runs every tick (1 Hz). For a 500-event / ~30-row run, each re-run is sub-millisecond. **Risk:** profile worst-case at 500 events + 50 rows on a slow terminal. If slow, add a dirty-flag that skips re-derivation when neither `events` identity nor active-retry-hint has changed AND elapsed-ms deltas fall within the 1 Hz rounding band. Deferred.
9. **`<App>` prop wiring.** Adding `engineState?: EngineState` to `<App>` is a minor surface extension; the existing `AppProps` expects production wiring in P6-T0. Until P6-T0, production sees `initialEngineState` (empty) and renders empty step tables. This is a graceful degradation consistent with P5-T3's empty-runs-feed approach.
