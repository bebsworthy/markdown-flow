## §2 Data model

### 2.1 Where `selectedRunId` and `runsCursor` live

Two related concepts, two distinct state fields, both on **`AppState` root** (not inside `runsTable` slice — there is no such slice today, and introducing one for two fields is overkill).

```ts
// state/types.ts (P5-T3 additions — only additions shown)
export interface AppState {
  readonly mode: Mode;
  readonly overlay: Overlay | null;
  readonly filter: string;
  readonly selectedWorkflowId: string | null;
  readonly selectedRunId: string | null;   // EXISTING — lifted into the cursor model
  readonly runsSort: RunsSortState;
  readonly runsFilter: RunsFilterState;
  readonly runsArchive: RunsArchivePolicy;
  readonly runsCursor: number;             // NEW — see 2.2
}
```

**`selectedRunId` (existing)** is the **logical selection**: the id the bottom pane follows, the id RUN mode zooms into, the id future overlays (approve, resume, cancel) target. It is a `string | null` — `null` when no runs exist at all.

**`runsCursor` (new)** is the **index into the sorted-filtered-archived row list**. An integer. Always clamped to `[0, rows.length - 1]` (or `0` when empty) by the reducer / cursor helpers. The component reads it directly to compute the window, to highlight the row, and to derive `selectedRunId` via `rows[runsCursor].id`.

**Why both?** Two reasons:
1. **Row-set changes must preserve selection when possible.** If the user selects run `abcd12`, applies a filter that includes `abcd12`, the cursor's numeric position may have changed (the row could be at a different index now) but the logical selection shouldn't jump to a different run. `reconcileCursorAfterRowsChange(prevCursor, prevRunId, nextRows)` (§4.5) finds the run by id and sets the cursor there.
2. **The bottom pane / RUN mode consumes `selectedRunId`, not the cursor.** A Phase-6 subscriber doesn't care about list geometry; it needs an id.

**Redundancy check.** These two fields are NOT fully independent — every reducer transition maintains the invariant `selectedRunId === rows[runsCursor]?.id` **when rows are non-empty**. When `rows` is empty, `runsCursor = 0` and `selectedRunId = null`. The reducer does not have access to `rows` (it's a pure function over `AppState + Action`), so the invariant is maintained **at the dispatch site** (the `<RunsTable>` component uses `useEffect` to emit `RUNS_SELECT` whenever the derived `rows[cursor].id` diverges from `state.selectedRunId`). See §6.

**Alternative considered.** Put cursor inside a `runsTable: { cursor, offset }` sub-object. Rejected — `offset` is already component-local (`useState` in `runs-table.tsx` line 94) because it's a render-time derivation of `(cursor, visibleRows, rowCount)` per `computeWindow`. Promoting `offset` to `AppState` would add noise without adding testability; `cursor` alone on the root keeps the action surface flat.

### 2.2 `runsCursor` initial value + invariants

- `runsCursor: 0` in `initialAppState`.
- Invariant (enforced by every action handler): `runsCursor >= 0`. We do NOT encode an upper bound in the reducer (it doesn't know `rows.length`). The **component** is responsible for emitting a `RUNS_CURSOR_JUMP` or `RUNS_CURSOR_HOME` when `rows.length` shrinks to make the current cursor invalid (§4.5).

### 2.3 Mode transition types — no new `Mode` variants

The task body says *"`Enter` zooms the bottom pane full-screen (switches to RUN mode) and hides the runs table. `Esc` returns."*

- "RUN mode" is **`Mode.kind === "viewing"`** — this is already the established convention (§0). The mockup's `[ RUN ]` pill is a cosmetic label over that kind. No new `Mode` variant is introduced.
- The Enter transition: `browsing.runs` + `selectedRunId !== null` → `viewing { runId: selectedRunId, focus: "graph" }`. Already modelled by existing `MODE_OPEN_RUN` action.
- The Esc transition: `viewing.*` → `browsing.runs`. Already modelled by existing `MODE_CLOSE_RUN`.

**Net.** No type additions for mode transitions themselves. Only new actions are the cursor/selection ones.

### 2.4 `Action` additions

```ts
// state/types.ts — appended to the Action union
| { readonly type: "RUNS_CURSOR_MOVE"; readonly delta: number }
| { readonly type: "RUNS_CURSOR_JUMP"; readonly index: number }
| { readonly type: "RUNS_CURSOR_HOME" }
| { readonly type: "RUNS_CURSOR_END"; readonly rowCount: number }
| {
    readonly type: "RUNS_CURSOR_PAGE";
    readonly direction: "up" | "down";
    readonly pageSize: number;
    readonly rowCount: number;
  }
| { readonly type: "RUNS_SELECT"; readonly runId: string | null };
```

Payload rationale:

- `RUNS_CURSOR_MOVE { delta }`. `delta` can be negative or positive. Reducer does `max(0, cursor + delta)`; upper-bound clamp is component-emitted via `RUNS_CURSOR_JUMP` on render (§4). We deliberately keep `delta` integer-unbounded — the component caller owns page-size.
- `RUNS_CURSOR_JUMP { index }`. Component emits it with an already-clamped value (the component knows `rowCount`). Reducer honours it as-is (still applies `max(0, index)`).
- `RUNS_CURSOR_HOME`. Always sets cursor to `0`. Handles the `g` / `Home` binding.
- `RUNS_CURSOR_END { rowCount }`. Sets cursor to `max(0, rowCount - 1)`. `rowCount` payload so the pure reducer knows the end — this is the one place we thread rowCount through an action rather than have the component emit `RUNS_CURSOR_JUMP` directly. **Decision:** payload-carrying variant is clearer than a component-computed jump; the alternative bloats the component and hides the intent.
- `RUNS_CURSOR_PAGE { direction, pageSize, rowCount }`. The component knows both `visibleRows` (== pageSize) and `rows.length`. Reducer computes `cursor ± pageSize` clamped to `[0, rowCount-1]`.
- `RUNS_SELECT { runId }`. Direct setter for `selectedRunId`. Fired by the component on cursor-change effect (§6). Also usable by future overlay-close flows to re-align selection.

**Why `rowCount` on some actions but not `RUNS_CURSOR_MOVE`.** The `MOVE` variant is bidirectional (delta) and in practice the upper bound is only hit when moving down; the component's follow-up `RUNS_CURSOR_JUMP` on re-render corrects overshoot cheaply. For `END` and `PAGE` the intent is inherently terminal-anchored, so carrying `rowCount` avoids a two-dispatch cycle.

### 2.5 Payload validation

All new actions are no-ops under these conditions:
- `RUNS_CURSOR_*` when `state.mode.kind !== "browsing"` OR `state.mode.pane !== "runs"` (cursor is runs-mode-scoped). Reducer guards — see §5.2.
- `MODE_OPEN_RUN` with `runId === ""` is a no-op today (not literally guarded — but the current reducer sets `selectedRunId = ""` which downstream would dispatch `RUNS_SELECT(null)` via the component effect; we add an explicit guard in the reducer to reject empty strings → no-op).
- `RUNS_SELECT { runId: null }` clears `selectedRunId`. This is fine in RUNS mode; in `viewing.*` it is a no-op (we don't want the bottom pane to lose its target mid-zoom).

### 3.2 No changes to existing transitions

- `MODE_SHOW_WORKFLOWS` / `MODE_SHOW_RUNS` / `FOCUS_BROWSING_PANE` / `FOCUS_VIEWING_PANE`: unchanged. Each preserves `selectedRunId` and `runsCursor` (they operate only on `mode`).

### 3.3 Side-effect: derived `selectedRunId`

No reducer action directly ties cursor movement to `selectedRunId`. The link is made in the component layer (§6). This keeps reducer tests trivial (no `rows` parameter leaking into the pure slice).

## §4 Cursor follow-selection semantics

### 4.1 Bounds

Cursor is always a non-negative integer. The reducer clamps to `>= 0` in every action handler. The *upper* bound is `rows.length - 1` — but `rows` lives outside reducer state, so the upper-bound enforcement happens in the component.

### 4.2 Empty-rows special case

When `rows.length === 0`:
- `runsCursor` stays at `0` (cannot be -1).
- Component skips rendering `<RunsTableRow>`s and the empty-state branch renders (`runs-table.tsx` lines 171–183).
- `selectedRunId` is forced to `null` via `RUNS_SELECT { runId: null }` by the component effect.
- Enter on an empty table is a no-op (component guards `rows[cursor]?.id == null`).

### 4.3 Clamp direction on filter/archive change

**Decision: when cursor points outside the filtered slice after a filter or archive change, clamp to `min(prevCursor, rows.length - 1)` — i.e., keep the cursor at the highest index still in range, never jump to 0.** The test matrix (§9.3) asserts this exact behaviour.

**Rationale.** Two options considered:

| Option | Behaviour | Why rejected / picked |
|---|---|---|
| (A) Reset to 0 | Filter change → cursor snaps to first visible row | Predictable but disorienting. A user who filtered out 1 of 50 rows does not expect their cursor to jump to the top. Rejected. |
| (B) **Clamp to last visible index** | Filter change → cursor stays put if still valid, otherwise slides up to the last remaining row | Matches VSCode / lazygit / htop behaviour. Keeps spatial context. **Picked.** |

Edge cases:
- `rows.length` shrinks to 0 → `runsCursor = 0` (no other choice). `selectedRunId → null`.
- Cursor was on a row whose id still appears after the filter change → `reconcileCursorAfterRowsChange` (§4.5) finds the id in the new list and places the cursor there (preserves selection identity over position).
- Cursor was on a row whose id is now filtered out → clamp to `min(prevCursor, rows.length - 1)` within the new list.

### 4.4 Derived `selectedRunId`

Pseudocode for the component effect:

```
// In <RunsTable> — after sorted rows are computed
const derivedSelected = rows.length > 0 ? rows[windowState.cursor].id : null;
useEffect(() => {
  if (derivedSelected !== selectedRunId) {
    dispatch({ type: "RUNS_SELECT", runId: derivedSelected });
  }
}, [derivedSelected, selectedRunId]);
```

The `windowState.cursor` value is the already-clamped cursor returned by `computeWindow` (`src/runs/window.ts` lines 29–51) — not `state.runsCursor` directly. This guarantees the derived id always maps to a real row.

### 4.5 `reconcileCursorAfterRowsChange`

```ts
// src/runs/cursor.ts
export function reconcileCursorAfterRowsChange(
  prevCursor: number,
  prevRunId: string | null,
  nextRows: ReadonlyArray<{ readonly id: string }>,
): number {
  if (nextRows.length === 0) return 0;
  if (prevRunId !== null) {
    const stillHere = nextRows.findIndex((r) => r.id === prevRunId);
    if (stillHere >= 0) return stillHere;
  }
  return Math.min(Math.max(0, prevCursor), nextRows.length - 1);
}
```

Called from the component inside a `useEffect` keyed on `rows` identity:

```
useEffect(() => {
  const next = reconcileCursorAfterRowsChange(state.runsCursor, state.selectedRunId, sortedRows);
  if (next !== state.runsCursor) {
    dispatch({ type: "RUNS_CURSOR_JUMP", index: next });
  }
}, [sortedRows, state.runsCursor, state.selectedRunId]);
```

Because this fires in a `useEffect` layer, a single filter-apply produces at most two reducer dispatches: (a) the `RUNS_FILTER_APPLY` itself, (b) the cursor reconciliation. The derived-selection effect (§4.4) is idempotent in the common case (the id didn't change, so no `RUNS_SELECT` fires).

Helper bodies:

```ts
function moveRunsCursor(state: AppState, delta: number): AppState {
  if (state.mode.kind !== "browsing" || state.mode.pane !== "runs") return state;
  if (delta === 0) return state;
  const next = Math.max(0, state.runsCursor + delta);
  return next === state.runsCursor ? state : { ...state, runsCursor: next };
}

function jumpRunsCursor(state: AppState, index: number): AppState {
  if (state.mode.kind !== "browsing" || state.mode.pane !== "runs") return state;
  const next = Math.max(0, Math.floor(index));
  return next === state.runsCursor ? state : { ...state, runsCursor: next };
}

function pageRunsCursor(state: AppState, a: { direction: "up"|"down"; pageSize: number; rowCount: number }): AppState {
  if (state.mode.kind !== "browsing" || state.mode.pane !== "runs") return state;
  if (a.pageSize <= 0 || a.rowCount <= 0) return state;
  const signed = a.direction === "up" ? -a.pageSize : a.pageSize;
  const next = Math.max(0, Math.min(a.rowCount - 1, state.runsCursor + signed));
  return next === state.runsCursor ? state : { ...state, runsCursor: next };
}

function selectRun(state: AppState, runId: string | null): AppState {
  // Guard: in viewing.*, never clear selection (preserves the zoom target).
  if (state.mode.kind === "viewing" && runId === null) return state;
  if (state.selectedRunId === runId) return state;
  return { ...state, selectedRunId: runId };
}
```

## §8 No-orphan-state rules

### 8.1 State preservation across zoom / unzoom

When `MODE_OPEN_RUN` fires:
- `mode` → `viewing { runId, focus: "graph" }`
- `selectedRunId` → `runId` (already `runId` if row was selected)
- `filter` (legacy) → `""` (existing behaviour — low-impact)
- `runsCursor` → **unchanged**
- `runsSort` → **unchanged**
- `runsFilter` → **unchanged** (draft/applied/open all preserved; `open: true` is legal but rare — see below)
- `runsArchive` → **unchanged**

When `MODE_CLOSE_RUN` fires:
- `mode` → `browsing.runs`
- All other slices → **unchanged** (matching today's reducer behaviour)

**Invariant.** For any pair (state before `MODE_OPEN_RUN`, state after `MODE_OPEN_RUN` + `MODE_CLOSE_RUN`), every field except `mode` and legacy `filter` is identical. Tested by a property-style unit test in §9.2.

### 8.2 Filter-bar open across zoom

If the user opens the filter bar (`runsFilter.open = true`), types nothing, then triggers `MODE_OPEN_RUN` via the tab row's F3 key (or via Enter on a different row if they click away) — what happens?

- Reducer does not close the filter bar on `MODE_OPEN_RUN`. On `MODE_CLOSE_RUN` it comes back with `open: true`.
- **Component layer** decides whether the filter bar renders. It only renders inside `<RunsTable>`, which is unmounted in `viewing.*`. So visually the bar disappears and reappears. No data loss.
- This is acceptable and tested by §9.4 case "open filter bar → zoom → unzoom preserves filter".

### 8.3 Cursor-past-end on return

When `MODE_CLOSE_RUN` fires, the cursor may point to a now-nonexistent row (if the backing data changed while in RUN mode — a run was archived / deleted). The reconciliation effect (§4.5) re-runs on mount and clamps. No UI flash because the effect fires before first paint of `<RunsTable>` in the new mode.

### 8.4 Zoom on a deleted row

**Risk.** User has `selectedRunId = "abcd"`, presses Enter. Between the dispatch and the next render, the runs-feed deletes `abcd`.

**Mitigation.**
- The reducer does NOT check `runExists` (it can't — rows are outside state). `MODE_OPEN_RUN` succeeds.
- `<RunDetailPlaceholder>` receives `runExists={false}` and renders the "no longer exists — Esc to return" copy.
- User hits Esc → `MODE_CLOSE_RUN` → back to `browsing.runs`.
- No state is orphaned. `selectedRunId` is still `"abcd"` but the component-level reconciliation fires: since the id is gone, `rows[cursor].id` gives some *other* id, and `RUNS_SELECT` is emitted to point at it (or `null` if rows is empty).

Test matrix covers this as `mode-transitions.test.tsx::"zoom on a run deleted mid-transition"`.

### 8.5 Repeated zoom/unzoom idempotency

Sequence: Enter, Esc, Enter, Esc, Enter, Esc. After each pair, `AppState` (except `mode` mid-sequence) is identical. Enforced by §9.2 reducer test.

### 8.6 Empty runs table, pressing Enter

- `rows.length === 0` → component Enter handler short-circuits (no dispatch).
- Even if dispatch sneaks through, reducer guards `runId === ""` → no-op.

5. **Esc unzoom.** `Esc` in `viewing.*` dispatches `MODE_CLOSE_RUN` and restores the table with every non-mode slice unchanged.
6. **Empty-table discipline.** Enter on an empty table is a no-op (no dispatch). No crash.
7. **Filter-clamp.** Applying a filter that removes the selected row slides the cursor to a valid index (last-visible rule, §4.3). The new `selectedRunId` is emitted via `RUNS_SELECT`.
8. **Cursor preserved across zoom.** After zoom + unzoom, `runsCursor` is exactly what it was before zoom.
9. **Vi keys.** `j`/`k`/`g`/`G` behave identically to `↓`/`↑`/`Home`/`End`.
10. **Suppression rules.** While the filter bar is open, none of the cursor/Enter keys fire. While an overlay is open, none fire. While in `viewing.*`, the table is unmounted so key routing is moot.
11. **Purity preserved.** `src/runs/cursor.ts` imports nothing from `react`/`ink`/`node:*` and appears in the purity test's `files[]`.
12. **Scope fence.** The bottom pane is a placeholder. No step table, no log pane, no detail tabs. The mockup's §4 step-table top half is blank in this task.

## §11 Risks / unknowns

### 11.1 Race: `runs.watch()` removes the currently-selected run mid-RUN-mode

**Symptom.** User is in `viewing { runId: "abcd" }`. The backing `runs/` directory deletes `abcd`'s meta.json. Next `runs.watch()` emission removes the row from the list. On Esc-return, cursor lands on a different row.

**Mitigation.** Handled by `reconcileCursorAfterRowsChange` on re-entry to `browsing.runs`. Placeholder renders `"run <id> no longer exists"` while still in RUN mode. Explicitly tested in §9.6.

**Open question.** Should we auto-exit RUN mode when the viewed run vanishes? **Decision (deferred to Phase 6):** No auto-exit — let the user hit Esc. An auto-transition during a mid-interaction render is likely to be surprising. Phase 6 when the detail tabs land can revisit.

### 11.2 `Esc` order-of-precedence ambiguity

**Symptom.** `Esc` has three possible handlers: overlay (if open), filter bar (if open), RUN mode (if zoomed). Which fires?

**Current resolution.** Ink's `useInput` is LIFO — the most-recently-mounted handler fires first. Overlay handlers mount inside `<App>`'s overlay render path (when `state.overlay !== null`). Filter bar handler mounts inside `<RunsTable>` via `<RunsFilterBar>`, only when `runsFilter.open`. Both are children of `<App>`, which mounts the RUN-mode Esc handler.

Mount order when all three are active: `<App>` Esc hook mounts first, then filter bar (deep inside `<RunsTable>` subtree) mounts after. The LIFO guarantees filter bar handles Esc first.

**Risk.** If Ink changes its LIFO guarantee, our `<App>`-level Esc handler could swallow Esc before the filter bar sees it. **Mitigation.** `<App>`-level handler guards on `!state.runsFilter.open` explicitly as a belt-and-suspenders check. Tested by §9.6 "Esc while filter bar is open".

### 11.3 Derived-selection effect re-entrancy

**Symptom.** Component effect dispatches `RUNS_SELECT` on cursor change. If the reducer / React scheduler re-renders synchronously, could we loop?

**Mitigation.** `RUNS_SELECT` reducer helper returns identical state when `runId === selectedRunId` (§5.2 `selectRun`). So the second render produces no new dispatch. Tested by asserting dispatch call-count in §9.3.

### 11.4 Empty-runs feed in `<App>`

**Symptom.** This task wires `<RunsTable rows={[]} />` pending the engine feed. Visual output in `browsing.runs` will show the empty-state branch from `runs-table.tsx` (line 171–183). Users won't see runs until P6-T0.

**Mitigation.** Acceptable for this phase. A TODO comment in `app.tsx` points to P6-T0. The empty-state text already exists and is tested.

### 11.5 `Enter` race between filter-apply and zoom

**Symptom.** Filter bar is open; user types `status:running` and hits Enter. The filter applies, bar closes. If the same Enter event bubbles up to `<RunsTable>`, a spurious `MODE_OPEN_RUN` would fire.

**Mitigation.** `<RunsTable>`'s `useInput` is gated on `!runsFilter.open`. At the moment Enter fires, `runsFilter.open === true` (the reducer updates synchronously after the filter bar dispatches, but the gate evaluates from the current render's state). Enter is consumed by the filter-bar handler only. Verified by §9.6 and a targeted test in §9.3.

### 11.6 `Home` / `End` key detection

**Symptom.** Ink 5's `useInput` exposes `key.pageDown` and `key.pageUp` but the `Home` / `End` keys arrive as raw escape sequences in some terminals.

**Mitigation.** Detect both the Ink `key.*` flag and the raw ANSI sequences (`\x1b[H`, `\x1b[F`, `\x1b[1~`, `\x1b[4~`). Pattern already used by `keyToMode` for F-keys (`src/components/app-shell-layout.ts` lines 52–64). Fallback on `g` / `G` regardless of platform.

### 11.7 React `useEffect` ordering

**Symptom.** Two effects: (a) reconciliation (`rows` change → jump cursor), (b) derived selection (`cursor` change → emit RUNS_SELECT). If (a) fires and dispatches `RUNS_CURSOR_JUMP`, the reducer updates synchronously but React batches renders — so (b) sees the jumped cursor on next render. No double-dispatch.

**Mitigation.** Tested indirectly by §9.3 filter-clamp tests that assert correct `RUNS_SELECT` + `RUNS_CURSOR_JUMP` dispatches and their order.

### 11.8 `pageSize` vs. `visibleRows`

**Decision.** `PgUp`/`PgDn` use `visibleRows - 1` for overlap (one row stays in view). Matches htop / less. Tested with a fixture where `visibleRows = 10`, starting cursor = 5, `PgDn` → 14 (not 15).

Actually: reconsidered — simpler to use `pageSize = visibleRows` exactly. Overlapping by one is a nice-to-have that complicates the pure `pageRunsCursor` math. **Picked:** full-page moves (`pageSize = visibleRows`). Tests assert this. Follow-up can add overlap if UX feedback requests it.
