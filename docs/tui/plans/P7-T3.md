### 0.2 Keybar fixtures already exist per mode

- Nine mode/focus combinations have fixtures in `src/components/keybar-fixtures/`: `workflows.ts`, `workflows-empty.ts`, `runs.ts`, `graph.ts`, `log.ts` (two variants), `events.ts`, `approval.ts`, `resume.ts`.

- **What is missing**: a registry that maps the current `(mode, focus, overlay)` tuple to the fixture that the keybar is rendering. Today `<App>` hard-picks the fixture per mode via an inline switch; to render the help overlay from the same source we need that mapping exposed as data.

### 0.3 What's missing

- **No component** renders either overlay. `app.tsx`'s overlay branch handles `addWorkflow`, `approval`, `resumeWizard` — not `commandPalette` or `help`.
- **No `:` / `?` bindings wired**. The `?` key is declared in several fixtures (`graph.ts` line 117, `workflows.ts`, `runs.ts`) with a deferred action; `:` is not declared at all. Neither key is handled by app-level `useInput`.

- **No fuzzy matcher**. Palette filter needs a small prefix+substring+bold-char matcher — not present in the codebase.
- **No keybar fixture** for COMMAND or HELP modes; `keybar-fixtures/` lacks `command.ts` and `help.ts`.
- **No keybar-fixture registry**. Required so the help overlay reads the same `Binding[]` that the keybar rendered for the current mode.

## §2 Exported names + signatures

### 2.1 `src/palette/types.ts`

```ts
/** Stable command identifiers — the enum the UI dispatches against. */
export type CommandId =
  | "run"
  | "resume"
  | "rerun"
  | "cancel"
  | "approve"
  | "pending"
  | "goto"
  | "theme"
  | "quit";

/** Catalogue row; the authoritative list lives in commands.ts. */
export interface Command {
  readonly id: CommandId;
  readonly name: string;          // "run", "resume", …  — what appears after `:`
  readonly usage: string;         // ":run <workflow>"
  readonly summary: string;       // "Start a new run of the selected workflow"
  readonly argRequired: boolean;
  /** Same signature as keybar bindings so we can reuse AppContext. */
  readonly when: (ctx: AppContext) => boolean;
}

/** Filter-match with character indices for bold rendering. */
export interface CommandMatch {
  readonly command: Command;
  readonly matchedIndices: readonly number[];   // indices into command.name
  readonly score: number;                       // higher == better
}

/** Local state for <CommandPaletteOverlay>. */
export interface PaletteState {
  readonly cursor: number;                      // index into filtered
  readonly fsm: "idle" | "running" | "error";
  readonly error: string | null;
}

export type PaletteAction =
  | { readonly type: "CURSOR_MOVE"; readonly delta: number }
  | { readonly type: "CURSOR_RESET_TO_FIRST" }
  | { readonly type: "RUN_START" }
  | { readonly type: "RUN_OK" }
  | { readonly type: "RUN_FAIL"; readonly error: string };

/** Side-effects the palette can request. Injected by <App> at render. */
export interface CommandExecContext {
  readonly state: AppState;                            // read-only snapshot
  readonly engineHints: EngineHints;                   // activeRun, counts …
  readonly dispatch: (action: AppAction) => void;      // reducer dispatch
  readonly runsDir: string;
  readonly runWorkflow: (workflowId: string) => Promise<CommandResult>;
  readonly resumeRun: typeof defaultResumeRun;
  readonly decideApproval: typeof defaultDecideApproval;
  readonly cancelRun: (runId: string) => Promise<CommandResult>;
  readonly rotateTheme: () => void;
  readonly quit: () => void;
}

/** Typed outcome of a command execution. */
export type CommandResult =
  | { readonly kind: "ok" }
  | { readonly kind: "usage"; readonly message: string }   // bad argument
  | { readonly kind: "unavailable"; readonly message: string } // when(ctx) false
  | { readonly kind: "error"; readonly message: string };
```

### 2.3 `src/palette/fuzzy.ts`

```ts
/** Pure. Returns null for no-match. Scoring: prefix > head-char > substring > scattered. */
export function matchCommand(
  query: string,
  command: Command,
): CommandMatch | null;

/** Pure. Filters + sorts by descending score; ties broken by catalogue order. */
export function filterCommands(
  query: string,
  commands: readonly Command[],
  ctx: AppContext,
): readonly CommandMatch[];
```

Rationale: the query has a leading `:` stripped first. Empty query returns all `when(ctx)` rows in catalogue order with `matchedIndices: []`. Matches must be order-preserving (no anagram matches) — prevents `:qr` from matching `:rerun`.

### 2.6 `src/palette/executeCommand` (in `src/palette/exec.ts` — pure-logic helper)

```ts
/** Pure dispatcher: picks the right closure on `ctx` based on the command id.
 *  Contains NO IO; the closures on `ctx` perform IO. Returns a Promise because
 *  some closures are async. */
export function executeCommand(
  match: CommandMatch,
  arg: string,
  ctx: CommandExecContext,
): Promise<CommandResult>;
```

Per-command mapping:

| id | Behaviour |
|---|---|
| `run` | `await ctx.runWorkflow(arg)`. Resolves workflow id via registry lookup on the caller. |
| `resume` | If arg provided → `ctx.resumeRun({ runId:arg, … })`; else dispatch `MODE_SHOW_RUNS` + filter `status:suspended`. |
| `rerun` | `ctx.resumeRun({ runId: state.mode.runId, rerunNodes:[arg], … })` — same impure path P7-T2 uses. |
| `cancel` | `ctx.cancelRun(runId)` (ctx reads it from `state.mode`). |
| `approve` | Opens `Overlay.approval` for the current pending gate (uses `findPendingApproval`). If `arg` matches an option, pre-decides via `ctx.decideApproval`. |
| `pending` | `ctx.dispatch({ type:"MODE_SHOW_RUNS" })` + a runs-filter apply `status:suspended`. |
| `goto` | `ctx.dispatch({ type:"FOCUS_VIEWING_PANE", focus:"events" })` + a local cursor dispatch (Events pane owns seq jumping; P6-T4 wired the follow cursor). |
| `theme` | `ctx.rotateTheme()`. |
| `quit` | `ctx.quit()`. |

After any `{kind:"ok"}`: `ctx.dispatch({ type:"OVERLAY_CLOSE" })`.

```
overlay === null && mode.kind === "browsing" && pane === "workflows" → WORKFLOWS_KEYBAR or WORKFLOWS_EMPTY_KEYBAR
overlay === null && mode.kind === "browsing" && pane === "runs"      → RUNS_KEYBAR
overlay === null && mode.kind === "viewing"                          →
    focus === "graph"  → GRAPH_KEYBAR
    focus === "detail" → GRAPH_KEYBAR              (detail reuses graph keybar per P6-T4)
    focus === "log"    → logFollowing ? LOG_FOLLOWING_KEYBAR : LOG_PAUSED_KEYBAR
    focus === "events" → EVENTS_KEYBAR
```

`modeLabel` comes from a small table (`"WORKFLOWS" | "RUNS" | "RUN" | "COMMAND" | "HELP" | "APPROVAL" | "RESUME"`). `focusLabel` matches `ViewingFocus` lowercased.

## §3 Data flow

### 3.1 Opening the overlays (UI → reducer)

1. User presses `:` in any non-overlay state. App-level `useInput`:
   ```ts
   if (input === ":" && state.overlay === null) {
     dispatch({ type: "OVERLAY_OPEN", overlay: { kind: "commandPalette", query: "" }});
   }
   ```
2. `<App>` routes render: `selectKeybarFixture` returns `COMMAND_KEYBAR` + `[COMMAND]` pill; overlay branch mounts `<CommandPaletteOverlay>` centred over the AppShell (same overlay slot as other modals).
3. `?` is symmetric: `OVERLAY_OPEN { kind:"help" }`; `<HelpOverlay>` mounts; keybar becomes `HELP_KEYBAR` + `[HELP]`.
4. `:` is **ignored** when `state.overlay !== null` (no overlay-stacking, D7). `?` inside other overlays is also ignored — their own fixtures own it (e.g. `APPROVAL_KEYBAR` has `? Help` but that's a deferred follow-up — see §7 in P7-T1).

### 3.2 Palette editing + execution

1. Printable keystrokes in `<CommandPaletteOverlay>` update `query` via `dispatch({ type:"COMMAND_PALETTE_QUERY", query: next })`. The existing reducer helper `updatePaletteQuery` covers this.
2. The overlay derives `matches = filterCommands(query, COMMANDS, ctx)` on every render — pure, O(9), no memoisation needed (D8).
3. `↑/↓` moves the local cursor (`useReducer(paletteReducer)`); `Tab` rewrites `query` via `onQueryChange` with the completion of the current match.
4. `Enter`: local `RUN_START` → `await executeCommand(matches[cursor], arg, exec)` → `RUN_OK`/`RUN_FAIL`.
5. `{kind:"ok"}` ⇒ `exec.dispatch(OVERLAY_CLOSE)`; the side-effecting closure may have already dispatched a mode change (e.g. `:pending`) — the close still fires (idempotent).
6. `{kind:"usage"|"unavailable"|"error"}` ⇒ modal stays open with the message in a red footer line; cursor + query preserved.

### 3.3 Help rendering (keybar registry → help overlay)

1. `<App>` memoises `selection = useMemo(() => selectKeybarFixture(...), [mode, overlay, logFollowing])`.
2. `<Keybar>` receives `selection.bindings` + `selection.modePill` + tier from `useStdout().cols`.
3. `<HelpOverlay>` — mounted **while the help overlay is the active overlay** — also reads from `selection`. Because opening `?` swaps the overlay to `{kind:"help"}`, `selectKeybarFixture` would now return `HELP_KEYBAR`. The help overlay therefore reads its bindings from a **snapshot taken before the overlay opened** (D9).
4. Snapshot mechanism: when `?` is pressed, `<App>` passes the pre-swap `KeybarSelection` to `<HelpOverlay>` via a ref set at dispatch time: `prevFixtureRef.current = selection; dispatch(OVERLAY_OPEN { kind:"help" });`. The ref is read (and cleared) by the next render's `<HelpOverlay>` prop. This keeps the "single source of truth" rule intact — the help overlay renders the **exact fixture the keybar was showing the instant the user asked for help**.
5. `<HelpOverlay>` calls `deriveHelpModel({ bindings: prevFixture.bindings, ctx, search })` — the same `ctx` the keybar uses. Hide-don't-grey is applied via the same `when(ctx)` predicates.
6. Because the help overlay shows categories, bindings without a `.category` are grouped under `"GLOBAL"`.

### 3.4 Global `?` / `:` handling precedence

- App-level `useInput` runs **before** component-owned `useInput`s in the Ink event model only when the app owner is mounted higher up the tree. `<CommandPaletteOverlay>` / `<HelpOverlay>` own their `useInput` once rendered; while either is active the app-level `?`/`:` handlers are guarded by `state.overlay === null`. The modal's own `Esc` handler closes — it does not re-trigger open.
- Within modals: `<CommandPaletteOverlay>` does NOT treat `?` as an escape hatch; `<HelpOverlay>` treats `?` as a close alias (D4, matches lazygit convention).

## §5 Acceptance criteria (verbatim + derived)

**Verbatim (plan.md).**
> Help content derived from the same keymap used by the keybar (single source of truth, features.md §5.6 rule 8).

**Derived sub-criteria.** All must hold to close the task.

1. **Help = keybar fixture.** A dedicated assertion in `test/app/help-overlay.test.tsx` verifies that for every `(mode, focus)` tuple in §15 (at least WORKFLOWS, RUNS, RUN-graph, LOG-following, LOG-paused, EVENTS), the `Binding[]` passed to `<HelpOverlay>` is `===`-identical (same array reference) to the one passed to `<Keybar>` in the previous render. This is the literal "single source of truth" proof.
2. **Hide-don't-grey in help.** Help rows for bindings whose `when(ctx) === false` are omitted entirely. No `.dim` or `.grey` style is applied to suppressed bindings; the row is absent.
3. **Palette filtering.** Typing `:re` shows `resume` then `rerun` (matching mockup §10 ordering). Matching characters are bold. `when(ctx)=false` commands never appear.
4. **Palette mode pill.** While `overlay.kind === "commandPalette"` the keybar renders `[COMMAND]` in inverse video at `full`+`short` tiers per §15 COMMAND row.
5. **Help mode pill.** While `overlay.kind === "help"` the keybar renders `[HELP]` in inverse video at `full`+`short` tiers per §15 HELP row.
6. **Palette keybar.** Matches §15 COMMAND row at all three tiers — `⏎ Run  ↑↓ Select  Tab Complete      Esc Cancel` at full, `⏎ ↑↓ Tab   Esc` at short/keys.
7. **Help keybar.** Matches §15 HELP row — `↑↓ Navigate   / Search   Esc Close` at full, `↑↓ / Esc` at short/keys.
8. **Command catalogue completeness.** `COMMANDS` contains exactly the nine ids listed in plan.md scope: `run, resume, rerun, cancel, approve, pending, goto, theme, quit`. Asserted by `test/palette/commands.test.ts`.
9. **Tab-complete.** Pressing `Tab` when exactly one match remains rewrites the input to the command's `usage`. With ≥2 matches Tab completes to the longest common prefix.
10. **No overlay stacking.** `:` and `?` dispatches are no-ops when `state.overlay !== null`. Asserted in `test/app/command-palette.test.tsx` case (d).
11. **Esc closes.** In both overlays, `Esc` dispatches `OVERLAY_CLOSE` and causes no side-effects. In `<HelpOverlay>`, `?` is an alias for `Esc`.
12. **Usage/unavailable handling.** A command that fails `when(ctx)` at run-time (race) returns `{kind:"unavailable"}` and the modal stays open with a red footer. A missing `argRequired` argument returns `{kind:"usage"}`.
13. **Help search hides bindings that don't match.** Typing `/app` in help shows only the approve row; typing `/zzz` shows an empty state (no sections, a muted `no bindings match` line).
14. **Live toggle labels in help.** The `f Follow`/`f Unfollow` binding in LOG-following renders with the currently-active label via `toggleLabel(ctx.toggleState)`.
15. **No engine edits.** `git diff` shows zero changes under `packages/markflow/`.
16. **No `.skip`.** `rg '\.skip\(' packages/markflow-tui/test` returns only pre-existing matches unchanged by this task.
17. **Mode transitions preserved.** Opening or closing the palette / help overlay does not mutate `state.mode`. Asserted by extending `test/app/mode-transitions.test.tsx`.
18. **Layout primitive.** Both overlays use the same overlay-slot convention as `<ApprovalModal>` / `<ResumeWizardModal>` — width/height props driven by `useStdout()`; 40-col floor; 110-col ceiling.
19. **Registry covers every current combination.** `selectKeybarFixture` handles every `(mode, focus, overlay)` combination reachable by the reducer's current actions. A test exhaustively enumerates them.
20. **Double-run guarded.** Rapid Enter presses in the palette produce at most one `executeCommand` invocation per run cycle; while `fsm==="running"` Enter is ignored.

## §6 Design decisions

| # | Decision | Rationale |
|---|---|---|
| D1 | **Command catalogue is a pure data module (`commands.ts`), not a runtime registration API.** | Matches the keybar-fixture convention (`Binding[]` literals). Discoverability via one `git grep` beats "what got registered by which module at startup". Any future plugin story can add an `extraCommands` prop to the overlay — non-breaking. |
| D2 | **Palette local state (cursor, fsm) via `useReducer`; only `query` on `AppState`.** | `query` is what the user would want preserved if the UI re-renders or hot-swaps the overlay in a future multi-overlay world; cursor/fsm are ephemeral. Mirrors P7-T1/P7-T2 D2. |
| D3 | **Help overlay is read-only — Enter does not run the highlighted binding.** | Running from Help duplicates the keybar's job, risks double-handling keys, and violates "one overlay at a time". If users want to run a binding they close Help (`Esc`/`?`) and press the key. Matches lazygit and k9s help screens. |
| D4 | **`?` toggles Help closed when open; `:` does NOT toggle palette.** | `?` is a stateless discovery aid — "show/hide the cheat sheet" reads naturally. `:` carries arguments; repeating `:` mid-typing would clobber the user's partial input. Asymmetric on purpose. |
| D5 | **`:` is never shown in keybar fixtures; it's an invisible global.** | §5.5 groups `:` under "Global" with `?`. Putting it in every mode's fixture would waste two columns. Users discover it through `?` help (which lists it as a GLOBAL row — see mockup §11 line 425) and the features doc. |
| D6 | **Help annotations (e.g. `(1 available)`) are table-driven in `deriveHelpModel`, not a per-binding property.** | Only two annotations exist today (approve count + re-run resumability). Growing this to a `binding.annotation?` field now would pollute every fixture for a minority concern. The table lives in `derive.ts` and is 3 lines; if it grows past 5 rules we extract. |
| D7 | **No overlay stacking — `:` and `?` are ignored while another overlay is open.** | Stacking would require a per-overlay focus stack and new lifecycle hooks. The mockups show no stacking. If a user is mid-approval and wants help, they `Esc` the approval first; the approval preserves overlay state via D2 of P7-T1. |
| D8 | **Re-derive matches on every render.** | Catalogue is 9 rows; `filterCommands` is O(9 × |query|). Negligible. `useMemo` on `(query, ctx)` would add surface without benefit because `ctx` allocates fresh each render anyway. |
| D9 | **Help reads a ref-snapshot of the fixture the keybar was showing right before help opened.** | The alternative — recomputing `selectKeybarFixture(prevMode, prevOverlay, ...)` inside the overlay — requires the component to know "what I was before I became active", which is awkward. A ref set at dispatch-time is three lines and makes the invariant ("help = fixture at `?`-press moment") testable (criterion 1). |
| D10 | **`selectKeybarFixture` replaces the inline switch in `app.tsx`.** | Canonicalises the mapping; eliminates drift between the keybar's choice and the help overlay's choice. Side benefit: the `app.tsx` render becomes shorter and the mapping becomes unit-testable. |
| D11 | **Commands route to existing engine bridges; no new impure module.** | `resumeRun`, `decideApproval`, (future) `cancelRun` already cover the surface. Creating `src/engine/commands.ts` would just fan out. `executeCommand` is the pure dispatcher; IO lives in the closures on `CommandExecContext`. |
| D12 | **Matcher is bespoke (~60 LoC), not a third-party fuzzy lib.** | The catalogue is nine rows and the algorithm is well-defined in §2.3. Adding `fuzzysort` / `fuse.js` adds a dependency and bundle weight for no measurable benefit. The hand-rolled matcher is easier to unit-test. |
| D13 | **`toggleLabel` is resolved inside `deriveHelpModel`, not by the overlay.** | Keeps the overlay a dumb renderer and lets the same derivation service drive both the help overlay and any future JSON-export-of-help subcommand without duplicating the resolution logic. |
| D14 | **Width/height as props**, same contract as `<ApprovalModal>` / `<ResumeWizardModal>`. | Deterministic test rendering. Production uses the same clamp formula as sibling overlays. |
| D15 | **`Tab`-complete reuses the current match's `command.name` (+ trailing space) for single match, longest-common-prefix for multi.** | Same semantics as shells and k9s palette — users expect it. Makes discovery of arg-taking commands (`:resume `) flow into arg typing without a second keystroke. |
